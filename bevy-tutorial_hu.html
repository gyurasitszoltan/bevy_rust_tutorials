0x00

Mi a Bevy? Telepítés és első projekt

A Bevy egy nyílt forráskódú, Rustban írt, adatvezérelt játékmotor. Nincs hozzá kötelező grafikus szerkesztő: a játékod kódból épül fel, és a motor minden része — a renderelő, a hangrendszer, a UI, sőt maga a játéklogikád is — ugyanazon az ECS (Entity Component System) architektúrán fut. Ez a tananyag pontosan ezért az ECS-re helyezi a hangsúlyt: aki az ECS-t érti, az a Bevyt érti.

Mire jó, és mire nem?

  • Erőssége: szimulációnehéz játékok (roguelike, kolóniaszim, stratégia), 2D és 3D egyaránt, webes (WASM) célplatform, teljes kontroll a kód felett.
  • Gyengéje: nincs érett vizuális szerkesztő, és kb. 3 havonta jelenik meg törő változásokat hozó új verzió — a régi internetes példakódok gyakran elavultak.
Figyelem — verziók

Ez a tananyag a Bevy 0.18-at követi (2026. január). Ha egy régebbi blogposztban Camera2dBundle-t, EventReader-t vagy SpriteBundle-t látsz, az korábbi verzió: a bundle-öket a kötelező komponensek (0x04), a pufferelt eseményeket a Message típus (0x0A) váltotta. Frissítéskor mindig a hivatalos migrációs útmutatót kövesd: bevy.org/learn/migration-guides.

0x01

Az első Bevy-app: App, pluginok, rendszerek

Minden Bevy-program egy App köré épül. Az App három dolgot fog össze: a World-öt (ebben él az összes adat), a Schedule-öket (ezek döntik el, mikor mi fut), és a pluginokat (ezek csomagolják a funkcionalitást). A legkisebb értelmes program:

src/main.rs
use bevy::prelude::*;

fn main() {
    App::new()
        .add_plugins(DefaultPlugins)   // ablak, renderelő, bemenet, hang, idő...
        .add_systems(Startup, koszones)
        .run();
}

fn koszones() {
    println!("Szia, Bevy!");
}

Futtatva megnyílik egy üres ablak, a terminálban pedig megjelenik a köszöntés. Két fontos dolog történt:

  • A DefaultPlugins plugingyűjtemény húzza be a motor szinte teljes funkcionalitását. Enélkül a Bevy csak egy csupasz ECS-futtató.
  • A koszones egy rendszer (system): közönséges Rust-függvény, amelyet a Startup schedule-be regisztráltunk, ezért pontosan egyszer fut le, induláskor.

A játékhurok

A Startup mellett a leggyakoribb schedule az Update: az ide regisztrált rendszerek minden képkockában lefutnak. Ez a Bevy játékhurka — nem te írod a while ciklust, hanem rendszereket adsz a motornak, és ő hívogatja őket:

src/main.rs
use bevy::prelude::*;

fn main() {
    App::new()
        .add_plugins(DefaultPlugins)
        .add_systems(Startup, koszones)
        .add_systems(Update, szamlalo)
        .run();
}

fn koszones() { println!("Indulás!"); }

fn szamlalo(mut keret: Local<u64>) {
    *keret += 1;
    if *keret % 60 == 0 {
        println!("{} képkocka telt el", *keret);
    }
}
Indulás! 60 képkocka telt el 120 képkocka telt el 180 képkocka telt el

A Local<u64> egy csak ehhez a rendszerhez tartozó, hívások közt megőrzött érték — részletesen a 0x07 fejezetben. A lényeg most az: a rendszerek paraméterei deklarálják, milyen adatra van szükségük, és a Bevy ezeket automatikusan beadja. Ez a minta — a paraméterlistából kiolvasott adatigény — az egész motor működésének kulcsa, és a következő részben járjuk körül alaposan.

0x02

2D alapok: kamera, sprite, koordinátarendszer

Rajzoljunk valamit. Két entitásra van szükségünk: egy kamerára és egy sprite-ra. Mindkettőt a Commands segítségével „spawnoljuk" be a világba:

src/main.rs
use bevy::prelude::*;

fn main() {
    App::new()
        .add_plugins(DefaultPlugins)
        .add_systems(Startup, szinpad)
        .run();
}

fn szinpad(mut commands: Commands) {
    // 2D kamera — enélkül semmi nem látszik
    commands.spawn(Camera2d);

    // Egy egyszínű téglalap sprite
    commands.spawn((
        Sprite {
            color: Color::srgb(0.95, 0.65, 0.25),
            custom_size: Some(Vec2::new(120.0, 120.0)),
            ..default()
        },
        Transform::from_xyz(0.0, 0.0, 0.0),
    ));
}

Az ablak közepén megjelenik egy narancssárga négyzet. Képfájl betöltését a 0x0C fejezetben látjuk majd — addig az egyszínű sprite tökéletes kísérleti alany.

A 2D koordinátarendszer

  • Az origó (0, 0) alapból a képernyő közepén van.
  • +X jobbra, +Y felfelé mutat (matematikai irány — nem úgy, mint sok más 2D motorban!).
  • A Z 2D-ben a rétegsorrendet adja: a nagyobb Z-jű sprite kerül felülre.
  • Az egység a „world unit", ami alapbeállításban 1 képernyőpixelnek felel meg.

A Transform

A Transform komponens hordozza a pozíciót (translation: Vec3), a forgatást (rotation: Quat) és a méretezést (scale: Vec3). Mozgatni úgy mozgatunk valamit, hogy egy rendszer minden képkockában módosítja a transformját:

src/main.rs — részlet
fn forgas(mut sprite_q: Query<&mut Transform, With<Sprite>>, time: Res<Time>) {
    for mut transform in &mut sprite_q {
        transform.rotate_z(1.5 * time.delta_secs());
    }
}

// main-ben: .add_systems(Update, forgas)

Itt már három ECS-fogalom dolgozik együtt: a Query (megkeresi az összes sprite-os transformot), a Res<Time> (globális idő-erőforrás), és a time.delta_secs() (az előző képkocka óta eltelt idő — ezzel szorozva a mozgás sebessége képkockasebesség-független lesz). Ha ez most még varázslatnak tűnik, jó helyen jársz: a következő rész pontosan ezt a gépezetet szedi szét.

0x03

Az ECS gondolkodásmód

Itt kezdődik a tananyag szíve. A következő kilenc fejezet kizárólag az ECS-sel foglalkozik, mert a Bevyben minden — a játékosod, a kamera, az ablak, egy UI-gomb, sőt egy hangeffekt is — ECS-adat. Aki itt szilárd alapot épít, annak a motor többi része már csak szótár.

Az OOP-reflex és a baja

Objektumorientált háttérrel az ember így modellezne egy játékot: van egy Jatekos osztály, amely örököl az Elolenytol, az meg a Mozgothatobol... Két jól ismert probléma jön elő:

  • Az öröklési rács szétesik. Mi van, ha kell egy szellem, ami élőlény, de nem ütközik? Egy szobor, ami ütközik, de nem él? Az egyszeres öröklés merev, a többszörös káosz.
  • A memória szétszórt. Ezer objektum ezer helyen a heapen — a processzor gyorsítótára (cache) folyamatosan hibázik, amikor végig kell menni rajtuk.

Az ECS válasza: kompozíció és táblázatok

Az ECS három fogalomra bontja a világot:

FogalomMi ez?Rust-megfelelő a Bevyben
Entity (entitás)Puszta azonosító. Nincs benne adat, nincs viselkedése — egy sorszám.Entity (lényegében egy index + generáció)
Component (komponens)Egyetlen adatszelet: pozíció, életerő, sebesség...#[derive(Component)] struct Eletero(f32);
System (rendszer)Logika, amely a komponenseken dolgozik.Közönséges fn, speciális paraméterekkel

Egy „játékos" tehát nem osztály, hanem egy entitás, amelyre komponenseket aggattunk: Jatekos + Eletero + Sebesseg + Transform + Sprite. A szellem? Ugyanez, ütköző nélkül. A szobor? Transform + Sprite + Utkozo, élet nélkül. Nincs öröklési fa — szabad kompozíció van.

Hogyan tárolja ezt a Bevy? Az archetípusok

A Bevy ECS-e archetípusokba rendezi az entitásokat: egy archetípus az azonos komponens-összetételű entitások csoportja. Minden archetípus egy táblázat, amelyben a komponensek oszloponként, folytonos tömbökben fekszenek:

Archetípus: [Transform, Sebesseg, Eletero]TransformSebessegEletero
Entity 7 (játékos)(0, 0)(3, 0)100
Entity 12 (ellenfél)(40, 9)(-1, 0)30
Entity 31 (ellenfél)(85, -4)(-2, 1)30
Archetípus: [Transform, Eletero]TransformEletero
Entity 19 (rombolható láda)(120, 0)10

Ennek két messzemenő következménye van:

  • Cache-barát bejárás. Ha egy rendszer az összes (Transform, Sebesseg) párt kéri, a motor archetípusonként, folytonos memórián söpör végig — ezért emlegetik az ECS-t „adatorientált" tervezésként. Ugyanezért tud egy Bevy-játék tízezres entitásszámmal is simán futni.
  • Komponens hozzáadása/elvétele = költöztetés. Ha az Entity 19-re ráteszünk egy Sebesseg-et, az entitás átköltözik a másik archetípus-táblába. Ezért nem szabad képkockánként tömegesen komponenst ki-be kapcsolgatni — erre jobb eszköz egy bool mező vagy egy enum a komponensen belül.
Mélység — honnan tudja a rendszer, mit kérhet?

Egy rendszer paraméterlistája (Query, Res, Commands...) valójában típusszinten leírt adathozzáférési deklaráció. A Bevy ütemezője ebből pontosan tudja, melyik rendszer mit olvas és mit ír — és amelyik kettő nem ütközik, azt automatikusan párhuzamosan futtatja, külön szálakon, mindenféle lock nélkül. A Rust borrow checker garanciái itt futásidejű ütemezési szabállyá lépnek elő: ez a Bevy egyik legszebb húzása.

A World

Az összes archetípus, entitás és erőforrás egyetlen konténerben él: ez a World. Rendszert írva ritkán nyúlsz hozzá közvetlenül — a paramétereiden át kapsz belőle szeleteket —, de jó tudni, hogy a háttérben mindig egyetlen, központi adatbázis áll. Az ECS-t emiatt szokták in-memory relációs adatbázishoz hasonlítani: az archetípusok a táblák, az entitások a sorok, a komponensek az oszlopok, a Query-k pedig a SELECT-ek.

0x04

Entitások és komponensek a gyakorlatban

Komponens definiálása

Komponens bármely Rust-típus lehet, amelyre rátesszük a #[derive(Component)]-et — struct, tuple struct, enum, akár üres struct is:

komponensek
use bevy::prelude::*;

#[derive(Component)]
struct Eletero {
    aktualis: f32,
    maximum: f32,
}

#[derive(Component)]
struct Sebesseg(Vec2);          // tuple struct — newtype minta

#[derive(Component)]
struct Jatekos;                 // üres "marker" komponens

#[derive(Component)]
enum Csapat { Piros, Kek }      // enum is lehet komponens
Tipp — a marker komponens a legfontosabb minta

Az adat nélküli marker komponens (Jatekos, Ellenfel, FoKamera) az ECS címkézőeszköze: a Query-kben szűrőként használjuk (With<Jatekos>), és gyakorlatilag ingyen van. Ne félj sokat csinálni belőlük — Bevy-kódban ez a normális stílus.

Spawn és despawn

Entitást a Commands-on keresztül hozunk létre. Több komponenst tuple-ként adunk át:

spawn
fn jatekos_spawn(mut commands: Commands) {
    let entity: Entity = commands.spawn((
        Jatekos,
        Eletero { aktualis: 100.0, maximum: 100.0 },
        Sebesseg(Vec2::ZERO),
        Sprite { color: Color::srgb(0.3, 0.7, 1.0),
                 custom_size: Some(Vec2::splat(32.0)), ..default() },
        Transform::from_xyz(0.0, -200.0, 1.0),
    )).id();                       // az .id() visszaadja az Entity azonosítót

    info!("Játékos spawnolva: {entity}");
}

Kötelező komponensek (required components)

Honnan tudta a sprite-os entitásunk, hogy kell neki láthatóság, globális transform és minden egyéb, amit sosem adtunk meg? A komponensek deklarálhatják, hogy milyen más komponenseket követelnek meg: amikor spawnoláskor ezek hiányoznak, a Bevy alapértékkel automatikusan beszúrja őket. A Sprite például megköveteli a Transform-ot és a Visibility-t. Saját típusainknál ugyanez a #[require(...)] attribútummal megy:

required components
#[derive(Component)]
#[require(Sebesseg, Sprite)]    // Jatekos-hoz mindig jár Sebesseg és Sprite
struct Jatekos;

#[derive(Component, Default)]
struct Sebesseg(Vec2);          // a require-hez kell a Default (vagy explicit érték)

fn spawn(mut commands: Commands) {
    // Elég ennyi — a Sebesseg, a Sprite, és a Sprite által megkövetelt
    // Transform + Visibility mind automatikusan rákerül:
    commands.spawn(Jatekos);
}

Ez a mechanizmus váltotta ki a régi „bundle" mintát: a komponens maga írja le, mi tartozik hozzá, a hívóhely pedig rövid marad. Konkrét alapérték is megadható: #[require(Sebesseg(Vec2::new(0.0, 120.0)))].

Komponens hozzáadása és elvétele utólag

insert / remove
#[derive(Component)]
struct Megfagyott;     // státuszeffekt markerrel

fn fagyasztas(mut commands: Commands, ellenfelek: Query<Entity, With<Ellenfel>>) {
    for entity in &ellenfelek {
        commands.entity(entity).insert(Megfagyott);
    }
}

fn olvadas(mut commands: Commands, fagyottak: Query<Entity, With<Megfagyott>>) {
    for entity in &fagyottak {
        commands.entity(entity).remove::<Megfagyott>();
    }
}
Figyelem — archetípus-költöztetés

Ahogy a 0x03-ban láttuk, az insert/remove az entitást másik archetípusba költözteti. Ritka eseményekhez (státuszeffekt fel/le) tökéletes; képkockánként, entitások ezrein viszont kerülendő — ott jobb egy bool/Option mező a komponensben.

0x05

Rendszerek és lekérdezések: a Query anatómiája

A Query az ECS lelke: vele kérdezzük le a világból az entitások egy szeletét. Két típusparamétere van: mit kérünk (adat) és mire szűrünk (filter):

Query<ADAT, SZŰRŐ>
// Minden entitás Transform-ja, AMELYEN van Jatekos komponens is:
Query<&Transform, With<Jatekos>>

// Írható hozzáférés + több komponens tuple-ben:
Query<(&mut Transform, &Sebesseg)>

// Az Entity azonosító is kérhető adatként:
Query<(Entity, &Eletero), With<Ellenfel>>

// Opcionális komponens — None, ha az adott entitáson nincs:
Query<(&Transform, Option<&Megfagyott>)>

Bejárás

mozgásrendszer
fn mozgas(mut mozgok: Query<(&mut Transform, &Sebesseg)>, time: Res<Time>) {
    for (mut transform, sebesseg) in &mut mozgok {
        transform.translation += sebesseg.0.extend(0.0) * time.delta_secs();
    }
}

Ez az egyetlen rendszer mozgat mindent, aminek van transformja és sebessége — játékost, ellenfelet, lövedéket. Ez az ECS ígérete a gyakorlatban: a logika nem típusokhoz, hanem adatalakzatokhoz kötődik.

Szűrők

SzűrőJelentés
With<T>csak ha az entitáson van T (de a T adatát nem kérjük)
Without<T>csak ha nincs rajta T
Or<(With<A>, With<B>)>ha A vagy B rajta van

Amikor pontosan egy entitást várunk: Single

A játékosból, főkamerából tipikusan egy darab van. Erre való a Single rendszerparaméter — ha nem pontosan egy találat lenne, a rendszer az adott képkockában egyszerűen nem fut le:

Egy konkrét entitás lekérése

Ha megvan valahonnan egy Entity (pl. ütközésvizsgálatból), a query-ből célzottan is kiolvasható: query.get(entity) / query.get_mut(entity) — ezek Result-ot adnak, mert az entitás időközben megszűnhetett, vagy nem illik a query alakjára.

Mélység — párhuzamos bejárás

Nagy entitásszámnál a bejárás maga is párhuzamosítható: mozgok.par_iter_mut().for_each(|(mut t, s)| ...) — a Bevy a munkát automatikusan szétosztja a szálkészletén. Tízezres nagyságrend alatt általában nem éri meg, fölötte ingyen sebesség.

0x06

Commands: halasztott műveletek és a sync pont

Feltűnhetett: spawnolni, törölni, komponenst fel-le rakni mindig a Commands-on keresztül szoktunk, sosem „közvetlenül". Ennek mély oka van.

Miért nem lehet azonnal?

A 0x03-ban láttuk, hogy a rendszerek párhuzamosan futnak, és hogy a strukturális változás (spawn, despawn, insert, remove) archetípus-táblákat rendez át. A kettő összeférhetetlen: nem lehet egy táblát átrendezni, miközben három másik szál épp iterál rajta. A megoldás: a Commands a műveletet nem végrehajtja, hanem felírja egy parancslistára, és a Bevy egy biztonságos pillanatban — úgynevezett sync pontban, tipikusan a schedule rendszerei közt — egyben lejátssza őket, amikor kizárólagos hozzáférése van a World-höz.

a leggyakoribb parancsok
fn pelda(mut commands: Commands) {
    // entitásműveletek
    let e = commands.spawn((Ellenfel, Transform::default())).id();
    commands.entity(e).insert(Megfagyott);          // komponens fel
    commands.entity(e).remove::<Megfagyott>();      // komponens le
    commands.entity(e).despawn();                   // törlés (gyerekekkel együtt)

    // erőforrás-műveletek (0x07)
    commands.insert_resource(Pontszam(0));
}

A halasztottság következménye

Mivel a parancs csak később fut le, ugyanabban a képkockában egy másik rendszer még nem látja az eredményét. Ha az A rendszer spawnol valamit, és B még ugyanebben a frame-ben query-zné, B üres kezekkel marad — a frissen spawnolt entitás csak a következő sync pont után létezik a query-k számára. Ez a Bevy-kezdők leggyakoribb „bugja", pedig nem bug: ez a modell. Megoldások:

  • Legtöbbször nem kell tenni semmit — a következő képkockában minden a helyén lesz, és egy 60 FPS-es játékban ez észrevehetetlen.
  • Ha tényleg azonnali sorrend kell, kösd össze a két rendszert: .add_systems(Update, (spawnolo, feldolgozo).chain()) — a chain sorrendet kényszerít, és a Bevy a kettő közé sync pontot tesz, ha kell (0x09).
Figyelem — eltűnt entitások

Fordított irányban is igaz: ha egy entitást ebben a frame-ben despawnoltál, egy később futó rendszer query-je még láthatja. Célzott query.get(entity) hívásnál ezért mindig kezeld a Result hibaágát ahelyett, hogy unwrap-olnál.

Mélység — amikor mégis azonnal kell: exkluzív rendszerek

Egy rendszer kérheti a teljes világot is: fn rendszer(world: &mut World). Az ilyen exkluzív rendszer egyedül fut, semmi mással nem párhuzamosan, cserébe azonnal hajthat végre strukturális változást. Mentés/betöltés, pályagenerálás jellemző terepe — játéklogikához ritkán éri meg feladni érte a párhuzamosságot.

Resource-ok és Local: a globális és a privát állapot

Nem minden adat tartozik entitáshoz. A pontszám, a játékbeállítások, a véletlenszám-generátor, az eltelt idő — ezek globális, egyetlen példányban létező adatok. Az ECS-ben ezek a Resource-ok: típusonként legfeljebb egy példány élhet belőlük a World-ben.

resource definiálása és használata
#[derive(Resource, Default)]
struct Pontszam(u32);

fn main() {
    App::new()
        .add_plugins(DefaultPlugins)
        .init_resource::<Pontszam>()          // Default-tal hozza létre
        // vagy konkrét értékkel: .insert_resource(Pontszam(100))
        .add_systems(Update, (pont_gyujtes, pont_kiiras))
        .run();
}

fn pont_gyujtes(mut pont: ResMut<Pontszam>) {
    pont.0 += 1;
}

fn pont_kiiras(pont: Res<Pontszam>) {
    if pont.0 % 100 == 0 { info!("Pontszám: {}", pont.0); }
}

A hozzáférés kétféle: Res<T> (olvasás) és ResMut<T> (írás). Az ütemező itt is a paraméterekből tudja a függéseket: két Res<Pontszam>-os rendszer futhat párhuzamosan, de ResMut mellett senki más nem nyúlhat ugyanahhoz a resource-hoz egy időben.

A motor maga is resource-okkal kommunikál: a Time, a ButtonInput<KeyCode>, az AssetServer — mind resource, ezért kérjük őket Res<...>-ként.

Figyelem — opcionális resource

Ha egy resource nem biztos, hogy létezik (pl. csak bizonyos játékmódban kerül be), kérd Option<Res<T>>-ként, különben a rendszer pánikol. A run_if(resource_exists::<T>) futási feltétel is megoldás (0x09).

Tipikus használat: számláló, „első futás" jelző, rendszeren belüli gyorsítótár. Ha két rendszernek is kellene → Resource. Ha entitásonként más → Component. Változásérzékelés: Changed, Added, RemovedComponents

„Csak akkor csinálj valamit, ha valami megváltozott" — a játéklogika fele ilyen. A Bevy ECS-ébe ez be van építve: minden komponensíráskor a motor automatikusan feljegyzi a változás idejét, és erre query-szűrőkkel lehet építeni.

Changed és Added
// Csak azokon fut, akiknek MOST változott az életereje:
fn eletcsik_frissites(serultek: Query<(&Eletero, &Children), Changed<Eletero>>) {
    for (elet, _gyerekek) in &serultek {
        info!("Életcsík újrarajzolása: {}/{}", elet.aktualis, elet.maximum);
    }
}

// Csak a frissen felkerült komponensekre — pl. spawn utáni inicializálás:
fn uj_ellenfel_koszontes(ujak: Query<Entity, Added<Ellenfel>>) {
    for entity in &ujak {
        info!("Új ellenfél a pályán: {entity}");
    }
}

Ez nem esemény-feliratkozás, hanem ugyanaz a query-mechanika: a Changed<T> azokat az entitásokat engedi át, amelyeknél a T komponens a rendszer előző lefutása óta íródott. Egy 10 000 entitásos pályán az életcsík-frissítő így képkockánként jellemzően 0–2 entitást dolgoz fel tízezer helyett.

Figyelem — az írás ténye számít, nem az érték

A változásérzékelés a mutábilis hozzáférést figyeli: ha egy &mut Eletero-t lekérsz és ugyanazt az értéket írod vissza, az is „változásnak" számít. Ha ez gond, írás előtt hasonlíts: if elet.aktualis != uj { elet.aktualis = uj; } — vagy használd a set_if_neq segédet ott, ahol elérhető.

Eltávolított komponensek

A „leszedték rólam a Megfagyott-at" esemény nem fogható meg query-vel — az entitás már másik archetípusban van. Erre a RemovedComponents paraméter való:

    for entity in levettek.read() {
        info!("{entity} kiolvadt — itt jöhetne részecskeeffekt");
    }
}
Mélység — hogyan működik belül?

Minden komponenscella mellett két számláló (tick) fekszik: mikor lett hozzáadva, mikor írták utoljára. Minden rendszer megjegyzi, mikor futott. A Changed<T> egyszerű összehasonlítás: utolso_iras > rendszer_elozo_futasa. Olcsó, ágmentes, és pont emiatt mindig be van kapcsolva — nem kell külön kérni.

Üzenetek és események: Message, Event, observerek

A rendszerek eddig komponenseken és resource-okon át kommunikáltak. Gyakran viszont nem állapotot, hanem történést akarunk átadni: „a játékos sérült", „érme felvéve", „hullám vége". Erre a Bevynek két, szándékosan különválasztott eszköze van.

Figyelem — 0.17 előtti anyagokban

Régebbi tutorialokban mindkét fogalmat „Event"-nek hívták (EventWriter/EventReader). A 0.17 óta a pufferelt fajta neve Message, az Event név pedig az observereké. A régi kód mintái egy az egyben átnevezhetők.

1. Message: pufferelt, sokan írják, sokan olvassák

A Message egy típusos postaláda: bármely rendszer írhat bele (MessageWriter), bármely másik kiolvashatja (MessageReader), és minden olvasó a sajátját olvassa — egymástól függetlenül.

messages
#[derive(Message)]
struct ErmeFelveve {
    ertek: u32,
}

fn main() {
    App::new()
        .add_plugins(DefaultPlugins)
        .add_message::<ErmeFelveve>()
        .add_systems(Update, (erme_utkozes, pontozas, hang_lejatszas))
        .run();
}

fn erme_utkozes(mut writer: MessageWriter<ErmeFelveve>) {
    // ...ütközésvizsgálat után:
    writer.write(ErmeFelveve { ertek: 10 });
}

fn pontozas(mut reader: MessageReader<ErmeFelveve>, mut pont: ResMut<Pontszam>) {
    for uzenet in reader.read() {
        pont.0 += uzenet.ertek;
    }
}

fn hang_lejatszas(mut reader: MessageReader<ErmeFelveve>) {
    for _ in reader.read() {
        info!("*csilingelés*");   // ugyanazt az üzenetet ő is megkapja
    }
}

Fontos tulajdonságok: az üzenetek két képkockáig élnek, aztán a motor eldobja őket (aki addig nem olvasta, lemaradt); a kézbesítés húzó jellegű — az olvasó rendszer a saját következő futásakor dolgozza fel; és a minta természeténél fogva szétkapcsol: az író nem tud az olvasókról.

2. Event + observer: azonnali, reaktív

Az observer fordított logika: nem egy rendszer kérdezi le az üzeneteket, hanem egy eseményre felakasztott megfigyelő fut le azonnal, amikor az eseményt valaki kiváltja (trigger):

observer
#[derive(Event)]
struct JatekVege;

fn main() {
    App::new()
        .add_plugins(DefaultPlugins)
        .add_observer(jatek_vege_kezeles)
        .add_systems(Update, eletfigyeles)
        .run();
}

fn eletfigyeles(jatekos: Single<&Eletero, With<Jatekos>>, mut commands: Commands) {
    if jatekos.aktualis <= 0.0 {
        commands.trigger(JatekVege);
    }
}

fn jatek_vege_kezeles(_esemeny: On<JatekVege>, mut commands: Commands) {
    info!("Vége a játéknak!");
    // az observer maga is teljes értékű rendszer: Commands, Query, Res — mind kérhető
    let _ = &mut commands;
}

Az observer entitásokhoz is köthető (commands.entity(e).observe(...)) — ilyenkor csak az adott entitást célzó eseményekre fut le. Ez a Bevy UI-interakcióinak (kattintás, hover) is az alapja.

Hierarchia és entitás-kapcsolatok

A játékvilág entitásai gyakran tartoznak össze: a kard a játékos kezében van, az életcsík az ellenfél feje fölött lebeg, a torony alkatrészei együtt mozognak. A Bevyben ezt a szülő–gyerek kapcsolat fejezi ki, amely maga is komponensekkel van megvalósítva: a gyereken egy ChildOf(szülő) komponens ül, a szülőn pedig a motor automatikusan karbantart egy Children listát.

Mit ad a hierarchia?

  • Transform-öröklés. A gyerek Transform-ja a szülőhöz képest értendő; a végleges világpozíciót a motor számolja ki a GlobalTransform komponensbe. Ha a torony elmozdul, az ágyú és az életcsík vele megy — kód nélkül.
  • Láthatóság-öröklés. A szülő elrejtése (Visibility::Hidden) a teljes részfát elrejti.
  • Együttes törlés. A despawn() az egész részfát eltakarítja.

const JATEKOS_SEBESSEG: f32 = 220.0;

fn iranyitas(
    billentyuk: Res<ButtonInput<KeyCode>>,
    mut jatekos: Single<&mut Sebesseg, With<Jatekos>>,
) {
    let mut irany = Vec2::ZERO;
    if billentyuk.pressed(KeyCode::KeyW) || billentyuk.pressed(KeyCode::ArrowUp)    { irany.y += 1.0; }
    if billentyuk.pressed(KeyCode::KeyS) || billentyuk.pressed(KeyCode::ArrowDown)  { irany.y -= 1.0; }
    if billentyuk.pressed(KeyCode::KeyA) || billentyuk.pressed(KeyCode::ArrowLeft)  { irany.x -= 1.0; }
    if billentyuk.pressed(KeyCode::KeyD) || billentyuk.pressed(KeyCode::ArrowRight) { irany.x += 1.0; }

    jatekos.0 = irany.normalize_or_zero() * JATEKOS_SEBESSEG;
}

A Time resource

A Res<Time>-ból két dolgot használunk lépten-nyomon: delta_secs() (az előző képkocka óta eltelt idő másodpercben — minden mozgást ezzel szorzunk, hogy 30 és 144 FPS-en is ugyanolyan gyors legyen a játék) és elapsed_secs() (a teljes futási idő — lüktetés, hullámzás, szinuszos animációk barátja).

FixedUpdate: a determinisztikus szív

Az Update annyiszor fut, ahány képkocka van — változó ütemben. Fizikának, determinisztikus szimulációnak ez méreg: ugyanaz a játékmenet más gépen máshogy alakulna. A FixedUpdate schedule ezzel szemben rögzített ütemben fut (alapból 64 Hz): ha a renderelés gyorsabb, kimarad; ha lassabb, többször fut egymás után, hogy behozza a lemaradást.

fizika a FixedUpdate-ben
App::new()
    .add_plugins(DefaultPlugins)
    .add_systems(FixedUpdate, (gravitacio, mozgas, utkozes).chain())
    .insert_resource(Time::<Fixed>::from_hz(64.0))   // ütem átállítása
    .run();

fn gravitacio(mut esok: Query<&mut Sebesseg>, time: Res<Time>) {
    // FixedUpdate-ben a Time automatikusan a fix lépésközt adja:
    for mut seb in &mut esok {
        seb.0.y -= 600.0 * time.delta_secs();
    }
}

Praktikus felosztás: bemenet és kamera az Update-ben (reszponzivitás), fizika és játéklogika a FixedUpdate-ben (determinizmus). A Res<Time> mindkét helyen „a megfelelő" deltát adja — a FixedUpdate-en belül mindig a fix lépésközt.

        .insert_resource(CsillagIdozito(Timer::from_seconds(0.9, TimerMode::Repeating)))
        .add_message::<CsillagElkapva>()
        .add_message::<CsillagLeesett>()
        .add_systems(Startup, szinpad)
        .add_systems(OnEnter(Allapot::Jatszik), jatek_inditas)
        .add_systems(OnEnter(Allapot::GameOver), game_over_felirat)
        .add_systems(Update, (
            (iranyitas, csillag_spawn, mozgas, elkapas_es_leeses, pontozas, hud_frissites)
                .chain()
                .run_if(in_state(Allapot::Jatszik)),
            ujrainditas.run_if(in_state(Allapot::GameOver)),
        ))
        .run();
}

fn iranyitas(
    billentyuk: Res<ButtonInput<KeyCode>>,
    mut jatekos: Single<&mut Sebesseg, With<Jatekos>>,
) {
    let mut x = 0.0;
    if billentyuk.pressed(KeyCode::ArrowLeft)  || billentyuk.pressed(KeyCode::KeyA) { x -= 1.0; }
    if billentyuk.pressed(KeyCode::ArrowRight) || billentyuk.pressed(KeyCode::KeyD) { x += 1.0; }
    jatekos.0 = Vec2::new(x * 420.0, 0.0);
}

fn csillag_spawn(
    mut commands: Commands,
    mut idozito: ResMut<CsillagIdozito>,
    time: Res<Time>,
) {
    idozito.0.tick(time.delta());
    if idozito.0.just_finished() {
        let mut rng = rand::rng();
        commands.spawn((
            Csillag,
            Sebesseg(Vec2::new(0.0, -rng.random_range(140.0..260.0))),
            DespawnOnExit(Allapot::Jatszik),
            Sprite { color: Color::srgb(1.0, 0.85, 0.2),
                     custom_size: Some(Vec2::splat(22.0)), ..default() },
            Transform::from_xyz(rng.random_range(-PALYA_FEL_SZELESSEG..PALYA_FEL_SZELESSEG), 320.0, 0.0),
        ));
    }
}

fn mozgas(mut mozgok: Query<(&mut Transform, &Sebesseg)>, time: Res<Time>) {
    for (mut t, s) in &mut mozgok {
        t.translation += s.0.extend(0.0) * time.delta_secs();
        t.translation.x = t.translation.x.clamp(-PALYA_FEL_SZELESSEG, PALYA_FEL_SZELESSEG);
    }
}

fn elkapas_es_leeses(
    mut commands: Commands,
    csillagok: Query<(Entity, &Transform), With<Csillag>>,
    jatekos: Single<&Transform, With<Jatekos>>,
    mut elkapva: MessageWriter<CsillagElkapva>,
    mut leesett: MessageWriter<CsillagLeesett>,
) {
    for (entity, csillag_t) in &csillagok {
        let tavolsag = csillag_t.translation.truncate()
            .distance(jatekos.translation.truncate());
        if tavolsag < 45.0 {
            commands.entity(entity).despawn();
            elkapva.write(CsillagElkapva);
        } else if csillag_t.translation.y < TALAJ - 40.0 {
            commands.entity(entity).despawn();
            leesett.write(CsillagLeesett);
        }
    }
}

fn pontozas(
    mut elkapva: MessageReader<CsillagElkapva>,
    mut leesett: MessageReader<CsillagLeesett>,
    mut pont: ResMut<Pontszam>,
    mut eletek: ResMut<Eletek>,
    mut kovetkezo: ResMut<NextState<Allapot>>,
) {
    for _ in elkapva.read() { pont.0 += 10; }
    for _ in leesett.read() {
        eletek.0 = eletek.0.saturating_sub(1);
        if eletek.0 == 0 { kovetkezo.set(Allapot::GameOver); }
    }
}

fn hud_frissites(
    pont: Res<Pontszam>, eletek: Res<Eletek>,
    mut kijelzo: Single<&mut Text, With<PontKijelzo>>,
) {
    if pont.is_changed() || eletek.is_changed() {
        kijelzo.0 = format!("Pontszám: {}   Életek: {}", pont.0, eletek.0);
    }
}

fn game_over_felirat(mut commands: Commands) {
    commands.spawn((
        DespawnOnExit(Allapot::GameOver),
        Node { width: Val::Percent(100.0), height: Val::Percent(100.0),
               justify_content: JustifyContent::Center,
               align_items: AlignItems::Center, ..default() },
        children![(
            Text::new("GAME OVER — Enter: újra"),
            TextFont { font_size: 42.0, ..default() },
            TextColor(Color::srgb(1.0, 0.4, 0.3)),
        )],
    ));
}

fn ujrainditas(
    billentyuk: Res<ButtonInput<KeyCode>>,
    mut kovetkezo: ResMut<NextState<Allapot>>,
) {
    if billentyuk.just_pressed(KeyCode::Enter) {
        kovetkezo.set(Allapot::Jatszik);
    }
}

Tipp — a verziókövetés stratégiája

A Bevy kb. 3 havonta ad ki törő verziót. Játékprojektben rögzítsd a verziót, és csak tudatosan, a migrációs útmutatóval a kézben frissíts — vagy maradj egy verzión a kiadásig. A tananyag fogalmai (entitás, komponens, query, schedule, message, observer) verziók közt stabilak; ami változik, az többnyire a felszíni API.

És ami a legfontosabb: az ECS-gondolkodás — adat és transzformáció szétválasztása, kompozíció öröklés helyett, deklarált adathozzáférés — nem csak Bevy-tudás. Aki innen néz egy szimulációs vagy adatfeldolgozó problémára, az másként, és gyakran tisztábban látja. Jó építkezést!
