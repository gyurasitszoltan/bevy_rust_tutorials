RESZ 00

Bevezető & projekt-setup

Mielőtt egyetlen gombot kirajzolnánk, érdemes tisztázni, hogyan gondolkodik a Bevy a UI-ról, és hogyan állítunk be egy pixel-art-barát projektet.

0.1Mit építünk és kinek szól

A tutorial egy felülnézetes (top-down), pixel-art dungeon-játék UI-jára van kihegyezve — kis felbontású sprite-ok, csontváz-ellenségek, egészség-szívek, érmék, inventory-rács. A példák önállóan is érthetők, de egymásra épülnek: az elején lefektetett mentális modellt (UI = entitások egy fában) végig használjuk.

Feltételezem, hogy van alap Rust-tudásod, és láttál már Bevy App-ot futni. A Bevy-specifikus dolgokat (ECS-fogalmak, komponensek, observerek, állapotok) menet közben elmagyarázom — ezeket kék «Bevy-fogalom» dobozokban emelem ki.

BEVY-FOGALOM · ECS

A Bevy egy ECS (Entity-Component-System) motor. Entity = egy id (mint egy sorszám). Component = adat, amit egy entityhez ragasztasz (pl. Transform, Health). System = függvény, ami komponenseken dolgozik. A UI nem kivétel: egy gomb is csak egy entity néhány komponenssel (Node, Button, BackgroundColor…).

0.2Cargo, feature-ök és a DefaultPlugins

A Cargo.toml-ban hozzáadjuk a Bevyt. A 0.18 egyik újdonsága a cargo feature-kollekciók: magas szintű kapcsolók (2d, ui, 3d), amikkel csak azt fordítod be, amire szükséged van — ez egy 2D-s pixel-játéknál érezhetően gyorsít a fordításon.

Cargo.toml
[dependencies]
# A "default-features = false" + szűkített lista gyorsabb fordítást ad.
bevy = { version = "0.18", default-features = false, features = [
    "2d",            # 2D renderer: Sprite, Mesh2d, Camera2d
    "ui",            # bevy_ui: Node, Text, Button, layout
    "bevy_ui_picking_backend", # UI elemek kattinthatóvá tétele
    "png",           # textúra-formátum a pixel-art atlaszokhoz
    "x11", "wayland",# Linux ablakkezelők (válaszd ki, ami kell)
] }

A program belépési pontja egy App. Erre rárakjuk a DefaultPlugins csomagot (ablak, renderer, input, asset-rendszer…), majd a saját rendszereinket.

src/main.rs
use bevy::prelude::*;

fn main() {
    App::new()
        // Pixel-artnál FONTOS: nearest (point) szűrés, hogy a textúrák
        // ne legyenek elmosva nagyításkor. Lásd 2.1 fejezet.
        .add_plugins(DefaultPlugins.set(ImagePlugin::default_nearest()))
        .add_systems(Startup, setup)   // egyszer fut, induláskor
        .run();
}

fn setup(mut commands: Commands) {
    // Minden 2D jelenethez kell egy kamera. A UI is ezen keresztül rajzolódik.
    commands.spawn(Camera2d);
}
BEVY-FOGALOM · PLUGIN & SYSTEM SCHEDULE

A Plugin egy újrahasznosítható csomag rendszerekből és erőforrásokból. A add_systems(Startup, …) egy rendszert az ütemezés (schedule) adott pontjához köt. A leggyakoribbak: Startup (egyszer), Update (minden képkockán), és az állapotokhoz kötött OnEnter/OnExit (lásd 5. rész).

TIPP

A Commands egy „parancssor”: vele hozol létre/törölsz entitásokat és rakhatsz rájuk komponenseket. A parancsok nem azonnal futnak le, hanem a rendszer végén alkalmazódnak — ezért biztonságos ugyanabban a rendszerben spawn-olni és lekérdezni.

RESZ 01

Tervezési szempontok

A jó játék-UI nem „ablakos szoftver”. Mielőtt kódot írunk, döntsük el, milyen felbontásban gondolkodunk, mi diegetikus, és hogyan rétegezünk.

1.1Pixel-art UI alapelvek

Pixel-artnál a legnagyobb hiba az inkonzisztens pixelméret: ha a sprite-ok 3×-os nagyításban élesek, de a UI 2,5×-ösben homályos, az azonnal „olcsónak” néz ki. Három alapelv vezessen:

  • Egész számú skálázás (integer scaling). A virtuális felbontást (pl. 320×180) mindig egész szorzóval nagyítsd a képernyőre (2×, 3×, 4×), különben a pixelek „táncolnak”.
  • Egységes pixelrács. Döntsd el, hány képernyő-pixel egy „art-pixel”, és minden UI-méret (padding, border, ikon) ennek a többszöröse legyen.
  • Olvashatóság mindenek felett. A retró bitmap-font hangulatos, de kis méretben nehéz olvasni. Tartsd a fontméretet az art-pixel egész többszörösén, és hagyj bőven kontrasztot.
TIPP · KÉT RÉTEG, KÉT FILOZÓFIA

A játékvilág (sprite-ok) jellemzően alacsony virtuális felbontáson, integer scalinggel rajzolódik. A UI-t viszont gyakran a teljes natív felbontáson érdemes rajzolni, és a pixel-hatást a fontok/ikonok eleve-pixeles voltával elérni — így a szöveg éles marad 1440p-n is. Mindkettő legitim; csak legyél benne következetes.

1.2Diegetikus vs. nem-diegetikus UI

A felülnézetes játékoknál különösen érdemes átgondolni, mi tartozik a világhoz és mi a HUD-hoz:

TípusHol élBevybenPélda
Nem-diegetikus HUDképernyő-térben, mindig fentNode (bevy_ui)szív-sáv, érme-számláló, minimap keret
World-space UIa világban, kamerával együtt mozogText2d / Spritefej feletti névtábla, sebződés-szám
Diegetikus felületa fikció részeSprite + világ-entityláda kinyitása, falra festett tábla

A különbség azért fontos, mert más rendszerrel rajzolódnak: a HUD a bevy_ui layout-motorján fut (flexbox-szerű), a world-space dolgok pedig a Transform-jukkal a kamerához viszonyítva. Ne keverd: ne próbálj fej-feletti életcsíkot bevy_ui node-ként pozicionálni — arra a Text2d + világkoordináta való.

1.3A UI-rétegek felépítése

Egy tipikus dungeon-játéknak több, egymásra rakott UI-rétege van. Érdemes ezeket már a tervezésnél nevesíteni, mert ebből lesz a kódstruktúra (és az állapotgép) is:

  1. HUD – mindig látszik játék közben (élet, erőforrás, gyorssáv).
  2. Inventory / karakterlap – egy gombra előjön, megállítja vagy lelassítja a játékot.
  3. Menük – főmenü, pause, beállítások (állapotvezérelt).
  4. Modálok / popupok – megerősítés, párbeszéd, „biztos kilépsz?”.
  5. Tranziens elemek – tooltipek, toast-értesítések, damage-számok.

Ezeknek a rétegsorrendje (mi van mi fölött) explicit döntés. Bevyben ezt a ZIndex/GlobalZIndex komponenssel és/vagy az entity-fa sorrendjével szabályozod (lásd 2.5).

1.4Mentális modell: a UI egy entity-fa

A legfontosabb gondolat az egész tutorialhoz: a Bevyben a UI nem egy külön retained-mode widget-fa, amit te kézzel szinkronban tartasz az adattal. A UI ugyanaz az ECS-világ, mint a játéklogika. Egy panel = egy szülő-entity Node-dal; a benne lévő elemek = gyerek-entitások. A megjelenést komponensek írják le, a viselkedést rendszerek és observerek.

BEVY-FOGALOM · A HIERARCHIA

A szülő-gyerek kapcsolatot a ChildOf / Children relációk adják. Spawnoláskor a children![…] makróval vagy a with_children(|p| …) / with_child(…) hívásokkal építed a fát. A layout (méret, igazítás) lefelé terjed a fán, mint a CSS flexboxnál.

a fa-építés két stílusa
// 1) Deklaratív, a children! makróval (Bevy 0.16+):
commands.spawn((
    Node { /* a panel layoutja */ ..default() },
    BackgroundColor(Color::srgb(0.10, 0.11, 0.15)),
    children![
        (Text::new("Inventory"), TextColor(Color::WHITE)),
        (Node { /* egy slot */ ..default() }, BackgroundColor(Color::BLACK)),
    ],
));

// 2) Imperatív, closure-rel (ha logika is kell közben):
commands
    .spawn((Node::default(), BackgroundColor(Color::BLACK)))
    .with_children(|parent| {
        for i in 0..5 {
            parent.spawn((SlotIndex(i), Node::default()));
        }
    });
RESZ 02

Renderelés

Hogyan lesz élesen pixeles a kép, és hogyan rak ki a Bevy UI motorja egy node-ot a képernyőre.

2.1Pixel-pontos renderelés

Két dolog kell az éles pixelekhez. Az első a textúra-szűrés: alapból a Bevy lineárisan (simítva) mintavételez, ami elmossa a pixel-artot. A ImagePlugin::default_nearest() globálisan „nearest” (legközelebbi szomszéd) szűrésre vált — ezt láttuk a setupban.

A második az integer scaling: a játékvilágot egy alacsony, fix virtuális felbontáson rajzolod, majd egész szorzóval nagyítod. Erre a tiszta megoldás egy ortografikus kamera, ami fix világ-magasságot tart.

kamera fix virtuális magassággal
use bevy::render::camera::ScalingMode;

fn setup_camera(mut commands: Commands) {
    commands.spawn((
        Camera2d,
        // A Projection egy komponens-enum. 2D-hez ortografikus vetítés kell.
        Projection::from(OrthographicProjection {
            // Mindig pontosan 180 art-pixel magas a látható terület,
            // a szélesség az ablak arányából jön -> a HUD igazodni tud.
            scaling_mode: ScalingMode::FixedVertical { viewport_height: 180.0 },
            ..OrthographicProjection::default_2d()
        }),
    ));
}
FIGYELEM

A pontosan egész szorzós megjelenítéshez az ablak méretét is illik egész többszörösre állítani, különben fél-pixeles „úszás” jelentkezhet kamera-mozgásnál. Gyakori megoldás, hogy a világot egy köztes RenderTarget::Image-re rajzolod (pl. 320×180), majd azt a textúrát nagyítod fel egész szorzóval a képernyőre. A HUD-ot pedig külön, a natív felbontáson teszed rá.

2.2UI node-ok és a layout

A UI minden építőeleme egy entity, amin van egy Node komponens. A Node a layoutot írja le egy flexbox-szerű modellel: méret, igazítás, irány, padding, margó, szegély.

BEVY-FOGALOM · NODE & VAL

A Node a régebbi Bevy Style komponens utódja. A méreteket a Val enum adja: Val::Px(48.0) (abszolút), Val::Percent(100.0) (a szülő arányában), Val::Auto, illetve Val::Vw/Val::Vh (viewport-arány). A 0.18-ban vannak rövidítő segédfüggvények is, pl. px(48), percent(100) — ugyanazt adják.

teljes képernyős, középre igazító konténer
commands.spawn(Node {
    width: Val::Percent(100.0),
    height: Val::Percent(100.0),
    // flex-irány: a gyerekek oszlopban (egymás alatt) sorakoznak
    flex_direction: FlexDirection::Column,
    // a fő tengely (függőleges) mentén középre
    justify_content: JustifyContent::Center,
    // a kereszt-tengely (vízszintes) mentén középre
    align_items: AlignItems::Center,
    row_gap: Val::Px(12.0),               // térköz a gyerekek között
    padding: UiRect::all(Val::Px(16.0)),  // belső térköz minden oldalon
    ..default()
});

A flexbox-mentális modell egy az egyben átültethető a CSS-ből: flex_direction az irány, justify_content a fő tengely menti igazítás, align_items a kereszt-tengelyé. Abszolút pozicionáláshoz pedig position_type: PositionType::Absolute + left/top/… — ez kell pl. egy sarokban ülő HUD-elemhez.

2.3Színek, szegélyek, képek

A Node csak a layout. A megjelenést külön komponensek adják, amiket ugyanarra az entityre rakunk. Ez az ECS-szerűség lényege: a kinézet komponensek kompozíciója.

0.18 / 0.17 VÁLTOZÁS · BorderColor

A BorderColor már oldalanként tárolja a színt (top/bottom/left/right). Egységes szegélyhez a BorderColor::all(szín) konstruktort használd — a régi BorderColor(szín) forma már nem fordul.

egy „panel” kinézet összerakása
commands.spawn((
    Node {
        width: Val::Px(220.0),
        height: Val::Px(140.0),
        border: UiRect::all(Val::Px(2.0)),   // a szegély VASTAGSÁGA itt van
        ..default()
    },
    BackgroundColor(Color::srgb(0.10, 0.11, 0.16)),
    BorderColor::all(Color::srgb(0.95, 0.71, 0.27)),  // arany keret
    BorderRadius::all(Val::Px(4.0)),
));

Ikonokhoz és pixel-art panelekhez a ImageNode kell. Ez tud egész képet vagy texture atlasz egy celláját megjeleníteni — utóbbi a tipikus, mert a pixel-art ikonok általában egy spritesheet-en élnek.

ikon egy atlaszból
fn spawn_icon(
    mut commands: Commands,
    assets: Res<AssetServer>,
    mut atlases: ResMut<Assets<TextureAtlasLayout>>,
) {
    let texture = assets.load("ui/icons.png");
    // 16x16-os cellák, 8 oszlop, 4 sor egy atlaszban:
    let layout = atlases.add(TextureAtlasLayout::from_grid(
        UVec2::splat(16), 8, 4, None, None,
    ));

    commands.spawn((
        Node { width: Val::Px(32.0), height: Val::Px(32.0), ..default() },
        ImageNode::from_atlas_image(texture, TextureAtlas { layout, index: 3 }),
    ));
}
TIPP · 9-SLICE PANELEK

A „dobozos” UI-keretekhez ne nyújts egy képet (eltorzul a pixel). Használj 9-slice-ot: az ImageNode-on beállítható egy ImageNodeSlicer (slice/tiling), ami a saroktextúrákat fixen tartja, és csak az éleket/közepet ismétli. Így a panel bármekkora lehet pixel-torzulás nélkül.

2.4Szöveg

A szöveg is komponensek kompozíciója. Egy UI-felirat: egy Text + a stílust adó TextFont és TextColor. A világban lebegő szöveghez (fej feletti név, sebződés-szám) a Text2d a megfelelő, mert az a kamerához igazodik.

UI-felirat saját fonttal
fn spawn_label(mut commands: Commands, assets: Res<AssetServer>) {
    commands.spawn((
        Text::new("Csontmező — 1. szint"),
        TextFont {
            font: assets.load("fonts/pixel.ttf"),
            font_size: 16.0,           // tartsd az art-pixel egész többszörösén!
            ..default()
        },
        TextColor(Color::srgb(0.93, 0.90, 0.84)),
        // több sornál a sorigazítás:
        TextLayout::new_with_justify(Justify::Center),
    ));
}
BEVY-FOGALOM · SZÖVEG-SZAKASZOK (TextSpan)

Egy feliraton belül több, eltérő stílusú szakaszt a Text entity TextSpan gyerekeivel adsz (pl. „Élet: ” fehéren, az érték pirosan). Így egy rendszer csak a számot frissíti, a többit békén hagyja — lásd a HUD-frissítést a 4.4-ben.

0.18 ÚJDONSÁG · FONT-VARIÁCIÓK

A 0.18 támogatja a variable font-okat (folytonos vastagság), valamint az aláhúzást és áthúzást, illetve OpenType feature-öket. Egy HUD-ban ez jól jön pl. áthúzott „régi ár” megjelenítéséhez vagy egy aktív menüpont kiemeléséhez vastagítással.

2.5Z-rendezés és UI-rétegek

Az 1.3-ban felsorolt rétegek sorrendjét két dolog dönti el. Alapból a fa-sorrend: a később spawnolt testvér-node a korábbi fölé rajzolódik. Finomabb vezérléshez:

  • ZIndex(i32) – lokális rendezés a testvérek között.
  • GlobalZIndex(i32) – az egész UI-fában érvényes, globális réteg. Ezt használd a modálokhoz/popupokhoz, hogy biztosan minden HUD fölé kerüljenek.
egy modál garantáltan legfelül
commands.spawn((
    Node { /* sötétítő, teljes képernyős overlay */ ..default() },
    BackgroundColor(Color::srgba(0.0, 0.0, 0.0, 0.6)),
    GlobalZIndex(100),   // minden HUD (pl. GlobalZIndex(10)) fölé
));
TIPP · TÖBB UI-KAMERA

Ha külön rétegként akarod kezelni a világot és a HUD-ot (pl. a HUD éles natív felbontáson, a világ skálázva), megteheted több kamerával és UiTargetCamera-val, ami egy UI-fát egy konkrét kamerához rendel. Kezdetben viszont maradj egy kameránál — a GlobalZIndex a legtöbb rétegzési igényt megoldja.

RESZ 03

Eseménykezelés

A kép kész — most hogyan reagáljon kattintásra, hover-re, billentyűre, gamepadre.

3.1Observerek és a picking rendszer

A modern Bevyben a UI-interakció gerince a picking rendszer és az observerek. A picking kiszámolja, melyik entity fölött van az egér, és Pointer<E> eseményeket küld (kattintás, hover, drag…). Az observer pedig egy függvény, amit egy konkrét entityhez (vagy globálisan) kötsz, és lefut, amikor az adott esemény bekövetkezik.

BEVY-FOGALOM · OBSERVER & On<E>

Az observer paramétere On<E> (a korábbi Trigger<E> 0.17-ben erre lett átnevezve). A .observe(…) egy entityre köti, így az csak arra az entityre fut. Az eseményből kiolvasható a célzott entity (ev.entity), és a bugyborékolást a ev.propagate(false) állítja le. A UI-elemek alapból „pickable”-ek, nem kell rájuk külön Pickable — sprite-okra viszont igen.

observer egy konkrét entityre
commands
    .spawn((Button, Node { ..default() }, BackgroundColor(Color::BLACK)))
    .observe(|click: On<Pointer<Click>>, mut commands: Commands| {
        // click.entity = a kattintott entity
        info!("Kattintás: {}", click.entity);
    });

A Pointer<E> esemény-családból a leggyakoribbak: Click, Over (rámegy az egér), Out (lemegy), Pressed/Released (lenyom/felenged), és a drag-család (DragStart, Drag, DragEnd, Drop) — ez utóbbiakra a 4.3-ban (drag & drop) építünk.

3.2Gomb-interakció és vizuális állapotok

Egy „rendes” gomb háromféleképp néz ki: alap, hover, lenyomva. Két út van: observerek (eseményalapú), vagy egy lekérdező rendszer az Interaction komponensre. Nézzük a lekérdezős utat, mert ez illusztrálja a klasszikus ECS-rendszert.

BEVY-FOGALOM · Interaction & Changed

Egy Button entityn a picking automatikusan karbantart egy Interaction komponenst: None, Hovered vagy Pressed. A Changed<Interaction> lekérdezés-szűrő gondoskodik róla, hogy a rendszer csak akkor fusson le egy gombra, amikor annak az állapota valóban megváltozott — nem minden képkockán feleslegesen.

gomb-stílus az Interaction alapján
fn style_buttons(
    mut q: Query<
        (&Interaction, &mut BackgroundColor, &mut BorderColor),
        (Changed<Interaction>, With<Button>),
    >,
) {
    for (interaction, mut bg, mut border) in &mut q {
        match *interaction {
            Interaction::Pressed => {
                *bg = BackgroundColor(Color::srgb(0.18, 0.14, 0.06));
                *border = BorderColor::all(Color::srgb(1.0, 0.85, 0.4));
            }
            Interaction::Hovered => {
                *bg = BackgroundColor(Color::srgb(0.15, 0.16, 0.22));
                *border = BorderColor::all(Color::srgb(0.95, 0.71, 0.27));
            }
            Interaction::None => {
                *bg = BackgroundColor(Color::srgb(0.10, 0.11, 0.16));
                *border = BorderColor::all(Color::srgb(0.35, 0.37, 0.45));
            }
        }
    }
}
// majd: .add_systems(Update, style_buttons)
TIPP · MELYIKET HASZNÁLD?

Vizuális visszajelzéshez (hover-szín) a lekérdező rendszer kényelmes, mert egyszerre kezeli az összes gombot. Egyedi akcióhoz („ez a gomb új játékot indít”) az observer tisztább, mert az adott entityhez tapad — nem kell marker-komponenssel szétválogatni, melyik gomb mit csinál.

3.3Standard widgetek (bevy_ui_widgets)

A 0.17 óta a Bevy szállít egy beépített, logikai widget-készletet a bevy::ui_widgets modulban: Button, Checkbox, Slider, RadioButton/RadioGroup, és a 0.18-ban új Popover és MenuPopup. Ezek a viselkedést adják (fókusz, billentyű-aktiválás, érték-változás esemény), a kinézetet neked kell rájuk tenni — ezért hívják őket „unthemed” (téma nélküli) widgeteknek.

BEVY-FOGALOM · ValueChange & UiWidgetsPlugins

A widgetek ValueChange<T> eseményt adnak ki, amikor az értékük változik (pl. a slider új értéke). A működésükhöz add hozzá a UiWidgetsPlugins-t, a fókusz-alapú billentyűkezeléshez pedig az InputDispatchPlugin-t.

slider érték-változás kezelése
use bevy::ui_widgets::{Slider, SliderValue, ValueChange};

// Spawnolás (a kinézetet külön komponensekkel adod hozzá):
commands

3.4Fókusz, Tab- és irány-navigáció

RESZ 04

Inventory

Egy rácsos tárgylista: adatmodell, a rács kirajzolása, drag & drop, és a reaktív frissítés.

4.1Az adatmodell

Először az adat, csak utána a megjelenítés. Tartsd szét a kettőt: az inventory tartalma egy adatszerkezet, a UI pedig ennek a nézete. Így a logika tesztelhető és független a kirajzolástól.

BEVY-FOGALOM · RESOURCE vs. COMPONENT

A Resource egy globális, egyetlen példányú adat (pl. a játékos inventoryja, ha egy játékos van). A Component entitásokhoz kötött. Ha több „táskás” entitásod lehet (játékos, ládák, NPC-k), tedd az inventoryt komponensbe; ha csak egy van, lehet Resource.

adat-réteg
#[derive(Clone, Copy, PartialEq)]
struct ItemStack { item: ItemKind, count: u16 }

#[derive(Clone, Copy, PartialEq)]
enum ItemKind { Sword, Potion, Coin, Bone }

#[derive(Resource, Default)]
struct Inventory {
    slots: [Option<ItemStack>; 20],   // 20 fix slot
}

4.2A slot-rács kirajzolása

A rácshoz a CSS Grid-mintát használjuk: a konténer Display::Grid, és megadjuk az oszlopokat. Minden slot egy gyerek-node, amit megjelölünk a saját indexével, hogy később tudjuk, melyik melyik.

5 oszlopos slot-rács
#[derive(Component)]
struct SlotUi(usize);   // melyik inventory-indexet jeleníti meg

4.3Drag & drop

A tárgyak áthúzása a picking drag-eseményeire épül. A minta: a DragStart-nál megjegyezzük, honnan indultunk és létrehozunk egy „szellem” ikont, ami az egeret követi (Drag alatt mozgatjuk), végül a célon a Drop eseményben végrehajtjuk a cserét az adatban.

BEVY-FOGALOM · A DRAG ESEMÉNY-CSALÁD

A húzott elemen: Pointer<DragStart> → több Pointer<Drag> (a .delta mutatja az elmozdulást) → Pointer<DragEnd>. A céltáblán pedig a Pointer<Drop> tüzel, és onnan elérhető, mit ejtettek rá. Mivel a UI-node-ok alapból pickable-ek, nem kell külön bekapcsolni.

drop a céltábla-sloton
// Ezt az observert minden slotra rákötjük (a spawnolásnál .observe(on_drop)).
fn on_drop(
    drop: On<Pointer<Drop>>,
    slots: Query<&SlotUi>,
    mut inv: ResMut<Inventory>,
) {
    // drop.entity = a céltábla (ahová ejtettünk)
    // drop.dropped = az elejtett entity (amit húztunk)
    let Ok(&SlotUi(to)) = slots.get(drop.entity) else { return; };
    let Ok(&SlotUi(from)) = slots.get(drop.dropped) else { return; };

    // A csere az ADATON történik; a UI-t a frissítő rendszer rajzolja újra.
    inv.slots.swap(from, to);
}
TIPP · A „SZELLEM” KÖVESSE AZ EGERET

A húzott ikont legkönnyebb egy abszolút pozícionált, GlobalZIndex-szel mindenek fölé emelt node-ként megvalósítani, amit a Pointer<Drag> observerben az aktuális kurzor- pozícióra mozgatsz. A Pickable::IGNORE-t tedd rá, hogy ne fogja el a saját kattintásaidat.

4.4Reaktív frissítés — az adat az igazság

A kulcsdöntés: a UI ne tárolja az állapotot, hanem tükrözze az adatot. Egy frissítő rendszer minden képkockán (vagy csak változáskor) szinkronizálja a slot-node-okat az Inventory tartalmával. Így a drag & drop, a felvétel, az elhasználás — mind csak az adatot bántja, a megjelenés magától követi.

BEVY-FOGALOM · CHANGE DETECTION

A Res<T> automatikusan jelzi, ha módosult: a inv.is_changed() igaz abban a képkockában, amikor az erőforrást írták. Ezzel elkerülöd, hogy minden képkockán feleslegesen újrarajzolj — csak akkor dolgozol, ha tényleg változott valami.

adat → UI szinkron
fn sync_inventory_ui(
    inv: Res<Inventory>,
    slots: Query<(&SlotUi, &Children)>,
    mut commands: Commands,
) {
    if !inv.is_changed() { return; }   // csak változáskor dolgozunk

    for (slot, children) in &slots {
        // a slot előző ikonját töröljük
        for &child in children.iter() {
            commands.entity(child).despawn();
        }
        // ha van benne tárgy, új ikont teszünk bele
        if let Some(stack) = inv.slots[slot.0] {
            commands.entity(/* a slot entity-je */).with_child((
                Node { width: Val::Px(28.0), height: Val::Px(28.0), ..default() },
                // ImageNode az atlaszból az item_kind alapján (lásd 2.3)
            ));
        }
    }
}
TIPP · SZÁMLÁLÓ-SZÖVEG FRISSÍTÉSE

A darabszám-feliratot (pl. „x12”) ne töröld-újraépítsd, hanem tartsd meg a Text entityt, és csak az értékét írd át: a lekérdezésben kérj &mut Text-et, és **text = format!("x{}", n). Ez olcsóbb, mint újraspawnolni.

RESZ 05

Menük

Az állapotgép a menürendszer gerince: ez dönti el, melyik UI látszik és melyik logika fut.

5.1Az állapotgép

Egy menürendszer természetes módon egy véges állapotgép: a játék vagy a főmenüben, vagy játszik, vagy szünetel, vagy az inventoryban van. A Bevy States rendszere pont erre való, és összeköti az állapotokat az ütemezéssel.

BEVY-FOGALOM · STATES, OnEnter/OnExit, in_state

Egy States-t megvalósító enumot az init_state-tel regisztrálsz. Az OnEnter(X)/OnExit(X) ütemezésekbe rakott rendszerek pontosan az állapotba lépéskor/onnan kilépéskor futnak (ide tesszük a menü fel-/leépítését). Az in_state(X) futásfeltétel pedig egy Update-rendszert csak az adott állapotban enged futni. Állapotot a NextState<X> erőforrással váltasz.

állapotok definíciója és bekötése
#[derive(States, Debug, Clone, PartialEq, Eq, Hash, Default)]
enum GameState {
    #[default]
    MainMenu,
    Playing,
    Paused,
    Inventory,
}

fn main() {
    App::new()
        .add_plugins(DefaultPlugins)
        .init_state::<GameState>()
        // a főmenü csak belépéskor épül fel, kilépéskor lebomlik:
        .add_systems(OnEnter(GameState::MainMenu), spawn_main_menu)
        // a hotkey-rendszer csak játék közben fusson:
        .add_systems(Update, hotkeys.run_if(in_state(GameState::Playing)))
        .run();
}

5.2Főmenü — felépítés és automatikus lebontás

A főmenüt az OnEnter(MainMenu)-ben spawnoljuk. A trükk: nem kell kézzel kitakarítani kilépéskor — megjelöljük a menü gyökerét egy állapot-élettartamú komponenssel, ami automatikusan despawnolja az egész fát, amikor elhagyjuk az állapotot.

A korábbi StateScoped komponens neve több körön át változott; a 0.18-ban DespawnOnExit<S> a neve (és van fordított párja, a DespawnOnEnter<S>). Ha régebbi tutorialban StateScoped-ot látsz, ez az utódja.

6.2Megerősítő dialógus

A leggyakoribb modál: „Biztosan kilépsz? — Igen / Mégse”. A trükk, hogy a dialógus ne kötődjön egy konkrét akcióhoz: küldjön egy általános eredmény-eseményt, amire a hívó reagál. Így ugyanaz a dialógus-komponens újrahasználható minden „biztos?” helyzethez.

BEVY-FOGALOM · SAJÁT ESEMÉNYEK

Saját üzenetet egy #[derive(Event)] (ill. cél-entityhez kötött #[derive(EntityEvent)]) típussal definiálsz. Globális eseményt MessageWriter/MessageReader-rel írsz/olvasol; entity-eseményt observerrel kapsz el. A megerősítő gombok ezt küldik, a hívó logika feliratkozik rá.

újrahasználható megerősítés
#[derive(Message)]
struct ConfirmResult { accepted: bool }

// az "Igen" gomb observere:
.observe(|_: On<Pointer<Click>>,
          mut writer: MessageWriter<ConfirmResult>,
          modal: Single<Entity, With<ModalRoot>>,
          mut commands: Commands| {
    writer.write(ConfirmResult { accepted: true });
    commands.entity(*modal).despawn();   // a modál bezárul
});

// a hívó oldalon valahol:
fn on_confirm(mut reader: MessageReader<ConfirmResult>,
              mut exit: MessageWriter<AppExit>) {
    for r in reader.read() {
        if r.accepted { exit.write(AppExit::Success); }
    }
}

6.3Tooltip és Popover

A tooltip a hover-eseményekre épül: Pointer<Over>-re megjelenítesz egy kis panelt a kurzor mellett, Pointer<Out>-ra eltünteted. A nehéz rész a pozicionálás: a tooltip ne lógjon le a képernyőről. Ezt a 0.18-ban már nem kell kézzel számolnod.

A Popover komponens egy abszolút pozícionált node-ra téve automatikus popup-elhelyezést ad (a népszerű floating-ui mintájára): megadsz preferált irányokat (fent/lent/balra/jobbra), és ha az ablak széle levágná, magától „átfordul” a legjobban látható oldalra. Ugyanez működik görgetett konténerben és ablak-átméretezéskor is. Dropdown menühöz az erre épülő MenuPopup ad kész nyitás/zárás eseményeket és billentyű-navigációt.

6.4Toast-értesítés — ami magától eltűnik

A „+5 érme” vagy „Mentve” típusú értesítés időzítve tűnik el. A minta: spawnolsz egy node-ot egy Timer-komponenssel, és egy rendszer a Time erőforrással lépteti az időzítőt, lejáratkor pedig despawnolja.

BEVY-FOGALOM · Time & Timer

A Res<Time> adja a két képkocka közt eltelt időt (time.delta()), keretsebességtől függetlenül. A Timer ezt gyűjti, és a tick(delta) után a just_finished() jelzi a lejáratot. Ez a helyes mód időzített dolgokra — soha ne számolj képkockákat, mert az gépfüggő.

eltűnő toast
#[derive(Component)]
struct Toast(Timer);

fn spawn_toast(commands: &mut Commands, msg: &str) {
    commands.spawn((
        Toast(Timer::from_seconds(2.0, TimerMode::Once)),
        Node { position_type: PositionType::Absolute, top: Val::Px(16.0),
               right: Val::Px(16.0), padding: UiRect::all(Val::Px(8.0)), ..default() },
        BackgroundColor(Color::srgb(0.10, 0.16, 0.12)),
        GlobalZIndex(400),
        children![ Text::new(msg.to_string()) ],
    ));
}

fn tick_toasts(time: Res<Time>, mut q: Query<(Entity, &mut Toast)>, mut commands: Commands) {
    for (e, mut toast) in &mut q {
        if toast.0.tick(time.delta()).just_finished() {
            commands.entity(e).despawn();
        }
    }
}

}

7.2Teljesítmény: mikor frissíts?

A leggyakoribb kezdő-hiba, hogy minden képkockán újraépítjük a UI-t. A Bevy eszközei pont ez ellen vannak:

  • Observer esemény helyett polling helyett. Egyedi akcióra (kattintás) observer, ne képkockánként figyelő rendszer.
  • Change detection. Changed<T> szűrő és is_changed() — csak akkor dolgozz, ha tényleg változott az adat.
  • Frissíts, ne újraépíts. Egy számláló-szöveget módosíts a helyén, ne despawnold és spawnold újra az egész panelt.
  • Futásfeltételek. A run_if(in_state(…)) egész rendszereket kapcsol ki, amikor nem relevánsak.
