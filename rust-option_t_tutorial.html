Option<T>
Amikor egy érték vagy van, vagy nincs

A Rust nem ismeri a null-t. Helyette egy típussal mondja ki az igazságot: egy érték vagy létezik (Some(érték)), vagy hiányzik (None) — és a fordító nem engedi, hogy erről megfeledkezz.

// A teljes Option a standard könyvtárban — ennyi az egész:
enum Option<T> {
    Some(T),  // van érték, és az T típusú
    None,     // nincs érték
}
// 01 — a probléma

Miért nem elég a null?

A legtöbb nyelvben (C, Java, JavaScript, Python a maga None-jával) bármelyik referencia lehet „üres”, és ezt a típusrendszer nem jelzi. Egy String típusú változó lehet, hogy string — de lehet, hogy null. Ezt csak futásidőben tudod meg, jellemzően egy NullPointerException vagy egy TypeError: cannot read property of undefined formájában.

Tony Hoare, a null referencia feltalálója ezt utólag „milliárd dolláros hibának” nevezte. A Rust válasza radikális: a hiány nem rejtett tulajdonság, hanem külön típus.

A kulcsgondolat

Rustban egy String mindig string. Ha egy érték hiányozhat, annak típusa nem String, hanem Option<String>. A két típus nem keverhető — a fordító kényszerít rá, hogy a „nincs érték” esetet kezelt állapotba hozd, mielőtt a belsejéhez nyúlnál.

// Ez le sem fordul — Option<i32> és i32 két különböző típus:
let x: i32 = 5;
let y: Option<i32> = Some(5);

let osszeg = x + y; // ❌ fordítási hiba: i32 + Option<i32> nem megy
// Előbb ki KELL bontanod y-t — és közben dönteni a None esetről is.
// 02 — egy valódi példa

Miért ad a Vec::pop() Option-t?

A Vec::pop() leveszi és visszaadja a vektor utolsó elemét. Csakhogy a lista lehet üres — és akkor nincs mit visszaadni. A visszatérési típus ezért nem T, hanem Option<T>: a függvény szignatúrája őszintén közli, hogy a hívás kudarcot is vallhat.

tilemap.rs
enum TileKind { Grass, Stone, Bone, Lava }

let mut tiles: Vec<TileKind> = vec![TileKind::Grass, TileKind::Stone];

let tile = tiles.pop(); // típusa: Option<TileKind> → Some(Stone)
let tile = tiles.pop(); // Some(Grass)
let tile = tiles.pop(); // None — a lista kiürült, és ez NEM hiba,
                          // hanem egy szabályos, típusban kódolt állapot
Próbáld ki: tiles.pop()
Vedd le a csempéket a stackről, és figyeld, mit ad vissza az Option.
tiles.pop() visszatérési értéke — még nem hívtad meg —

Figyeld meg: amikor a stack kiürül, a pop() nem dob kivételt és nem omlik össze — egyszerűen None-t ad, amit a kódod ágként kezel.

// 03 — kibontás biztonságosan

Kibontás = ellenőrzés egy lépésben

Az Option belsejéhez nem férsz hozzá „csak úgy”. A kibontás mintaillesztéssel történik, és ez a lényeg: a „van-e érték?” kérdés és az érték kinyerése egyetlen, szétválaszthatatlan művelet. Nem fordulhat elő, hogy ellenőrzöl, aztán két sorral lejjebb mégis a hiányzó értékhez nyúlsz.

1. match — minden eset kötelező

A match a legteljesebb forma: a fordító kikényszeríti, hogy a Some és a None ágat is megírd. Ha az egyiket kihagyod, a kód le sem fordul.

match tiles.pop() {
    Some(tile) => {
        // itt a `tile` már sima TileKind — garantáltan létezik
        spawn_tile_sprite(&mut commands, tile);
    }
    None => {
        info!("A pálya elfogyott, nincs több csempe.");
    }
}

2. if let — amikor csak az egyik ág érdekel

Ha a None esetben nincs teendőd, az if let tömörebb. A minta (Some(tile)) illeszkedését vizsgálja, és siker esetén rögtön be is köti a tile változót.

if let Some(tile) = tiles.pop() {
    // csak akkor futunk ide, ha tényleg volt elem
    spawn_tile_sprite(&mut commands, tile);
}
// None esetén egyszerűen átugorjuk a blokkot

3. let ... else — korai kilépés

Gyakori minta függvényekben és Bevy-rendszerekben: ha nincs érték, lépj ki korán, a függvény többi része pedig már a kibontott értékkel dolgozhat, behúzás nélkül.

fn place_next_tile(tiles: &mut Vec<TileKind>) {
    let Some(tile) = tiles.pop() else {
        return; // nincs több csempe → csendben kilépünk
    };

    // innentől a `tile` kibontva él a függvény végéig
    println!("Lerakom: {tile:?}");
}

4. while let — ürítés ciklusban

Bónusz: a pop() + while let páros a klasszikus „dolgozd fel az összes elemet” minta. A ciklus pontosan addig fut, amíg a pop() Some-ot ad.

while let Some(tile) = tiles.pop() {
    process_tile(tile);
}
// a ciklus után a vektor garantáltan üres
// 04 — a vészkijárat

Az unwrap() és társai

Az unwrap() azt mondja: „tudom, hogy itt van érték, add ide”. Ha igazad van, megkapod a T-t. Ha tévedsz, és az Option None, a program pánikkal leáll — pontosan az a fajta futásidejű összeomlás, amit az Option egyébként megelőzne.

let tile = tiles.pop().unwrap();
// üres vektornál:
// thread 'main' panicked at 'called `Option::unwrap()` on a `None` value'
Ökölszabály

Az unwrap() nem kezeli a hiányt — csak letagadja. Éles, normál kódban a match, az if let vagy a let ... else a helyes választás. Az unwrap() elfogadható prototípusban, tesztben, vagy ott, ahol egy invariáns miatt bizonyíthatóan nem lehet None — de ilyenkor is jobb az expect().

expect() — unwrap üzenettel

Ugyanúgy pánikol None esetén, de te adod meg a hibaüzenetet, ami a naplóban azonnal elárulja, melyik feltételezésed dőlt meg.

let tile = tiles.pop()
    .expect("a pályagenerátor mindig legalább 1 csempét tesz a listába");

Szelídebb alternatívák

// Alapérték, ha None:
let tile = tiles.pop().unwrap_or(TileKind::Grass);

// Alapérték lustán (csak None esetén fut le a closure):
let tile = tiles.pop().unwrap_or_else(|| default_tile());

// Transzformáció kibontás nélkül — az Option "belsejében" dolgozol:
let nev: Option<String> = tiles.pop().map(|t| format!("{t:?}"));

// Gyors logikai kérdések:
if tile.is_some() { /* ... */ }
if tile.is_none() { /* ... */ }
// 05 — puska

A nagy kép

Az Option<T> a hiány lehetőségét a típusba költözteti. Más nyelvekben a null egy időzített bomba, ami bárhol felrobbanhat; Rustban a hiányzó érték egy fordításidőben látható, kötelezően kezelendő eset. Ezért nincs Rustban NullPointerException — a megfelelője le sem fordul.
