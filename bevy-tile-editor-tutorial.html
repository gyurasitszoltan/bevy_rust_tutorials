Saját pálya-editor — a játékon belül, tisztán elválasztva

Pixelart top-down játéknál a legjobb editor a játékon belül van: ugyanabban a render-pipeline-ban szerkeszted a pályát, ahogy a játékos látni fogja. De hogyan teszed ezt be úgy, hogy ne keveredjen a játékkóddal, egy gombbal váltható legyen, és release-ben ne is legyen ott? Három ortogonális kapcsoló a válasz — state · plugin · feature flag —, plusz egy közös adatmodell: a LevelData.

↑ Élő minta: ugyanaz a pálya editor módban (rács, paletta, kurzor) és játék módban (tiszta kép, sétáló karakter) — a render azonos. A váltás: Ctrl+E.

1. fejezet

In-game vs külső editor

A kérdés egyszerű: csináljak saját pálya-editort a játékon belül, vagy használjak külsőt (pl. Tiled)? Top-down pixelart játéknál a saját, beépített editornak van egy döntő előnye: szerkesztés közben pontosan azt látod, amit a játékos is fog. Ugyanaz a kamera, zoom, pixel-perfect renderelés, ugyanazok a fények, árnyékok, shaderek — a végleges kinézetben.

Próbáld ki: fess a palettával, kapcsold be a collision-réteget, majd nyomj ▶ Play-t — a rács és a kurzor eltűnik, és pontosan a játékbeli képet kapod. Ez ugyanaz a LevelData, ugyanaz a render.

Bal-katt (húzva is): fest · jobb-katt: radíroz. A collision a nem-járható csempéket pirossal jelöli — pont, ahogy a játék is számolja.

Egy csempe ugyanis top-down játékban nem csak grafika. Egyetlen tile gyakran egyszerre hordoz:

⊞ Mit jelent EGY csempe?
   grafika  +  collision  +  gameplay-szabály  +  layer  +  interakció
   (sprite)    (járható?)     (sebzés? lassít?)      (föld/   (ajtó, kapcsoló,
                                                      tető)    trigger-zóna)
Külső editorban (Tiled) ezek egy része kezelhető, de mindig van egy absztrakciós réteg az editor és a játék között — export/import, eltérő tile-fogalom. Saját in-game editorban pont azt szerkeszted, amit a motor használ.

A hátrány viszont valós: egy editor önmagában is fejlesztési projekt (UI, mentés/betöltés, undo/redo, layer-kezelés, paletta, selection, brush, fill, copy-paste). Ezért a jó kompromisszum elsőre egy minimális in-game editor:

  • csempe-választás palettából · bal-katt: lerak · jobb-katt: töröl
  • layer-váltás · collision-overlay kapcsoló · rács
  • mentés/betöltés saját formátumba (JSON) · Ctrl+E a játék/editor váltás
▲ A legfontosabb architekturális döntés
A pálya formátuma legyen független az editortól. A játék ugyanazt a level.json-t töltse be, amit az editor ment. Így a belső és egy esetleges külső editor is ugyanarra a formátumra dolgozhat — és pont ez visz át a következő fejezetekre: mi az a közös formátum, és hogyan ül rá a kódszervezés.
2. fejezet

Három kapcsoló: state · plugin · feature

A célod három, látszólag ellentmondó dolog egyszerre: az editor (1) legyen ott és élőben kapcsolható, (2) ne keveredjen a játékkóddal, (3) release-ben ne zavarjon. A trükk: ne egy eszközzel próbáld megoldni mindhármat, hanem három független kapcsolóval, amelyek mindegyike a maga problémáját intézi.

⊞ A három ortogonális kapcsoló
  STATE     AppMode::Game / Editor   →  mi fut MOST?      (futásidő)
  PLUGIN    GamePlugin / EditorPlugin →  HOL van a kód?     (szervezés)
  FEATURE   #[cfg(feature = "editor")] →  benne van a buildben? (fordítás)
A state dönti el, mely rendszerek futnak; a plugin tartja egy helyen az editor kódját; a feature flag dönti el, hogy a release binárisba egyáltalán belekerül-e. Külön tengelyek — együtt adják ki, amit szeretnél.

A fő kapcsoló egy State: játszunk vagy szerkesztünk. Érdemes nem egyetlen óriás állapotba zsúfolni mindent — a játék belső állapota (loading, pause) külön géppé válik, így az editor nem borítja szét a menü/pause logikát.

src/app_state.rs — két, független állapotgép
use bevy::prelude::*;

/// A FŐ kapcsoló: játszunk vagy szerkesztünk?
#[derive(States, Default, Copy, Clone, Debug, PartialEq, Eq, Hash)]
pub enum AppMode {
    #[default]
    Game,
    Editor,
}

/// A játék BELSŐ állapota — ehhez az editor nem nyúl.
#[derive(States, Default, Copy, Clone, Debug, PartialEq, Eq, Hash)]
pub enum GameState {
    #[default]
    Loading,
    Playing,
    Paused,
}
⚙ Bevy fogalom — States, NextState, in_state

Az init_state::<AppMode>() létrehozza az állapotgépet a #[default] változattal indulva. Az aktuálisat a Res<State<AppMode>> (.get()), a váltást a ResMut<NextState<AppMode>> (.set(..)) adja. A run_if(in_state(AppMode::Editor)) egy rendszert csak abban a módban enged futni. Két külön States nyugodtan élhet egymás mellett — ortogonális tengelyek.

3. fejezet

Az igazságforrás: LevelData

Ez a tutorial legfontosabb mondata. Ne így gondolkodj: „az editor módosítja a tile-sprite entitásokat, a játék meg majd kiolvassa belőlük”. Hanem így: van egy közös LevelData; az editor azt módosítja, a játék abból renderel és épít collisiont, a mentés azt szerializálja. A fenti játszótér editora pontosan ezt a struktúrát írja.

src/shared/level_data.rs — a közös adatmodell
use bevy::prelude::*;

/// Egy csempe azonosítója az atlaszban.
#[derive(Clone, Copy, Debug, PartialEq, Eq, Hash)]
pub struct TileId(pub u32);

/// Egy réteg: csempék + ütközik-e a réteg.
pub struct TileLayer {
    pub name: String,
    pub tiles: Vec<Option<TileId>>,   // None = üres cella
    pub collision: bool,
}

/// Egy elhelyezett objektum (spawn, ajtó, láda, NPC, trigger-zóna).
pub struct LevelObject {
    pub id: String,
    pub position: IVec2,
    pub kind: ObjectKind,
}

pub enum ObjectKind { PlayerSpawn, Door, Chest, Npc, TriggerZone }

/// A teljes pálya — EZ az igazságforrás.
pub struct LevelData {
    pub width: u32,
    pub height: u32,
    pub layers: Vec<TileLayer>,
    pub objects: Vec<LevelObject>,
}

impl LevelData {
    pub fn set_tile(&mut self, layer: usize, x: u32, y: u32, tile: Option<TileId>) {
        let i = (y * self.width + x) as usize;
        self.layers[layer].tiles[i] = tile;
    }
    pub fn get_tile(&self, layer: usize, x: u32, y: u32) -> Option<TileId> {
        self.layers[layer].tiles[(y * self.width + x) as usize]
    }
}

/// A betöltött pálya resource-ként. A `dirty` jelzi: újra kell rajzolni a vizuált.
#[derive(Resource)]
pub struct CurrentLevel {
    pub data: LevelData,
    pub dirty: bool,
}

Ez a shared/ rétegben él, és se a game, se az editor belső logikájától nem függ. Mindkét oldal ezt használja:

⊞ Ki mit csinál a LevelData-val?
  game/map.rs         → LevelData alapján spawnol / renderel
  game/collision.rs   → LevelData alapján collision-map
  editor/brush.rs      → LevelData módosítása (lerak/töröl)
  shared/save_load.rs → LevelData mentése / betöltése (level.json)
Ez garantálja, hogy amit editorban látsz, az ugyanaz, amit a játék használ — mert nincs két külön „igazság”, csak egy.
▲ Formátum > editor
Mivel az editor nem birtokolja a formátumot (csak egy a LevelData íróiból), később akár külső szerkesztőt is használhatsz ugyanarra a level.json-ra — vagy generálhatsz pályát kóddal. Az adat a központ, az editor csak egy ráírt eszköz.
4. fejezet

Adat ⇄ vizuális: a dirty-sync

Ha a LevelData az igazság, akkor a képernyőn látható csempe-entitások csak a vizuális tükörképe. A folyam egyirányú: az editor a adatot írja, egy rendszer pedig a vizuált az adathoz igazítja — de csak akkor, ha tényleg változott (dirty). Így nem rajzolunk újra fölöslegesen minden frame-ben.

▶ JÁTSZÓTÉR — LevelData → vizuál (dirty-sync)

Bal oldalt a LevelData (az igazság), jobb oldalt a renderelt csempék. Az editor írja az adatot → a cella dirty lesz (sárga keret) → a sync_tilemap_visuals átmásolja a vizuálba és törli a dirty-jelet.

dirty cellák: 0

src/game/map.rs — a vizuál az adatot követi
use bevy::prelude::*;
use crate::shared::level_data::CurrentLevel;

#[derive(Component)]
struct TileSprite;     // marker: ez egy vizuális csempe-entitás

/// A LevelData a forrás; ez tükrözi vizuálisan — CSAK ha változott.
fn sync_tilemap_visuals(
    mut level: ResMut<CurrentLevel>,
    mut commands: Commands,
    old: Query<Entity, With<TileSprite>>,
    // assets: a TileId → atlasz-index leképezés...
) {
    if !level.dirty {
        return;                       // nincs változás → nincs munka
    }

    // EGYSZERŰ verzió: töröld és építsd újra az egész tilemapet
    for e in &old {
        commands.entity(e).despawn();
    }
    for (li, layer) in level.data.layers.iter().enumerate() {
        for (i, cell) in layer.tiles.iter().enumerate() {
            let Some(tile) = cell else { continue };   // None = üres
            let x = (i as u32 % level.data.width) as f32;
            let y = (i as u32 / level.data.width) as f32;
            commands.spawn((
                TileSprite,
                // Sprite::from_atlas_image(...) a tile.0 indexszel,
                Transform::from_xyz(x * 16.0, y * 16.0, li as f32),
            ));
            let _ = tile;
        }
    }

    level.dirty = false;              // elvégeztük → tiszta
}
⚠ Kezdj az egyszerűvel, optimalizálj később
A teljes újrarajzolás elsőre tökéletes. Ha lassú lesz, vezess be dirty cellákat: egy DirtyTiles { cells: Vec<(u32, u32, usize)> } erőforrásba gyűjtöd a megváltozott (x, y, layer) hármasokat, és a sync csak azokat frissíti. A Res::is_changed() is segít — de a saját dirty flag explicitebb, és pont a fenti demó logikája.

A modul-fa: game · editor · shared

A kód három fő mappára oszlik, és ez a felosztás a teljes architektúra gerince: game/ = a játék futtatása, editor/ = szerkesztő eszközök, shared/ = a közös adatmodell, amit mindkettő használ.

⊞ A projekt fa
  src/
  ├─ main.rs            App + DefaultPlugins + GamePlugin (+ EditorPlugin feature mögött)
  ├─ app_state.rs       AppMode · GameState
  │
  ├─ shared/            KÖZÖS — nem függ se game-től, se editortól
  │  ├─ mod.rs
  │  ├─ level_data.rs     LevelData · TileId · TileLayer · LevelObject · CurrentLevel
  │  ├─ tile_defs.rs      csempe-definíciók (járható? atlasz-index?)
  │  └─ save_load.rs      level.json mentés/betöltés
  │
  ├─ game/              A JÁTÉK
  │  ├─ plugin.rs         GamePlugin
  │  ├─ player.rs         camera.rs  enemy.rs  interaction.rs
  │  ├─ map.rs            LevelData → sprite-ok (sync_tilemap_visuals)
  │  └─ collision.rs      LevelData → collision-map
  │
  └─ editor/            A SZERKESZTŐ (feature mögött)
     ├─ plugin.rs        EditorPlugin · EditorOnly · toggle_editor_mode
     ├─ state.rs          EditorState · EditorTool
     ├─ input.rs  brush.rs  palette.rs
     ├─ ui.rs             editor UI gyökér
     └─ overlays.rs       rács · collision-overlay · kurzor
A shared a közepén áll, a game és az editor köré szerveződik. Az editor egy új csempe-eszköze egyetlen fájl változása — a játékhoz hozzá sem kell nyúlni.

A vezérelv egy mondatban: az editor ne manipulálja közvetlenül a játék belső logikáját. Inkább legyen egy közös LevelData, amit az editor módosít, a game pedig renderel és használ. A találkozási pont az adat, nem a kód.

⊞ A megengedett irányok
   game  ───►  shared  ◄───  editor        ✔ megengedett
   game  ──✗──  editor                       ✘ TILOS

⚠ Miért pont ez az irány?
Mert ez teszi lehetővé, hogy a game az editor nélkül is leforduljon és fusson (release build). Ha a game bárhol importálna az editor-ból, a feature flag mögé rejtett editor kihúzná a talajt a játék alól. A „shared befelé, semmi kifelé” szabály a feltétele a 8. fejezet feature-flagjének.
7. fejezet

A négy réteg

Másik nézőpontból ugyanez négy funkcionális rétegre bomlik. Ez segít eldönteni, egy új rendszer hová kerül — és főleg: mi fut mindkét módban.

⊞ A négy réteg
  1. DATA          LevelData · TileId · ObjectKind · map-formátum
  2. RUNTIME       render · collision · gameplay · player · enemies
  3. EDITOR        tools · UI · overlays · brush · palette
  4. PERSISTENCE   save/load JSON · asset-betöltés
A DATA és a PERSISTENCE közös. A RUNTIME-ból a render rész közös (editorban is látni akarod a pályát!), a gameplay rész csak játék módban fut. Az EDITOR réteg csak editor módban.
RétegTartalomMikor fut?

8. fejezet

AppMode + main.rs + feature flag

A main rövid marad: a beépített pluginok mellé a GamePlugin mindig bekerül, az EditorPlugin viszont csak az editor feature mögött. Ez a harmadik kapcsoló: fordításidőben dönti el, benne van-e az editor a binárisban.

src/main.rs — a belépési pont
use bevy::prelude::*;

mod app_state;
mod shared;
mod game;
#[cfg(feature = "editor")]
mod editor;

use app_state::AppMode;
use game::GamePlugin;
#[cfg(feature = "editor")]
use editor::EditorPlugin;

fn main() {
    let mut app = App::new();

    app.add_plugins(DefaultPlugins.set(ImagePlugin::default_nearest()))
        .init_state::<AppMode>()       // a FŐ kapcsoló állapotgépe
        .add_plugins(GamePlugin);       // a játék MINDIG bekerül

    // az editor CSAK akkor fordul a buildbe, ha kéred:
    #[cfg(feature = "editor")]
    app.add_plugins(EditorPlugin);

    app.run();
}
Cargo.toml — a feature definíciója
[features]
default = []
editor = []
⊞ Buildek
  cargo run --features editor          # fejlesztés: Ctrl+E él, editor benne
  cargo build --release                # release JÁTÉK: editor SEHOL
  cargo build --release --features editor   # belső teszt-build editorral
Release-ben az ImagePlugin::default_nearest() adja a pixel-perfect (nearest) skálázást — ugyanaz a render editorban és játékban.
⚙ Bevy / Cargo fogalom — feature flag

A #[cfg(feature = "editor")] attribútum a Rust feltételes fordítása: a mögötte álló mod/use/sor csak akkor kerül a buildbe, ha a feature aktív. Így az editor kódja, függőségei és binárismérete a release játékból teljesen kimaradnak — miközben a forrásban végig ott van.

A GamePlugin — mi fut editorban?

A GamePlugin a 7. fejezet táblázatát kódolja: a gameplay rendszerek run_if(in_state(AppMode::Game)), a render rendszerek viszont feltétel nélkül futnak — mert editorban is a végleges pályát akarod látni.

src/game/plugin.rs
use bevy::prelude::*;
use crate::app_state::AppMode;

pub struct GamePlugin;

impl Plugin for GamePlugin {
    fn build(&self, app: &mut App) {
        app
            .add_systems(Startup, setup_game)
            // CSAK játék módban: a tényleges gameplay
            .add_systems(
                Update,
                (
                    player_input,
                    player_movement,
                    camera_follow_player,
                    enemy_ai,
                    interaction,
                )
                    .run_if(in_state(AppMode::Game)),

            .add_systems(
                Update,
                (
                    sync_tilemap_visuals,   // LevelData → sprite-ok
                    render_objects,
                    update_animated_tiles,
                ),
            );
    }
}

10. fejezet

Az EditorPlugin — setup/cleanup

Az EditorPlugin minden editor-viselkedést egy helyen tart. A belépéskor (OnEnter) felépíti az editor UI-t és overlayeket, kilépéskor (OnExit) eltakarítja őket. Az editor-specifikus entitások egy EditorOnly markert kapnak — így a takarítás egyszerű.

src/editor/mod.rs
pub mod plugin;
pub mod state;
pub mod input;
pub mod brush;
pub mod ui;
pub mod overlays;

pub use plugin::EditorPlugin;
src/editor/plugin.rs
use bevy::prelude::*;
use crate::app_state::AppMode;
use super::state::EditorState;

pub struct EditorPlugin;

impl Plugin for EditorPlugin {
    fn build(&self, app: &mut App) {
        app
            .init_resource::<EditorState>()
            // a kapcsoló MINDIG fut (de csak az `editor` feature-rel fordul be):
            .add_systems(Update, toggle_editor_mode)
            // belépés/kilépés: az editor-réteg fel- és leépítése
            .add_systems(OnEnter(AppMode::Editor), setup_editor)
            .add_systems(OnExit(AppMode::Editor), cleanup_editor)
            // CSAK editor módban: az eszközök
            .add_systems(
                Update,
                (
                    editor_camera_controls,
                    editor_mouse_input,
                    editor_brush_system,
                    editor_palette_ui,
                    editor_overlay_system,
                    editor_save_shortcut,
                )
                    .run_if(in_state(AppMode::Editor)),
            );
    }
}

/// Minden editor-specifikus entitás (UI, rács, kurzor) ezt kapja.
#[derive(Component)]
pub struct EditorOnly;

fn setup_editor(mut commands: Commands) {
    // editor UI-gyökér, rács-overlay, csempe-kurzor, paletta...
    commands.spawn((EditorOnly, Name::new("Editor Root")));
}

fn cleanup_editor(mut commands: Commands, q: Query<Entity, With<EditorOnly>>) {
    for e in &q {
        commands.entity(e).despawn();   // 0.18: a leszármazottakkal együtt
    }
}

fn editor_save_shortcut() {}
⚙ Bevy fogalom — OnEnter / OnExit (+ DespawnOnExit)

Az OnEnter(state) egyszer fut belépéskor (setup), az OnExit(state) egyszer kilépéskor (teardown). Az EditorOnly markeres kézi takarítás explicit; alternatívaként a spawnolt entitásokra tehetsz DespawnOnExit(AppMode::Editor)-t, és akkor maguktól eltűnnek kilépéskor — ugyanaz az elv, mint a scene-eknél.

11. fejezet

Mód-váltás: Ctrl+E

A váltás egyetlen kis rendszer: figyeli a Ctrl+E-t, és átállítja a NextState<AppMode>-ot. Ez a rendszer mindig fut (nincs run_if), hiszen mindkét irányba kell — de mivel az EditorPlugin-ban él, a feature flag mögött a release-ből kimarad.

src/editor/plugin.rs — a kapcsoló
use bevy::prelude::*;
use crate::app_state::AppMode;

fn toggle_editor_mode(
    keys: Res<ButtonInput<KeyCode>>,
    mode: Res<State<AppMode>>,
    mut next: ResMut<NextState<AppMode>>,
) {
    let ctrl = keys.pressed(KeyCode::ControlLeft)
        || keys.pressed(KeyCode::ControlRight);

    if ctrl && keys.just_pressed(KeyCode::KeyE) {
        next.set(match mode.get() {
            AppMode::Game => AppMode::Editor,
            AppMode::Editor => AppMode::Game,
        });
    }
}
▲ Ez az egyetlen „mindig fut” editor-darab
Minden más editor-rendszer run_if(in_state(AppMode::Editor)) mögött van. A toggle_editor_mode a kivétel — de mivel az egész EditorPlugin a #[cfg(feature = "editor")] mögött kerül be, release-ben ez a sor sem létezik. A játékos nem tud véletlenül editorba kapcsolni.
12. fejezet

EditorState + tile-lerakás

Az editor állapotát egy EditorState resource hordozza: melyik réteg aktív, melyik csempe van kijelölve, melyik eszköz, és a kapcsolók (rács, collision). A 4. fejezet szerint az ecset a LevelData-t írja, és dirty-re állítja — a vizuál onnantól magától követi.

src/editor/state.rs
use bevy::prelude::*;
use crate::shared::level_data::TileId;

#[derive(Resource)]
pub struct EditorState {
    pub active_layer: usize,
    pub selected_tile: Option<TileId>,
    pub tool: EditorTool,
    pub show_collision: bool,
    pub show_grid: bool,
}

impl Default for EditorState {
    fn default() -> Self {
        Self {
            active_layer: 0,
            selected_tile: Some(TileId(1)),
            tool: EditorTool::Brush,
            show_collision: true,
            show_grid: true,
        }
    }
}

#[derive(Clone, Copy, Debug, PartialEq, Eq)]
pub enum EditorTool { Brush, Eraser, Fill, Select, ObjectPlace }
src/editor/brush.rs — egér → cella → LevelData
use bevy::prelude::*;
use crate::editor::state::{EditorState, EditorTool};
use crate::shared::level_data::CurrentLevel;

const TILE: f32 = 16.0;

fn editor_brush_system(
    windows: Query<&Window>,
    camera: Query<(&Camera, &GlobalTransform)>,
    mouse: Res<ButtonInput<MouseButton>>,
    editor: Res<EditorState>,
    mut level: ResMut<CurrentLevel>,
) {
    let Ok(window) = windows.single() else { return };
    let Some(cursor) = window.cursor_position() else { return };
    let Ok((cam, cam_tf)) = camera.single() else { return };
    let Ok(world) = cam.viewport_to_world_2d(cam_tf, cursor) else { return };

    // világ-koordináta → rács-cella
    let gx = (world.x / TILE).floor() as i32;
    let gy = (world.y / TILE).floor() as i32;
    if gx < 0 || gy < 0
        || gx as u32 >= level.data.width
        || gy as u32 >= level.data.height {
        return;
    }
    let (x, y, layer) = (gx as u32, gy as u32, editor.active_layer);

    // bal-katt: fest (radír esetén None) · jobb-katt: radíroz
    if mouse.pressed(MouseButton::Left) {
        let tile = match editor.tool {
            EditorTool::Eraser => None,
            _ => editor.selected_tile,
        };
        level.data.set_tile(layer, x, y, tile);
        level.dirty = true;          // <- innen a sync újraépíti a vizuált
    } else if mouse.pressed(MouseButton::Right) {
        level.data.set_tile(layer, x, y, None);
        level.dirty = true;
    }
}
⚙ Bevy fogalom — viewport_to_world_2d

A Camera::viewport_to_world_2d(cam_tf, cursor) a képernyő-pixelt világ-koordinátává alakítja (figyelembe véve a kamera pozícióját és zoomját). Innen az osztás TILE-lal + floor() adja a rács-cellát. A single() 0.18-ban Result-ot ad, ezért a let Ok(..) = ... else { return } minta.

14. fejezet

Release & merre tovább?

Az editor „benne van, de ne zavarjon” három, egyre erősebb szinten oldható meg — kombinálhatók:

⊞ Három szint az editor kihagyására
  1. Runtime kikapcsolás   az editor bent van, de csak AppMode::Editor alatt fut
                          + egyszerű   − a release binárisban is benne van
  2. Feature flag          #[cfg(feature = "editor")] — csak kérésre fordul be
                          + a release-ből KIMARAD   ← ez az alap-megoldás
  3. Külön bináris         src/bin/editor.rs — teljesen önálló editor-build
                          + teljes izoláció   − nincs élő Ctrl+E váltás
A te igényedhez — élőben, billentyűvel váltható — a 2. (feature flag) + 1. (runtime state) párosa kell. A 3. akkor jön, ha az editor önálló alkalmazássá nő.

Merre tovább?

  • Undo/redo. Mivel minden szerkesztés a LevelData-n megy át, egy parancs-verem (Vec<EditCommand>) elég: minden ecsetvonás egy visszafordítható parancs. Az adat-központú modell ezt szinte ingyen adja.
  • Eszközök. A EditorTool bővítése: Fill (floodfill), Select + copy-paste, ObjectPlace spawn/ajtó/láda lerakásra — mind a LevelData-t írja.
  • Object-inspector. Kijelölt LevelObject mezőinek szerkesztése (trigger-zóna mérete, ajtó célja) egy oldalsó UI-panelen.
  • Autosave & hot-reload. A dirty flagre kötött mentés, és a level.json figyelése — szerkeszd kívülről is, a játék újratölti.
  • GameState-integráció. A 2. fejezet GameState (Loading/Playing/Paused) most kapcsolódik be: a pálya-betöltés Loading, a pause overlay Paused — az editor ezekhez nem nyúl, külön tengely.
⚙ Az igazi tanulság

Ne az editor legyen a játék része, hanem a játék és az editor legyen két külön plugin ugyanarra a közös LevelData adatmodellre építve. Ha ez megvan, a három kapcsoló (state · plugin · feature) mindent megold: Ctrl+E élő váltás, tiszta kódszeparáció, release-ből kihagyható editor — és pontosan azt látod szerkesztés közben, amit a játékos.
