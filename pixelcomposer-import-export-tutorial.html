█ I. rész — Import
1. fejezet

Mit importál a PXC?

A PXC két irányban dolgozik: be importál képeket (ezeket alakítja/dúsítja effekt-tel) és ki exportál Surface-eket (PNG, spritesheet, GIF). Előbb nézzük a bejövő oldalt, mert az export megértéséhez tudni kell, honnan jön az input.

FormátumMit csinál veleTipikus forrás
PNG / JPGEgyetlen képként tölti be (statikus Surface).Bármilyen képszerkesztő, Aseprite export, 3D render.
.asepriteKözvetlenül beolvassa — rétegeket, frame-eket, palettát is.Aseprite (nem kell előbb exportálni!).
GIFAnimált képként — minden frame külön Surface frame.Kész animáció, webes forrás.
drag & dropBármelyik fenti formátum behúzható a Graph-ra.Fájlkezelőből egyenesen a PXC-be.
▸ Aseprite közvetlen — ez nagy dolog

A PXC közvetlenül olvassa az Aseprite fájlokat — nem kell előbb PNG-be exportálnod. Az Image node-nak csak a .aseprite fájlt adod meg Path-nak, és a PXC beolvassa a rétegeket, a frame-eket, sőt a palettát is. Így az Aseprite-ban szerkesztett sprite-ot módosítva a PXC gráf automatikusan frissül.

2. fejezet

Aseprite és PNG — melyiket mikor?

A gyakori kérdés: „Aseprite fájlt adjak meg, vagy előbb PNG-be exportáljak?" A válasz a munkafolyamattól függ:

⊞ Import döntésfa
  A sprite-om még változik (Aseprite-ban szerkesztem)?
    ├─ IGEN  → add meg a .aseprite fájlt közvetlenül
    │           ▸ előny: automatikus frissítés, rétegek beolvasása
    │           ▸ hátrány: a PXC projekt az Aseprite fájltól függ
    │
    └─ NEM (kész, „fagyasztott" sprite)
                → exportálj PNG-be, azt add meg
                  ▸ előny: önálló, hordozható projekt
                  ▸ hátrány: ha módosul a sprite, újra export kell
Tanulság: fejlesztés közben Aseprite-ot használj (gyors iteráció), archiváláskor/megosztáskor PNG-t. A PXC projekt (.pxc) csak a Path-ot tárolja — magát a képet nem, így mindig kell a forrásfájl a projekt mellé.
▲ „Nem találom a képet" — projekt áthelyezése
Ha áthelyezed vagy megosztod a .pxc projektet, a képeket veled kell vinned. A PXC csak a fájlútvonalat tárolja, nem ágyazza be a képet. Ha az útvonal megszakad (pl. másik gépen más a mappa), az Image node „pirosra" vált — újra ki kell választani a Path-ot. Pro tipp: a projekt mellé tegyél egy assets/ mappát, és onnan hivatkozz.
█ II. rész — Export
3. fejezet

Az Export node

Az Export node (zöld fejléc, Render/Export kategória) a gráf végére kerül — az utolsó Surface-t kiírja fájlba. Ez az a pont, ahol a gráfod „kikerül" a fájlrendszerbe.

Az Export node egy bemenettel rendelkezik (Surface) — erre az utolsó Preview/effekt kimenete csatlakozik. Az Inspector ▸ Path adja meg a kimeneti fájlt, a Template a névsablont. A ▶ EXPORT gomb (zöld kiemelés) a tényleges kiírást indítja.
  1. Jobb-klikk → Add Node ▸ Export ▸ Export.
  2. Húzd az utolsó node (általában Preview, vagy a Blend/Transform kimenete) kimenetét az Export bemenetére.
  3. Jelöld ki az Export node-ot. Az Inspector-ban állítsd be a Path-ot (hova kerüljön a fájl).
  4. Állítsd a Template-et (a fájlnév sablon, lásd 5. fejezet).
  5. Nyomd meg a ▶ EXPORT gombot az Inspector tetején (a zöld kiemelés).
▸ „A gráf végén" — mit jelent?
Az Export node-nak azt a Surface-t adjad, amit ki akarsz menteni — vagyis a gráfod végleges kimenetét. Ha van egy Preview-öd, az is jó (a Preview csak megjelenít, de kimenete ugyanaz a Surface, mint a bemenete). Ha több ágon több dolgot akarsz menteni, tegyél több Export node-ot — minden ágra egyet.
4. fejezet

A ▶ EXPORT gomb — a #1 kezdő csapda

Itt van a Pixel Composer leggyakoribb kezdő problémája — a fórumokon, itch.io community-n többször visszatérő kérdés:

▲ A leggyakoribb kezdő csapda
Az Export node nem automatikus. Hiába csinálod meg a gráfot, hiába módosítod utána — a fájl csak akkor íródik ki, ha az Inspector-ban az ▶ EXPORT gombot manuálisan megnyomod. Sokan várnak, hogy „mentésre" írja ki (mint egy normál program Save-je), de a File ▸ Save csak a .pxc projektet menti — a kimeneti PNG-t nem.
⊞ Mentés vs. export — a két külön dolog
  File ▸ Save  (Ctrl+S)   →  a .pxc projekt-fájl
                                   (gráf, node-ok, paraméterek, animáció)
                                   — a munkádat menti, nem a kimenetet

  Inspector ▸ ▶ EXPORT     →  a kimeneti fájl (PNG / spritesheet / GIF)
                                   — ezt minden módosítás után újra kell nyomni!
Ez a két dolog teljesen független. Menthetsz projektet anélkül, hogy exportálnál; exportálhatsz anélkül, hogy mentenél. A jó szokás: módosítasz a gráfon → Ctrl+S (projekt mentés) → ▶ EXPORT (kimenet frissítése).
▸ „De legalább látszik a változás?"

Igen — a Preview node él, az automatikusan frissül, ahogy módosítod a gráfot. Tehát látod, mit csinálsz. De a fájlrendszerbe (a lemezre) csak a ▶ EXPORT gomb ír. Tehát: fejlesztés közben a Preview-t nézed, és csak akkor exportálsz, ha az eredmény már kész — vagy ha új frame kell a spritesheet-be.

✓ Batch export — animációnál
Animációnál nem kell 24-szer megnyomni a gombot. Az Export node-nak van „Export all frames" opciója (vagy a Template %d jelölője automatikusan generálja az összes frame-et). Esetleg a külön Sprite Sheet node (lásd 6. fejezet) egyetlen PNG-t csinál az összes frame-ből — azt elég egyszer exportálni.
5. fejezet

Template és %d — fájlnév-sablon

A Template mező egy fájlnév-sablont tartalmaz. Statikus képnél egyszerű (sprite.png), de animációnál vagy több kimenetnél speciális jelölőket használhatsz:

JelölőMit helyettesítPélda
%dA frame sorszáma (0-tól vagy 1-től).spr_%d.png → spr_0.png, spr_1.png…
%04dFrame sorszáma, nulla-padded 4 jegyre.spr_%04d.png → spr_0000.png, spr_0001.png…
{name}A node vagy kép neve.out_{name}.png
▸ Miért fontos a nulla-padding?
A %04d formátum (4 jegy, elé nullák) nem „kellék" — a legtöbb játékmotor és atlas-packelő ábécé-sorrendben tölti be a fájlokat. spr_1.png, spr_10.png, spr_2.png rossz sorrendet ad; spr_0001.png, spr_0002.png, …, spr_0010.png helyeset. Szokj rá a %04d-re.
█ III. rész — Formátumok
6. fejezet

PNG vs spritesheet

Animáció exportjánál két fő formátum közül választasz: egyedi PNG-k (frame-enként egy fájlt) vagy spritesheet (az összes frame egy PNG-ben). A különbség hatalmas — a játékmotorok nagy része a spritesheet-et preferálja.

① Egyedi PNG-k: frame-enként külön fájl (spr_000.png…). Könnyen szerkeszthető, cserélhető, de sok kis fájl. ② Spritesheet: egyetlen PNG, a frame-ek vízszintesen (vagy rácsban) elrendezve. Egy fájl, motor-barát — ez az ajánlott.
Egyedi PNG-kSpritesheet
fájlokSok (24 frame = 24 fájl)Egyetlen PNG
betöltésManuális, egyesévelEgyszer, atlas-ként
memóriaTöbb foglalás, fragmentáltEgybefüggő, kompakt
atlas-packKülön lépés kellBeépítve
frame-csereEgy fájlt cserélszÚjra export kell
ajánlottPrototípus, egyedi frameGame-dev pipeline

Sprite Sheet node

A spritesheet-hez nem (csak) az Export node kell — van egy külön Sprite Sheet node (Generate vagy Render kategória). Ez a node veszi az animált Surface-t, és elrendezi a frame-eket egyetlen képre (vízszintesen vagy rácsban). Ennek a kimenetét azután egy Export node kiírja.

⊞ Spritesheet-pipeline gráfja
  Image ─▸ Blend (animált Opacity) ─▸ Sprite Sheet ─▸ Export
                                              ▲
                                    az animáció minden frame-jét
                                    vízszintesen egymás mellé rakja
                                    → egyetlen PNG, pl. 256×64 px
                                       (4 frame × 64 px széles)
A Sprite Sheet node paraméterei: frame-ek száma, elrendezés (horizontal/vertical/grid), padding (keret a frame-ek közt, hogy elkerüld a „vérzés” — bleeding — jelenséget). A kimenetet egy Export node-tal írod ki — egyszeri ▶ EXPORT.
▲ Padding — a „vérzés" (bleeding) ellen
Spritesheet-nél előfordulhat, hogy egy frame széle „átfolyik" a szomszéd frame-be (a motor lineáris mintavételezése miatt). Ezt úgy kerülöd el, hogy a Sprite Sheet node-on beállítod a padding-et (1–2 pixel keret minden frame körül). A modern motorok „nearest" mintavételezéssel ezt nem igénylik, de ha forgatást is használsz, a padding menthet.

GIF és videó — egyéb formátumok

A PNG és spritesheet mellett a PXC más formátumokat is tud:

FormátumMire jóMikor használd
GIF (animált)Egyetlen fájl, beágyazott animáció.Demó/megmutató megosztása, social media, README-ben.
MP4 / WebMVideó, tömörített.Hosszabb animáció, videós demó.
JSON (atlas metaadat)A spritesheet elrendezésének leírása (frame méretek, pozíciók).Motortól függ — Godot/Spine/Bevy atlas-packelés.

Ha valakinek meg akarod mutatni az effektedet (Discordon, fórumon), a GIF a leggyorsabb. De game-dev-be ne GIF-et használj — a GIF csak 256 színt támogat (indexed palette), ami rontja a minőséget. A játékhoz spritesheet PNG + külön metaadat a helyes út.

8. fejezet

A teljes pipeline — PXC-től motorig

Tegyük össze az egészet. A teljes munkafolyamat egy ábrán:

A teljes pipeline: PXC gráf (az 1–3. részből) → export → spritesheet.png (egy PNG, frame-ek vízszintesen) → load → játékmotor (Godot / Bevy / GameMaker), ami atlas-ként tölti be és lépteti a frame-et.
⊞ A metaadat — mit kell tudnia a motornak
  spritesheet.png  ──▸  szélesség: 256 px   (4 frame × 64 px)
                       magasság: 64 px
                       frame-db: 4
                       frame-méret: 64 × 64 px
                       fps: 24  (vagy amennyit állítottál)
                       loop: igen

  A motor ezeket a metaadatokat külön beállítja (nem a PNG-ből olvassa ki).
  Ezért fontos, hogy jól jegyezd fel — vagy JSON metaadatot exportálj.
A PXC-ben a Canvas mérete határozza meg a frame méretét. Ha 4 frame-t tettél vízszintesen a Sprite Sheet node-ban, és a Canvas 64×64 px, akkor a kimeneti PNG 256 × 64 px lesz — ezt a 4 számot (szélesség, magasság, frame-db, fps) jegyezd fel a motorhoz.
9. fejezet

Godot / Bevy / GameMaker

Három gyakori motorral mutatjuk be a betöltést. A PXC-ből a spritesheet.png + metaadat jön ki; a motorban ezeket állítod be.

① Godot (4.x)

  1. Húzd be a spritesheet.png-t a res:// mappába.
  2. Kattints rá az FileSystem-ben → az Inspector-ban a Import fülön válaszd 2D Pixel presetet (a „nearest" filter miatt) és nyomd meg a Reimport gombot.
  3. A Scene-ben hozz létre egy Sprite2D node-ot, és állítsd be a Texture-t a spritesheet-re.
  4. A Sprite2D Inspector-ban nyisd le a Animation szekciót: Hframes = 4 (vízszintes frame-ek száma), Vframes = 1.
  5. Adj hozzá egy AnimationPlayer vagy szkriptet a frame léptetéséhez.

② Bevy (Rust)

A Bevy-ben a TextureAtlas a megfelelő eszköz. A spritesheet PNG-t betöltöd, és egy TextureAtlasLayout-tal megadod a frame-ek rácsát:

use bevy::sprite::{TextureAtlasLayout, Sprite};

fn setup(
    mut commands: Commands,
    asset_server: Res<AssetServer>,
    mut layouts: ResMut<Assets<TextureAtlasLayout>>,
) {
    // a PXC-ből exportált spritesheet.png
    let texture = asset_server.load("spritesheet.png");
    let layout = layouts.add(TextureAtlasLayout::from_grid(
        UVec2::new(64, 64),  // frame mérete
        4,                // vízszintes frame-ek
        1,                // függőleges
        None, None,
    ));
    commands.spawn((
        Sprite {
            texture: Some(texture.clone()),
            texture_atlas: Some(layout.into()),
            ..default()
        },
        // Animation léptető komponens
        AnimationTimer::default(),
    ));
}

Az AnimationTimer egy egyedi komponens, ami Timer::from_seconds(1.0/24.0, TimerMode::Repeating)-ként lépteti a frame indexet.

▸ Bevy + pixel art — fontos beállítás

A Bevy alapból lineáris mintavételezést használ, ami elmosja a pixel art-ot. Állítsd a ImagePlugin::default_nearest()-re a default_plugins-ben — ez a teljes app-ra érvényesíti a „nearest" filtert. Ezzel a sprite-ok élesen, pixesen maradnak, ahogy a PXC-ben is láttad.

③ GameMaker

  1. Sprites ▸ New Sprite → nevezd el, importáld be a spritesheet.png-t.
  2. A Sprite Editor-ban: Modify Mask ▸ Manual, állítsd a Width-re a frame szélességére (64), a Height-ra a magasságra.
  3. A Sprite Editor-ban Frame ▸ Strip Frame → szélesség 64 → a GameMaker automatikusan vágja a frame-eket.
  4. Állítsd be az image_speed-et (a léptetés sebessége; 1.0 = a game-maker alap fps).

▸ A „kötelező házi"

Vedd elő a 3. részes pulzáló glow projektedet. Rakj rá egy Sprite Sheet node-ot, kösd rá az Export-ot, és ments ki egy glow_sheet.png-t (4 vagy 6 frame). Aztán töltsd be a választott motorodba (Godot/Bevy/GameMaker) és léptesd — látnod kell a lüktető glow-t a játékodban is. Ez az első tényleges game-dev pipeline végigviteled.
