00

Miért procedurális?

A procedurális generálás azt jelenti, hogy a pályát nem kézzel rakod ki egy editorban, hanem egy algoritmus állítja elő futásidőben — minden indításkor más barlangot, más szigetet, más kazamatát. Ez adja a roguelike-ok újrajátszhatóságát, egy survival játék végtelen világát, vagy csak azt a kényelmet, hogy nem kell 200 pályát kézzel megrajzolnod.

A munka két, élesen elkülönülő fázisból áll, és a tutorial végig ezt a kettőt választja szét:

  1. Generálás (adat). Egy algoritmus eldönti, mi van hol: fal vagy padló, víz vagy föld. Az eredmény egy egyszerű rács — számok egy tömbben. Itt dől el a játszhatóság.
  2. Autotiling (megjelenítés). A nyers rácsból szép kép lesz: a szomszédság alapján minden cellához a megfelelő él-, sarok- vagy átmenet-csempét választjuk. Itt dől el, hogy jól néz-e ki.

Ez a szétválasztás nem stiláris döntés, hanem a téma gerince. Az ütközés, a pathfinding és az AI mind az adatból dolgozik; az autotiling csak a felszín. Ha összekevered a kettőt, később minden megjelenítési finomítás eltöri a játékmenetet.

Mi épül és mi nem

Ez a tutorial a generálásra fókuszál — ez az, amiről a sorozatban még nem volt szó. Az autotilingot (09) a generált adatra alkalmazott záró lépésként mutatom be; a maszk-számítás mély részleteit a csempe-bitmaszkolás és a dual-grid tutorial tárgyalja.

01

A térkép mint adat

Mielőtt bármit generálnánk, kell egy hely, ahova generálunk. A térkép a források forrása (single source of truth): egy lapos rács, amibe minden algoritmus ír, és amiből a megjelenítés meg az ütközés is olvas.

Egy 2D rácsot a leghatékonyabb egyetlen lapos Vec-ben tárolni, és az (x, y) koordinátát egy indexszé alakítani. Ez gyorsabb és egyszerűbb, mint a Vec<Vec<T>>:

use bevy::prelude::*;

#[derive(Clone, Copy, PartialEq, Eq, Debug)]
enum Tile {
    Wall,   // tömör — itt nem lehet járni
    Floor,  // járható
}

#[derive(Resource, Clone)]
struct Map {
    width: i32,
    height: i32,
    tiles: Vec<Tile>,   // hossza = width * height
}

impl Map {
    fn new(width: i32, height: i32) -> Self {
        // alapból minden fal — a generálás majd kivájja a padlót
        Self { width, height, tiles: vec![Tile::Wall; (width * height) as usize] }
    }

    // (x, y) -> lapos index. Ez a kulcs-képlet: sor * szélesség + oszlop
    fn idx(&self, x: i32, y: i32) -> usize {
        (y * self.width + x) as usize
    }

    fn in_bounds(&self, x: i32, y: i32) -> bool {
        x >= 0 && y >= 0 && x < self.width && y < self.height
    }

    fn set(&mut self, x: i32, y: i32, t: Tile) {
        if self.in_bounds(x, y) {
            let i = self.idx(x, y);
            self.tiles[i] = t;
        }
    }
}

A legfontosabb döntés a get-ben rejlik. Mit adjon vissza, ha a kért cella a rácson kívül van? Ha azt mondjuk, „a rácson túl minden fal", akkor a pálya széle automatikusan tömör keretet kap, és minden szomszéd-vizsgáló algoritmus (sejtautomata, autotiling) természetesen viselkedik a szélen — nincs külön peremeset-kód.

impl Map {
    // A határon túl FAL. Ez egyetlen sor, de fél tutorialnyi peremesetet spórol meg.
    fn get(&self, x: i32, y: i32) -> Tile {
        if self.in_bounds(x, y) {
            self.tiles[self.idx(x, y)]
        } else {
            Tile::Wall
        }
    }
}
Miért i32, nem usize?

A koordinátákat i32- vel tartjuk, mert a szomszéd-vizsgálatok folyton számolnak x - 1-et és y - 1-et. usize-zal egy 0 - 1 alulcsordulna és pánikolna; i32-vel egyszerűen -1 lesz, amit az in_bounds szépen kiszűr. Csak az indexelés pillanatában váltunk usize-ra.

02

Seed és determinizmus

A „véletlen" a proc-genben ravasz fogalom. Nem igazi véletlent akarsz, hanem reprodukálható véletlent: ugyanaz a seed (kezdőszám) mindig pontosan ugyanazt a pályát adja. Ez nélkülözhetetlen, mert:

  • Hibajelentés. „A 8472-es seeden a játékos beragad" — és te pontosan ugyanazt a pályát kapod vissza.
  • Daily run / megosztás. Egy szám elég, hogy két játékos ugyanazt a világot kapja.
  • Mentés. Elég a seedet elmenteni, nem a teljes térképet — a generálás újra előállítja.

A rand crate StdRng-je pont ezt adja: egy seedelhető generátort, amit egy resource-ban hordozol körbe.

use rand::prelude::*;
use rand::rngs::StdRng;
use rand::SeedableRng;

#[derive(Resource)]
struct WorldRng(StdRng);

// Egy konkrét seedből — ugyanaz a szám, ugyanaz a pálya, mindig.
let seed: u64 = 8472;
let mut rng = StdRng::seed_from_u64(seed);

// A három leggyakoribb hívás generáláshoz:
let p: f64  = rng.gen_range(0.0..1.0);   // valószínűséghez
let coin    = rng.gen_bool(0.45);        // 45% esély true-ra
let x: i32  = rng.gen_range(0..80);      // egész tartomány [0, 80)
rand 0.9-es átnevezés

A fenti az elterjedt rand 0.8 API. A rand 0.9-ben a gen_range → random_range, a gen_bool → random_bool lett (mert a gen foglalt kulcsszó lesz). A logika ugyanaz, csak a nevek mások — nézd meg, melyik verziót húzza be a Bevyd.

Egy RNG, sorrendben

Egyetlen RNG-t használj az egész pályához, és mindig ugyanabban a sorrendben kérj tőle számokat. Ha menet közben átrendezed a generálási lépéseket, ugyanaz a seed más pályát ad — ez meglepő tud lenni. A determinizmus a seedből és a hívási sorrendből együtt jön.

03

Fehér zaj — és miért rossz út

A legnaivabb ötlet: menj végig minden cellán, és dobj egy érmét, hogy fal legyen-e vagy padló. Ezt hívják fehér zajnak (white noise) — minden cella független a szomszédjától.

fn random_fill(map: &mut Map, rng: &mut StdRng, wall_chance: f64) {
    for y in 0..map.height {
        for x in 0..map.width {
            let t = if rng.gen_bool(wall_chance) { Tile::Wall } else { Tile::Floor };
            map.set(x, y, t);
        }
    }
}

Futtasd le, és azonnal látod a bajt: „só és bors" — szétszórt pontok, semmi szerkezet. Nincsenek barlangok, járatok, szobák; nincs hely, ahol a karakter járhatna, mert minden járható cellát azonnal körbevesz egy fal. Játszhatatlan.

Ettől még nem haszontalan: a fehér zaj a kiindulópont sok algoritmushoz. A sejtautomata (05) éppen ezt a sót-borsot simítja organikus barlanggá. De önmagában a fehér zaj soha nem pálya.

04

Zaj-alapú világtérkép

A fehér zaj baja a függetlenség. A megoldás a koherens zaj (Perlin, Simplex, value noise): olyan függvény, ahol a közeli pontok hasonló értéket adnak, a távoliak meg eltérőt — folytonos, sima dombok és völgyek egy értékmezőben.

A trükk: a zaj egy magasságtérképet ad (minden cellához egy 0..1 szám), te pedig küszöbökkel vágod biome-okra. Tengerszint alatt víz, fölötte fű, magasan hegy, a tetején hó:

use noise::{NoiseFn, Perlin};

fn generate_heightmap(map: &mut Map, seed: u32, scale: f64, sea_level: f64) {
    let perlin = Perlin::new(seed);
    for y in 0..map.height {
        for x in 0..map.width {
            // a skála "nagyítja" a zajt: kisebb = nagyobb, lágyabb formák
            let nx = x as f64 * scale;
            let ny = y as f64 * scale;
            // a Perlin -1..1 között ad; toljuk 0..1-be
            let h = perlin.get([nx, ny]) * 0.5 + 0.5;

            let t = if h < sea_level { Tile::Water } else { Tile::Floor };
            map.set(x, y, t);
        }
    }
}

Egyetlen Perlin-réteg sima, de kicsit „buborékos". Igazi terephez több oktávot rétegzel egymásra (fractal Brownian motion, fBm): minden oktáv kétszer akkora frekvencián és fele akkora amplitúdón ad részletet — a nagy formákra finomabb érdesség kerül.

use noise::{Fbm, Perlin, MultiFractal, NoiseFn};

fn make_fbm(seed: u32, octaves: usize, scale: f64) -> Fbm<Perlin> {
    Fbm::<Perlin>::new(seed)
        .set_octaves(octaves)     // hány réteg részlet (3–6 jó)
        .set_frequency(scale)     // alap "nagyítás"
        .set_persistence(0.5)     // minden oktáv fele olyan erős
        .set_lacunarity(2.0)      // minden oktáv kétszer sűrűbb
}

Ugyanezt a technikát használhatod nedvesség-, hőmérséklet- vagy érc-térképhez is, külön seeddel. Két-három zajréteg keresztezéséből (magasság × nedvesség) jönnek a klasszikus biome-mátrixok: forró+száraz = sivatag, hideg+nedves = tundra.

05

Sejtautomata — organikus barlangok

Ez a kedvencem, mert döbbenetesen egyszerű és mégis gyönyörű eredményt ad. A sejtautomata (cellular automata) a 03-as fehér zajból indul, majd néhány simítási lépéssel organikus barlanggá olvasztja. A szabály egyetlen mondat:

A „4-5 szabály"

Minden cellát átírunk a 8 szomszédja alapján: ha 5 vagy több szomszédja fal, legyen fal; különben padló. A fal vonzza a falat, a nyílt tér a nyílt teret — a zaj néhány lépés alatt összeáll tömbökké.

Két dolog kritikus. Először: egyszerre olvasol és írsz, ezért dupla bufferre van szükség — egy másolatból olvasod a szomszédokat, és az új térképbe írsz. Ha helyben módosítanál, a már átírt cellák meghamisítanák a többi számolását.

// Hány fal van a 8 szomszéd között? A get() a határon túl falat ad,
// így a pálya széle magától tömör marad — nincs peremeset.
fn count_wall_neighbors(map: &Map, x: i32, y: i32) -> u8 {
    let mut count = 0;
    for dy in -1..=1 {
        for dx in -1..=1 {
            if dx == 0 && dy == 0 { continue; }       // önmagát nem
            if map.get(x + dx, y + dy) == Tile::Wall { count += 1; }
        }
    }
    count
}

// Egy simítási lépés: ÚJ térképet ad vissza (dupla buffer!).
fn smooth_step(map: &Map) -> Map {
    let mut next = map.clone();
    for y in 0..map.height {
        for x in 0..map.width {
            let walls = count_wall_neighbors(map, x, y);
            let t = if walls >= 5 { Tile::Wall } else { Tile::Floor };
            next.set(x, y, t);
        }
    }
    next
}

Másodszor: a teljes recept fehér zaj + néhány lépés. A kezdeti fal-arány és a lépésszám hangolja az eredményt — kevés lépés érdes, sok lépés gömbölyű:

fn generate_cave(map: &mut Map, rng: &mut StdRng) {
    // 1) só-bors kiindulás, ~45% fal
    random_fill(map, rng, 0.45);
    // 2) 4–5 simítás összeolvasztja barlanggá
    for _ in 0..5 {
        *map = smooth_step(map);
    }
}

06

Részeg séta — garantáltan összefüggő

A sejtautomata szép, de nem garantálja, hogy a barlangok össze vannak kötve (erről szól a 08). A részeg séta (drunkard's walk) ezt a problémát kerüli meg: egy „ásó" elindul a térkép közepéről, és véletlenszerű irányokba botorkálva kivájja maga után a padlót. Mivel egyetlen folytonos úton halad, a kivájt tér mindig összefüggő.

fn drunkards_walk(map: &mut Map, rng: &mut StdRng, steps: u32) {
    // középről indul
    let mut x = map.width / 2;
    let mut y = map.height / 2;

    for _ in 0..steps {
        map.set(x, y, Tile::Floor);        // kivájjuk az aktuális cellát
        match rng.gen_range(0..4) {         // véletlen lépés 4 irányba
            0 => x += 1,
            1 => x -= 1,
            2 => y += 1,
            _ => y -= 1,
        }
        // bent maradunk, 1 cellás kerettel a szélen
        x = x.clamp(1, map.width - 2);
        y = y.clamp(1, map.height - 2);
    }
}

A lépésszám szabja meg, mennyit váj ki: a térkép celláinak nagyjából 40–50%-a kellemes „barlang-folyosó" érzetet ad. A jellegzetes szerves, kanyargós alakzat a séta sajátja.

Variációk

Több ásó párhuzamosan (mind a középről) szélesebb termeket nyit. Ha az ásó néha „súlyozottan" tartja az előző irányát, hosszabb, egyenesebb folyosók lesznek. Ha az irányváltáskor néha 2×2-t váj, vastagabb járatokat kapsz. Ezek apró módosítások ugyanazon a cikluson.

07

BSP dungeon — szobák és folyosók

A barlang-algoritmusok organikusak. De ha klasszikus, szögletes kazamatát akarsz — különálló szobák folyosókkal összekötve —, a BSP (binary space partitioning, bináris térfelosztás) a bevált eszköz. A logika rekurzív:

  1. Vedd az egész térképet egy téglalapként.
  2. Vágd ketté — véletlenszerűen vízszintesen vagy függőlegesen, véletlen helyen.
  3. Ismételd mindkét félre, amíg elég kicsik nem lesznek.
  4. A levél-téglalapokba rajzolj egy-egy szobát (kicsit kisebbet a téglalapnál, hogy legyen fal-keret).
  5. A testvér-csomópontok szobáit kösd össze egy folyosóval.

Az eredmény egy fa, aminek a levelei a szobák — sosem fednek át, és a fa garantálja az összeköttetést.

#[derive(Clone, Copy)]
struct Rect { x: i32, y: i32, w: i32, h: i32 }

fn split(r: Rect, rng: &mut StdRng, depth: u32, rooms: &mut Vec<Rect>) {
    // megállás: elég mély vagy elég kicsi -> ez egy levél, ide jön szoba
    if depth == 0 || (r.w < 14 && r.h < 14) {
        rooms.push(carve_room(r, rng));
        return;
    }
    // döntsd el az irányt; a jóval hosszabb oldal mentén vágj
    let vertical = if r.w as f32 / r.h as f32 > 1.25 { true }
                   else if r.h as f32 / r.w as f32 > 1.25 { false }
                   else { rng.gen_bool(0.5) };

    if vertical {
        let cut = rng.gen_range(r.w / 3 ..= r.w * 2 / 3);
        split(Rect { w: cut, ..r }, rng, depth - 1, rooms);
        split(Rect { x: r.x + cut, w: r.w - cut, ..r }, rng, depth - 1, rooms);
    } else {
        let cut = rng.gen_range(r.h / 3 ..= r.h * 2 / 3);
        split(Rect { h: cut, ..r }, rng, depth - 1, rooms);
        split(Rect { y: r.y + cut, h: r.h - cut, ..r }, rng, depth - 1, rooms);
    }
}

A szoba kivájása egyszerű: a téglalapon belül egy véletlen, kicsit kisebb terület padlóra állítása. A folyosó pedig két szoba középpontját köti össze egy „L" alakú (előbb vízszintes, majd függőleges) vájattal:

fn carve_corridor(map: &mut Map, a: IVec2, b: IVec2) {
    // vízszintes szakasz a.x -> b.x az a.y soron
    for x in a.x.min(b.x) ..= a.x.max(b.x) { map.set(x, a.y, Tile::Floor); }
    // függőleges szakasz a.y -> b.y a b.x oszlopon
    for y in a.y.min(b.y) ..= a.y.max(b.y) { map.set(b.x, y, Tile::Floor); }
}
Mikor melyik algoritmus?

Sejtautomata → természetes barlang, bánya. Részeg séta → szerves, garantáltan összefüggő üreg. BSP → épített, szobás kazamata, börtön, épület. Zaj → szabadtéri világtérkép, szigetek, biome-ok. Sok játék kombinálja őket: BSP a szobákra, majd sejtautomata-„kopás" az éleken, hogy ne legyenek tökéletesen szögletesek.

08

Összefüggőség — flood fill

A sejtautomata és a zaj egy alattomos hibát rejt: a pálya szigetekre szakadhat. Lehet két gyönyörű barlangod, amik közt nincs átjárás — a játékos beragad, a kulcs elérhetetlen. A generálás után mindig ellenőrizd, hogy a járható tér összefügg-e.

Az eszköz a flood fill (elöntés): egy padlócellából indulva BFS-sel bejárod az összes elérhető padlót. Ezzel megtalálod a régiókat, és a legnagyobbon kívül mindent betömhetsz fallal — vagy folyosóval összekötheted őket.

use std::collections::VecDeque;

// Egy régió (összefüggő padló-folt) feltérképezése egy kezdőpontból.
fn flood_region(map: &Map, start: IVec2, seen: &mut Vec<bool>) -> Vec<IVec2> {
    let mut region = Vec::new();
    let mut queue = VecDeque::new();
    queue.push_back(start);
    seen[map.idx(start.x, start.y)] = true;

    while let Some(p) = queue.pop_front() {
        region.push(p);
        // 4 fő irány (nem átló — a járás 4-szomszédos)
        for d in [IVec2::new(1,0), IVec2::new(-1,0), IVec2::new(0,1), IVec2::new(0,-1)] {
            let n = p + d;
            if map.get(n.x, n.y) == Tile::Floor && !seen[map.idx(n.x, n.y)] {
                seen[map.idx(n.x, n.y)] = true;
                queue.push_back(n);
            }
        }
    }
    region
}

// Tartsd meg a legnagyobb régiót, a többit töltsd fel fallal.
fn keep_largest_region(map: &mut Map) {
    let mut seen = vec![false; map.tiles.len()];
    let mut regions: Vec<Vec<IVec2>> = Vec::new();

    for y in 0..map.height {
        for x in 0..map.width {
            if map.get(x, y) == Tile::Floor && !seen[map.idx(x, y)] {
                regions.push(flood_region(map, IVec2::new(x, y), &mut seen));
            }
        }
    }
    // a legnagyobb régió indexe
    if let Some(biggest) = regions.iter().enumerate().max_by_key(|(_, r)| r.len()).map(|(i, _)| i) {
        for (i, region) in regions.iter().enumerate() {
            if i != biggest {
                for p in region { map.set(p.x, p.y, Tile::Wall); }
            }
        }
    }
}
Kapcsolat a pathfindinggel

Ugyanez a bejárás bizonyítja, hogy két konkrét pont elérhető-e egymásból — pont, amit a pathfinding tutorial A*-ja is használ. A generálási fázisban olcsó BFS-szel garantálod, hogy a játszhatóság feltételei (start, kijárat, kulcsok egy régióban legyenek) teljesüljenek — még mielőtt egyetlen csempét is kirajzolnál.

A „garantáld vagy dobd el" minta

Néha egyszerűbb a régiókat összekötni (legközelebbi padlópárokat folyosóval), néha a leghatékonyabb a regen-amíg-jó: generálj, ellenőrizd a feltételeket (pl. legnagyobb régió > a térkép 40%-a), és ha nem felel meg, új seeddel próbáld újra. Pár iteráció alatt mindig kijön egy jó pálya.

09

Kitüntetett objektumok elhelyezése

A procedurális alapréteg mindenhol „egyforma" — de a játékhoz gyakran pontosan N kitüntetett dolog kell: 5 bánya, 1 boss-szoba, a kijárat, néhány kincs. A kulcs, hogy ezt ne ugyanabban a lépésben oldd meg, mint az alapterepet, hanem egy későbbi passzban, a már kész és validált rácsra.

A passz-modell

A generálás passzok sorozata, és a feature-ök egy későbbi rétegben kerülnek a kész terepre — azután, hogy az összefüggőség (08) már garantált:

1. Alapterep         (CA / noise / BSP)   -> mi fal, mi padló
2. Összefüggőség      (flood fill, 08)      -> biztosan járható
3. Feature-placement  (bányák, POI-k)       <- EZ a passz
4. Spawn / loot / ellenség
5. Megjelenítés       (autotiling, 10)

Hol éljen a feature az adatban?

Két szint, és általában mindkettőt használod:

  • Strukturális rész → a tile-grid. Amit a bánya kiváj (egy szoba, egy akna), az Tile írás a Map-be — esetleg új típus (Tile::OreVein), ha az autotiling vagy az ütközés másképp kezeli.
  • Logikai rész → külön réteg. Maga a „bánya mint objektum" (mennyi érc, interakció) nem tile — ez egy entitás saját komponenssel. A tile-grid maradjon szűken „mi járható".

Hüvelykujjszabály: ami a falakat/padlót befolyásolja → grid; aminek saját állapota vagy viselkedése van → entitás.

Reject sampling — próbáld és dobd el

A „pontosan N darab, szétszórva, megszorításokkal" esetre a bevált eszköz a reject sampling: húzz egy véletlen pozíciót a seedelt RNG-ből, ellenőrizd a feltételeket, fogadd el vagy próbálj újat — egy felső próbaszám-korláttal, hogy ne pörögjön örökké, ha nincs elég hely.

use rand::Rng;

/// Pontosan `count` bánya elhelyezése. Feltételek: padló, elérhetőség,
/// és minimális távolság a már elhelyezett bányáktól.
fn place_mines(
    map: &Map,
    rng: &mut StdRng,
    reachable: &[bool],   // a 08-as flood fill `seen` tömbje: elérhető-e a cella
    count: usize,
    min_dist: i32,
) -> Vec<IVec2> {
    let mut mines: Vec<IVec2> = Vec::new();
    let max_attempts = count * 200;     // védőkorlát a végtelen ciklus ellen

    for _ in 0..max_attempts {
        if mines.len() == count { break; }

        let p = IVec2::new(
            rng.gen_range(0..map.width),
            rng.gen_range(0..map.height),
        );

        if map.get(p.x, p.y) != Tile::Floor { continue; }    // 1) járható padló
        if !reachable[map.idx(p.x, p.y)] { continue; }        // 2) nem zárt zseb
        if mines.iter().any(|m| m.distance_squared(p) < min_dist * min_dist) {
            continue;                                         // 3) ne csomósodjon
        }
        mines.push(p);
    }
    mines   // ha < count jött ki: lazíts a min_dist-en, vagy regenerálj
}

A három continue a lényeg — mindegyik egy megszorítás. A min_dist adja a szép szétterítést, különben a bányák egymásra csúsznának. A 2. feltétel külön figyelmet érdemel: ha a 08-ban már a legnagyobb régiót tartottad meg, akkor minden Floor elérhető, és a reachable ellenőrzés elhagyható; ha viszont meghagytál kisebb zsebeket, ez akadályozza meg, hogy egy bánya elérhetetlen üregbe kerüljön.

Néhány hasznos variáció ugyanezen a cikluson:

  • Súlyozott elhelyezés. Ha a bánya logikusan kőhöz/hegyhez tartozik, ne uniform húzz: használd a 04-es zaj magasság-rétegét alkalmassági pontszámként, és a köves cellákat preferáld. Így a feature összenő az alapréteggel, nem ül rajta idegenül.
  • Régió-kvóta. Oszd a térképet 3×3 zónára, és zónánként engedj max 1-2 bányát — garantáltan minden tájra jut, nem csomósodik egy sarokba.
  • Poisson-disk / blue noise. Csak ha sok (50+) feature-t szórsz egyenletesen (fák, ércfoltok). 5 bányához a reject sampling bőven elég és egyszerűbb.

Több csempés feature: prefab-stamp

Ha egy bánya nem egy cella, hanem egy kis kézzel tervezett szoba, akkor prefab-stamp-elsz: van egy minta, ellenőrzöd hogy elfér, beírod a gridbe, majd bekötöd egy folyosóval.

// '#'=fal, '.'=padló, 'O'=ércpont (ide spawnolsz interaktív entitást),
// '^'=bejárat (innen vájj folyosót a legközelebbi padlóig)
const MINE_PREFAB: &[&str] = &[
    "#######",
    "#.....#",
    "#.O.O.#",
    "#..^..#",
    "###.###",
];

fn stamp_prefab(map: &mut Map, top_left: IVec2, prefab: &[&str]) {
    for (dy, row) in prefab.iter().enumerate() {
        for (dx, ch) in row.chars().enumerate() {
            let x = top_left.x + dx as i32;
            let y = top_left.y + dy as i32;
            match ch {
                '#' => map.set(x, y, Tile::Wall),
                _   => map.set(x, y, Tile::Floor),   // '.', 'O', '^' mind járható
            }
        }
    }
}

A teljes folyamat négy lépés: (1) az elhelyezés előtt nézd meg, hogy a prefab teljes téglalapja elfér-e és nem ír-e felül egy másik fontos feature-t; (2) stamp-eld be; (3) vájj folyosót a ^ bejárattól a legközelebbi járható padlóig; (4) futtasd újra a flood fillt — egy beágyazott fal-blokk könnyen elzárhatott egy átjárót. Így a szoba mindig ugyanaz marad (kézi design), a helye viszont procedurális.

Determinizmus: a feature-placement ugyanabból a seedelt RNG-ből húzzon, a generálási lánc fix pontján (02). Autotiling: a nyers adatból szép kép

Itt találkozik a két fázis. A generálás után a térképed cellái „fal" vagy „padló" — de ha ezt nyers blokként rajzolod ki, lépcsős, otromba lesz. Az autotiling minden cellához a szomszédai alapján választ csempét: egyenes fal-szakasz, belső sarok, külső sarok, vég. A játékos sima körvonalat lát, te meg egyetlen adat-rácsot tartasz karban.

A mag egy bitmaszk: minden cellához egy kis számot számolsz, ahol minden bit egy szomszéd „ugyanolyan terep-e?" kérdésére felel. A 4 fő irányból egy 0..15 közötti szám jön, ami egyenesen indexel egy 16 elemű csempe-táblát:

// 4-bites él-maszk: É, K, D, NY szomszéd ugyanolyan-e, mint a cella.
fn four_bit_mask(map: &Map, x: i32, y: i32) -> u8 {
    let me = map.get(x, y);
    let same = |nx: i32, ny: i32| (map.get(nx, ny) == me) as u8;
    (same(x, y - 1) << 0)   // észak  -> 1. bit
  | (same(x + 1, y) << 1)   // kelet  -> 2. bit
  | (same(x, y + 1) << 2)   // dél    -> 4. bit
  | (same(x - 1, y) << 3)   // nyugat -> 8. bit
}

// A 0..15 maszk az atlasz egy konkrét csempéjére mutat.
// (A 16 -> index táblát a csempe-atlaszod elrendezése adja.)
fn tile_index(mask: u8) -> usize {
    TILE_LOOKUP[mask as usize]
}

Bevyben a kirajzolás ekkor egy atlasz-sprite, aminek az indexét a maszk adja:

Sprite::from_atlas_image(
    atlas_image.clone(),
    TextureAtlas { layout: layout.clone(), index: tile_index(mask) },
)
Itt csak a kapcsolódás van

A 4-bites maszk a belépő. A teljes kép — a 8-bites (sarokcsempés) maszk, a 47/256 redukció, a több tereptípus közti átmenetek, a komplett érték→csempe tábla — a csempe-bitmaszkolás tutorialban van. Egy elegáns alternatíva, ami csak 16 csempével ad sima eredményt, a dual-grid (és a PRO változata több tereppel). Mindkettő ugyanarra a generált adatra épül, amit itt előállítottál.

Bevy integráció

Most kösd be a Bevybe. A kulcsdöntés a mikor és a szétválasztás: a generálás egyszeri, nehéz művelet, ezért a Startup-ba vagy egy állapot OnEnter-jébe való — soha nem az Update-be vagy FixedUpdate-be. A Map resource a tiszta adat; a sprite-ok ettől külön, levezetett megjelenítés.

const TILE: f32 = 16.0;

#[derive(Component)]
struct MapTile;   // jelölő, hogy egyszerre tudjuk despawnolni

// 1) GENERÁLÁS: csak adatot állít elő, semmit nem rajzol.
fn generate_world(mut commands: Commands) {
    let mut rng = StdRng::seed_from_u64(8472);
    let mut map = Map::new(80, 50);

    random_fill(&mut map, &mut rng, 0.45);
    for _ in 0..5 { map = smooth_step(&map); }
    keep_largest_region(&mut map);   // játszhatóság garantálva

    commands.insert_resource(map);   // ez a forrás mindenki másnak
}

// 2) MEGJELENÍTÉS: a Map-ból sprite-okat told ki.
fn spawn_tiles(mut commands: Commands, map: Res<Map>) {
    for y in 0..map.height {
        for x in 0..map.width {
            let color = match map.get(x, y) {
                Tile::Wall  => Color::srgb(0.16, 0.16, 0.22),
                Tile::Floor => Color::srgb(0.45, 0.62, 0.40),
            };
            commands.spawn((
                Sprite::from_color(color, Vec2::splat(TILE)),
                Transform::from_xyz(x as f32 * TILE, y as f32 * TILE, 0.0),
                MapTile,
            ));
        }
    }
}

Komplett minta — összerakva

Egy minimál, de teljes váz: seedelt sejtautomata-barlang, összefüggőség-garancia, színes render, és R-re újragenerálás. Ez a csontváz skálázható egy igazi roguelike-ká.

use bevy::prelude::*;
use rand::{rngs::StdRng, SeedableRng, Rng};

const W: i32 = 80;
const H: i32 = 50;
const TILE: f32 = 16.0;

#[derive(Clone, Copy, PartialEq, Eq)]
enum Tile { Wall, Floor }

#[derive(Resource, Clone)]
struct Map { width: i32, height: i32, tiles: Vec<Tile> }

impl Map {
    fn new(w: i32, h: i32) -> Self { Self { width: w, height: h, tiles: vec![Tile::Wall; (w*h) as usize] } }
    fn idx(&self, x: i32, y: i32) -> usize { (y * self.width + x) as usize }
    fn in_bounds(&self, x: i32, y: i32) -> bool { x >= 0 && y >= 0 && x < self.width && y < self.height }
    fn get(&self, x: i32, y: i32) -> Tile { if self.in_bounds(x,y) { self.tiles[self.idx(x,y)] } else { Tile::Wall } }
    fn set(&mut self, x: i32, y: i32, t: Tile) { if self.in_bounds(x,y) { let i = self.idx(x,y); self.tiles[i] = t; } }
}

#[derive(Component)] struct MapTile;
#[derive(Resource)] struct Seed(u64);

fn main() {
    App::new()
        .add_plugins(DefaultPlugins)
        .insert_resource(Seed(8472))
        .add_systems(Startup, (setup_camera, regen).chain())
        .add_systems(Update, on_keypress)
        .run();
}

fn setup_camera(mut commands: Commands) {
    commands.spawn((Camera2d, Transform::from_xyz(W as f32*TILE/2.0, H as f32*TILE/2.0, 0.0)));
}

// teljes regenerálás: adat + kép, egy helyen
fn regen(mut commands: Commands, seed: Res<Seed>, old: Query<Entity, With<MapTile>>) {
    for e in &old { commands.entity(e).despawn(); }

    let mut rng = StdRng::seed_from_u64(seed.0);
    let mut map = Map::new(W, H);

    // 1) só-bors
    for y in 0..H { for x in 0..W {
        map.set(x, y, if rng.gen_bool(0.45) { Tile::Wall } else { Tile::Floor });
    }}
    // 2) simítás
    for _ in 0..5 {
        let mut next = map.clone();
        for y in 0..H { for x in 0..W {
            let mut walls = 0;
            for dy in -1..=1 { for dx in -1..=1 {
                if (dx, dy) != (0, 0) && map.get(x+dx, y+dy) == Tile::Wall { walls += 1; }
            }}
            next.set(x, y, if walls >= 5 { Tile::Wall } else { Tile::Floor });
        }}
        map = next;
    }

    // 3) kirajzolás
    for y in 0..H { for x in 0..W {
        let color = match map.get(x, y) {
            Tile::Wall  => Color::srgb(0.16, 0.16, 0.22),
            Tile::Floor => Color::srgb(0.45, 0.62, 0.40),
        };
        commands.spawn((
            Sprite::from_color(color, Vec2::splat(TILE)),
            Transform::from_xyz(x as f32*TILE, y as f32*TILE, 0.0),
            MapTile,
        ));
    }}

    commands.insert_resource(map);
}

// R -> új seed -> újragenerálás
fn on_keypress(keys: Res<ButtonInput<KeyCode>>, mut seed: ResMut<Seed>,
               commands: Commands, old: Query<Entity, With<MapTile>>) {
    if keys.just_pressed(KeyCode::KeyR) {
        seed.0 += 1;
        regen(commands, Res::from(seed.into_inner_ref()), old); // illusztráció — lásd a jegyzetet
    }
}
A regen újrahívásáról

Egy rendszert nem hívunk meg kézzel egy másikból (a paraméterei a sémához kötöttek). Élesben a tiszta minta: tedd a generálást egy AppState::Generating állapot OnEnter-jébe, és R-re csak állítsd NextState-tel oda vissza. Az állapotgép-mintát a scene-váltás tutorial részletezi; ott a generálás egyszerűen egy „jelenet", amibe be- és kiléphetsz.

Gyakorlati tippek & buktatók

Tedd

A generálás a Startup-ba vagy egy állapot OnEnter-jébe való — egyszer fut, nem frame-enként. Lásd az időzítés tutorialt a schedule-ök különbségéről.

Ne

Ne keverd az adatot és a megjelenítést. Az ütközés, a pathfinding és az AI a Map-ből dolgozik; az autotiling csak a felszín. Ha a logikád a kirajzolt csempe-indexet kérdezi, később minden vizuális finomítás eltöri a játékmenetet.

Tedd

Mindig garantáld a játszhatóságot generálás után: flood fillel ellenőrizd az összefüggőséget (08), és ha a feltételek nem teljesülnek, köss össze régiókat vagy regenerálj új seeddel.

Vigyázz

Egy sprite / csempe nem skálázódik — nagy pályán bevy_ecs_tilemap (chunkolt mesh). A naiv megközelítés 80×50-ig elmegy tanulni, fölötte fékez.

Tedd

Mentsd a seedet, ne a kész térképet. Egy u64 elég, hogy bárki reprodukálja a pályát — hibajelentéshez, daily runhoz, mentéshez egyaránt.

Vigyázz

A determinizmus a seedből és a hívási sorrendből együtt jön. Ha átrendezed a generálási lépéseket vagy beszúrsz egy új RNG-hívást, ugyanaz a seed más pályát ad. Tartsd stabilan a sorrendet.

Kombinálj

A legjobb pályák több algoritmust rétegeznek: BSP a szobákra + sejtautomata-kopás az éleken; zaj a biome-okra + részeg séta a barlangbejáratokra. Mindegyik ugyanabba a Map-be ír — ez a szétválasztás haszna.

Tedd

A keretet zárd le: a pálya legszélső cellái legyenek mindig fal. A get() „határon túl = fal" trükkje ezt a vizsgálatoknál megadja, de a tényleges szélső cellákat is állítsd fixen falra, hogy a játékos ne sétálhasson le a térképről.

TL;DR

A lényeg egy kártyán

  • Két fázis: generálás (adat: mi van hol) és autotiling (megjelenítés: hogy néz ki). Soha ne keverd.
  • Map = lapos Vec<Tile>, idx = y*w + x; a get() a határon túl falat ad — fél tutorialnyi peremesetet spórol.
  • Seed = reprodukálhatóság: StdRng::seed_from_u64; mentsd a seedet, ne a térképet.
  • Fehér zaj önmagában játszhatatlan — de jó kiindulás a sejtautomatához.
