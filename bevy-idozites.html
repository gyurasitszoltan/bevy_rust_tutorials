Időzítés Bevyben:
Time, delta_secs() és FixedUpdate

Minden, amit a frame-független mozgásról, a fix időlépésről, az interpolációról és a klasszikus buktatókról tudni érdemes — két interaktív demóval, amik a böngészőben mutatják meg, mi történik valójában.

Time<Real> Time<Virtual> Time<Fixed> overstep_fraction() Timer · Stopwatch
00

Miért számít az idő?

Egy játék fő ciklusa nem fut állandó sebességgel. Egy gyengébb gépen 30, egy 144 Hz-es monitoron 144 frame készülhet másodpercenként. Ha minden frame-ben ugyanannyit lépsz, akkor a mozgás sebessége a hardvertől függ — a karakter háromszor gyorsabb lesz egy erős gépen.

A megoldás, hogy a mozgást idővel skálázod, nem frame-mel. Két alapeszközöd van rá, és nem egymás alternatívái:

  • Time::delta_secs() az Update-ben — változó időlépés, frame-enként egyszer. A sebességet megszorzod a delta idővel, és kész a frame-független mozgás.
  • FixedUpdate — fix időlépés, egy frame alatt akár nullaszor vagy többször is lefut. Determinisztikus, stabil szimulációhoz.
Verzió

A tutorial Bevy 0.18-ra készült. A metódusnevek (delta_secs(), elapsed_secs()) a 0.15 óta érvényesek — régebbi kódban delta_seconds() szerepelhet.

01

A Time óra-rendszer

Bevyben nem egy óra van, hanem három, mind Time<T> resource formában. Mindegyik mást mér:

ÓraMit mérPause / lassítás hat rá?
Time<Real>A valós, fali óra ideje. A nyers eltelt idő.Nem. Mindig pörög.
Time<Virtual>A „játékidő". A Real-ből származik, de szüneteltethető és skálázható.Igen.
Time<Fixed>Fix lépésközű idő. A Virtual időt fogyasztja egy akkumulátorból.Igen (a Virtual-on keresztül).

A lánc: a Real idő hajtja a Virtual időt (skálázva, vágva, esetleg megállítva), a Virtual pedig táplálja a Fixed akkumulátort. Ezért elég a Virtual-t megállítanod ahhoz, hogy az egész szimuláció megálljon — a fizika is.

// A három órát explicit típussal kéred le:
fn debug_clocks(real: Res<Time<Real>>, virt: Res<Time<Virtual>>) {
    info!("valós dt: {}", real.delta_secs());   // mindig pörög
    info!("játék dt: {}", virt.delta_secs());   // 0, ha szünetel
}
02

Update + delta_secs()

Az Update schedule frame-enként pontosan egyszer fut. A delta_secs() az előző frame óta eltelt másodperceket adja vissza f32-ként. Az aranyszabály: amit mozgatsz, szorozd meg a delta idővel.

// ROSSZ — FPS-függő. 144 Hz-en 2,4x gyorsabb, mint 60 Hz-en.
fn move_naiv(mut q: Query<&mut Transform, With<Player>>) {
    for mut t in &mut q {
        t.translation.x += 4.0;
    }
}

// JÓ — frame-független. Mindig 200 px/másodperc, akármilyen FPS-en.
fn move_jo(time: Res<Time>, mut q: Query<&mut Transform, With<Player>>) {
    let speed = 200.0; // px / másodperc
    for mut t in &mut q {
        t.translation.x += speed * time.delta_secs();
    }
}

A mértékegység gondolkodás segít: a sebesség px/másodperc, a delta másodperc, a szorzat px — pont egy frame-nyi elmozdulás. Ez független attól, hány frame van egy másodpercben.

A delta nulla lehet az első frame-ben vagy szünet alatt. Ha osztasz vele (pl. sebesség becsléshez), mindig védd le: if dt > 0.0 { ... }.

03

FixedUpdate

A FixedUpdate nem frame-enként fut, hanem fix időközönként — alapból 64-szer másodpercenként (64 Hz). Bevy egy akkumulátorban gyűjti az eltelt időt, és amíg az akkumulátorban van legalább egy fix lépésnyi idő, lefuttatja a FixedUpdate-et és levon belőle egy lépést.

Ennek a következménye, hogy egy renderelt frame alatt a FixedUpdate lefuthat nullaszor, egyszer, vagy többször is:

  • Gyors frame (pl. 144 Hz): néha 0 fix lépés (még nem gyűlt össze elég idő).
  • Normál frame: 1 fix lépés.
  • Lassú/akadó frame: 2-3 fix lépés egyszerre, hogy „behozza" a lemaradást.
// FixedUpdate-ben futó rendszer. Itt a delta MINDIG ugyanaz (a fix lépésköz).
fn apply_gravity(time: Res<Time>, mut q: Query<&mut Velocity>) {
    let g = -9.81;
    for mut v in &mut q {
        v.0.y += g * time.delta_secs(); // delta = 1/64 s, állandó
    }
}

app.add_systems(FixedUpdate, apply_gravity);
Miért szorzunk delta-val, ha úgyis fix?

Mert így hordozható marad: ha később 64 Hz-ről 128 Hz-re váltasz, a kód egy sor változtatás nélkül helyes marad. A delta a fizikában a „mindig fix" érték, de attól még a képletbe tartozik.

Mikor melyiket?

A döntés magja: kell-e determinizmus és stabilitás, vagy elég, hogy simán nézzen ki?

FeladatHovaMiért

Ha nincs komoly fizika, a játékos mozgását nyugodtan csináld Update-ben delta-val — bőven elég sima. Ha viszont avian2d-t használsz, az alapból FixedUpdate-ben fut, így a mozgásod is oda való, különben a fizika és a karakter eltérő ütemben halad, és bizonytalan ütközések lesznek.

05

A generikus Time automatikus feloldása

Itt egy kulcsfontosságú apróság, amit sokan félreértenek. A típus nélküli Res<Time> nem egy negyedik óra — ez egy kontextusfüggő mutató a megfelelő órára:

  • Update-ben a Res<Time> a Time<Virtual> értékét adja.
  • FixedUpdate-ben ugyanaz a Res<Time> a Time<Fixed> értékét adja.

Ez azt jelenti, hogy ugyanazt a time.delta_secs() hívást használhatod mindkét helyen, és mindig a helyes delta jön vissza. Ezért szoktuk a rendszerekben a generikus Res<Time>-ot kérni — a rendszer attól lesz „hordozható" a két schedule között.

// Ugyanaz a rendszer mindkét schedule-ben helyesen viselkedik:
fn integrate(time: Res<Time>, mut q: Query<(&mut Transform, &Velocity)>) {
    for (mut t, v) in &mut q {
        t.translation += v.0 * time.delta_secs();
    }
}

// Update-ben: delta = változó frame-idő (Virtual)
// FixedUpdate-ben: delta = fix lépésköz (Fixed)
Vigyázz

Csak akkor kérj explicit Res<Time<Fixed>>-t, ha tényleg a fix óra speciális adataira van szükséged (pl. overstep_fraction()). Ha az Update-ben kérsz Time<Fixed>-et, a frissítés ütemén kívüli, „beragadt" deltákat kaphatsz.

06

Interpoláció: sima render fix lépéseknél

Probléma: ha a fizika 64 Hz-en fut, de a monitor 144 Hz-en renderel, akkor a fizikai pozíció „ugrál" — több frame ugyanazt a pozíciót mutatja, aztán egyet ugrik. Ez látható remegés (jitter) lehet, főleg kamera-követésnél.

Megoldás: tárold a előző és a jelenlegi fizikai pozíciót, és a render frame-ben interpolálj közöttük. Az interpolációs tényezőt a Time<Fixed>::overstep_fraction() adja — ez egy 0..1 érték, ami megmondja, „mennyire vagyunk benne" a következő fix lépésben.

#[derive(Component)]
struct Physical { previous: Vec3, current: Vec3 }

// FixedUpdate: a fizika lépteti a logikai pozíciót, és eltolja previous-t
fn step_physics(mut q: Query<(&mut Physical, &Velocity)>, time: Res<Time>) {
    for (mut p, v) in &mut q {
        p.previous = p.current;
        p.current += v.0 * time.delta_secs();
    }
}

// Update: a renderelt Transform interpolál a két logikai pozíció között
fn interpolate(
    fixed: Res<Time<Fixed>>,
    mut q: Query<(&mut Transform, &Physical)>,
) {
    let alpha = fixed.overstep_fraction();
    for (mut t, p) in &mut q {
        t.translation = p.previous.lerp(p.current, alpha);
    }
}

app.add_systems(FixedUpdate, step_physics)
   .add_systems(Update, interpolate);
Mikor érdemes?

Csak akkor építsd be, ha tényleg látsz remegést, vagy ha pontos determinizmus + sima kép is kell. Egyszerűbb játéknál (és ha a mozgás úgyis Update-ben van) felesleges bonyolítás.

07

Input + FixedUpdate — a klasszikus buktató

Ez talán a leggyakoribb hiba. A ButtonInput él-detektálása (just_pressed, just_released) frame-enként resetelődik, az Update ütemében. Ha közvetlenül FixedUpdate-ben olvasod:

  • Lemaradhat egy gombnyomás, ha a FixedUpdate épp 0-szor fut abban a frame-ben.
  • Duplázódhat, ha a FixedUpdate kétszer fut, mire a flag törlődik.

A megoldás: az input él-eseményeit az Update-ben olvasd be egy bufferbe, és a FixedUpdate-ben azt a buffert fogyaszd el.

#[derive(Resource, Default)]
struct InputBuffer { jump_queued: bool }

// Update: itt megbízható az él-detektálás
fn read_input(keys: Res<ButtonInput<KeyCode>>, mut buf: ResMut<InputBuffer>) {
    if keys.just_pressed(KeyCode::Space) {
        buf.jump_queued = true; // bufferbe rakjuk
    }
}

// FixedUpdate: itt használjuk fel, pontosan egyszer
fn apply_jump(mut buf: ResMut<InputBuffer>, mut q: Query<&mut Velocity, With<Player>>) {
    if buf.jump_queued {
        for mut v in &mut q { v.0.y = 300.0; }
        buf.jump_queued = false; // elfogyasztva
    }
}

app.init_resource::<InputBuffer>()
   .add_systems(Update, read_input)
   .add_systems(FixedUpdate, apply_jump);
Tipp

A folyamatos állapot (keys.pressed(...), pl. tartott irány) viszont biztonságosan olvasható FixedUpdate-ben is — az nem reseteldik él alapján, hanem a gomb aktuális állapotát tükrözi.

08

Idő-manipuláció: pause, lassítás, gyorsítás

A Time<Virtual> az, amit szabadon manipulálhatsz. Mivel a Fixed óra ebből táplálkozik, ha megállítod a Virtual-t, a fizika is megáll — de az Update és a Time<Real> tovább pörög, így a szünet-menü UI-ja működik.

// Pause / unpause (pl. ESC-re)
fn toggle_pause(keys: Res<ButtonInput<KeyCode>>, mut time: ResMut<Time<Virtual>>) {
    if keys.just_pressed(KeyCode::Escape) {
        if time.is_paused() { time.unpause(); } else { time.pause(); }
    }
}

// Lassítás (slow-mo) és gyorsítás
fn set_speed(mut time: ResMut<Time<Virtual>>) {
    time.set_relative_speed(0.5); // fél sebesség — bullet time
    // time.set_relative_speed(2.0); // dupla sebesség
}
Fontos

A szünet-menü UI-ja és a „valódi" időt igénylő dolgok (FPS-mérés, teljesítmény, valós idejű timeout) a Time<Real>-t használják — különben a szünet alatt befagynának.

09

Timer és Stopwatch

A nyers delta jó a mozgáshoz, de eseményekhez (cooldown, spawn-időzítő, ideiglenes effekt) a Timer és a Stopwatch kényelmesebb. Mindkettőt tick(delta)-kal lépteted.

Timer — visszaszámláló, ismétlődhet

#[derive(Component)]
struct AttackCooldown(Timer);

// pl. 0.8 mp-es cooldown, egyszeri
// AttackCooldown(Timer::from_seconds(0.8, TimerMode::Once))

fn tick_cooldown(time: Res<Time>, mut q: Query<&mut AttackCooldown>) {
    for mut cd in &mut q {
        cd.0.tick(time.delta());
        if cd.0.finished() {
            // kész, lehet újra támadni
        }
    }
}

Ismétlődő timer — pl. ellenség-spawn

#[derive(Resource)]
struct SpawnTimer(Timer);

// SpawnTimer(Timer::from_seconds(2.0, TimerMode::Repeating))

fn spawn_enemies(time: Res<Time>, mut timer: ResMut<SpawnTimer>, mut commands: Commands) {
    timer.0.tick(time.delta());
    if timer.0.just_finished() {
        commands.spawn(/* enemy bundle */);
    }
}
Hasznos metódusok

just_finished() — pont ebben a tickben fejeződött be (él). finished() — már lejárt. fraction() — 0..1 haladás (jó UI sávhoz). reset() — újraindít. A Stopwatch ezzel szemben csak felfelé számol, nincs célideje — eltelt időt mérsz vele.

10

FixedUpdate hangolása

A fix lépésközt egy resource beszúrásával állítod. Alap a 64 Hz, de pl. retro-érzethez vagy hálózathoz más is jó lehet.

// Fix ráta beállítása indításkor
app.insert_resource(Time::<Fixed>::from_hz(60.0));
// vagy: Time::<Fixed>::from_seconds(1.0 / 50.0)

A „halálspirál" (spiral of death)

Ha egy frame túl sokáig tart (pl. nagy GC-szünet, betöltés), sok fix lépés gyűlik fel. Ezek lefuttatása megint sok időt vesz, ami még több lépést halmoz — a játék befagyhat. A védelem: a Time<Virtual> egy frame alatt felvehető maximális deltáját korlátozod.

fn clamp_frame_time(mut time: ResMut<Time<Virtual>>) {
    // egy frame max 0.25 mp-nyi időt adhat tovább — efölött "lassul",
    // de nem fagy be a fix lépések lavinájától
    time.set_max_delta(std::time::Duration::from_millis(250));
}
Mi a kompromisszum?

A vágott idő azt jelenti, hogy egy hatalmas akadás után a játék kicsit „lassított felvételben" hozza be a lemaradást, ahelyett hogy hirtelen nagyot ugrana vagy befagyna. Ez szinte mindig a kívánt viselkedés.

11

Gyakorlati tippek & buktatók

Tedd

A rendszereidben kérj generikus Res<Time>-ot, hacsak nincs konkrét okod a típusos órára. Így a rendszer mindkét schedule-ben helyesen viselkedik.

Ne

Ne olvass just_pressed él-eseményt közvetlenül FixedUpdate-ben — bufferelj az Update-ben (lásd 07).

Ne

Ne tegyél render-simaságot FixedUpdate-be, és determinisztikus fizikát ne tegyél Update-be. A két cél keverése okozza a legtöbb remegést és bugot.

Tedd

Oszd ketté: szimuláció FixedUpdate-ben, megjelenítés Update-ben. Ha kamerád egy FixedUpdate-ben mozgatott testet követ, az interpoláció (06) megszünteti a remegést.

Vigyázz

A delta lehet 0 (első frame, szünet). Osztás előtt mindig ellenőrizd. A nagyon nagy delta (akadás) is gond lehet — ezt a set_max_delta kezeli.

Tedd

Cooldownhoz, spawnhoz, ideiglenes effekthez használj Timer-t a kézi delta-számolgatás helyett — olvashatóbb és kevésbé hibázható.

Debug

FPS-mérésre, profilozásra, valós idejű timeoutra mindig Time<Real>-t használj, hogy a pause és a slow-mo ne torzítsa.

12

Komplett minta — összerakva

Egy minimál, de helyesen szétválasztott váz: input bufferelés, fizika fix lépésközön, és (egyszerű esetben) közvetlen Update-mozgás. Ez a csontváz skálázható egy igazi pixel-art top-down játékká.

use bevy::prelude::*;

#[derive(Component)] struct Player;
#[derive(Component)] struct Velocity(Vec3);
#[derive(Resource, Default)] struct InputBuffer { jump: bool }

fn main() {
    App::new()
        .add_plugins(DefaultPlugins)
        .insert_resource(Time::<Fixed>::from_hz(64.0))
        .init_resource::<InputBuffer>()
        // --- Update: input + sima vizuál ---
        .add_systems(Update, (read_input, camera_follow))
        // --- FixedUpdate: determinisztikus szimuláció ---
        .add_systems(FixedUpdate, (apply_jump, apply_gravity, integrate).chain())
        .run();
}

// Update: megbízható él-detektálás, bufferbe
fn read_input(keys: Res<ButtonInput<KeyCode>>, mut buf: ResMut<InputBuffer>) {
    if keys.just_pressed(KeyCode::Space) { buf.jump = true; }
}

// FixedUpdate: a bufferből, pontosan egyszer
fn apply_jump(mut buf: ResMut<InputBuffer>, mut q: Query<&mut Velocity, With<Player>>) {
    if buf.jump {
        for mut v in &mut q { v.0.y = 320.0; }
        buf.jump = false;
    }
}

// FixedUpdate: stabil lépésköz → stabil fizika
fn apply_gravity(time: Res<Time>, mut q: Query<&mut Velocity>) {
    for mut v in &mut q { v.0.y += -980.0 * time.delta_secs(); }
}

fn integrate(time: Res<Time>, mut q: Query<(&mut Transform, &Velocity)>) {
    for (mut t, v) in &mut q { t.translation += v.0 * time.delta_secs(); }
}

// Update: a kamera simán követ minden render frame-ben
fn camera_follow(/* ... */) { /* delta-alapú lerp a játékos felé */ }
13

TL;DR

A lényeg egy kártyán

  • Mozgás Update-ben: mindig * time.delta_secs() — különben FPS-függő.
  • FixedUpdate: fix lépésköz, frame-enként 0..n lefutás, determinisztikus fizikához.
  • Generikus Res<Time> automatikusan a helyes órát adja: Update → Virtual, FixedUpdate → Fixed.
  • Három óra: Real (sosem áll meg), Virtual (pause/slow-mo), Fixed (a Virtual-ból táplálkozik).
  • Input él (just_pressed): Update-ben olvasd, bufferbe, FixedUpdate-ben fogyaszd.
  • Remegés fix lépésnél? Interpolálj overstep_fraction()-nel.
  • Pause: állítsd meg a Time<Virtual>-t — a fizika is megáll, az UI nem.
  • Befagyás akadásnál? set_max_delta a halálspirál ellen.
  • Pixel-art top-down: nincs fizika → Update+delta elég; van avian2d → FixedUpdate.
