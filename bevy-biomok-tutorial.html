Biomok — a procedurális világ
identitása és progressziója

A nyers zaj még csak terep. A biom az, ami értelmet ad neki: minden tájnak saját arcot, saját nehézséget és saját jutalmat. Ebben a leckében nem új térképet generálunk — a procedurális térkép tutorial erre megtanított —, hanem egy réteget építünk fölé, ami a véletlent irányított, felfedezésre hívó világgá szervezi. Öt interaktív demóval.

Biome enum + def-tábla hőmérséklet × nedvesség domain warping nehézség-progresszió súlyozott spawn data-driven
00

Miért biomok?

A procedurális generálás kétélű fegyver. A véletlen adja az újrajátszhatóságot — minden indításkor más világ —, de ha semmi nem fogja meg, a világ céltalanná válik: végtelen, egyforma terep, ahol nincs hova tartani, és nincs miért. A játékos elindul egy irányba, és ugyanazt látja, mint az ellenkező irányban.

A biom erre a gyógyír. Egy biom nem más, mint a térkép egy címkézett régiója, amihez saját identitás (paletta, csempék, növényzet, hangulat), saját nehézség és saját tartalom (lények, nyersanyagok, struktúrák) tartozik. Három dolgot old meg egyszerre:

  • Változatosság. A havas fenyves nem ugyanaz az élmény, mint a forró sivatag. A szem és a játékmenet is felfrissül a határ átlépésekor.
  • Progresszió. Ha a biomok növekvő nehézségűek, és a játékos nagyjából a helyes sorrendben találkozik velük, a céltalan bóklászásból irányított utazás lesz — felfedezés cél felé.
  • Jutalom. Ha a nehezebb biomok jobb nyersanyagot adnak, a felfedezés kockázat–jutalom döntéssé válik. Megéri-e tovább menni? Ez tartja mozgásban a játékost.
A lecke gerince

Egy biom-réteg három, élesen szétváló kérdésre felel — és a tutorial végig ezt a hármat választja szét: (1) Mező — milyen folytonos jellemzők alapján döntünk (hőmérséklet, nedvesség, nehézség)? (2) Identitás — a címkéből hogyan lesz szín, csempe, lény? (3) Progresszió — hogyan rendezzük el a biomokat úgy, hogy a felfedezés iránnyal bírjon, mégse legyen unalmasan lineáris?

Mire épül & függőségek

Ez a lecke a procedurális generálást feltételezi (zaj/fBm, seed-determinizmus, flood fill, feature-placement) — innen a Map, a StdRng és az fBm. A kód Bevy 0.18-ra készült, ugyanazokkal a crate-ekkel:

[dependencies]
bevy  = "0.18"
rand  = "0.8"   # seedelhető RNG (spawn-táblákhoz)
noise = "0.9"   # Perlin / fBm — hőmérséklet- és nedvesség-mezőkhöz
01

A biom mint adat

Mielőtt eldöntenénk, hol van melyik biom, döntsük el, mi egy biom. A kulcs ugyanaz a szétválasztás, mint a térkép-generálásnál: a biom egy sima címke (egy enum), és minden, ami abból következik — szín, nehézség, lények —, egy külön adattáblában él. Ez a „data-driven" megközelítés: új biom = egy enum-variáns + egy sor a táblában, kódlogika átírása nélkül.

use bevy::prelude::*;

/// A 7 biom — a világ "identitás-címkéi". Olcsó, Copy enum:
/// könnyű másolni, match-elni, tömbben tárolni.
#[derive(Clone, Copy, PartialEq, Eq, Debug)]
enum Biome {
    EasyRainforest,                  // spawn-közeli, könnyű
    Meadow, Rainforest, Shrubland,   // középső gyűrű
    Snowy,  DarkForest, Desert,      // külső gyűrű, nehéz
}

/// Minden biomhoz EGY adatrekord tartozik. Ez vezérli a teljes
/// "lefelé" irányt: a render, az ütközés és a spawn ebből olvas.
struct BiomeDef {
    name:       &'static str,
    ground:     Color,   // alap talajszín — a megjelenítés innen olvas
    difficulty: u8,      // 0 = spawn-közeli, 2 = külső gyűrű
    decoration: f32,     // dekoráció-sűrűség 0..1 (fű, kő, fa)
}

fn biome_def(b: Biome) -> BiomeDef {
    use Biome::*;
    match b {
        EasyRainforest => BiomeDef { name: "Easy Rainforest", ground: Color::srgb(0.51,0.65,0.34), difficulty: 0, decoration: 0.55 },
        Meadow         => BiomeDef { name: "Meadow",          ground: Color::srgb(0.60,0.63,0.29), difficulty: 1, decoration: 0.45 },
        Rainforest     => BiomeDef { name: "Rainforest",      ground: Color::srgb(0.30,0.48,0.25), difficulty: 1, decoration: 0.80 },
        Shrubland      => BiomeDef { name: "Shrubland",       ground: Color::srgb(0.76,0.64,0.37), difficulty: 1, decoration: 0.35 },
        Snowy          => BiomeDef { name: "Snowy",           ground: Color::srgb(0.78,0.83,0.84), difficulty: 2, decoration: 0.25 },
        DarkForest     => BiomeDef { name: "Dark Forest",     ground: Color::srgb(0.20,0.27,0.19), difficulty: 2, decoration: 0.95 },
        Desert         => BiomeDef { name: "Desert",          ground: Color::srgb(0.85,0.77,0.53), difficulty: 2, decoration: 0.15 },
    }
}

A térképhez pedig egy biom-réteget tartunk — a tile-Map testvérét: ugyanakkora rács, de cellánként egy Biome-címke. A két réteg külön él, mert külön kérdésre felel: a Map azt mondja meg, „itt fal van vagy padló", a BiomeMap azt, „ez itt a sivatag".

/// A biom-térkép a tile-Map TESTVÉRE. Cellánként egy biom-címke.
#[derive(Resource)]
struct BiomeMap {
    width:  i32,
    height: i32,
    cells:  Vec<Biome>,
}

impl BiomeMap {
    fn get(&self, x: i32, y: i32) -> Biome {
        self.cells[(y * self.width + x) as usize]
    }
}
Miért külön réteg?

Csábító egy Tile-ba zsúfolni a biomot is (Tile::DesertFloor, Tile::SnowWall…), de ez kombinatorikus robbanás: minden csempe × minden biom. Tartsd a Tile-t szűken (járható-e?), a Biome-ot meg külön rétegben — a render majd a kettő kombinációjából választ csempét. Ugyanaz az adat-vs-megjelenítés elv, mint a tile-editor leckében.

02

Egy tengely kevés

A legkézenfekvőbb ötlet: vedd a zaj-magasságtérképet, és vágd küszöbökkel biomokra — víz, part, fű, erdő, hegy, hó. Ezt a procedurális-térkép lecke már bemutatta, és kiváló terephez: a magasság természetes módon ad óceánt és hegycsúcsot.

De biom-identitáshoz az egytengelyes magasság hamar elfogy. Egyetlen szám csak egyetlen, monoton sorrendet ad: alacsony → magas. Nem tudja megkülönböztetni a forró alföldet a hideg alföldtől, a száraz dombot a nedvestől. Márpedig pont ez teszi a sivatagot sivataggá és a havast havassá — nem a magasság, hanem a klíma.

A felismerés

A biom nem a magasság függvénye, hanem több, független jellemzőé. A klasszikus modell (Whittaker biom-diagramja) kettőt használ: hőmérséklet és csapadék. Ezekből egy 2D mátrix nyílik — és ott már elfér a tundra, a sivatag, az esőerdő egymás mellett, nem egymás fölött. A következő szakasz ezt építi fel.

A magasság ettől nem felesleges: kiváló harmadik tengely (víz a mélyben, kopár szikla a csúcson, a klíma a kettő között), vagy külön vizuális réteg. De a biom gerince a klíma-mátrix lesz.

03

Klíma-mátrix: hőmérséklet × nedvesség

A trükk meglepően egyszerű: generálj két független zajmezőt külön seeddel — az egyik a hőmérséklet, a másik a nedvesség —, majd a kettő párosából egy táblázattal olvasd ki a biomot. Ez a Whittaker-modell játék-változata.

use noise::{Fbm, Perlin, MultiFractal, NoiseFn};

/// Két FÜGGETLEN zajmező. A +9000 seed-eltolás garantálja, hogy a
/// hőmérséklet és a nedvesség ne korreláljon. (fBm: lásd a proc-gen leckét.)
fn climate_fields(seed: u32) -> (Fbm<Perlin>, Fbm<Perlin>) {
    let temp  = Fbm::<Perlin>::new(seed)
        .set_octaves(4).set_frequency(0.03);
    let moist = Fbm::<Perlin>::new(seed + 9_000)
        .set_octaves(4).set_frequency(0.03);
    (temp, moist)
}

/// A mátrix: (hőmérséklet, nedvesség) -> biom. A küszöbök hangolhatók;
/// a sorrend számít — a korábbi ágak "elkapják" az eseteket.
fn biome_from_climate(temp: f64, moist: f64) -> Biome {
    use Biome::*;
    match (temp, moist) {
        (t, _) if t < 0.30             => Snowy,       // hideg: minden fagyos
        (t, m) if t > 0.70 && m < 0.40  => Desert,      // forró + száraz
        (t, _) if t > 0.70             => Shrubland,    // forró + nyirkosabb
        (_, m) if m > 0.65             => Rainforest,   // mérsékelt + nedves
        (_, m) if m < 0.35             => Meadow,       // mérsékelt + száraz
        _                              => DarkForest,   // mérsékelt + közepes
    }
}

A mintát cellánként így olvasod ki — a Perlin -1..1 között ad, ezért 0..1-be toljuk:

let t = temp.get([x as f64 * 0.03, y as f64 * 0.03]) * 0.5 + 0.5;
let m = moist.get([x as f64 * 0.03, y as f64 * 0.03]) * 0.5 + 0.5;
let biome = biome_from_climate(t, m);

Egy 3×3-as nézet a leghasznosabb térkép a fejedben: a sorok a hőmérséklet, az oszlopok a nedvesség. Minden cella egy biom:

▶ Demó: klíma-mátrix
Két független zajmező (hőmérséklet + nedvesség) párosa indexel egy 3×3 biom-mátrixot. Told a hőmérséklet- vagy a nedvesség-eltolást, és nézd, ahogy a világ a forró/hideg, illetve a száraz/nedves sarok felé csúszik — a mátrix jobb alsó sarkából (dzsungel) a bal felsőbe (tundra).
Független ≠ egyenletes

A két mező legyen független (más seed), különben a forró mindig nedves lenne, és odaveszne a mátrix fele. De nyugodtan adj a hőmérséklethez globális trendet — pl. észak felé hűljön —, hogy a sarkok ne véletlenszerűen szóródjanak. A demó temp-eltolása pont egy ilyen globális trend.

04

Természetes határok — domain warping

Van egy árulkodó jel, amiről azonnal kiszúrod a gyenge biom-rendszert: a határok egyenesek. Ha a hőmérsékletet pusztán a kelet–nyugati pozícióból veszed, a sivatag–cserjés határ tökéletes függőleges vonal lesz. A természet nem így néz ki — a határoknak öböleik, félszigeteik, foltjaik vannak.

A megoldás a domain warping (tartomány-torzítás): mielőtt lekérdeznéd a klímamezőt, told el a mintavételi pontot egy másik zajjal. Vagyis nem azt kérdezed, „milyen a klíma itt", hanem „milyen a klíma egy kicsit arrébb — és arra, amerre a torzító-zaj mutat". Az egyenes küszöbből hullámos, organikus határ lesz, plusz kódlogika nélkül.

/// A pontot egy Perlin-zajjal told el, MIELŐTT lekérdezed a klímát.
/// `strength` szabja a hullámzás amplitúdóját (cellában).
fn warp(p: Vec2, w: &Perlin, strength: f32) -> Vec2 {
    let s = 0.04;
    let ox = w.get([(p.x * s) as f64,         (p.y * s) as f64        ]) as f32;
    let oy = w.get([(p.x * s + 100.0) as f64, (p.y * s + 100.0) as f64]) as f32;
    p + Vec2::new(ox, oy) * strength
}

// használat: a klímát a TORZÍTOTT koordinátán kérdezd le
let wp = warp(Vec2::new(x as f32, y as f32), &warp_noise, 18.0);
let t  = temp.get([wp.x as f64 * 0.03, wp.y as f64 * 0.03]) * 0.5 + 0.5;

A +100.0 eltolás a két lekérdezés között azért kell, hogy az x- és y-eltolás a zaj különböző részéből jöjjön — különben átlósan torzulna minden. A torzítás ugyanazt a mezőt használja, csak más helyen mintavételezve, így determinisztikus marad: ugyanaz a seed, ugyanazok a határok.

▶ Demó: egyenes vs. torzított határok
Három biom-sáv, pusztán a kelet–nyugati pozícióból. Nulla torzításnál mesterséges, függőleges csíkok. Told fel a torzítást — ugyanaz a három biom, de a határok öblökké, félszigetekké, leszakadt foltokká hullámzanak. Ez egyetlen extra zajréteg.
05

A progresszió problémája

A klíma-mátrix gyönyörű, változatos világot ad — de céltalant. A biomok véletlenszerűen szóródnak; a sivatag ott lehet a spawn mellett, a könnyű rét meg a térkép túlvégén. A felfedezésnek nincs íve. Itt lép be a progresszió: a biomokat nehézség szerint kell elrendezni, és gondoskodni róla, hogy a játékos nagyjából a helyes sorrendben találkozzon velük.

Az első ösztön a lineáris réteg: a spawn legyen alul, fölötte a könnyű biom, fölötte a közepes, legfelül a nehéz. A progresszió kristálytiszta… de a felfedezés halálosan unalmas lesz. Egyetlen tengely mentén haladsz, oda-vissza; balra és jobbra ugyanaz a sáv. A világ egy folyosóvá lapul.

A jobb megoldás a zónás (radiális) elrendezés: a spawn-pont köré szervezd a nehézséget. A spawn-hoz közel minden könnyű; minél messzebb mész — bármelyik irányba —, annál nehezebb. Így a felfedezés kétdimenzióssá válik: minden irány új biomot kínál, de a nehézség mégis kiszámíthatóan nő. (Ezt a mintát népszerűsítette pl. a Core Keeper.)

A kulcsgondolat

Ne a biomot helyezd el közvetlenül, hanem egy nehézség-mezőt — egy folytonos „mennyire veszélyes itt" skalárt a térkép fölött. A biom ennek a mezőnek és a klímának a kombinációjából esik ki. Így a progresszió (kontrollált, radiális) és a változatosság (zajos klíma) egymástól függetlenül hangolható. A következő szakasz ezt írja kódba.

06

Nehézség-mező + a 7 biom elrendezése

A nehézség-mező egyetlen függvény: minden cellához ad egy 0..1 „veszélyesség" értéket. A radiális változat a spawntól (a térkép alja-közepe) mért, normált távolság — közel 0, a sarkokban 1:

/// RADIÁLIS nehézség: a spawntól mért távolság. Minden irányban nő.
fn difficulty_radial(p: IVec2, spawn: Vec2, max_dist: f32) -> f32 {
    (p.as_vec2().distance(spawn) / max_dist).clamp(0.0, 1.0)
}

/// LINEÁRIS nehézség (az "első ötlet"): csak a függőleges pozíció.
/// Tiszta progresszió, de a felfedezés egyetlen tengelyre lapul.
fn difficulty_linear(p: IVec2, height: i32) -> f32 {
    1.0 - p.y as f32 / height as f32
}

A 7 biom elrendezése most már két tengely metszete: a kontrollált nehézség-gyűrű (közép → külső) és a zajos hőmérséklet-sáv (hideg → forró). Pontosan ez adja a referencia-térkép szerkezetét: spawn körül a könnyű esőerdő, középső gyűrűben rét / esőerdő / cserjés, a külső gyűrűben havas / sötét erdő / sivatag.

/// nehézség-gyűrű × hőmérséklet-sáv -> a 7 biom egyike.
/// Ez köti össze a KONTROLLÁLT tengelyt a ZAJOSSAL.
fn biome_for(difficulty: f32, temp: f64) -> Biome {
    use Biome::*;
    let tier = if difficulty < 0.34 { 0 } else if difficulty < 0.67 { 1 } else { 2 };
    let band = if temp < 0.40 { 0 } else if temp < 0.66 { 1 } else { 2 }; // hideg/mérs/forró
    match (tier, band) {
        (0, _) => EasyRainforest,   // spawn körül MINDIG könnyű
        (1, 0) => Meadow,           // közép + hideg
        (1, 1) => Rainforest,       // közép + mérsékelt
        (1, _) => Shrubland,        // közép + forró
        (2, 0) => Snowy,            // külső + hideg
        (2, 1) => DarkForest,       // külső + mérsékelt
        (2, _) => Desert,           // külső + forró
        _      => Rainforest,
    }
}

Figyeld meg a (0, _) ágat: a spawn körüli gyűrűben a hőmérséklet nem számít — mindig a könnyű biom jön. Ez szándékos: a kezdő-zóna legyen biztonságos és egységes, függetlenül attól, merre lép ki a játékos. A nehézség a fő tengely; a klíma csak ízesíti a gyűrűket.

▶ Demó: a 7-biom világ — lineáris vs. zónás
A teljes elrendezés, ahogy a kód kiadja. A piros pötty a spawn (alja-közepe). Válts a zónás (radiális nehézség) és a lineáris (rétegek) elrendezés között — érezd a különbséget. A nehézség-hőtérkép a mögöttes mezőt mutatja (zöld = könnyű, piros = nehéz). A határok domain warpinggal hullámzanak.
Keverd a kettőt

A tiszta radiális is lehet jósolható. Élesben gyakran kombinálnak: a radiális alap-nehézséghez adnak egy lassú zaj-réteget („veszély-foltok"), így néha egy nehéz sziget bukkan fel a könnyű zónában — váratlan kihívás és jutalom. A difficulty ettől még 0..1 marad; csak a forrása gazdagabb.

07

Biom-identitás: a címkéből kép

A BiomeMap kész — most jön az identitás: a címkéből látható világ. Mivel mindent a biome_def tábla vezérel (01), a megjelenítés egyirányú levezetés: olvasd a biomot, kérd le a defjét, rajzolj. A render soha nem dönt biomról; csak megjeleníti.

const TILE: f32 = 16.0;

#[derive(Component)] struct MapTile;

/// A talaj kifestése: minden cella a biomja alapszínét kapja.
fn paint_tiles(mut commands: Commands, biomes: Res<BiomeMap>) {
    for y in 0..biomes.height {
        for x in 0..biomes.width {
            let def = biome_def(biomes.get(x, y));
            commands.spawn((
                Sprite::from_color(def.ground, Vec2::splat(TILE)),
                Transform::from_xyz(x as f32 * TILE, y as f32 * TILE, 0.0),
                MapTile,
            ));
        }
    }
}

Az identitás persze több a puszta színnél. A decoration mező vezérli a dekoráció-sűrűséget: a sötét erdő tele van fával és aljnövényzettel (0.95), a sivatag majdnem kopár (0.15). Ez ugyanaz a reject-sampling / szórás-passz, mint a feature-placement — csak a sűrűséget a biom adja.

Mivel az identitás tisztán adat, az egész világ „átöltöztethető" egyetlen tábla cseréjével — ősz, alkonyat vagy idegen-bolygó paletta, ugyanarra a generált mezőre. Ezt mutatja a demó:

▶ Demó: ugyanaz az adat, más bőr
Ugyanaz a BiomeMap, kétféleképp. Az adat nézet a nyers biom-régiókat mutatja (lapos kategória-színek); a render a biome_def palettáját + a biomonként eltérő dekoráció-sűrűséget. Válts palettát — a világ szerkezete nem változik, csak a bőre. Ez a data-driven identitás haszna.

08

Mi tölti meg? — súlyozott spawn-táblák

Egy biom akkor él, ha más lény les rád benne, és más nyersanyagot hagy maga után. Ez köti össze a nehézséget a jutalommal: a veszélyesebb biom tömöttebb és erősebb lényekkel jár, cserébe ritkább, értékesebb nyersanyaggal — így a felfedezés kockázat–jutalom döntés lesz.

Az eszköz a súlyozott spawn-tábla: bionként egy lista, amiben minden bejegyzéshez egy súly tartozik. Húzol egy súlyozott véletlent, és az adja, mi kerül oda (vagy semmi). A Nothing nagy súlya tartja ritkán a tartalmat — nem akarsz minden cellára egy bogarat.

#[derive(Clone, Copy)]
enum Spawn { Bug, Beetle, ClayPale, ClayRed, Flower, Nothing }

struct Weighted { what: Spawn, weight: u32 }
const fn w(what: Spawn, weight: u32) -> Weighted { Weighted { what, weight } }

/// Bionként más a "kit/mit találsz". A nehezebb biomok több és erősebb
/// lényt adnak, cserébe jobb nyersanyagot — ez a felfedezés jutalma.
fn spawn_table(b: Biome) -> &'static [Weighted] {
    use Biome::*; use Spawn::*;
    match b {
        EasyRainforest => &[w(Bug,2),   w(Flower,3),               w(Nothing,12)],
        Meadow         => &[w(Bug,3),   w(Flower,4), w(ClayPale,2), w(Nothing,10)],
        Rainforest     => &[w(Bug,5),   w(Beetle,2), w(Flower,3),  w(Nothing,8)],
        Shrubland      => &[w(Beetle,4),w(ClayPale,3),             w(Nothing,8)],
        Snowy          => &[w(Bug,4),   w(Beetle,3),               w(Nothing,7)],
        DarkForest     => &[w(Beetle,7),w(Flower,2),               w(Nothing,5)],
        Desert         => &[w(Beetle,6),w(ClayRed,4),             w(Nothing,6)],
    }
}

/// Egy súlyozott húzás: a súlyok összegéből egy szám, majd "leszámoljuk".
fn weighted_pick(table: &[Weighted], rng: &mut StdRng) -> Spawn {
    let total: u32 = table.iter().map(|e| e.weight).sum();
    let mut roll = rng.gen_range(0..total);
    for e in table {
        if roll < e.weight { return e.what; }
        roll -= e.weight;
    }
    Spawn::Nothing
}

A betöltő-passz ugyanaz a szórás, mint a dekorációnál: menj végig a cellákon (vagy mintavételezz pontokat), nézd meg a helyi biomot, húzz a táblájából, és ami nem Nothing, azt spawnold be entitásként a csempék fölé (z = 1).

fn populate_world(
    mut commands: Commands,
    biomes: Res<BiomeMap>,
    mut rng: ResMut<WorldRng>,
) {
    for y in 0..biomes.height {
        for x in 0..biomes.width {
            // ritkítás: a cellák töredékén próbálkozunk csak
            if !rng.0.gen_bool(0.12) { continue; }
            match weighted_pick(spawn_table(biomes.get(x, y)), &mut rng.0) {
                Spawn::Nothing => {}
                other => spawn_entity(&mut commands, other, x, y),
            }
        }
    }
}
A táblák a "tervezői" felület

A spawn-tábla az a hely, ahol a játéktervezés történik — nem a kódban, hanem az adatban. Élesben ezt gyakran egy RON/asset fájlba teszik, hogy a tervező a kód újrafordítása nélkül hangolhassa. A nehézség–jutalom görbe (mennyivel jobb a sivatagi nyersanyag, mint a réti?) tisztán a súlyokban lakik. A run-alapú jutalom-logikáról lásd a roguelite-rendszerek leckét.

Átmenetek & szegélyek

A domain warping organikus határvonalat ad, de a határ két oldala még mindig élesen vált: a sötét erdő utolsó cellája mellett rögtön a havas első cellája áll. A profi érzethez két dolog kell — átmeneti sáv és szegélycsempe.

Átmeneti (blend) sáv

A határ közelében keverd a két biom jellemzőit a távolság arányában: a talajszín lerpeljen az egyikből a másikba, a dekoráció-sűrűség is. Így nincs éles vágás, hanem fokozatos átolvadás — pont, mint a valódi ökotónok.

// két biom talajszínének keverése a határ-közelség (0..1) szerint
let a = biome_def(here).ground;
let b = biome_def(neighbour).ground;
let ground = a.mix(&b, blend);   // Bevy Color::mix — lineáris interpoláció

Szegélycsempe (autotiling a biomok közt)

A keskeny, jellegzetes perem — homokpart a víz mellé, fűsáv az erdő szélére — ugyanaz az autotiling, mint a térkép-leckében, csak a „terep" most a biom. A maszkot a szomszédos cellák biomjából számolod: ahol a szomszéd más biom, oda perem-csempe kerül.

// 4-bites maszk: hány fő-irányú szomszéd MÁS biom, mint én?
fn border_mask(biomes: &BiomeMap, x: i32, y: i32) -> u8 {
    let me = biomes.get(x, y);
    let diff = |nx, ny| (biomes.get(nx, ny) != me) as u8;
    (diff(x, y-1) << 0) | (diff(x+1, y) << 1)
  | (diff(x, y+1) << 2) | (diff(x-1, y) << 3)
}
Ne építsd újra — kösd be

A maszk → csempe-index leképezés teljes története (8-bites sarkok, 47/256 redukció, több terep közti átmenetek) a csempe-bitmaszkolás és a dual-grid (+ PRO) leckékben van. A biom-réteg pontosan ugyanazt a gépezetet hajtja, csak a bemenete a Biome a Tile helyett.

Magasság mint perem

Egy elegáns trükk a „hamis szintkülönbséghez": a magasság-tengelyt (02) is kezelheted szegélyként — a magasabb biom pereme egy lefelé „lelógó" sziklacsempét kap, amit a karakter és a növényzet mindig eltakar (a sprite-ok z-je nagyobb). Így a 2D top-down világ is kap egy réteg mélységet, külön Y-sorting logika nélkül.

Bevy integráció

Most fűzd láncba. A teljes pipeline ugyanaz a „előbb adat, aztán kép" sorrend, mint a térkép-generálásnál — csak most a BiomeMap az adat, és a render + a populálás a levezetés. Az egész egyszer fut: Startup-ban vagy egy állapot OnEnter-jében (lásd az időzítés leckét), soha nem az Update-ben.

#[derive(Resource)] struct WorldSeed(u32);
#[derive(Resource)] struct WorldRng(StdRng);

#[derive(States, Default, Clone, PartialEq, Eq, Hash, Debug)]
enum AppState { #[default] Generating, Playing }

fn main() {
    App::new()
        .add_plugins(DefaultPlugins)
        .insert_resource(WorldSeed(8472))
        .init_state::<AppState>()
        .add_systems(OnEnter(AppState::Generating), (
            setup_camera,
            assign_biomes,    // klímamezők + nehézség -> BiomeMap   (ADAT)
            paint_tiles,      // BiomeMap -> talaj-sprite-ok          (KÉP)
            populate_world,   // BiomeMap -> súlyozott spawn          (TARTALOM)
        ).chain())
        .run();
}

A szív az assign_biomes: itt fut össze minden eddigi rész — a klímamező (03), a domain warping (04), a radiális nehézség (06) és a biome_for lookup. A kimenet egyetlen BiomeMap resource:

fn assign_biomes(mut commands: Commands, seed: Res<WorldSeed>) {
    let (w, h) = (96, 64);
    let (temp, _moist) = climate_fields(seed.0);
    let warp_noise = Perlin::new(seed.0 + 1);

    let spawn = Vec2::new(w as f32 / 2.0, h as f32); // alja-közepe
    let max_dist = spawn.length();                   // ~a legtávolabbi sarok

    let mut cells = Vec::with_capacity((w * h) as usize);
    for y in 0..h {
        for x in 0..w {
            let p  = IVec2::new(x, y);
            let wp = warp(p.as_vec2(), &warp_noise, 18.0);          // hullámos határ
            let t  = temp.get([wp.x as f64 * 0.03, wp.y as f64 * 0.03]) * 0.5 + 0.5;
            let d  = difficulty_radial(p, spawn, max_dist);         // radiális progresszió
            cells.push(biome_for(d, t));
        }
    }

    commands.insert_resource(BiomeMap { width: w, height: h, cells });
    // a WorldRng-t a populáláshoz külön resource-ként ugyanebből a seedből hozd
    commands.insert_resource(WorldRng(StdRng::seed_from_u64(seed.0 as u64)));
}
Egy seed, egy sorrend

Ahogy a térkép-generálásnál: a determinizmus a seedből és a hívási sorrendből együtt jön. A klíma, a warp és a populáló RNG mind a WorldSeed-ből származzon (akár eltolt seedekkel), és a passzok sorrendje legyen stabil. Ha átrendezed a láncot, ugyanaz a seed más világot ad — mentsd a seedet, ne a kész BiomeMap-et.

Generálás mint jelenet

Az OnEnter(Generating) minta egyenes folytatása a scene-váltás leckének: a világépítés egy „jelenet", amibe belépsz, lefuttatod a láncot, majd NextState(Playing)-gel továbblépsz. Újrageneráláshoz csak lépj vissza Generating-be — a DespawnOnExit letakarítja a régi csempéket.

Komplett minta — összerakva

Egy minimál, de teljes biom-réteg: klímamező + warp + radiális nehézség → BiomeMap, majd talaj-render. A Map (fal/padló) generálását a térkép-lecke adja; ez a csontváz a fölé kerülő réteg.

use bevy::prelude::*;
use noise::{Fbm, Perlin, MultiFractal, NoiseFn};

const TILE: f32 = 16.0;
const W: i32 = 96;
const H: i32 = 64;

#[derive(Clone, Copy, PartialEq, Eq)]
enum Biome { EasyRainforest, Meadow, Rainforest, Shrubland, Snowy, DarkForest, Desert }

#[derive(Resource)]
struct BiomeMap { width: i32, height: i32, cells: Vec<Biome> }
impl BiomeMap { fn get(&self, x: i32, y: i32) -> Biome { self.cells[(y*self.width + x) as usize] } }

#[derive(Component)] struct MapTile;

fn ground(b: Biome) -> Color {
    use Biome::*;
    match b {
        EasyRainforest => Color::srgb(0.51,0.65,0.34),
        Meadow         => Color::srgb(0.60,0.63,0.29),
        Rainforest     => Color::srgb(0.30,0.48,0.25),
        Shrubland      => Color::srgb(0.76,0.64,0.37),
        Snowy          => Color::srgb(0.78,0.83,0.84),
        DarkForest     => Color::srgb(0.20,0.27,0.19),
        Desert         => Color::srgb(0.85,0.77,0.53),
    }
}

fn warp(p: Vec2, w: &Perlin, strength: f32) -> Vec2 {
    let s = 0.04;
    let ox = w.get([(p.x*s) as f64,        (p.y*s) as f64       ]) as f32;
    let oy = w.get([(p.x*s+100.0) as f64,  (p.y*s+100.0) as f64 ]) as f32;
    p + Vec2::new(ox, oy) * strength
}

fn biome_for(diff: f32, temp: f64) -> Biome {
    use Biome::*;
    let tier = if diff < 0.34 { 0 } else if diff < 0.67 { 1 } else { 2 };
    let band = if temp < 0.40 { 0 } else if temp < 0.66 { 1 } else { 2 };
    match (tier, band) {
        (0, _) => EasyRainforest,
        (1, 0) => Meadow, (1, 1) => Rainforest, (1, _) => Shrubland,
        (2, 0) => Snowy,  (2, 1) => DarkForest, (2, _) => Desert,
        _      => Rainforest,
    }
}

fn assign_biomes(mut commands: Commands) {
    let seed = 8472u32;
    let temp = Fbm::<Perlin>::new(seed).set_octaves(4).set_frequency(0.03);
    let wn   = Perlin::new(seed + 1);
    let spawn = Vec2::new(W as f32/2.0, H as f32);
    let max_d = spawn.length();

    let mut cells = Vec::with_capacity((W*H) as usize);
    for y in 0..H { for x in 0..W {
        let p  = IVec2::new(x, y);
        let wp = warp(p.as_vec2(), &wn, 18.0);
        let t  = temp.get([wp.x as f64*0.03, wp.y as f64*0.03]) * 0.5 + 0.5;
        let d  = (p.as_vec2().distance(spawn) / max_d).clamp(0.0, 1.0);
        cells.push(biome_for(d, t));
    }}
    commands.insert_resource(BiomeMap { width: W, height: H, cells });
}

fn paint_tiles(mut commands: Commands, biomes: Res<BiomeMap>) {
    for y in 0..biomes.height { for x in 0..biomes.width {
        commands.spawn((
            Sprite::from_color(ground(biomes.get(x, y)), Vec2::splat(TILE)),
            Transform::from_xyz(x as f32*TILE, y as f32*TILE, 0.0),
            MapTile,
        ));
    }}
}

fn setup_camera(mut commands: Commands) {
    commands.spawn((Camera2d, Transform::from_xyz(W as f32*TILE/2.0, H as f32*TILE/2.0, 0.0)));
}

fn main() {
    App::new()
        .add_plugins(DefaultPlugins)
        .add_systems(Startup, (setup_camera, assign_biomes, paint_tiles).chain())
        .run();
}
Skálázás

Csempénként egy entitás 96×64-ig kiváló tanulni, de egy igazi nyílt világhoz bevy_ecs_tilemap (chunkolt mesh) kell — és a BiomeMap-et chunkokra bontva, a játékos körül be/kitöltve generálod. Az adatmodell ettől nem változik, csak a megjelenítő réteg.

Gyakorlati tippek & buktatók

Tedd

Tartsd a biomot külön rétegben (BiomeMap), ne olvaszd a Tile-ba. A Tile a járhatóság, a Biome az identitás — a render a kettő kombinációjából választ.

Ne

Ne döntsd el a biomot a magasságból egyedül. Egy tengely csak monoton sávokat ad; a klíma-mátrix (hőmérséklet × nedvesség) kell a valódi változatossághoz.

Tedd

Válaszd szét a kontrollált (nehézség) és a zajos (klíma) tengelyt. A progresszió és a változatosság így egymástól függetlenül hangolható — ez a radiális elrendezés titka.

Ne

Ne hagyd egyenesen a határokat. Egy pozícióból vett küszöb függőleges csíkokat ad; egy domain warp réteg organikus peremmé töri.

Tedd

Kösd a jutalmat a nehézséghez a spawn-táblákban: nehezebb biom = több/erősebb lény + ritkább/jobb nyersanyag. Ez teszi a felfedezést kockázat–jutalom döntéssé.

Vigyázz

A determinizmus a seedből és a hívási sorrendből jön. A klíma, a warp és a spawn RNG mind a világ-seedből származzon; a passzok sorrendje legyen stabil. Mentsd a seedet, ne a kész térképet.

Kezdő-zóna

A spawn körüli gyűrűben hagyd a klímát „elnyomva" — mindig a könnyű biom jöjjön, függetlenül a hőmérséklettől. A játékos első percei legyenek biztonságosak és egységesek, bármerre indul.

TL;DR

A lényeg egy kártyán

  • A biom = réteg a terep fölött: a nyers zaj terep, a biom adja az identitást, a nehézséget és a tartalmat.
  • Három kérdés: mező (mi alapján döntünk?), identitás (címkéből kép?), progresszió (hogyan rendezzük el?).
  • Biom mint adat: sima enum + BiomeDef tábla; külön BiomeMap a tile-Map testvéreként.
  • Egy tengely kevés: a magasság monoton sávokat ad; a klíma-mátrix (hőmérséklet × nedvesség) kell a változatossághoz.
  • Klíma-mátrix: két független zajmező párosa egy 3×3 táblát indexel — tundrától a dzsungelig.
  • Domain warping: a mintavételi pontot egy másik zajjal told el → egyenes helyett organikus határ.
  • Progresszió: lineáris réteg = unalmas folyosó; radiális (spawntól mért távolság) = 2D felfedezés tiszta ívvel.
  • A 7 biom: nehézség-gyűrű × hőmérséklet-sáv lookup — kontrollált tengely metszi a zajosat.
  • Identitás = adat: a render egyirányú levezetés a biome_def-ből; az egész világ átöltöztethető egy tábla cseréjével.
  • Tartalom: bionkénti súlyozott spawn-tábla; nehézség ↔ jutalom; szórás-passz a kész BiomeMap-re.
  • Átmenetek: blend-sáv (szín-lerp) + szegély-autotiling a szomszéd-biom maszkjából. Részletek: bitmaszkolás, dual-grid.
  • Bevy: OnEnter(Generating) lánc — adat → kép → tartalom; egy seed, stabil sorrend.
