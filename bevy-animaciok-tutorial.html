Animációk Bevyben — állapotgép, átvezetések, minden réteg

Hogyan vált egy karakter tisztán futásból ugrásba, ugrásból esésbe, esésből landolásba? Hogyan animálsz sok NPC-t, ezer csempét és a UI-t egy átlátható rendszerrel? A kulcs: a klip (adat) + a közös lejátszó + a vezérlő (ami eldönti, mi szóljon) szétválasztása — rá egy állapotgép az átvezetésekhez. Végigvesszük mind a négy réteget, élő demókkal.

↑ Élő minta: a karakter magától körbejárja az állapotait — idle → fut → ugrik → esik → land → támad —, és minden váltás egy átvezetés. Lent a jelenlegi állapot neve.
█ I. rész — Alapok & adatmodell
1. fejezet

Az animáció anatómiája

Egy 2D sprite-animáció Bevyben három, jól szétválasztható dolog. Ez a szétválasztás az egész tutorial gerince — ha ezt elrontod, az animációs kód összegabalyodik a játéklogikával.

⊞ A három réteg
  1. KLIP (adat)      „mely képkockák, milyen tempóban, hogyan ismétlődik”
                       — AnimClip { first, last, fps, mode }

  2. LEJÁTSZÓ (közös)  lépteti a képkockát az idő szerint, beállítja az atlas-indexet
                       — egyetlen rendszer MINDEN sprite-animációhoz

  3. VEZÉRLŐ (dönt)    „MELYIK klip szóljon most?” — bemenet/fizika/AI alapján
                       — entitás-típusonként más (player input · NPC AI)
A lejátszó nem tudja, miért fut a karakter — csak lépteti a kockákat. A vezérlő nem tudja, hogyan rajzolódik — csak kiválasztja a klipet. A kettő közt a klip az interfész.

A nyers frame-léptetést (atlas-index + Timer) a sprite-animáció vs transzformáció tutorialban már láttad — most ezt rendszerré emeljük: adatmodell, közös lejátszó, állapotgép, átvezetések. A végén egy AnimationPlugin fogja össze.

▲ A legfontosabb elv
Válaszd szét a „mi szóljon”-t a „hogyan játszódik le”-től. A vezérlő csak állapotot/klipet állít; a lejátszó csak léptet. Így ugyanaz a lejátszó hajtja a karaktert, az NPC-t és a tile-t — és a UI-t külön, tween-nel kezeljük (11. fejezet).
2. fejezet

Az animációs klip

A klip a legkisebb adat-egység: melyik képkockák tartoznak hozzá (first..=last az atlaszban), milyen fps-sel pörögnek, és hogyan ismétlődik. Az ismétlés-mód három tipikus eset:

  • Loop — körbeér, újrakezd (járás, idle, víz-csillámlás).
  • Once — egyszer lejátszódik, majd megáll a végén (támadás, robbanás). A „vége” jelzi az átvezetést.
  • PingPong — oda-vissza (lebegés, lélegzés, néhány idle).

Próbáld ki a három módot, és állítsd az fps-t — figyeld a lejátszófejet a képkocka-csíkon:

▶ JÁTSZÓTÉR — klip lejátszó & ismétlés-módok
ismétlés-mód
fps8

frame 0 / 5

A felső sor a klip képkockái (atlasz), a keret a lejátszófej. Az Once a végén megáll — ezt használjuk az átvezetés kioldására.

src/anim/clip.rs — a klip-adatmodell + könyvtár
use bevy::prelude::*;
use std::collections::HashMap;

#[derive(Clone, Copy, PartialEq, Eq)]
pub enum LoopMode { Loop, Once, PingPong }

/// Egy animációs klip: mely képkockák, milyen tempóban, hogyan ismétlődik.
#[derive(Clone, Copy)]
pub struct AnimClip {
    pub first: usize,
    pub last: usize,
    pub fps: f32,
    pub mode: LoopMode,
}

/// A klip-könyvtár: névvel elérhető klipek (egy helyen definiálva).
#[derive(Resource, Default)]
pub struct AnimLibrary {
    pub clips: HashMap<&'static str, AnimClip>,
}

fn build_clip_library(mut lib: ResMut<AnimLibrary>) {
    let mut add = |name, first, last, fps, mode| {
        lib.clips.insert(name, AnimClip { first, last, fps, mode });
    };
    add("hero_idle",   0, 3, 6.0,  LoopMode::Loop);
    add("hero_run",    8, 13, 12.0, LoopMode::Loop);
    add("hero_jump",   16, 17, 10.0, LoopMode::Once);
    add("hero_fall",   18, 19, 10.0, LoopMode::Loop);
    add("hero_attack", 24, 28, 16.0, LoopMode::Once);
    // ... a tile/NPC klipek is ide kerülnek
}
3. fejezet

A lejátszó rendszer

A lejátszó egyetlen komponens (SpriteAnimator) + egyetlen rendszer (advance_animation). Ez minden sprite-animációt hajt — karaktert, NPC-t, tile-t. Nem tudja, miért fut épp ez a klip; csak lépteti a kockát, és kezeli a három ismétlés-módot.

src/anim/player.rs — a közös lejátszó
use bevy::prelude::*;
use std::time::Duration;
use super::clip::{AnimClip, LoopMode};

#[derive(Component)]
pub struct SpriteAnimator {
    pub clip: AnimClip,
    pub frame: usize,
    pub timer: Timer,
    pub dir: i32,        // PingPong irány
    pub finished: bool,  // Once végén true → átvezetés kioldása
}

impl SpriteAnimator {
    pub fn new(clip: AnimClip) -> Self {
        Self {
            frame: clip.first, dir: 1, finished: false, clip,
            timer: Timer::from_seconds(1.0 / clip.fps, TimerMode::Repeating),
        }
    }

    /// Klip-váltás — de CSAK ha tényleg más; különben ne reseteljük a fázist!
    pub fn play(&mut self, clip: AnimClip) {
        if clip.first == self.clip.first && clip.last == self.clip.last { return; }
        self.clip = clip;
        self.frame = clip.first;
        self.dir = 1;
        self.finished = false;
        self.timer.set_duration(Duration::from_secs_f32(1.0 / clip.fps));
        self.timer.reset();
    }
}

fn advance_animation(time: Res<Time>, mut q: Query<(&mut SpriteAnimator, &mut Sprite)>) {
    for (mut anim, mut sprite) in &mut q {
        anim.timer.tick(time.delta());
        if !anim.timer.just_finished() { continue; }
        let (first, last) = (anim.clip.first, anim.clip.last);
        match anim.clip.mode {
            LoopMode::Loop => anim.frame = if anim.frame >= last { first } else { anim.frame + 1 },
            LoopMode::Once => if anim.frame < last { anim.frame += 1; } else { anim.finished = true; },
            LoopMode::PingPong => {
                if anim.frame == last { anim.dir = -1; } else if anim.frame == first { anim.dir = 1; }
                anim.frame = (anim.frame as i32 + anim.dir) as usize;
            }
        }
        if let Some(atlas) = &mut sprite.texture_atlas {
            atlas.index = anim.frame;
        }
    }
}
⚠ A leggyakoribb animációs bug
A play() elején az „ugyanaz a klip? → return” őr a lényeg. Ha minden frame-ben meghívnád play("hero_run")-t, a frame örökké 0-on ragadna — a futás „beragad" az első kockára. Csak állapot-váltáskor váltsunk klipet (lásd a Changed szűrőt a következő fejezetben).

Az állapotgép

Egy karakter nem „valamelyik klipet" játssza — egy állapotban van (Idle, Run, Jump, Fall, Attack, Hurt), és az állapothoz tartozik egy klip. A mozgás lényege az átvezetés: az állapotok közti váltás. Ezt egy egyszerű enum + a hozzá kötött szabályok modellezik.

src/anim/character.rs — az állapot
use bevy::prelude::*;

#[derive(Component, Clone, Copy, PartialEq, Eq, Debug)]
pub enum CharState { Idle, Run, Jump, Fall, Attack, Hurt }

#[derive(Component)] pub struct Player;
#[derive(Component)] pub struct Velocity(pub Vec2);
#[derive(Component)] pub struct Grounded(pub bool);
⊞ Az állapotok & tipikus átvezetések
            ┌──────────────────────── Attack ◀── (J gomb, bármikor)
            │                            │ (klip vége)
            ▼                            ▼
   Idle ⇄ Run ───(elrugaszkodás)──► Jump ──(v.y<0)──► Fall ──(talaj)──► Idle/Run
     ▲                                                                    │
     └────────────────────────── Hurt ◀── (sebzés) ───────────────┘
Idle↔Run a vízszintes sebességtől; a Jump/Fall a levegőben-léttől; az Attack/Hurt prioritásos állapot, ami felülír — és a klip végén visszatér. A 6. fejezet demója interaktívan végigjárható.
5. fejezet

A vezérlő: állapot → klip

A vezérlő a fizikából/inputból kiszámolja az állapotot — nem nyúl a kliphez. Próbáld ki: mozgasd a karaktert, és nézd, hogy a state magától vált (és vele a klip), átvezetésekkel:

▶ JÁTSZÓTÉR — játszható karakter (state a fizikából)

állapot: Idle

átvezetés-napló

Kattints a vászonra a ← → szóköz J billentyűkhöz is. A state-et a sebesség + talajon-lét adja — a klip követi.

src/anim/character.rs — a vezérlő (állapot a fizikából)
fn drive_character_state(
    mut q: Query<(&Velocity, &Grounded, &mut CharState), With<Player>>,
) {
    for (vel, grounded, mut state) in &mut q {
        // a prioritásos állapotokat (Attack/Hurt) NEM írjuk felül itt:
        if matches!(*state, CharState::Attack | CharState::Hurt) { continue; }

        let next = if !grounded.0 {
            if vel.0.y > 0.0 { CharState::Jump } else { CharState::Fall }
        } else if vel.0.x.abs() > 10.0 {
            CharState::Run
        } else {
            CharState::Idle
        };

        if *state != next { *state = next; }   // csak tényleges váltáskor írunk
    }
}
⚙ Bevy fogalom — a vezérlő csak állapotot ír

Figyeld meg: a drive_character_state nem ismeri a SpriteAnimator-t, az atlaszt vagy a képkockákat. Csak a CharState-et állítja. Így a játéklogika (fizika → állapot) teljesen független az animáció lejátszásától — más rendszer (NPC-AI) ugyanezt a CharState-et más bemenetből számolja.

6. fejezet

Átvezetések

Az átvezetés három dolgot jelent: (1) állapot-váltáskor a megfelelő klipre váltunk (és csak akkor!), (2) az egyszer-lejátszós (Once) állapotok a klip végén visszatérnek, (3) a prioritás eldönti, mi írhat felül mit. Kattints a csúcsokra — figyeld a megengedett átmeneteket és az auto-visszatérést:

▶ JÁTSZÓTÉR — átvezetés-gráf (prioritás & auto-return)

Kattints egy állapotra a váltáshoz. Az Attack/Hurt prioritásos: bárhonnan elérhető, és a klip végén magától visszatér.

— válassz egy állapotot —

A kötés (állapot → klip) egyetlen rendszer, és a Changed<CharState> szűrő miatt csak váltáskor fut — pont ez akadályozza meg a „beragadt első kocka" bugot:

src/anim/character.rs — kötés + auto-return
use super::clip::AnimLibrary;
use super::player::SpriteAnimator;

/// Állapot → klip. A `Changed` szűrő miatt CSAK állapot-váltáskor fut.
fn apply_state_clip(
    lib: Res<AnimLibrary>,
    mut q: Query<(&CharState, &mut SpriteAnimator), Changed<CharState>>,
) {
    for (state, mut anim) in &mut q {
        let name = match state {
            CharState::Idle => "hero_idle",
            CharState::Run => "hero_run",
            CharState::Jump => "hero_jump",
            CharState::Fall => "hero_fall",
            CharState::Attack => "hero_attack",
            CharState::Hurt => "hero_hurt",
        };
        anim.play(lib.clips[name]);     // a play() őre miatt is biztonságos
    }
}

/// Once-klip vége → a prioritásos állapot visszatér (a driver innen újraszámol).
fn finish_oneshot(mut q: Query<(&mut CharState, &SpriteAnimator)>) {
    for (mut state, anim) in &mut q {
        if matches!(*state, CharState::Attack | CharState::Hurt) && anim.finished {
            *state = CharState::Idle;   // a driver a következő frame-en korrigál
        }
    }
}

/// Prioritásos belépés (input/observer): felülírja az aktuális állapotot.
fn start_attack(keys: Res<ButtonInput<KeyCode>>, mut q: Query<&mut CharState, With<Player>>) {
    if keys.just_pressed(KeyCode::KeyJ) {
        for mut s in &mut q { *s = CharState::Attack; }
    }
}
⚙ Bevy fogalom — Changed<T> szűrő

A Query<.., Changed<CharState>> csak azokat az entitásokat adja, amelyeknek a CharState-je az utolsó futás óta változott. Így a klip-váltás reagáló, nem pollozó: nem minden frame-ben fut, csak a tényleges átvezetéskor — pont ettől nem reseteli a fázist fölöslegesen.

7. fejezet

Lockolás & átvezető klipek

Két finomítás, amitől az átvezetések profivá válnak:

Lockolás — egy klip „kisajátítja" a karaktert

Támadás közben a játékos ne tudjon csak úgy futásba váltani. A megoldás: a vezérlő hagyja békén a prioritásos állapotot (ezt már láttad a matches!(... Attack | Hurt) { continue } sorral). A lock = „amíg a Once-klip le nem játszódott, a driver nem nyúl az állapothoz". A mozgás-bemenetet pufferelheted, és a lock feloldásakor alkalmazod.

Átvezető (in-between) klipek

A „kemény" váltás (Idle→Jump azonnal) néha rángat. Tehetsz közé egy rövid átvezető klipet: pl. jump_takeoff (elrugaszkodás) az Idle és a Jump közé, vagy land a Fall és az Idle közé. Ez egy Once-klip, ami a végén a cél-állapotba lép — pont mint az Attack.

⊞ Átvezető klipek a kemény váltások helyett
   kemény:   Idle ─────────────────► Jump            (rángat)
   sima:     Idle ─► Takeoff ─(vége)─► Jump      (elrugaszkodás-póz)

   kemény:   Fall ─────────────────► Idle
   sima:     Fall ─► Land ─(vége)─► Idle         (becsapódás-póz, kis squash)
A Takeoff/Land ugyanaz a minta, mint az Attack: Once-klip, ami a finished jelre a cél-állapotba lép. A 2D-ben nincs valódi blend (a kockák diszkrétek) — az átvezető klip helyettesíti a crossfade-et.
▲ 2D ≠ 3D blend
Csontvázas 3D-ben az átmenet blend (két póz keverése, AnimationGraph). 2D sprite-nál a kockák diszkrétek, nem keverhetők — helyette átvezető klipet teszel be, vagy a frame-anim fölé transzformációt raksz (squash, dőlés) a folytonosság érzetéért.
8. fejezet

Irány & juice-réteg

Két dolog, ami a frame-animáció fölé kerül, és nem igényel új kockát:

Irány — tükrözés, nem külön klip

Balra nézéshez ne rajzolj külön klipet: tükrözd a sprite-ot. A Sprite.flip_x a sebesség előjeléből.

Juice — transzformáció a frame-anim tetején

Landoláskor egy kis scale-squash, találatkor fehér tint, ugráskor enyhe dőlés — ezek transzformációk a meglévő klip fölött (lásd a transzformációs tutorialt). Együtt élővé teszik a diszkrét kockákat.

src/anim/character.rs — irány + landoló squash
/// Irány: a vízszintes sebesség előjele tükrözi a sprite-ot.
fn face_direction(mut q: Query<(&Velocity, &mut Sprite)>) {
    for (vel, mut sprite) in &mut q {
        if vel.0.x.abs() > 1.0 {
            sprite.flip_x = vel.0.x < 0.0;
        }
    }
}

/// Juice: a frame-anim FÖLÉ tett transzformáció (itt: landoló squash).
/// A landolás-eseményre indul egy rövid, csillapodó scale-rezgés.
fn land_squash(time: Res<Time>, mut q: Query<(&mut Transform, &mut LandSquash)>) {
    for (mut tf, mut s) in &mut q {
        s.timer.tick(time.delta());
        let k = (1.0 - s.timer.fraction()) * (s.timer.fraction() * 22.0).sin() * 0.3;
        tf.scale = Vec3::new(1.0 + k, 1.0 - k, 1.0);   // térfogat-tartó
    }
}

#[derive(Component)] struct LandSquash { timer: Timer }
▲ Rétegek, nem helyettesítés
A flip_x és a scale-juice nem avatkozik a klip-lejátszásba — a SpriteAnimator továbbra is az indexet lépteti, a Sprite/Transform mezők meg külön réteget tesznek rá. Tiszta felelősség-megosztás.

NPC-animáció

A jó hír: az NPC-hez semmi újat nem kell. Ugyanaz a CharState, ugyanaz a SpriteAnimator, ugyanaz az apply_state_clip és advance_animation — egyetlen dolog más: a vezérlő. A karaktert input hajtja, az NPC-t az AI. Ennyi.

src/anim/npc.rs — ugyanaz a gép, más vezérlő
use bevy::prelude::*;
use super::character::{CharState, Velocity};

#[derive(Component)] pub struct Npc;

/// Az NPC ugyanazt a CharState-et használja — csak a bemenet AI, nem input.
fn drive_npc_state(mut q: Query<(&Velocity, &mut CharState), With<Npc>>) {
    for (vel, mut state) in &mut q {
        if matches!(*state, CharState::Attack | CharState::Hurt) { continue; }
        let next = if vel.0.x.abs() > 5.0 { CharState::Run } else { CharState::Idle };
        if *state != next { *state = next; }
    }
}

Sok NPC-nél egy gond van: ha mind egyszerre spawnol, az idle-animációjuk szinkronban jár — gépiesen lüktet az egész tömeg. A trükk: spawnkor adj minden NPC-nek véletlen fázis-eltolást. Kapcsold át, és nézd a különbséget:

▶ JÁTSZÓTÉR — NPC-tömeg & fázis-eltolás

változatos (élethű)

Eltolás nélkül az egész tömeg egyszerre pattan — gépies. Egy kis véletlen kezdő-fázis, és máris él a csapat.

src/anim/npc.rs — fázis-eltolás spawnkor
use std::time::Duration;
use super::clip::AnimLibrary;
use super::player::SpriteAnimator;

fn spawn_npcs(mut commands: Commands, lib: Res<AnimLibrary>) {
    for i in 0..20 {
        let mut anim = SpriteAnimator::new(lib.clips["slime_idle"]);
        // KULCS: véletlen kezdő-fázis, hogy ne szinkronban lüktessenek
        let offset = (i as f32 * 0.137).fract() * 0.6;       // 0..0.6 s, szórt
        anim.timer.set_elapsed(Duration::from_secs_f32(offset));

        commands.spawn((
            Npc, CharState::Idle, Velocity(Vec2::ZERO), anim,
            // Sprite::from_atlas_image(...),
            Transform::from_xyz(-200.0 + i as f32 * 22.0, 0.0, 0.0),
        ));
    }
}
10. fejezet

Tile-animáció

Animált csempék (víz, láva, fáklya, virág) ezrével lehetnek a pályán. Ha mindegyikre teszel egy Timer-t, az pazarlás. A bevált minta: egyetlen globális óra, és a képkockát számítjuk belőle — nincs per-tile állapot, nincs ezernyi Timer.

A demó egy csempe-rács, mind egy órából animálva. Kapcsold a fázis-eltolást: nélküle minden víz/fáklya egyszerre lép (rács-hatás), vele természetes:

▶ JÁTSZÓTÉR — ezer csempe, egy óra

offset-tel: hullámzik

Víz (csillámlás), fáklya (lobogás), virág (ringás) — mind a (elapsed · fps) % n képletből, egyetlen rendszerből.

src/anim/tiles.rs — egy óra, akárhány csempe
use bevy::prelude::*;

#[derive(Component)]
pub struct AnimatedTile {
    pub frames: &'static [usize],
    pub fps: f32,
    pub offset: f32,        // fázis-eltolás a változatosságért
}

/// EGY rendszer, EGY óra, akárhány csempe — NINCS per-tile Timer.
fn animate_tiles(time: Res<Time>, mut q: Query<(&AnimatedTile, &mut Sprite)>) {
    let now = time.elapsed_secs();
    for (tile, mut sprite) in &mut q {
        let n = tile.frames.len();
        // a képkockát SZÁMÍTJUK az időből, nem léptetjük:
        let i = (((now + tile.offset) * tile.fps) as usize) % n;
        if let Some(atlas) = &mut sprite.texture_atlas {
            atlas.index = tile.frames[i];
        }
    }
}
⚙ Bevy / crate — bevy_ecs_tilemap AnimatedTile

Ha a bevy_ecs_tilemap crate-et használod, van beépített AnimatedTile { start, end, speed } komponense, ami pont ezt csinálja a chunk-renderelőbe építve (GPU-barát, sok ezer csempére). A saját kézi verzió jó néhány száz csempéig és tanulásra; nagy tilemaphoz a crate skálázódik.

UI-animáció

A UI más műfaj: itt nincsenek képkockák. Egy gomb-pop, egy panel-becsúszás, egy élet-csík — ezek egy tulajdonság (méret, pozíció, szín, scale) interpolációi idő szerint. És itt lép be az easings.net: az easing az, ami a mozgást nem gépiessé teszi.

◆ Az easing lényege (easings.net)

Az easing egy függvény, ami egy normalizált időt alakít át görbített haladássá:

t:         0.0 ───────────────► 1.0
easing(t): 0.0 ──görbe szerint──► 1.0

Vagyis nem közvetlenül „animációt" kapsz, hanem egy értéket, amivel interpolálsz:

current = start + (target - start) * easing(t)

Az easing függvény az animáció sebességváltozását írja le — hogy ne lineárisan, gépiesen induljon és álljon meg a mozgás. Bevyben ezt az EaseFunction + EasingCurve adja (ami pont a fenti képletet számolja).

Válassz easinget, és nézd, ahogy ugyanaz a UI-elem máshogy „érződik" — pop túllövéssel (BackOut), rugósan (ElasticOut), pattanva (BounceOut):

A gomb-pop, a panel-becsúszás, a toast — mind current = start + (target−start)·easing(t). Az élet-csík folytonos lerp-pel közelít.

src/ui/anim.rs — tween easinggel + élet-csík lerp
use bevy::prelude::*;

/// Általános UI-pop: a Transform.scale-t interpolálja 0→1-ig, easinggel.
/// (UI-node Transformján a scale és rotation érvényesül; a translation-t a layout adja.)
#[derive(Component)]
struct UiPop { timer: Timer, ease: EaseFunction }

fn animate_ui_pop(time: Res<Time>, mut q: Query<(&mut Transform, &mut UiPop)>) {
    for (mut tf, mut pop) in &mut q {
        pop.timer.tick(time.delta());
        let t = pop.timer.fraction();                 // normalizált idő 0→1
        // easings.net-képlet Bevyben: EasingCurve = start→target az ease szerint.
        // belül pontosan ezt számolja: start + (target - start) * ease(t)
        let scale = EasingCurve::new(0.0, 1.0, pop.ease).sample_clamped(t);
        tf.scale = Vec3::splat(scale);
    }
}

/// Gomb hover/press: cél-scale az interakcióból (élesben tweenelve, ne ugorva).
fn button_feedback(
    mut q: Query<(&Interaction, &mut Transform), (Changed<Interaction>, With<Button>)>,
) {
    for (interaction, mut tf) in &mut q {
        let target = match interaction {
            Interaction::Hovered => 1.08,
            Interaction::Pressed => 0.94,
            Interaction::None => 1.0,
        };
        tf.scale = Vec3::splat(target);
    }
}

/// Élet-csík: a Node.width-et FOLYTONOSAN közelíti a cél felé (exponenciális lerp).
fn animate_health_bar(
    time: Res<Time>,
    health: Res<PlayerHealth>,
    mut q: Query<&mut Node, With<HealthBarFill>>,
) {
    for mut node in &mut q {
        let target = health.fraction() * 100.0;
        if let Val::Percent(w) = node.width {
            let nw = w + (target - w) * (8.0 * time.delta_secs()).min(1.0);
            node.width = Val::Percent(nw);          // simán „lecsúszik" a sebzésnél
        }
    }
}
▲ Fix-idejű tween vs folytonos lerp
Két mintát látsz: a fix-idejű tween (Timer 0→1 + easing) a megjelenésekre/pop-okra; a folytonos lerp (minden frame közelítünk a cél felé) az élet-csíkra, kamerára, követésre — ahol a cél menet közben is változhat. Mindkettő ugyanazt a lerp-magot használja, csak más a „t".

Animációs események

Sokszor egy konkrét képkockán kell történnie valaminek: a 3. futó-kockán lépészaj, a támadás 2–4. kockáján aktív a hitbox. Ezt úgy oldjuk meg, hogy figyeljük, mikor lép be az animáció egy megjelölt képkockába, és olyankor eseményt küldünk.

src/anim/events.rs — képkocka-eseményt küld
use bevy::prelude::*;
use super::player::SpriteAnimator;

#[derive(Event)] struct FootstepEvent(Entity);
#[derive(Event)] struct HitboxActive(Entity);

#[derive(Component)]
struct FrameTriggers {
    footstep_frames: Vec<usize>,
    hitbox_frames: Vec<usize>,
    last_frame: usize,
}

/// Eseményt küld, amikor az animáció BELÉP egy megjelölt képkockába.
fn emit_frame_events(
    mut q: Query<(Entity, &SpriteAnimator, &mut FrameTriggers)>,
    mut footsteps: EventWriter<FootstepEvent>,
    mut hits: EventWriter<HitboxActive>,
) {
    for (e, anim, mut tr) in &mut q {
        if anim.frame == tr.last_frame { continue; }   // csak képkocka-VÁLTÁSKOR
        tr.last_frame = anim.frame;
        if tr.footstep_frames.contains(&anim.frame) { footsteps.write(FootstepEvent(e)); }
        if tr.hitbox_frames.contains(&anim.frame)   { hits.write(HitboxActive(e)); }
    }
}
▲ A „belépés" a lényeg, nem a „benne van"
A last_frame eltárolásával csak akkor sül el az esemény, amikor a kocka megváltozik az adott indexre — nem minden frame-ben, amíg azon a kockán áll. Így egy lépészaj egyszer szól lépésenként, nem 8×.

Az AnimationPlugin

Most összeáll a kép. Egyetlen plugin regisztrálja a klip-könyvtárat és a rendszereket — határozott sorrendben: előbb a vezérlők döntenek, aztán a kötés választ klipet, végül a közös lejátszó léptet. A .chain() garantálja a sorrendet.

src/anim/mod.rs — az AnimationPlugin
use bevy::prelude::*;

pub struct AnimationPlugin;

impl Plugin for AnimationPlugin {
    fn build(&self, app: &mut App) {
        app.init_resource::<AnimLibrary>()
            .add_event::<FootstepEvent>()
            .add_event::<HitboxActive>()
            .add_systems(Startup, build_clip_library)
            .add_systems(Update, (
                // 1) VEZÉRLŐK — eldöntik az állapotot (entitás-típusonként):
                (drive_character_state, drive_npc_state, start_attack),
                // 2) KÖTÉS — állapot → klip (Changed), + Once-auto-return:
                (apply_state_clip, finish_oneshot),
                // 3) LEJÁTSZÁS — közös, MINDENKIRE:
                (advance_animation, animate_tiles),
                // 4) ESEMÉNYEK + juice-réteg:
                (emit_frame_events, face_direction, land_squash),
            ).chain());
    }
}

Teljesítmény & merre tovább?

Buktatók (a tutorialból)

  • Ne reseteld a fázist minden frame-ben. A play() „ugyanaz a klip? → return" őre + a Changed<CharState> szűrő együtt biztosítja, hogy a klip csak váltáskor váltson.
  • Tile: egy óra, ne ezer Timer. A képkockát számítsd az időből ((elapsed·fps) % n); fázis-offsettel adj változatosságot.
  • Tömeg: fázis-eltolás. Sok NPC-nek véletlen kezdő-fázis, különben gépiesen szinkronban lüktet.
  • Takarítás. Az animált entitásokat DespawnOnExit(State)-tel vagy markerrel takarítsd jelenet-váltáskor (lásd a scene-váltás tutorialt).

Merre tovább?

  • Easing katalógus — easings.net. A teljes görbe-készlet vizuálisan; mindegyikhez van EaseFunction párja Bevyben (EasingCurve-on mintázva).
  • bevy_tweening. Láncolt / ismétlő / oda-vissza tweenekhez (UI és transzformáció) — megspórolja a kézi Timer-t.
  • bevy_ecs_tilemap AnimatedTile. Nagy, GPU-gyorsított animált tilemaphoz.
  • Aseprite-import. A bevy_aseprite_ultra-féle crate-ek beolvassák a tagelt animációkat és layoutot — a klip-könyvtárad az eszközből jön.
  • 3D ≠ 2D. Csontvázas 3D-ben az AnimationGraph + AnimationPlayer valódi blendet ad (pózok keverése). A 2D sprite itteni mintája diszkrét kockás — más állat, de a vezérlő/állapotgép réteg ugyanúgy érvényes.
⚙ Az igazi tanulság

Válaszd szét a három réteget: a klip az adat, a lejátszó közös, a vezérlő dönt. Egyetlen gép hajtja a karaktert, az NPC-t és a tile-t — csak a vezérlő más; a UI pedig easing-tween (current = start + (target−start)·easing(t)), nem képkocka. Az átvezetés egy állapotgép, ahol a Once-klipek a végükön visszatérnek és a prioritás dönti el, mi írhat felül mit. Ha ez a négy a helyén van, a futás→ugrás→esés→land tisztán, rángás nélkül vált.
