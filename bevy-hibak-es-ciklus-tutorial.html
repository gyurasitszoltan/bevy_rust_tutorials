A fordító elengedett, a játék mégis elszáll induláskor — vagy némán nem csinál semmit? Itt a Bevy-specifikus hibák referenciája, és az eszköztár, amivel naponta dolgozol.

0x00

A határ: fordító vs. futásidő

A fordítóhibák modulban láttad, hogy a Rust-fordító a fordítás idején elkapja a kölcsönzési hibákat. A Bevy ECS-ében viszont a rendszerek párhuzamosan futnak, és hogy ezt biztonságosan megtehesse, a borrow checker néhány garanciáját az ütemező futásidőre csúsztatja. Ezért van egy második hibaosztály, amit a fordító már nem lát — ezt kapod meg B0001-féle kódokkal, többnyire rögtön induláskor.

Hogyan néz ki egy Bevy-hiba?

Egy közönséges Rust-pánik, kiegészítve egy B#### kóddal és egy linkkel a bevy.org/learn/errors oldalra. A kód a kulcs: minden B####-höz tartozik egy magyarázó oldal. A RUST_BACKTRACE=1 itt is ad hívási láncot, az ECS-belső eseményekhez pedig: RUST_LOG=bevy_ecs=debug.

Az I. rész a hat hivatalos hibakódot veszi sorra (a leggyakoribbakat mélyebben), a II. rész pedig a napi fejlesztői ciklust gyorsítja fel.

0x01

B0001: query-konfliktus

Ez a leggyakoribb Bevy-hiba — és pontosan a Rust E0499 (két módosító kölcsön) futásidejű megfelelője. Akkor jön, ha egy rendszer két lekérdezése úgy fér hozzá ugyanahhoz a komponenshez, hogy az ütemező nem tudja garantálni: nem ugyanazt az entitást módosítják.

src/main.rs
// Mindkét lekérdezés &mut Transform-ot kér — a Bevy óvatos: nem bizonyított,
// hogy a két halmaz diszjunkt, ezért elutasítja.
fn rossz_rendszer(
    jatekosok:  Query<&mut Transform, With<Jatekos>>,
    ellensegek: Query<&mut Transform, With<Ellenseg>>,
) {
    // ...
}
thread 'main' panicked at bevy_ecs/src/system/...: error[B0001]: Query<&mut Transform, With<Jatekos>> in system rossz_rendszer accesses component(s) Transform in a way that conflicts with a previous system parameter. Consider using `Without<T>` to create disjoint Queries or merging conflicting Queries into a `ParamSet`. See: https://bevy.org/learn/errors/b0001

Javítás 1 — bizonyítsd a diszjunktságot Without-tal

src/main.rs
fn jo_rendszer(
    mut jatekosok:  Query<&mut Transform, (With<Jatekos>,  Without<Ellenseg>)>,
    mut ellensegek: Query<&mut Transform, (With<Ellenseg>, Without<Jatekos>)>,
) {
    // a Without garantálja: egyetlen entitás sem eshet mindkét halmazba
}

Javítás 2 — ParamSet, ha tényleg átfedhetnek

src/main.rs
// A ParamSet megengedi az ütköző lekérdezéseket, de garantálja, hogy
// egyszerre csak az EGYIKET használod (p0(), p1(), ...).
fn jo_rendszer2(mut keszlet: ParamSet<(
    Query<&mut Transform, With<Jatekos>>,
    Query<&mut Transform, With<Ellenseg>>,
)>) {
    for mut t in keszlet.p0().iter_mut() { /* játékosok */ }
    for mut t in keszlet.p1().iter_mut() { /* ellenségek */ }
}
⚙ Gyakorlat 0x01

Ez a rendszer B0001-gyel elszáll induláskor. Javítsd ki Without-tal úgy, hogy mindkét halmaz módosítható maradjon.

hibás kód
fn mozgat(
    mut baratok:    Query<&mut Transform, With<Barat>>,
    mut ellenfelek: Query<&mut Transform, With<Ellenfel>>,
) { /* ... */ }
Megoldás
megoldás
fn mozgat(
    mut baratok:    Query<&mut Transform, (With<Barat>,    Without<Ellenfel>)>,
    mut ellenfelek: Query<&mut Transform, (With<Ellenfel>, Without<Barat>)>,
) { /* ... */ }
0x02

B0002: resource-konfliktus

Ugyanaz az elv, csak resource-okra: egy rendszer nem férhet hozzá ugyanahhoz a resource-hoz egyszerre Res-ként (olvasó) és ResMut-ként (módosító) — ez sértené a Rust kölcsönzési szabályait.

src/main.rs
#[derive(Resource)]
struct Pontszam(u32);

// Ugyanaz a resource Res-ként ÉS ResMut-ként — ütközik.
fn rossz(olvas: Res<Pontszam>, mut ir: ResMut<Pontszam>) {
    // ...
}
thread 'main' panicked at bevy_ecs/src/system/...: error[B0002]: ResMut<Pontszam> in system rossz conflicts with a previous Res<Pontszam> access. Remove the duplicate access. See: https://bevy.org/learn/errors/b0002

A javítás egyszerű: a ResMut-ból olvasni is lehet, így a külön Res fölösleges — töröld.

src/main.rs
fn jo(mut pont: ResMut<Pontszam>) {
    let jelenlegi = pont.0; // a ResMut olvasásra is jó
    pont.0 = jelenlegi + 10;
}
0x03

B0003: despawn-olt entitás

A Commands műveletei halasztottak: nem azonnal hajtódnak végre, hanem a következő szinkronizációs pontnál. Ezért megtörténhet, hogy az egyik rendszer despawnol egy entitást, miközben egy másik még műveletet sorol be ugyanarra — mire az lefutna, az entitás már nincs.

src/main.rs
// Az egyik rendszer eltávolítja a lejárt entitásokat...
fn takarit(mut commands: Commands, q: Query<Entity, With<Lejart>>) {
    for e in &q {
        commands.entity(e).despawn();
    }
}

// ...a másik (ugyanabban a frame-ben) még komponenst tenne rájuk:
fn frissit(mut commands: Commands, q: Query<Entity, With<Lejart>>) {
    for e in &q {
        commands.entity(e).insert(Pont(0)); // lehet, hogy már despawnolták
    }
}
warning[B0003]: Could not insert a bundle (of type `Pont`) for entity 42v1 because it doesn't exist in this World. note: configure RUST_LOG=bevy_ecs=debug to find the system that despawned it. See: https://bevy.org/learn/errors/b0003

A javítás: rendezd a rendszereket, hogy az entitást használó fusson le előbb, és csak utána a despawn — a .chain() determinisztikus sorrendet ad:

src/main.rs
app.add_systems(Update, (frissit, takarit).chain());
// frissit ELŐBB fut, takarit (despawn) UTÁNA — nincs verseny
Miért csak warning?

A Bevy a despawn-olt entitásra adott parancsot alapból csak figyelmezteti, nem ejti össze a játékot — a hibás parancs egyszerűen elvész. Ez kényelmes, de alattomos: ha „néha nem történik meg" valami, keress B0003 figyelmeztetést a logban.

0x04

B0004 és a néma bukások

B0004 — megszakadt hierarchia

Ha egy gyerek-entitás örökölhető komponenst hordoz (pl. InheritedVisibility), de a szülőjén nincs meg a megfelelő komponens (Visibility), az öröklési lánc megszakad — és a gyerek pl. láthatatlan marad.

src/main.rs
// A szülőn nincs Visibility, csak Transform:
let szulo = commands.spawn(Transform::default()).id();
commands.spawn((Sprite::from_image(kep), ChildOf(szulo))); // a sprite nem látszik
warning[B0004]: An entity with `InheritedVisibility` has an ancestor without `Visibility`, which breaks visibility inheritance. See: https://bevy.org/learn/errors/b0004

Javítás: adj a szülőnek Visibility-t — ez (mint „required component") magával hozza a hiányzó láthatósági komponenseket:

src/main.rs
let szulo = commands.spawn((Transform::default(), Visibility::default())).id();

A néma bukások — amik még csak nem is figyelmeztetnek

A legtöbb időt nem a pánikok viszik el, hanem a csendes „miért nem történik semmi" hibák. A három leggyakoribb:

1. A single() elbukik (0 vagy 2+ entitás). Bevy 0.18-ban Result-ot ad — kezeld, ne pánikolj rá:

src/main.rs
fn kamera_kovet(jatekos: Query<&Transform, With<Jatekos>>) {
    // ha 0 vagy 2+ játékos van, szépen kilépünk a frame-ből (nem panic):
    let Ok(t) = jatekos.single() else { return; };
    // ... t használata
}

2. A lekérdezés semmire sem illik. Ilyenkor a for törzse egyszerűen nem fut — a rendszer némán nem csinál semmit. Derítsd ki egy gyors számlálással:

src/main.rs
fn debug_rendszer(q: Query<Entity, With<Ellenseg>>) {
    info!("ellenségek száma: {}", q.iter().count()); // 0? akkor a spawn vagy a szűrő a baj
}

3. Az asset még nem töltött be. A Handle azonnal érvényes, de az adat csak néhány frame múlva érhető el — addig az Assets::get None-t ad:

src/main.rs
fn hasznal_kepet(kepek: Res<Assets<Image>>, handle: Res<AktualisKep>) {
    let Some(kep) = kepek.get(&handle.0) else { return; }; // még tölt → kilépünk
    // ... a kép használata
}
⚙ Gyakorlat 0x04

Ez a rendszer unwrap()-pal pánikol, ha nincs (vagy több) játékos. Tedd biztonságossá let ... else-szel, hogy ilyenkor egyszerűen kilépjen a frame-ből.

hibás kód
fn hud_frissit(jatekos: Query<&Eletero, With<Jatekos>>) {
    let eletero = jatekos.single().unwrap(); // pánik, ha 0 vagy 2+ játékos
    // ... HUD frissítése
}
Megoldás
megoldás
fn hud_frissit(jatekos: Query<&Eletero, With<Jatekos>>) {
    let Ok(eletero) = jatekos.single() else { return; };
    // ... HUD frissítése
}
0x05

Gyors fordítás

A leggyakoribb Bevy-panasz: „lassú a fordítás". A hideg első build tényleg hosszú, de a napi inkrementális fordítás drámaian gyorsítható. Három beállítás, amit minden Bevy-projektben érdemes bekapcsolni dev közben.

1. Dynamic linking (a legnagyobb nyereség)

Cargo.toml
[dependencies]
bevy = { version = "0.18", features = ["dynamic_linking"] }
Csak fejlesztéshez!

A dynamic_linking a Bevyt megosztott könyvtárként linkeli — sokkal gyorsabb újrafordítás, de a kész játék mellé oda kell adni a .so/.dll-t. Kiadáshoz kapcsold ki (pl. tedd egy feature mögé, vagy fordíts --no-default-features nélküli release-t).

2. Gyorsabb linker

.cargo/config.toml — példa (Linux)
# A linkelés gyakran a szűk keresztmetszet. Linuxon az lld vagy a mold sokat gyorsít.
[target.x86_64-unknown-linux-gnu]
linker = "clang"
rustflags = ["-C", "link-arg=-fuse-ld=lld"]
# macOS és Windows beállítását lásd: bevy.org/learn (Fast Compiles)

3. A függőségek optimalizálva, a saját kódod gyorsan

Cargo.toml
# A te kódod gyorsan fordul (opt-level 1), de a függőségek optimalizáltak (3),
# így a játék dev-módban is élvezhetően fut.
[profile.dev]
opt-level = 1

[profile.dev.package."*"]
opt-level = 3
0x06

Asset hot-reload

Sprite-ot, hangot, shadert szerkeszteni a játék újraindítása nélkül — ez óriási gyorsítás. Kapcsold be az asset-figyelést, és minden mentett változás magától újratöltődik.

Cargo.toml
bevy = { version = "0.18", features = ["file_watcher"] }
src/main.rs
use bevy::prelude::*;

fn main() {
    App::new()
        .add_plugins(DefaultPlugins.set(AssetPlugin {
            watch_for_changes_override: Some(true),
            ..default()
        }))
        .run();
}

Mostantól, ha az assets/ mappában átrajzolsz egy PNG-t és mented, a futó játékban azonnal frissül — kódmódosítás és újraindítás nélkül.

0x07

Élő ECS-inspector

A dbg! jó, de az ECS-állapotot a legkényelmesebben egy élő böngészővel nézed: a bevy-inspector-egui egy oldalpanelt ad, amiben futás közben látod minden entitás komponenseit, és a számértékeket élőben át is állíthatod. Ez a debugger megfelelője az ECS-világban.

terminál
cargo add bevy-inspector-egui
src/main.rs
use bevy::prelude::*;
use bevy_inspector_egui::quick::WorldInspectorPlugin;

fn main() {
    App::new()
        .add_plugins(DefaultPlugins)
        .add_plugins(WorldInspectorPlugin::new()) // élő világ-böngésző panel
        .run();
}
Tipp

Az inspectort érdemes egy feature vagy egy #[cfg(debug_assertions)] mögé tenni, hogy a kiadott build ne tartalmazza. A komponenseid akkor jelennek meg szépen, ha levezeted rájuk a Reflect-et: #[derive(Component, Reflect)] + app.register_type::<T>().

Az ökoszisztéma

A Bevy szándékosan kicsi mag — sok mindent külső crate ad hozzá. Nem kell mindent magadnak megírni; íme a bevett választások.

docs.rs/bevy — a teljes API. A Bevy examples/ mappája (a repóban / a bevy.org-on) — a kanonikus „hogyan csináljam" referencia, témánként. bevy-cheatbook — gyakorlati receptek. Migration guide-ok (bevy.org/learn) — verzióváltáskor aranyat érnek, mert a Bevy API gyorsan változik. És a Discord a közösségi segítséghez.

Verziófüggőség

Egy külső crate csak akkor működik, ha a te Bevy-verziódhoz készült. Telepítés előtt nézd meg a crate Bevy-kompatibilitási tábláját — egy 0.17-re írt plugin nem feltétlenül fordul 0.18-cal.

A néma bukások ellenszere

  • „Néha nem történik meg" → keress B0003 warningot a logban (RUST_LOG=bevy_ecs=debug).
  • „Nem látszik a sprite" → B0004, vagy hiányzó Visibility/kamera/z-koordináta.
  • „A rendszer nem csinál semmit" → a lekérdezés 0 entitásra illik (q.iter().count()), vagy a single() Err.
  • „Pánikol indítás után pár frame-mel" → asset, ami még nem töltött be (Assets::get = None).

Merre tovább?

Most már nem a nyelv vagy a motor van az utadban, hanem csak a játék, amit meg akarsz csinálni. Fogj egy kis ötletet, fordítsd gyorsra a buildet, kapcsold be a hot-reloadot és az inspectort — és építsd meg. A hibák már nem ellenségek: tudod olvasni őket. 🦀
