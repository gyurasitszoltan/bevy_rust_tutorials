Játékérzés — game juice
egy top-down világban

A game feel (vagy game juice) az a réteg, ami egy mechanikailag helyes játékot élvezetessé tesz. A mozgás kódja lehet tökéletes — ha a karakter azonnal full sebességre kap és azonnal megáll, „robotos" lesz. Ha ugyanaz a sebesség egy kis gyorsulással és súrlódással érkezik, a karakternek súlya lesz. Ez a lecke a klasszikus platformeres technikákat (gyorsulás, squish, hit-stop, coyote time) top-down játékra adaptálja — ahol nincs gravitáció és nincs ugrás, de van irány, lendület, támadás és találat. Minden technikát külön körbejárunk Bevy 0.18 kóddal és interaktív demóval.

00

Mi a game feel, és miért top-down máshogy?

A kifejezés — game feel, game juice, punch — arra a megfoghatatlan minőségre utal, amit a Celeste, a Hollow Knight vagy a Hades érezetet, egy „halott halacska" rajzolatú prototípus pedig nem. A meglepő hír: a különbség nem a grafikában és nem is a kód helyességében van. Ugyanaz a mozgás, ugyanaz az ütközés — csak a másikhoz egy tucat apró réteg van hozzáadva: gyorsulás, sprite-nyújtás, porfelhő, apró rázkódás, egy pár frame fagyás, hang. Mindegyik önmagában szinte észrevétlen; együtt drámai különbséget csinálnak.

Ez a lecke az eredetileg platformerekre kidolgozott technikákat top-down környezetre fordítja. Miért kell külön? Mert a top-down játék fizikája más:

  • Nincs gravitáció. A karakter nem „érkezik" — így a „land dust" és a „landing squish" helyét a támadásütközés és a dash vége veszi át.
  • Négy irányú mozgás. A gyorsulás nem egy tengelyen megy — egy Vec2-ben kell simítani, és a squish a mozgásirányba nyújt, nem felfelé.
  • A „talaj" mindenhol. Nincs „lelépés a peremről" — de a coyote time logikája (egy kis türelmi ablak) átvihető a dash-re és az input buffering-ra.
  • A kamera felsőnézetből. A screen shake más irányba hat — és a „súly" érzését a zoom-pulse adja, nem a függőleges rázkódás.

A videó, ami ezt a leckét inspirálta (egy platformer „megjuiceolását" mutatja be), ezt a hat+két technikát veszi végig: (1) gyorsulás és súrlódás, (2) squish & stretch, (3) porfelhő és apró screen shake érkezéskor, (4) finom játék-fagyasztás (hit-stop), (5) láthatatlan türelmi ablakok (coyote + buffer), (6) hang pitch-randomizálással, és bónuszként (7) pickup-polish (érme-lebegés, sparkle, score-punch). Ezt mind a saját top-down kontextusunkban építjük fel.

Mire épül & függőségek

Ez a lecke feltételezi a Bevy ECS-alapokat (entitás/komponens/system, Resource, állapotok) — lásd a Bevy alapok tutorialt. A mozgás és ütközés hátteréhez az ütközések lecke, az animációhoz a animációk, a hanghoz a hang & zene a párdokumentumok. A kód Bevy 0.18-ra készült; a DefaultPlugins mindent tartalmaz, külön crate nem kell:

A legfontosabb alapelv

A juice nem változtatja a játékszabályt. A squish csak a sprite skáláját módosítja, nem a hitboxot. A hit-stop pár frame-re lassítja az időt, de a mozgás logikája ugyanaz. A screen shake a kamerát mozgatja, nem a játékost. Ez a megkülönböztetés kritikus: a játékos sosem büntethető a juice miatt (ne kezdjen el egy shake miatt kiesni egy peremről), és a hálózati/szimulációs logika nem férhet hozzá a juice adatokhoz. A juice egy vizuális + auditív + időbeli réteg, ami a fizika fölé van rakva.

01

Mozgás: gyorsulás & súrlódás

A legelső és legnagyobb hatású juice-elem. A „robotos" mozgás oka: a karakter azonnal a maximális sebességre ugrik (gomb lenyom → velocity = max), és gomb felenged → velocity = 0. A való életben semmi sem mozog így — minden gyorsul és lassul. Egy kis gyorsulás (acceleration) induláskor és súrlódás (friction) megálláskor azonnal „súlyt" ad a karakternek: néhány frame alatt ér csak fel a teljes sebességre, és néhány frame alatt áll le.

A trükk lényege, amit a videó kiemel: a maximális sebesség ugyanaz marad. A karakter nem lassabb — csak lassabban ér el oda. Ez pont elég ahhoz, hogy „súlyosnak" hasson, anélkül, hogy a játékos lassabbnak érezné a kontrollt. A Super Meat Boy extrém gyors gyorsulást használ (csúszós, „lebegős" érzés); a Celeste szorosra hangolt, precíz de súlyos érzést.

A platformeres alap → top-down

Platformerben egy move_toward(velocity.x, target, accel * dt) hívás elég. Top-down-ban a mozgás Vec2 — a gyorsulás is kétdimenziós. A bevált minta: az inputból először egy célsebességet számolsz (max sebesség × irányvektor), aztán a jelenlegi sebességet move_toward-dal (vektoronként) közelíted hozzá. Külön accel és decel — induláskor és megálláskor más értékkel.

use bevy::prelude::*;

#[derive(Component)]
struct Player {
    velocity: Vec2,
    max_speed: f32,   // a top speed — nem változik
    accel: f32,       // induláskor milyen gyorsan gyorsul (px/s²)
    decel: f32,       // megálláskor milyen gyorsan fékez (súrlódás)
}

impl Default for Player {
    fn default() -> Self {
        Self { velocity: Vec2::ZERO, max_speed: 180.0, accel: 1200.0, decel: 1600.0 }
    }
}

fn player_movement(
    keys: Res<ButtonInput<KeyCode>>,
    time: Res<Time>,
    mut q: Query<(&mut Player, &mut Transform)>,
) {
    let dt = time.delta_secs();
    for (mut p, mut tr) in &mut q {
        // 1) input → célsebesség vektor (max_speed × egységnyi irány)
        let mut dir = Vec2::ZERO;
        if keys.pressed(KeyCode::KeyW) { dir.y -= 1.0; }
        if keys.pressed(KeyCode::KeyS) { dir.y += 1.0; }
        if keys.pressed(KeyCode::KeyA) { dir.x -= 1.0; }
        if keys.pressed(KeyCode::KeyD) { dir.x += 1.0; }
        // normalizálás: átlós gombnyomás ne legyen gyorsabb (√2)
        let target = if dir != Vec2::ZERO {
            dir.normalize() * p.max_speed
        } else { Vec2::ZERO };

        // 2) gyorsulás/súrlódás komponensenként — a lényeg!
        //    ha van input → accel; ha nincs → decel (súrlódás)
        let rate = if dir != Vec2::ZERO { p.accel } else { p.decel };
        p.velocity.x = p.velocity.x.move_toward(target.x, rate * dt);
        p.velocity.y = p.velocity.y.move_toward(target.y, rate * dt);

        // 3) pozíció frissítése az aktuális (még nem teljes) sebességgel
        tr.translation.x += p.velocity.x * dt;
        tr.translation.y += p.velocity.y * dt;
    }
}

A move_toward(current, target, max_delta) függvény (a Rust f32-en) current-et target felé mozgatja legfeljebb max_delta lépéssel — pontosan az, amit egy platformeres lerp-easing csinál, csak lépéskorlátos. Ez a „súrlódásos" közérzet alapja: a gomb felengedése után a karakter még csúszik egy kicsit.

Külön accel és decel — miért?

A decel rendszerint nagyobb, mint az accel: a játékos azt akarja, hogy a karakter gyorsan álljon meg, ha elengedi a gombot, de kellemesen gyorsuljon, ha megnyomja. Ha a kettő egyenlő, a megállás túl „lebegős"; ha a decel sokkal nagyobb, „tapadós" lesz a kontroll. A fenti 1200/1600 arány egy jó kiindulópont — finomhangold a karakter egyéniségéhez.

Top-down vs platformer — a különbség

Platformerben a súrlódás csak vízszintesen van (a gravitáció függőlegesen „megállítja" a karaktert). Top-down-ban mindkét tengelyen kell. A normalizálás azért kritikus, mert különben az átlós mozgás (√2 ≈ 1.41×) gyorsabb lenne, mint az egyenes — a játékos „átlósúszást" tanulna ki a sebességmaxoláshoz, ami rontja a játékérzést.

02

Squish & stretch

A második leghatékonyabb juice-elem, és az egyik legolcsóbb. A karakter sprite-ját nyújtod (stretch) a mozgásirányba és összenyomod (squish), amikor megáll vagy ütközik — aztán egy rugós (elastic) tweennel visszaállítod az eredeti méretre. Ez „életet" lehel a karakterbe: súlya, lendülete, ütése van. A kulcs: a videó ismételten hangsúlyozza az impact szót — a squish az ütés/érkezés pillanatának megerősítése.

A trükk lényege, és ami a juice-elvek közül a legfontosabb: csak a sprite skáláját módosítod, nem a hitboxot. A játék logikája (ütközés, mozgás) változatlan marad; csak a vizuális megjelenés „lélegzik". Erre érdemes egy külön komponenst fenntartani, ami a vizuális transformot vezérli, fizikaitól függetlenül.

Top-down squish — a mozgásirányba nyújtás

Platformerben a squish függőleges: ugráskor magasabb/kisebb szélesség, érkezéskor szélesebb/kisebb magasság. Top-down-ban a mozgásirányba nyújtunk, és merőlegesen squishelünk — terelődik a karakter alakja. Megálláskor a squish „túlcsordul" a mozgás irányába (mint egy fékezést szimuláló rugó), és visszaugrik.

use bevy::prelude::*;

// A vizuális scale-t külön komponensen tartjuk — a fizika transform nem lélegzik.
#[derive(Component)]
struct VisualScale {
    base: Vec2,       // alap sprite-méret (1.0, 1.0)
    current: Vec2,    // aktuális vizuális skála
}

impl Default for VisualScale {
    fn default() -> Self { Self { base: Vec2::ONE, current: Vec2::ONE } }
}

// Squish-parancs: a mozgásirány + erő megadja, mennyit nyújtsunk.
// A rendszer a squish után rugósan visszaállítja az alapméretre.
#[derive(Component, Default)]
struct Squish {
    target: Vec2,    // pillanatnyi cél-skála (pl. (1.3, 0.8) = nyúlt vízszintesen)
    spring: f32,     // rugó keménysége — magasabb = gyorsabb visszaugrás
    damp: f32,       // csillapítás — 1.0 = tökéletes rugó, <1 = lassabb, >1 = ingadozik
}

fn apply_squish(
    time: Res<Time>,
    mut q: Query<(&mut Squish, &mut VisualScale)>,
) {
    let dt = time.delta_secs();
    for (mut sq, mut vs) in &mut q {
        // rugós közelítés a célhoz — ez adja az „elastic" visszapattanást
        let k = sq.spring * dt;
        vs.current.x += (sq.target.x - vs.current.x) * k.min(1.0);
        vs.current.y += (sq.target.y - vs.current.y) * k.min(1.0);
        // a cél visszatér az (1,1) alap felé — így a squish „elpárolog"
        sq.target.x += (1.0 - sq.target.x) * (sq.damp * dt).min(1.0);
        sq.target.y += (1.0 - sq.target.y) * (sq.damp * dt).min(1.0);
    }
}

// a vizuális scale-t a fizika transformtól FÜGGETLEN child-en tesszük
fn sync_visual_scale(
    q: Query<(&VisualScale, &Children), Changed<VisualScale>>,
    mut sprites: Query<&mut Sprite>,
) {
    for (vs, children) in &q {
        for child in children {
            if let Ok(mut sprite) = sprites.get_mut(*child) {
                // Sprite::custom_size vagy a Transform scale — mindkettő működik
                // a fontos: a szülő (fizika) transformot nem piszkáljuk
                sprite.custom_size = Some(Vec2::new(32.0 * vs.current.x, 32.0 * vs.current.y));
            }
        }
    }
}

// megállás érzékelése → squish indítása a mozgás irányába
fn squish_on_stop(
    mut q: Query<(&Player, &mut Squish), Changed<Player>>,
) {
    for (p, mut sq) in &mut q {
        // ha a sebesség majdnem 0, de előzőleg volt lendület —> squish
        if p.velocity.length_squared() < 5.0 {
            // a mozgásirány meghatározása a sebességből (előző frame)
            if let Some(dir) = p.velocity.try_normalize() {
                // nyújtás mozgásirányba, squish merőlegesen
                // (itt egyszerűség kedvéért csak a főirányt nézzük)
                let stretch = if dir.x.abs() > dir.y.abs() {
                    Vec2::new(1.25, 0.8)   // vízszintes nyújtás
                } else {
                    Vec2::new(0.8, 1.25)   // függőleges nyújtás
                };
                sq.target = stretch;
                sq.spring = 12.0;
                sq.damp = 6.0;
            }
        }
    }
}
Miért külön VisualScale?

A Bevy-ben egy entity Transform-ját használja a fizika és a renderelés is. Ha a fizika scale-t piszkálod, a hitbox is nő — ami megváltoztatja a játékot. A tiszta megoldás: a fizika-entitás Transform-ja csak pozíciót és forgást visz (scale = 1), és a sprite-on egy gyermek entity viszi a vizuális skálát. A VisualScale ezt a gyermek-skálát vezérli — a hitbox sértetlen marad. Ugyanez a minta a sprite-anim vs transform leckén szerepel.

Ne vidd túlzásba

A videó figyelmeztet: ha az összes juice-effektust egyszerre maximumon kapcsolod be, szédítő lesz. A squish legyen finom — 1.2–1.3× nyújtás, nem 2×. A cél az „észrevehetetlen, de hiányzik, ha nincs ott" érzés, nem a karikatúra. Ha a játékos észreveszi a squishet minden ütésnél, az már sok — csak a nagy pillanatokban (megállás, ütközés, támadás) hangsúlyos.

03

Nyomkövetés: porfelhő & lábnyom

A platformer „land dust" (érkezéskor pukkanó porfelhő) top-down megfelelője: mozgás közbeni porfelhő és lábnyomok. Mivel nincs „érkezés", a vizuális nyomkövetés a mozgás folyamatos jele — elmondja a játékosnak, hogy a karakter súlyos, és a talajon van, nem lebeg. Ez a legkönnyebben „hiányzik, ha nincs ott": egy porfelhő nélküli top-down karakter úgy néz ki, mint egy slide-en csúszó ikon.

Két réteg:

  • Porfelhő: mozgás közben (elsősorban gyorsuláskor vagy irányváltáskor) pukkan egy kis felhő a karakter talajánál. Ez a mozgás „lenyomata".
  • Lábnyom: ritkábban (lépésenként) egy halvány folt marad a talajon, ami lassan elhalványul. Ez térbeli folytonosságot ad — a játékos látja, hol járt.

Bevy-ben: spawn + életciklus

Mindkettő rövid életű entitás: spawnolod, rajzolod, pár másodperc múlva despawnol. A Bevy ECS-ben ez természetes: egy Lifetime komponens, egy rendszer ami lecsökkenti és eltávolítja, és egy másik ami a vizuális megjelenést (átlátszóság, méret) az életkorhoz köti.

use bevy::prelude::*;

// egy rövid életű vizuális effekt: porfelhő vagy lábnyom
#[derive(Component)]
struct Lifetime {
    remaining: f32,    // másodperc
    total: f32,        // eredeti — az alpha/scala interpolálásához
}

#[derive(Component)]
struct DustPuff {
    start_size: f32,
    end_size: f32,
}

#[derive(Component)]
struct Footprint {
    base_alpha: f32,
}

fn tick_lifetime(
    time: Res<Time>,
    mut commands: Commands,
    mut q: Query<(Entity, &mut Lifetime)>,
) {
    let dt = time.delta_secs();
    for (e, mut life) in &mut q {
        life.remaining -= dt;
        if life.remaining <= 0.0 {
            commands.entity(e).despawn();
        }
    }
}

// porfelhő vizuális frissítése: nő + halványul
fn animate_dust(
    mut q: Query<(&mut DustPuff, &Lifetime, &mut Sprite, &mut Transform)>,
) {
    for (mut dust, life, mut sprite, mut tr) in &mut q {
        let t = 1.0 - (life.remaining / life.total).max(0.0);  // 0 → 1 az idő múlásával
        let size = dust.start_size + (dust.end_size - dust.start_size) * t;
        if let Some(cs) = sprite.custom_size.as_mut() {
            cs.x = size; cs.y = size;
        }
        // alpha halványul
        sprite.color.set_alpha((1.0 - t).max(0.0) * 0.6);
    }
}

// porfelhő spawnolása mozgáskor (ritkábban, hogy ne spammeljük)
#[derive(Resource, Default)]
struct DustTimer(f32);

fn spawn_dust_while_moving(
    time: Res<Time>,
    mut timer: ResMut<DustTimer>,
    q: Query<&Transform, With<Player>>,
    p: Res<Player>,  // egyszerűsítés: a Player adata
    mut commands: Commands,
) {
    timer.0 -= time.delta_secs();
    if p.velocity.length() < 30.0 || timer.0 > 0.0 { return; }
    timer.0 = 0.08;  // 80ms a két porfelhő közt — ritkítja a spawnolást

    for tr in &q {
        // a karakter talajánál spawnolunk (kicsit lejjebb, hogy a „talajon" legyen)
        commands.spawn((
            Sprite::from_color(Color::srgba(0.8, 0.8, 0.85, 0.6), Vec2::new(8.0, 8.0)),
            Transform::from_xyz(tr.translation.x, tr.translation.y + 8.0, -1.0),
            DustPuff { start_size: 6.0, end_size: 18.0 },
            Lifetime { remaining: 0.35, total: 0.35 },
        ));
    }
}
Spawn-ritkítás — miért timer?

Ha mozgáskor minden frame-ben spawnolnál egy porfelhőt, 60 FPS-nél másodpercenként 60 entitás keletkezne — a képernyő tele lenne, és az ECS is szenvedne. A DustTimer resource egy egyszerű cooldown: csak 0.08s (80 ms) elteltével spawnol újabbat. Ez az egyszerű „pacing" minta minden juice-effektnél megjelenik (screen shake, hit-stop spark, lábnyom) — mindig timer mögé tedd a spawnolást, sose spammelj.

Lábnyom: szárazon vagy olajosan?

A lábnyomok akkor működnek a legjobban, ha a környezet is reagál: havon sötét nyom, sárban mélyebb, kövén alig látható. A technika ugyanaz (spawn + halványulás), de az base_alpha a biome-tól függ. Ha a pályád biomokból épül, a biome adja a lábnyom színét/alfáját — egy fillér extra adat, de hatalmas különbség a „helyérzésben".

▶ Demó: porfelhő + lábnyom
Mozgasd a karaktert (W/A/S/D vagy katt). A porfelhő a talajnál pukkan és nő/halványul; a lábnyomok lépésenként maradnak a talajon és lassan eltűnnek. Kapcsold ki mindkettőt — nézd, mennyire „lebegős" lesz a karakter nélkülük.
04

Screen shake

A screen shake a leggyakrabban túlzásba vitt juice-elem — és éppen ezért a videó ennyire hangsúlyozza a „tiny" szót. Egy apró rázkódás (1–3 px, pár frame) súlyt és ütést ad egy eseménynek; egy nagy rázkódás (10+ px, másfél másodperc) már szédítő és zavaró. A kulcs: arányos az eseménnyel. Egy lépésnek ne legyen shake-e; egy találatnak 2px; egy robbanásnak 8px; a halálnak 12px. A kontraszt a fontos, nem az abszolút érték.

Top-down-ban a screen shake más, mint platformerben: a kamera felsőnézetből mozog, így a rázkódás Vec2-es eltolás a kamera pozíción — gyakran egy trauma-alapú modell, ami nem lineárisan csillapodik, hanem exponenciálisan, és minden frame-ben új véletlens irányt vesz. Ez az, ami a „természetes" rázkódást adja — egy fix irányú eltolás gépies lenne.

A trauma-modell

A bevált módszer (a „Scream of the Bullet" / GDC talk óta standard): egy 0..1 trauma érték, amit események hozzáadnak (nem beállítanak), és ami exponenciálisan csillapodik. A vizuális eltolás trauma² × max amplitúdó × véletlens irány — a négyzetre emelés azt adja, hogy kis trauma alig látszik, nagy trauma drámai, és az átmenet nem egyenletes.

use bevy::prelude::*;

#[derive(Resource, Default)]
struct CameraShake {
    trauma: f32,        // 0..1, események hozzáadják
    max_offset: f32,    // max amplitúdó pixelben (pl. 8.0)
    decay: f32,         // exponenciális csillapodás sebessége (pl. 5.0)
}

impl CameraShake {
    fn add(&mut self, amount: f32) {
        // hozzáadás, felülírás helyett — több ütés halmozódhat
        self.trauma = (self.trauma + amount).min(1.0);
    }
}

// esemény: küldő rendszerek ezt írják
#[derive(Message)]
struct ShakeEvent { trauma: f32 }

fn read_shake_events(
    mut ev: MessageReader<ShakeEvent>,
    mut shake: ResMut<CameraShake>,
) {
    for e in ev.read() {
        shake.add(e.trauma);
    }
}

// kamera rendszer: traumából offset, exponenciális csillapodás
fn apply_camera_shake(
    time: Res<Time>,
    mut shake: ResMut<CameraShake>,
    mut q: Query<&mut Transform, With<Camera>>,
) {
    let dt = time.delta_secs();
    // exponenciális csillapodás: trauma *= e^(-decay * dt)
    shake.trauma *= (-shake.decay * dt).exp();
    if shake.trauma < 0.001 { shake.trauma = 0.0; }

    // offset = trauma² × max × véletlen irány
    let amp = shake.trauma * shake.trauma * shake.max_offset;
    let angle = rand::random::<f32>() * std::f32::consts::TAU;
    let offset = Vec2::new(angle.cos(), angle.sin()) * amp;

    for mut tr in &mut q {
        // a kamera alap-pozíciója fix; a shake csak eltolás
        tr.translation.x = offset.x;
        tr.translation.y = offset.y;
    }
}

// használat: találat → 0.3 trauma, robbanás → 0.8, halál → 1.0
fn on_hit(mut ev: MessageWriter<ShakeEvent>) {
    // ...találat detektálása...
    ev.write(ShakeEvent { trauma: 0.3 });
}
Egy abszolút szabály: skálázható + kikapcsolható

A screen shake egy hozzáférhetőségi kérdés is. Sok játékos (mozgásbetegség, vestibularis érzékenység) nem bírja. Mindig legyen egy „screen shake" csúszka a beállításokban, ami a max_offset-et skálázza (0 = kikapcsolva). A prefers-reduced-motion böngészőbeállítás detektálható (webes játék), és alapból lecsökkentheti a shake-et. Ez nem opció — kötelező.

Trauma² miért?

Ha lineáris lenne az amplitúdó (trauma × max), a kis értékek túl „hangosak", a nagyok pedig nem elég drámaiak. A trauma² miatt: 0.3 trauma → 0.09 amplitúdó (alig látszik), 0.7 → 0.49 (feltűnő), 1.0 → 1.0 (max). A átmenet „túlcsorduló" — pont az, amit egy ütésnél akarsz. Ugyanez az elv a hit-stop időtartamánál is.

Hit-stop / freeze frame

Ez a „nagyon finom, de érezhető" technika, amit a videó úgy ír: „megfagyasztod a játékot, nagyon rövid időre". Amikor valami fontos történik — egy találat, egy ütközés, egy támadás kezdete — a játék pár frame-re lelassul vagy megáll, aztán visszamegy a normál sebességre. Ezt a játékos alig veszi észre tudatosan, de érzi: az ütésnek „súlya" lesz, a tap-ból igazi ütés lesz. Ez a „impact", amit a videó a squish-nél említ — a hit-stop az időbeli megfelelője a vizuális squish-nek.

A platformeres „freeze a jump pillanatában" top-down megfelelője: freeze a találat/támadás pillanatában. Egy jól időzített 40–80 ms-os fagyás elképesztő különbséget csinál — a „katt" érzetből „csatt" lesz.

Bevy-ben: TimeScale a Time<Virtual>-ön

A Bevy Time<Virtual> erőforrás (a Time virtuális változata) rendelkezik egy relatív sebességgel, amit set_relative_speed-del állíthatsz. Ha a mozgásrendszered a virtuális időt használja (alapból a Time = Time<Virtual>), akkor egy rendszer, ami 0-ra állítja a sebességet, pár frame múlva 1-re vissza — kész a hit-stop. A fix timestep fizika ettől függetlenül mehet, de a vizuális mozgás megfagy.

use bevy::prelude::*;
use bevy::time::Time;

#[derive(Resource, Default)]
struct HitStop {
    timer: Timer,           // hátralévő fagyási idő
    speed_target: f32,       // fagyás alatt milyen sebességgel menjen (0 = teljes stop)
}

// esemény: küldő rendszerek indítják
#[derive(Message)]
struct HitStopEvent {
    duration: f32,    // másodperc (pl. 0.06 = 60ms)
    speed: f32,        // fagyás alatti time scale (0.0 = teljes stop, 0.1 = lassított)
}

fn read_hitstop_events(
    mut ev: MessageReader<HitStopEvent>,
    mut hs: ResMut<HitStop>,
) {
    for e in ev.read() {
        // a leghosszabb/megerősebb fagyás nyer — ne „törölje" egy gyengébb
        if e.duration > hs.timer.remaining_secs() {
            hs.timer.set_duration(std::time::Duration::from_secs_f32(e.duration));
            hs.timer.reset();
            hs.speed_target = e.speed;
        }
    }
}

fn apply_hitstop(
    time: Res<Time>,
    mut virtual_time: ResMut<Time<Virtual>>,
    mut hs: ResMut<HitStop>,
) {
    hs.timer.tick(time.delta());  // valós idővel tickel — nem a fagyottal!
    if hs.timer.just_finished() {
        // fagyás vége: vissza a normál sebességre
        virtual_time.set_relative_speed(1.0);
    } else if !hs.timer.finished() {
        virtual_time.set_relative_speed(hs.speed_target);
    }
}

// használat: találat → 60ms-os teljes stop
fn on_sword_hit(mut ev: MessageWriter<HitStopEvent>) {
    // ...találat detektálása...
    ev.write(HitStopEvent { duration: 0.06, speed: 0.0 });
}
A timer-t valósidejűen tickeld!

A leggyakoribb hiba: a hit-stop timert a virtuális idővel tickelik (mert a többi rendszer azt használja), de a time scale 0 — így a timer sosem megy tovább, és a játék örökre megfagy. A fenti kód time.delta()-t használ (a valós Time-ot, ami a virtuálistól független), így a hit-stop akkor is lejár, ha a virtuális idő megállt. Ez egy apró, de végzetes részlet.

A „lassított" hit-stop

A speed: 0.0 teljes fagyást ad — ez a leggyakoribb és legütősebb. De néha jobb a speed: 0.1 (10%-os sebesség): a játék nem áll meg teljesen, csak nagyon lelassul, ami egy „bullet-time" pillanatot ad. Ez különösen hatásos nagy találatoknál (boss-ütés, fejlövés) — a sima találatnál maradj a 0.0. A kettő keverése (kis ütés → 0.0/40ms, nagy ütés → 0.1/120ms) gazdag skálát ad.

Mozgásrendszer idősíkjának szétválasztása

A hit-stop csak akkor működik, ha a vizuális mozgás (játékos, ellenségek, projektil) a virtuális időt használja. Ha egy rendszer Fixed<T>-t használ (fix timestep fizika), az tovább megy a fagyás alatt is — ami lehet, hogy akarod (a fizika ne omoljon), vagy nem (a hit-stop „átjárja" a fizikát). A tiszta megoldás: a időzítés leckén részletezett módon a fizikát fix, a vizualitást virtuális időre tedd. A hit-stop a virtuálisat állítja.

Input buffering & leniency

A platformer klasszikus coyote time és jump buffering két „láthatatlan" türelmi ablak, amit a videó kiemel: coyote time = ha a játékos lelép egy peremről, még pár tizedmásodpercig ugorhat; jump buffering = ha egy kicsit azelőtt nyom ugrást, mielőtt földet érne, az ugrás „pufferel" és végrehajtódik, amint lehet. Ezek teszik, hogy a Celeste igazságosabbnak érződjön annál, amilyen nehéz — a játékos nem a frame-pontos időzítésért küzd, hanem a döntésért.

Top-down-ban nincs „perem" és nincs „földet érés". De a logika átvihető: minden olyan cselekvésre, ami csak bizonyos feltételekkel hajtható végre (dash csak ha nincs cooldown, támadás csak ha fegyver készen, interakció csak ha célpont közelben), adhatsz egy türelmi ablakot. Két minta:

  • Input buffer: a játékos megnyomja a gombot, de a cselekvés még nem lehetséges (cooldown) → a gombnyomás „vár" egy rövid ideig, és ha időben lehetőség nyílik, végrehajtódik. „Korai gombnyomás" = nem büntet.
  • Leniency window: a cselekvés épp most volt lehetséges, de a játékos egy kicsit késik → még egy pár tizedmásodpercig elérhető. „Késői gombnyomás" = nem büntet. Ez a coyote time analógia.

Dash-buffer: a leggyakoribb top-down minta

A dash (roham, görbe) egy cooldown-alapú cselekvés. Ha a játékos a cooldown alatt nyomja a gombot, a hagyományos megközelítés elveti a gombnyomást — frusztráló. A bufferelt megközelítés megjegyzi a gombnyomást, és a cooldown végén automatikusan elindítja a dasht — feltéve, hogy a gombnyomás elég friss.

use bevy::prelude::*;

#[derive(Component)]
struct DashController {
    cooldown: Timer,         // hátralévő várakozás
    buffer: Option<Timer>,   // pufferelt gombnyomás (ha van) + hátralévő élettartam
    dash_duration: Timer,    // ha dashel, mennyi ideig tart
    buffer_window: f32,      // mennyi ideig él a puffer (pl. 0.15 = 150ms)
}

fn dash_input(
    keys: Res<ButtonInput<KeyCode>>,
    time: Res<Time>,
    mut q: Query<&mut DashController>,
) {
    let dt = time.delta_secs();
    for mut d in &mut q {
        // 1) timer-ek tickelése
        d.cooldown.tick(time.delta());
        d.dash_duration.tick(time.delta());

        // 2) buffer lejár — ha a pufferelt gombnyomás túl régi, eldobjuk
        if let Some(buf) = &mut d.buffer {
            buf.tick(time.delta());
            if buf.finished() { d.buffer = None; }
        }

        // 3) új gombnyomás: ha dashelne, de cooldown van → pufferba
        if keys.just_pressed(KeyCode::ShiftLeft) {
            if d.cooldown.finished() && d.dash_duration.finished() {
                // azonnal dashelhet — itt elindítod a dasht
                start_dash(&mut d);
            } else {
                // puffer: megjegyezzük, amíg a cooldown le nem jár
                d.buffer = Some(Timer::from_seconds(d.buffer_window, TimerMode::Once));
            }
        }

        // 4) bufferelt gombnyomás végrehajtása, amint lehet
        if d.buffer.is_some() && d.cooldown.finished() && d.dash_duration.finished() {
            start_dash(&mut d);
            d.buffer = None;  // elfogyott
        }
    }
}

fn start_dash(d: &mut DashController) {
    d.dash_duration = Timer::from_seconds(0.18, TimerMode::Once);
    d.cooldown = Timer::from_seconds(0.6, TimerMode::Once);
    // a tényleges dash-sebesség adását a mozgásrendszer végzi, ha dash_duration megy
}
A türelmi ablak mérete — érzés kérdése

A videó ezt nem részletezi, de a gyakorlati érték: 100–200 ms. Rövidebb (50 ms) alatt a játékos nem érzi a segítséget; hosszabb (300+ ms) alatt „dupla dash" bugok jelennek meg (egy gombnyomás kétszer hajtódik végre). A Celeste coyote-ja 6 frame (~100 ms 60 FPS-nél), a jump buffer 8 frame (~133 ms). Ez a tartomány a „eszrevétlen segítség" — kezdésnek a 150 ms jó kompromisszum.

Mikor NE legyen buffer?

Nem minden inputot kell pufferelni. A mozgás input (W/A/S/D) sosem pufferel — az valós idejű. A committal cselekvések (nagy támadás, amibe belementél) esetén is gondolkozz: ha a buffer miatt a játékos „véletlenül" elindít egy második támadást, amit nem akart, az rosszabb, mint a frusztráció a kihagyott gombnyomás miatt. A buffer a gyors, ismétlődő cselekvéseknél (dash, ugrás, kis támadás) működik a legjobban.

Hang: pitch & audio control

A videó szerint — és itt a legerősebb az állítása — a hang a legfontosabb juice-elem, és a leggyakrabban elhanyagolt. Egy ugrás hang nélkül csak egy pozícióváltozás; hanggal súlya van. Egy találat hang nélkül egy animáció; hanggal „csatt" lesz. A hang az a réteg, ami a testnek szól — a másik kettő (vizualitás, idő) a szemnek és az érzékelésnek.

De a videó nem csak „legyen hang"-ot mond. A konkrét technika, amit kiemel: pitch randomizálás. Ugyanaz az efekt, minden lejátszáskor egy kicsit más magassággal — elkerüli a mechanikus ismétlődésérzést. A szimmetria, az állandóság a játékban rossz érzés; a láthatatlan randomizáció adja meg az „életérzetet". Ez egy apró, de drámai különbséget csinál: egy „katt" vs. egy „katt, katt, katt" ugyanabból a fájlból.

Bevy-ben: AudioSink + pitch

A Bevy 0.18-ban a hang entitás-alapú: spawnolsz egy AudioPlayer-t, Bevy ráteszi az AudioSink-et, és azon vezérled a hangerőt, sebességet (ami pitch-et is ad!), szünetet. A pitch-randomizáláshoz a set_speed-et használjuk — ami egyúttal magasságot is változtat. Egy kis ±5–10% ingadozás elég.

use bevy::prelude::*;
use bevy::audio::{AudioPlayer, AudioSink, AudioSinkPlayback, PlaybackSettings, PlaybackMode};

#[derive(Resource)]
struct SfxLibrary {
    step: Handle<AudioSource>,
    hit:  Handle<AudioSource>,
}

// marker: csak ezekre az entitásokra akarunk pitch-randomizációt (a zenére pl. nem)
#[derive(Component)]
struct RandomizePitch;

// pitch-randomizáló rendszer: amint Bevy ráteszi az AudioSink-et (a spawn UTÁNI
// frame-ben!), beállítjuk a véletlen magasságot. Added<AudioSink> kell, NEM
// Added<AudioPlayer>: a sink a spawn pillanatában még nem létezik, így a régi
// `get_mut`-os változat sosem találta volna meg.
fn randomize_pitch(
    mut sinks: Query<&mut AudioSink, (Added<AudioSink>, With<RandomizePitch>)>,
) {
    for mut sink in &mut sinks {
        // véletlenszerű pitch: 0.9–1.1 (±10%) — a set_speed pitch-et is ad
        let pitch = 0.9 + rand::random::<f32>() * 0.2;
        sink.set_speed(pitch);
    }
}

// helper: efekt lejátszása (Despawn mód — öntisztító)
fn sfx(commands: &mut Commands, handle: Handle<AudioSource>, volume: f32) {
    commands.spawn((
        AudioPlayer::new(handle),
        PlaybackSettings {
            mode: PlaybackMode::Despawn,
            volume: volume.into(),
            ..default()
        },
        RandomizePitch,   // ez kapja majd a véletlen pitch-et, amint elindul
    ));
}

// használat: lépéshang (halk, gyakori), találat (hangos, ritka)
fn on_step(mut commands: Commands, sfx: Res<SfxLibrary>) {
    sfx(&mut commands, sfx.step.clone(), 0.3);  // halkabb
}
fn on_hit(mut commands: Commands, sfx: Res<SfxLibrary>) {
    sfx(&mut commands, sfx.hit.clone(), 0.8);   // hangosabb
}
Hangerő is randomizálható

A pitch mellett a hangerő enyhe randomizálása (±5%) is segít — különösen, ha ugyanaz az efekt gyakran szól (pl. lépéshang). A kettő együtt azt az érzést adja, hogy „ugyanaz az esemény, de sosem pontosan ugyanúgy" — ami pont az, amit a való világ tesz. De vigyázat: túl sok randomizáció chaotikussá válik. A ±10% pitch + ±5% hangerő egy jó kiindulópont.

set_speed ≠ valódi pitch-shift

A set_speed a lejátszás sebességét változtatja, ami magával hozza a pitch-et — de a hossz is változik (gyorsabb = rövidebb). Egy rövid effektnél (kattanás, csatt) ez észrevehetetlen, és pont az, amit akarsz. De hosszabb effektnél (egy zenés motívum) a gyorsítás torzít. Ha „lassított, de eredeti magasságú" effekt kell (pl. lassított halál-megfagyás), arra a Bevy beépítetten nem ad — külső DSP/crate kell. A részleteket a hang & zene lecke 04-es szakasza tárgyalja.

A három hangerő-szint — mixer

Minden juice-hangnak három hangerő-rétege van: (1) master (a játék fő hangerője, a beállításokból), (2) csatorna (SFX/zene/ambient külön csúszka), (3) saját (az egyes efekt helyi hangerője, pl. lépés halk, találat hangos). A végleges hangerő a három szorzata. Ez a mixer modell — nélküle a játékos nem tudja külön állítani a zenét és az effektt, ami hozzáférhetőségi probléma. Korán érdemes beépíteni.

Pickup & UI polish

A videó utolsó, bónuszpontja: az apró vizuális flourishes, amik a játék „kész" érzetét adják. Három konkrét elemet emel ki:

  • Érme lebegése. Az érme nem csak eltűnik — felugrik egy picit, méretezik, és „elúszik" felfelé, mielőtt eltűnne. Egy tween a pozíción és a méreten.
  • Sparkle burst. Begyűjtéskor egy kis szikra-robbanás (néhány gyors, kicsi, szétáló partikula) jelenik meg az érme helyén.
  • Score punch. A pontszám-címke skálája „megüt" (1.3× lesz), aztán visszaáll — a szám nem csak nő, hanem üt.

Ezek mind rövid, automatikus animációk — a játékos nem csinálja, csak látja. A Bevy-ben mindegyik egy-egy Timer-vezérelt rendszer, ami egy komponenst interpolál. A lényeg ugyanaz, mint a squish-nél: egy célállapot felé közelítés, ami aztán visszaáll.

A három elem egyben

use bevy::prelude::*;

// 1) Pickup animáció: az érme felugrik, nő, halványul, eltűnik
#[derive(Component)]
struct PickupAnim {
    timer: Timer,
    start_pos: Vec3,
}

fn animate_pickup(
    time: Res<Time>,
    mut commands: Commands,
    mut q: Query<(Entity, &mut PickupAnim, &mut Transform, &mut Sprite)>,
) {
    for (e, mut anim, mut tr, mut sprite) in &mut q {
        anim.timer.tick(time.delta());
        let t = anim.timer.fraction();  // 0 → 1 az idő múlásával

        // felugrás: a pozíció felfelé megy (egy íven)
        tr.translation.y = anim.start_pos.y + 24.0 * (1.0 - (1.0 - t).powi(2));
        // méret: először nő, aztán zsugorodik
        let scale = if t < 0.3 { 1.0 + t * 1.5 } else { 1.45 - (t - 0.3) * 2.0 };
        tr.scale = Vec3::splat(scale.max(0.0));
        // alpha: végig halványul
        sprite.color.set_alpha(1.0 - t);

        if anim.timer.just_finished() {
            commands.entity(e).despawn();
        }
    }
}

// 2) Sparkle: rövid életű partikulák, sugarasan szét
#[derive(Component)]
struct Sparkle {
    vel: Vec2,
    timer: Timer,
}

fn animate_sparkles(
    time: Res<Time>,
    mut commands: Commands,
    mut q: Query<(Entity, &mut Sparkle, &mut Transform, &mut Sprite)>,
) {
    for (e, mut sp, mut tr, mut sprite) in &mut q {
        sp.timer.tick(time.delta());
        let t = sp.timer.fraction();
        // mozgás a sebességvektorral (lassuló)
        tr.translation.x += sp.vel.x * time.delta_secs() * (1.0 - t);
        tr.translation.y += sp.vel.y * time.delta_secs() * (1.0 - t);
        tr.scale = Vec3::splat((1.0 - t).max(0.0));
        sprite.color.set_alpha((1.0 - t).max(0.0));
        if sp.timer.just_finished() {
            commands.entity(e).despawn();
        }
    }
}

// 3) Score punch: a UI-címke skálája „üt" és visszaáll
#[derive(Component)]
struct ScorePunch {
    timer: Timer,
    max_scale: f32,
}

fn animate_score_punch(
    time: Res<Time>,
    mut q: Query<(&ScorePunch, &mut Transform)>,
) {
    for (punch, mut tr) in &mut q {
        let t = punch.timer.fraction();  // 0 → 1
        // először felfelé (üt), aztán vissza — egy „spring" görbe
        let scale = if t < 0.3 {
            1.0 + (punch.max_scale - 1.0) * (t / 0.3)
        } else {
            punch.max_scale - (punch.max_scale - 1.0) * ((t - 0.3) / 0.7)
        };
        tr.scale = Vec3::splat(scale);
    }
}

// gyűjtő: mindhárom elindítása
fn on_coin_collected(
    coin_pos: Vec3,
    mut commands: Commands,
    score_query: Single<Entity, With<ScorePunch>>,
    mut score_punch: Query<&mut ScorePunch>,
) {
    // 1) pickup animáció az érmén
    commands.spawn((
        Sprite::from_color(Color::srgba(1.0, 0.85, 0.3, 1.0), Vec2::new(12.0, 12.0)),
        Transform::from_translation(coin_pos),
        PickupAnim { timer: Timer::from_seconds(0.3, TimerMode::Once), start_pos: coin_pos },
    ));
    // 2) 6 sparkle sugarasan
    for i in 0..6 {
        let angle = (i as f32 / 6.0) * std::f32::consts::TAU;
        commands.spawn((
            Sprite::from_color(Color::srgba(1.0, 1.0, 0.6, 1.0), Vec2::new(4.0, 4.0)),
            Transform::from_translation(coin_pos),
            Sparkle {
                vel: Vec2::new(angle.cos(), angle.sin()) * 80.0,
                timer: Timer::from_seconds(0.25, TimerMode::Once),
            },
        ));
    }
    // 3) score punch
    if let Ok(mut punch) = score_punch.get_mut(*score_query) {
        punch.timer.reset();
    }
}
A „spring" görbe — miért nem lineáris?

A score punch és a pickup méret is először felfelé, aztán vissza megy. A lineáris átmenet (egyenletes növekedés, aztán egyenletes csökkenés) „mesterségesen" hat — a való világban a rugó lassulva ér a végébe. A fenti kód az t < 0.3 fázisban lineárisan felmegy (gyors „ütés"), aztán 0.3–1.0 között lineárisan vissza — ez egy leegyszerűsített spring. A valódi (easing-függvényes) változat jobban néz ki, de bonyolultabb. A Bevy bevy::math::cubic_spline vagy az ease crate segíthet.

A pickup = mini-event

A pickup polish eseményvezérelt — egy CoinCollected event indítja mind a három animációt. Ez a minta (event → több rendszer reagál) a Bevy idiomaticus módja az ilyen „szétcsapódó" reakcióknak. Az üzenetek lecke és az observerek lecke részletezi — a juice-rendszerek természetesen illeszkednek az esemény-architektúrába.

Bevy integráció: egyben a plugin

A fenti elemek mindegyike önálló rendszer. A Bevy idiomaticus módja, hogy egy plugin fogja össze őket — egy GameJuicePlugin, amit egyetlen sorral hozzáadsz az apphoz, és kész. Ez a plugin tartalmazza a screen shake-et, hit-stop-ot, squish-et, a hang-randomizációt és a pickup-rendszert. Az plugin-rendszer lecke részletezi a mintát; itt a teljes kép.

use bevy::prelude::*;

pub struct GameJuicePlugin;
impl Plugin for GameJuicePlugin {
    fn build(&self, app: &mut App) {
        app
            // erőforrások
            .init_resource::<CameraShake>()
            .init_resource::<HitStop>()
            .init_resource::<DustTimer>()
            // események
            .add_message::<ShakeEvent>()
            .add_message::<HitStopEvent>()
            // rendszerek
            .add_systems(Update, (
                read_shake_events,
                read_hitstop_events,
                apply_camera_shake,
                apply_hitstop,
                apply_squish,
                sync_visual_scale,
                squish_on_stop,
                spawn_dust_while_moving,
                tick_lifetime,
                animate_dust,
                animate_pickup,
                animate_sparkles,
                animate_score_punch,
                randomize_pitch,
            ).chain());
    }
}

A .chain() — miért fontos?

A rendszer-sorrend itt számít. Ha az apply_camera_shake előtt fut az apply_hit_stop (ami a time scale-t 0-ra állítja), a shake-rendszer leáll a hit-stop alatt — ami nem akarod (a shake épp az ütés pillanatában legyen erős). A .chain() garantálja a sorrendet. De vigyázat: ha a hit-stop megfagyasztja a virtuális időt, minden Time<Virtual>-et használó rendszer megáll — így a screen shake-nek vagy a valós időt kell használnia, vagy külön kell kezelni. Ez egy finom, de fontos architekturális döntés.

RendszerIdősíkMikor fusson
apply_hitstopvalós (Time)mindig — még a fagyás alatt is
apply_camera_shakevalósmindig — a shake pont a fagyás alatt legerősebb
player_movementvirtuáliscsak ha nem fagyott — a játékos megáll
enemy_aivirtuáliscsak ha nem fagyott — az ellenség is megáll
animate_dustvirtuálismegáll a hit-stop alatt — a por is megfagy
animate_pickupvalósmindig — a pickup animáció ne fagyjon
A valós vs virtuális osztás

A fenti tábla a kulcs: a juice-renderelési rendszerek (shake, sparkle, pickup animáció, score punch) a valós időt használják — nem fagynak a hit-stop alatt. A játékmenet (mozgás, AI, fizika) a virtuális időt — az megáll. Ez a kettőválasztás teszi, hogy a hit-stop „súlyt" ad: a játék megáll, de a vizuális flourishes (sparkle, shake) pont akkor a legaktívabbak. Ha mindkettő fagyna, az ütés üres lenne; ha egyik sem, nem lenne hit-stop.

Tippek & buktatók

Gyakori hibák

  • Túl sok juice egyszerre. A videó figyelmeztetése: ha mindent maxon kapcsolod be, szédítő lesz. A juice legyen arányos az eseménnyel — egy lépésnek ne legyen shake-e, csak porfelhő; egy találatnak shake + hit-stop + squish; egy halálnak minden maximalisan. A kontraszt a fontos, nem az abszolút mennyiség.
  • Squish a hitboxon. Ha a squish a fizika transformot módosítja, a hitbox nő/zsugorodik — a játékszabály megváltozik. Védekezés: külön VisualScale komponens gyermek entity-n; a fizika transform scale = 1 fix.
  • Hit-stop örökfagyás. Ha a hit-stop timert a virtuális idővel tickeled (ami 0), sosem jár le. Védekezés: time.delta() (valós idő) a timer tickelésére.
  • Screen shake fix irányba. Egy fix (offset_x, offset_y) rázkódás „gépies" — nem természetes. Védekezés: trauma-modell, minden frame-ben új véletlen irány.
  • Screen shake nem skálázható. Ha nincs „shake intensity" beállítás, a mozgásérzékeny játékosok nem tudják kikapcsolni. Védekezés: max_offset csúszka a beállításokban, prefers-reduced-motion tiszteletben tartása.
  • Spawn-spam. Ha minden frame-ben porfelhőt/sparkle-t spawnolsz, az ECS tele lesz, és a vizuális is kaotikus. Védekezés: timer-vezérelt spawnolás (cooldown), és a Lifetime komponens mindig takarít.
  • Hang szimmetrikus. Ugyanaz a pitch minden lejátszásnál = gépies. Védekezés: set_speed(0.9 + rand * 0.2) — ±10% elég.
  • Input buffer túl hosszú. 300+ ms buffer → „dupla dash" bugok. Védekezés: 100–200 ms; teszteld gyors gombnyomással.
  • Buffer commitmentális cselekvésnél. A nagy támadásnál a buffer miatt a játékos „véletlenül" második támadást indít. Védekezés: csak a gyors, ismétlődő cselekvéseknél (dash, kis támadás).
  • A juice büntet. Ha a screen shake miatt a játékos kiesik egy peremről, vagy a hit-stop miatt elvéti a következő támadást — a juice kárt okoz. Védekezés: a juice sosem módosítja a játéklogikát; csak vizuális/auditív/időbeli.

Checklista

  • Gyorsulás & súrlódás. A mozgás nem azonnali; külön accel/decel; átlós normalizálás.
  • Squish külön entitáson. A fizika transform scale = 1 fix; a vizuális gyermek-skála „lélegzik".
  • Spawn-cooldown. Minden gyakori effekt (por, sparkle) timer-vezérelt, nem spam.
  • Lifetime takarít. Minden rövid életű entitás Lifetime-tel; egy rendszer despawnol.
  • Screen shake trauma-modell. trauma² × max × véletlen irány; exponenciális csillapodás.
  • Screen shake skálázható. max_offset csúszka; prefers-reduced-motion tisztelet.
  • Hit-stop valós idővel tickel. A timer time.delta()-t használ, nem virtuálist.
  • Hit-stop csak a virtuális időt állítja. A juice-renderelés (shake, sparkle) a valós időt használja, tovább megy.
  • Input buffer 100–200 ms. Csak gyors, ismétlődő cselekvéseknél; nem commitmentálisnál.
  • Pitch randomizáció ±10%. Minden gyakori efektnél; hangerő ±5% kiegészítés.
  • Három hangerő-szint. Master × csatorna × saját — a játékos tudja állítani.
  • Egy plugin fogja össze. GameJuicePlugin egy sorral hozzáadható; rendszer-sorrend .chain()-nel.
◆ Hol kezdj?

Ha mindent egyszerre akarsz betenni, az túl sok. A javasolt sorrend (a legnagyobb hatás/hányadviszony szerint): (1) gyorsulás/súrlódás (a legnagyobb egyetlen különbség), (2) squish (második legnagyobb, olcsó), (3) hang pitch-randomizáció (a videó szerint a legfontosabb, és a leggyakrabban hiányzik), (4) screen shake (trauma-modell), (5) hit-stop (a legfinomabb, de a legütősebb az ütéseknél), (6) porfelhő + lábnyom, (7) input buffer, (8) pickup polish. Az első három már drámai különbséget csinál; a többi finomítás. Teszteld az egészet prefers-reduced-motion-nel is.

TL;DR

A game juice top-down összefoglaló

  • A game feel egy vizuális + auditív + időbeli réteg a fizika fölött — nem változtatja a játékszabályt, csak az érzést.
  • Gyorsulás/súrlódás: move_toward(velocity, target, accel*dt); külön accel/decel; átlós normalizálás. A max sebesség ugyanaz, csak lassabban éri el.
  • Squish & stretch: csak a sprite skálája, külön VisualScale gyermeken; rugós visszaállás; a hitbox érintetlen.
  • Porfelhő + lábnyom: rövid életű entitások Lifetime-tel; timer-vezérelt spawn, nem spam.
  • Screen shake: trauma-modell, trauma² × max × véletlen irány, exponenciális csillapodás; skálázható + kikapcsolható.
  • Hit-stop: Time<Virtual>::set_relative_speed(0) pár frame-re; a timer valós idővel tickel; csak a virtuális időt fagyasztja, a juice-renderelés megy tovább.
  • Input buffer: 100–200 ms türelmi ablak; csak gyors, ismétlődő cselekvéseknél; nem commitmentálisnál.
  • Hang: a legfontosabb (a videó szerint); set_speed(0.9 + rand*0.2) pitch-randomizáció; három hangerő-szint (master × csatorna × saját).
  • Pickup polish: érme felugrás + sparkle burst + score punch — mindegyik rövid Timer-vezérelt animáció, eseményből indítva.
  • Plugin: egy GameJuicePlugin fogja össze; rendszer-sorrend .chain()-nel; valós vs virtuális idő szétválasztása kritikus.
  • A legfőbb elv: a juice arányos az eseménnyel, nem maximális. A kontraszt a fontos — lépésnek por, találatnak shake+stop, halálnak minden.
