1. rész · Alapok

Bevezetés: két világ találkozik

Egy top-down játékban valójában két párhuzamos világ él egymás fölött, és a kettő teljesen másképp gondolkodik:

  • A tile-világ egy diszkrét rács. Minden cellának egész koordinátája van ((x: u32, y: u32)), a tartalma pedig adat: fű, fal, víz. Olcsó tárolni, gyors lekérdezni, és tökéletes ütközési térképnek.
  • Az entitás-világ folytonos. A játékos, az NPC-k, a ládák mind ECS-entitások, amelyeknek Transform-juk van: lebegőpontos világpozíció (Vec3), ami bármilyen érték lehet — két tile között is.

A tutorial gerince pontosan ez a határfelület: hogyan fordítunk a két világ között oda-vissza. Spawnoláskor tile-koordinátából világkoordinátát számolunk; ütközésnél fordítva, az entitás világpozíciójából kérdezzük le, melyik tile-okon áll éppen.

Fontos tervezési döntés: a tile-ok nem entitások ebben a megközelítésben — a tilemap egy erőforrás (adat + egyetlen renderelt réteg). Léteznek könyvtárak (pl. bevy_ecs_tilemap), ahol minden tile külön entitás; az itt tanult elvek ott is ugyanígy működnek, csak a tárolás más.

Mit építünk fel a fejezetek alatt?

  1. Koordináta-konverziós segédfüggvények (tile_to_world, world_to_tile).
  2. Játékos, NPC és statikus objektumok spawnolása marker komponensekkel.
  3. Billentyűzet-vezérelt mozgás Time-alapú delta idővel.
  4. Ütközési térkép a tilemapból, AABB-ütközés tengelyenként szétválasztva (a „falon csúszás” trükkje).
  5. Entitás–entitás közelség: NPC-interakció trigger-zónával.
  6. Y-rendezés, hogy a fa mögé be lehessen sétálni.
1. rész · Alapok

Bevy ECS gyorstalpaló

Mielőtt kódot írnánk, rögzítsük a négy alapfogalmat, amire minden további fejezet épül. Ha az ECS már ismerős, ugorhatsz a koordinátákhoz — de a dobozokra később is visszahivatkozunk.

Bevy fogalom

Entity · Component · System

Entity: egy puszta azonosító (gyakorlatilag egy index + generáció). Önmagában semmit sem tud.

Component: sima Rust struct vagy enum, amit egy entitásra „ráakasztunk” — ez az adat. A #[derive(Component)] attribútum teszi komponenssé.

System: egy közönséges Rust függvény, amelynek paraméterei megmondják a Bevy-nek, milyen adatokhoz akar hozzáférni. A Bevy ütemezője (scheduler) hívogatja őket — ahol lehet, párhuzamosan.

Bevy fogalom

Resource

Globális, egyetlen példányban létező adat — nem entitáshoz kötött. Ilyen lesz nálunk az ütközési térkép (CollisionMap), és ilyen a beépített Time is. Rendszerben Res<T> (olvasás) vagy ResMut<T> (írás) paraméterrel kérhető.

Bevy fogalom

Commands

Entitások létrehozása/törlése és komponensek hozzáadása nem történhet azonnal, miközben más rendszerek párhuzamosan futnak. A Commands ezért egy parancs-puffer: a commands.spawn((...)) hívásaid összegyűlnek, és a frame egy biztonságos pontján futnak le egyben. Ezért nem kapod vissza azonnal „a kész entitást” — csak az Entity azonosítót.

Bevy fogalom

Query

A Query<(&A, &mut B), With<C>> azt jelenti: „adj minden entitást, amelynek van A és B komponense (B-t írnám is), és van rajta C is — de a C adatára nem vagyok kíváncsi”. A With tehát csak szűrő. A query-k a Bevy lelke: ezekből tudja a scheduler, mely rendszerek futhatnak párhuzamosan.

Így néz ki mindez együtt egy minimális vázban — ezt fogjuk fejezetről fejezetre kitölteni:

src/main.rsrust
use bevy::prelude::*;

fn main() {
    App::new()
        .add_plugins(DefaultPlugins.set(ImagePlugin::default_nearest()))
        .add_plugins(ActorsPlugin) // ezt írjuk meg ebben a tutorialban
        .run();
}

pub struct ActorsPlugin;

impl Plugin for ActorsPlugin {
    fn build(&self, app: &mut App) {
        app.add_systems(Startup, (spawn_player, spawn_from_map))
            .add_systems(Update, (read_input, apply_movement, y_sort).chain());
    }
}
Bevy fogalom

Plugin, Schedule és .chain()

A Plugin a Bevy moduláris egysége: egy csomagba rendezi az összetartozó rendszereket és erőforrásokat. A Startup ütemezés egyszer fut az induláskor, az Update minden frame-ben. A .chain() sorrendet kényszerít: input → mozgás → y-rendezés, mert alapból a Bevy nem garantál sorrendet a rendszerek között.

Pixel art tipp: az ImagePlugin::default_nearest() kapcsolja ki a textúra-szűrést (bilineáris elmosást) — enélkül a pixel artod homályos lesz. Top-down pixel játéknál ez gyakorlatilag kötelező első sor.

Koordináta-rendszerek és konverzió

A Bevy 2D-ben a világ Y-tengelye felfelé nő, az origó pedig ott van, ahova te teszed (a kamera alapból a (0,0)-t nézi). A tile-rácsunkat érdemes úgy lefektetni, hogy a (0,0) tile bal alsó sarka legyen a világ origójában — így a konverzió triviális marad.

A két irányú konverzió, plusz egy TilePos típus, amit komponensként is használunk majd:

src/grid.rsrust
use bevy::prelude::*;

pub const TILE_SIZE: f32 = 16.0;

/// Diszkrét rács-pozíció. Komponensként is felkerülhet entitásokra.
#[derive(Component, Clone, Copy, PartialEq, Eq, Hash, Debug)]
pub struct TilePos {
    pub x: u32,
    pub y: u32,
}

/// Tile -> világ: a tile KÖZÉPPONTJÁT adja vissza.
pub fn tile_to_world(pos: TilePos) -> Vec2 {
    Vec2::new(
        pos.x as f32 * TILE_SIZE + TILE_SIZE * 0.5,
        pos.y as f32 * TILE_SIZE + TILE_SIZE * 0.5,
    )
}

/// Világ -> tile: melyik cellába esik egy világpont?
/// i32-t adunk vissza, mert a pont a pályán KÍVÜL is lehet (negatív index).
pub fn world_to_tile(world: Vec2) -> (i32, i32) {
    (
        (world.x / TILE_SIZE).floor() as i32,
        (world.y / TILE_SIZE).floor() as i32,
    )
}
  • Miért középpont? A spritejaink alapból középre horgonyzottak, így ha az entitás Transform-ját a tile közepére tesszük, a sprite pontosan a cellát fedi.
  • Miért floor() és nem cast? Az as i32 a nulla felé csonkít, így a -0.3 as i32 == 0 lenne — a negatív oldalon hibás cellát kapnánk. A floor() mindig lefelé kerekít, ami rács-logikához kell.
  • Miért i32 a visszatérés? Ütközésvizsgálatnál rendszeresen rákérdezünk pályán kívüli pontokra is; jobb ezt típus-szinten megengedni és az ütközési térképben kezelni.
Bevy fogalom

Transform és GlobalTransform

A Transform az entitás szülőhöz viszonyított pozíciója, forgatása és skálája; a GlobalTransform a Bevy által ebből kiszámolt világbeli érték. Gyerek-szülő hierarchia nélkül a kettő megegyezik. Te mindig a Transform-ot írod, és többnyire azt is olvasod — a GlobalTransform akkor kell, ha beágyazott entitás abszolút pozíciójára vagy kíváncsi.

1. rész · Alapok

Z-rétegek és mélység (y-sort)

2D-ben a Transform Z-koordinátája dönti el a kirajzolási sorrendet: a nagyobb Z van felül. Top-down játékban két szinten kell ezt kezelni:

RétegZ-tartományMi kerül ide
Talaj0.0a tilemap alapréteg (fű, út, víz)
Talaj-dekor1.0virág, repedés — entitás nélkül, a tilemap 2. rétege
Szereplők10.0 ± y·εjátékos, NPC-k, fák, ládák — y-rendezve
Fák lombja, tetők20.0ami mindig a szereplők fölött van
UI / effektek100.0+kijelölés, lebegő szövegek

Az y-rendezés a top-down illúzió kulcsa: aki a képernyőn lejjebb áll, az van közelebb a kamerához, tehát nagyobb Z-t kap. Így tudsz besétálni egy fa mögé. Egy apró rendszer minden frame-ben frissíti az érintett entitások Z-jét:

src/depth.rsrust
use bevy::prelude::*;

pub const Z_ACTORS: f32 = 10.0;

/// Marker: ezt az entitást y szerint kell rendezni.
#[derive(Component)]
pub struct YSorted;

pub fn y_sort(mut query: Query<&mut Transform, With<YSorted>>) {
    for mut transform in &mut query {
        // Minél kisebb az y (lejjebb áll), annál nagyobb a z.
        transform.translation.z = Z_ACTORS - transform.translation.y * 0.001;
    }
}
Horgony-tipp: y-rendezésnél a sprite-okat érdemes alsó-közép horgonyra (Anchor::BOTTOM_CENTER) állítani, így a „lábánál” hasonlítod össze a szereplőket, nem a sprite közepénél — egy magas fa és egy alacsony karakter így is jól rendeződik. (Bevy 0.17 előtt az anchor a Sprite struct mezője volt; 0.17-től önálló Anchor komponens.)

A sandboxban (4. rész) van egy fa pontosan ezért — próbáld majd ki, hogy elé és mögé is odaállsz.

A játékos spawnolása

A játékos egy entitás lesz, három saját komponenssel: egy marker (Player — üres struct, csak címkének), egy sebesség, és egy ütközőtest (ezt a 3. részben töltjük meg élettel). Ehhez jön a Bevy beépített Sprite és Transform komponense.

src/player.rsrust
use bevy::prelude::*;
use bevy::sprite::Anchor;
use crate::grid::{tile_to_world, TilePos, TILE_SIZE};
use crate::depth::{YSorted, Z_ACTORS};

#[derive(Component)]
pub struct Player; // marker: nincs adata, csak címke

#[derive(Component)]
pub struct Speed(pub f32); // pixel / másodperc

#[derive(Component)]
pub struct Collider {
    pub half: Vec2,   // fél-szélesség, fél-magasság
    pub offset: Vec2, // eltolás a Transformhoz képest
}

pub fn spawn_player(mut commands: Commands, assets: Res<AssetServer>) {
    let start = TilePos { x: 5, y: 7 };

    commands.spawn((
        Player,
        Speed(80.0),
        Collider {
            // Szándékosan KISEBB, mint a 16x16-os sprite:
            // csak a karakter "lába" ütközik, a feje belelóghat a fal tile-ba.
            half: Vec2::new(5.0, 4.0),
            offset: Vec2::new(0.0, 4.0),
        },
        Sprite {
            image: assets.load("sprites/skeleton.png"),
            ..default()
        },
        Anchor::BOTTOM_CENTER,
        Transform::from_translation(
            (tile_to_world(start) - Vec2::Y * TILE_SIZE * 0.5).extend(Z_ACTORS),
        ),
        YSorted,
    ));
}
Bevy fogalom

commands.spawn((...)) és a required components

A spawn egy tuple-t kap: minden eleme egy komponens, amit az új entitás megkap. A modern Bevy-ben (0.15+) nem kell „bundle”-öket keresgélni: a Sprite komponens required components mechanizmussal automatikusan maga mellé húzza a hiányzó Transform, Visibility stb. komponenseket alapértékekkel. Mi a Transform-ot azért adjuk meg kézzel, mert nem az alapértéket (origó) akarjuk.

Bevy fogalom

AssetServer és a Handle

Az assets.load("sprites/skeleton.png") nem blokkol: azonnal visszaad egy Handle<Image>-t (egy könnyű hivatkozást), a betöltés a háttérben történik az assets/ mappából. Amíg tölt, a sprite egyszerűen nem látszik — induló projektnél ez tökéletes, később jöhet a bevy_asset_loader a betöltőképernyőhöz.

Két apró, de fontos döntés van a fenti kódban:

  • Az ütközőtest kisebb a sprite-nál. Top-down játékokban a karakternek csak a „talplenyomata” ütközik (kb. az alsó fele) — így a feje vizuálisan átfedhet egy mögötte lévő fal-tile-lal, ami sokkal természetesebben néz ki. Ezt a sandboxban be tudod majd kapcsolva nézni.
  • Alsó-közép horgony + fél tile lefelé tolás: így a karakter lába kerül a kiválasztott tile aljára, és az y-sort is a lábpozícióval dolgozik.

NPC-k és statikus elemek

Az NPC és a láda szerkezetileg ugyanaz, mint a játékos — entitás komponensekkel —, csak más markereket és adatokat kap. Pontosan ez az ECS ereje: nem öröklődési hierarchiát építünk (GameObject → Character → NPC), hanem komponenseket komponálunk.

EntitásKomponensekMiért
JátékosPlayer, Speed, Collider, Sprite, YSortedmozog, ütközik, y-rendezett
NPCNpc, Speed, Collider, Sprite, YSorted, Namemint a játékos, de input helyett AI mozgatja
LádaChest, Collider, Sprite, YSorted, TilePosnem mozog → nincs Speed; rácshoz kötött
FaSolid, Sprite, YSorted, TilePosaz ütközését a tile-térkép kezeli (lásd 3. rész)
src/npc.rsrust
use bevy::prelude::*;
use crate::grid::{tile_to_world, TilePos};
use crate::depth::{YSorted, Z_ACTORS};
use crate::player::{Collider, Speed};

#[derive(Component)]
pub struct Npc {
    pub dialog: String,
}

#[derive(Component)]
pub struct Chest {
    pub opened: bool,
}

pub fn spawn_npc(mut commands: Commands, assets: Res<AssetServer>) {
    commands.spawn((
        Npc { dialog: "Üdv, vándor! A híd keletre van.".into() },
        Name::new("Boltos"),
        Speed(40.0),
        Collider { half: Vec2::new(5.0, 4.0), offset: Vec2::new(0.0, 4.0) },
        Sprite { image: assets.load("sprites/villager.png"), ..default() },
        Transform::from_translation(tile_to_world(TilePos { x: 12, y: 6 }).extend(Z_ACTORS)),
        YSorted,
    ));
}
Bevy fogalom

Name

A Name a Bevy beépített „címke” komponense. A játéklogikának nem kell, de a debug-eszközök (pl. bevy-inspector-egui) ezt jelenítik meg az entitáslistában — a 4096-os Entity(4096v1) helyett azt látod: Boltos. Vedd fel korán a szokást.

Statikus elem: rácshoz kötve vagy szabadon?

Statikus objektumoknál két iskola létezik, és érdemes tudatosan választani:

  • Rácshoz kötött (láda, ajtó, fa): kap egy TilePos komponenst is, és az ütközését nem AABB-vel, hanem az ütközési térképben kezeljük — spawnoláskor egyszerűen „fallá” jelöljük a celláját. Olcsó és robusztus.
  • Szabad pozíciójú (eldobott tárgy, lövedék): nincs TilePos, AABB-vel vagy körrel ütközik, mint a mozgó szereplők.

A hüvelykujjszabály: ami sosem mozdul és cellára esik, az a tile-világ ütközési adatába tartozik, még akkor is, ha vizuálisan entitásként (saját sprite-tal) jelenik meg. A két világ tehát itt is összeér: a fa sprite-ja entitás (hogy y-rendezhető legyen), az ütközése tile-adat.

Spawnolás térkép-adatból

Kézzel spawnolgatni csak prototípusban szabad. A pályagenerátorod (vagy pályaszerkesztőd) ideális esetben nemcsak tile-okat ad, hanem egy spawn-listát is: „ide játékos, oda két NPC, amoda láda”. Az LDtk és a Tiled szerkesztők ezt entity layer-nek hívják — procedurális generálásnál te magad állítod elő ugyanezt a listát.

src/map.rsrust
use bevy::prelude::*;
use crate::grid::TilePos;

#[derive(Clone, Copy, Debug)]
pub enum SpawnKind {
    Player,
    Npc,
    Chest,
    Tree,
}

pub struct MapSpawn {
    pub kind: SpawnKind,
    pub tile: TilePos,
}

/// Ezt az erőforrást a (már kész) tilemap-generátorod tölti fel.
#[derive(Resource)]
pub struct MapData {
    pub width: u32,
    pub height: u32,
    pub solid_tiles: Vec<bool>,   // soronként: width * height elem
    pub spawns: Vec<MapSpawn>,
}

A spawnoló rendszer ezután egyetlen match: adatból entitás.

src/map.rs (folytatás)rust
pub fn spawn_from_map(
    mut commands: Commands,
    map: Res<MapData>,
    assets: Res<AssetServer>,
) {
    for spawn in &map.spawns {
        match spawn.kind {
            SpawnKind::Player => spawn_player_at(&mut commands, &assets, spawn.tile),
            SpawnKind::Npc    => spawn_npc_at(&mut commands, &assets, spawn.tile),
            SpawnKind::Chest  => spawn_chest_at(&mut commands, &assets, spawn.tile),
            SpawnKind::Tree   => spawn_tree_at(&mut commands, &assets, spawn.tile),
        }
    }
}
Miért jó ez a felépítés? A pálya adat, az entitások viselkedés. Ha később LDtk-ra vagy Tiled-re váltasz (bevy_ecs_ldtk / bevy_ecs_tiled), csak a MapData feltöltése változik — a spawn-rendszered és minden, ami utána jön, marad. Ugyanez igaz visszafelé: a procedurális generátorod kimenetét elég erre az egy struktúrára illeszteni.
Bevy fogalom

Res<MapData> és a futási sorrend

A Res<MapData> paraméter miatt ez a rendszer pánikol, ha az erőforrás még nem létezik. Két megoldás: az erőforrást még az App felépítésekor szúrd be (app.insert_resource(map_data)), vagy a rendszerek közé tegyél sorrendet: spawn_from_map.after(generate_map). Opcionális erőforráshoz a paraméter Option<Res<MapData>> is lehet.

3. rész · Mozgás és ütközés

Input és mozgás

A mozgást két rendszerre bontjuk: az egyik az inputból szándékot csinál (egy irányvektort), a másik a szándékból elmozdulást — ütközésvizsgálattal. Ez a szétválasztás később ajándék: az NPC-AI ugyanabba a „szándék” komponensbe ír, mint a billentyűzet, és a mozgató rendszer nem tudja (nem is kell tudnia), ki vezérli az entitást.

src/movement.rsrust
use bevy::prelude::*;
use crate::player::Player;

/// "Szándék": merre akar menni az entitás ebben a frame-ben. (-1..1 komponensek)
#[derive(Component, Default)]
pub struct MoveIntent(pub Vec2);

pub fn read_input(
    keys: Res<ButtonInput<KeyCode>>,
    mut query: Query<&mut MoveIntent, With<Player>>,
) {
    let Ok(mut intent) = query.single_mut() else { return };

    let mut dir = Vec2::ZERO;
    if keys.pressed(KeyCode::KeyW) || keys.pressed(KeyCode::ArrowUp)    { dir.y += 1.0; }
    if keys.pressed(KeyCode::KeyS) || keys.pressed(KeyCode::ArrowDown)  { dir.y -= 1.0; }
    if keys.pressed(KeyCode::KeyA) || keys.pressed(KeyCode::ArrowLeft)  { dir.x -= 1.0; }
    if keys.pressed(KeyCode::KeyD) || keys.pressed(KeyCode::ArrowRight) { dir.x += 1.0; }

    // Átló: normalizálás nélkül √2-szer gyorsabb lenne.
    intent.0 = dir.normalize_or_zero();
}
Bevy fogalom

ButtonInput<KeyCode>

Beépített erőforrás, a Bevy minden frame-ben frissíti. Három fő kérdés tehető fel neki: pressed() — éppen lenyomva tartva; just_pressed() — ebben a frame-ben nyomták le (interakcióhoz, ugráshoz); just_released() — most engedték fel. A KeyCode fizikai billentyűt jelöl (a QWERTY „W” helye), tehát magyar kiosztáson is jó helyen lesz a WASD.

Bevy fogalom

query.single_mut() és a let-else

Ha egy query-ről tudod, hogy pontosan egy entitást érint (egy játékos van), a single_mut() kényelmes — Result-ot ad, ami hibázik, ha 0 vagy több találat van. A let Ok(..) = .. else { return } minta csendben kilép, amíg pl. a játékos még nem spawnolt. (Bevy 0.15-ben létezik a Single<T> paramétertípus is, ami ugyanezt deklaratívan oldja meg: a rendszer egyszerűen nem fut, ha nincs pont egy találat.)

Az elmozdulást a Time-ból számoljuk — még ütközés nélkül (azt a következő fejezet adja hozzá):

src/movement.rs (első verzió, még ütközés nélkül)rust
use crate::player::Speed;

pub fn apply_movement(
    time: Res<Time>,
    mut query: Query<(&mut Transform, &MoveIntent, &Speed)>,
) {
    for (mut transform, intent, speed) in &mut query {
        let delta = intent.0 * speed.0 * time.delta_secs();
        transform.translation += delta.extend(0.0);
    }
}
Bevy fogalom

Time és a delta idő

A time.delta_secs() az előző frame óta eltelt idő másodpercben. A sebesség × delta szorzat teszi a mozgást képkockasebesség-függetlenné: 60 és 144 FPS-en ugyanannyi pixel/másodperc lesz a tempó. Determinisztikusabb fizikához létezik a FixedUpdate ütemezés (fix, pl. 64 Hz-es lépésköz) — egy egyszerű top-down játéknak az Update + delta idő bőven elég.

Az ütközési térkép

Itt fizetődik ki igazán a tile-világ. Nem kell minden falnak külön ütközőtest-entitás: a tilemapból egyszer legeneráljuk a tömör cellák bittérképét, és onnantól bármely világpontról O(1) időben megmondható, falban van-e.

src/collision.rsrust
use bevy::prelude::*;
use crate::grid::TILE_SIZE;
use crate::map::MapData;

#[derive(Resource)]
pub struct CollisionMap {
    pub width: u32,
    pub height: u32,
    pub solid: Vec<bool>, // soronként tárolva: index = y * width + x
}

impl CollisionMap {
    pub fn from_map(map: &MapData) -> Self {
        Self {
            width: map.width,
            height: map.height,
            solid: map.solid_tiles.clone(),
        }
    }

    /// Tömör-e az (x, y) tile? A pályán kívül MINDEN tömör –
    /// így a pálya széle automatikusan fal, külön határellenőrzés nélkül.
    pub fn is_solid(&self, x: i32, y: i32) -> bool {
        if x < 0 || y < 0 || x >= self.width as i32 || y >= self.height as i32 {
            return true;
        }
        self.solid[(y as u32 * self.width + x as u32) as usize]
    }

    /// Tömör tile alá esik-e egy világbeli PONT?
    pub fn is_solid_at_world(&self, p: Vec2) -> bool {
        let tx = (p.x / TILE_SIZE).floor() as i32;
        let ty = (p.y / TILE_SIZE).floor() as i32;
        self.is_solid(tx, ty)
    }
}
  • „Pályán kívül = fal” — ez az apró döntés rengeteg határeset-hibától véd meg, és a játékos sosem eshet le a világ pereméről.
  • Az erőforrást a térkép elkészülte után egyszer hozzuk létre: commands.insert_resource(CollisionMap::from_map(&map)). Ha a pálya futás közben változhat (ajtó kinyílik, fát kivágnak), ugyanitt kell átbillenteni a megfelelő cellát — pontosan ezt csinálja a sandbox, amikor kattintással falat raksz le.
  • Nagyobb pályán a Vec<bool> cserélhető bitmaszkra vagy HashSet<(i32,i32)>-re ritkás falaknál — az interfész (is_solid) marad.

AABB a tile-rács ellen

A szereplőink ütközőteste egy AABB (axis-aligned bounding box — tengelyekkel párhuzamos téglalap). Az ütközésvizsgálat ötlete egyszerű: nézzük meg, hogy az AABB által lefedett összes tile között van-e tömör. Mivel az ütközőtest kisebb egy tile-nál vagy alig nagyobb, ez tipikusan 1–4 cella ellenőrzése.

src/collision.rs (folytatás)rust
/// Ütközik-e egy AABB (középpont + fél-méretek) bármely tömör tile-lal?
pub fn aabb_hits_solid(map: &CollisionMap, center: Vec2, half: Vec2) -> bool {
    let min = center - half;
    let max = center + half;

    // A lefedett cellatartomány. A max oldalon egy pici epsilont levonunk,
    // hogy a pontosan cellahatáron álló doboz ne "lógjon át" a szomszédba.
    let x0 = (min.x / TILE_SIZE).floor() as i32;
    let y0 = (min.y / TILE_SIZE).floor() as i32;
    let x1 = ((max.x - 0.001) / TILE_SIZE).floor() as i32;
    let y1 = ((max.y - 0.001) / TILE_SIZE).floor() as i32;

    for ty in y0..=y1 {
        for tx in x0..=x1 {
            if map.is_solid(tx, ty) {
                return true;
            }
        }
    }
    false
}

A „falon csúszás” trükkje: tengelyek szétválasztása

Ha az elmozdulást egyben próbálnánk (és ütközésnél egyszerűen eldobnánk), a játékos átlósan nekifutva a falnak megállna — pedig az elvárt viselkedés, hogy a fal mentén csússzon. A megoldás meglepően egyszerű: az X és Y elmozdulást külön-külön próbáljuk meg. Ha X-ben ütköznénk, csak az X-et dobjuk el; az Y még átmehet.

src/movement.rs (végleges)rust
pub fn apply_movement(
    time: Res<Time>,
    map: Res<CollisionMap>,
    mut query: Query<(&mut Transform, &MoveIntent, &Speed, &Collider)>,
) {
    for (mut transform, intent, speed, collider) in &mut query {
        let delta = intent.0 * speed.0 * time.delta_secs();
        let mut pos = transform.translation.truncate();

        // 1) X tengely külön
        let try_x = pos + Vec2::new(delta.x, 0.0);
        if !aabb_hits_solid(&map, try_x + collider.offset, collider.half) {
            pos = try_x;
        }

        // 2) Y tengely külön (a már frissített X-szel!)
        let try_y = pos + Vec2::new(0.0, delta.y);
        if !aabb_hits_solid(&map, try_y + collider.offset, collider.half) {
            pos = try_y;
        }

        transform.translation.x = pos.x;
        transform.translation.y = pos.y;
    }
}
Korlát, amit érdemes ismerni: ez az „elmozdulok, ha nem ütköznék” módszer diszkrét — nagyon nagy sebességnél (egy frame alatt egy teljes tile-nyi út) az entitás átugorhatna egy vékony falon (tunneling). 16 px-es tile-nál és ~100 px/s sebességnél ez 60 FPS-en kizárt (100/60 ≈ 1,7 px/frame), de ha valaha lövedékeket vagy dash-t adsz hozzá, az elmozdulást több kis lépésre kell bontani, vagy sugárral (raycast) ellenőrizni.
Bevy fogalom

Change Detection — ingyen kapott optimalizáció

A Bevy minden komponens-írást nyilvántart. A Query<.., Changed<Transform>> szűrővel egy rendszer csak azokon az entitásokon fut, amelyek Transformja tényleg változott — a fenti y_sort például így csak a mozgó szereplőket rendezné újra. Fontos finomság: a &mut hozzáférés már önmagában „changed”-nek jelöl, ezért a forró ciklusokban csak akkor kérj &mut-ot, ha tényleg írsz.

Entitás–entitás ütközés és trigger-zónák

A tile-ütközés a világ ellen véd; az entitások egymás elleni vizsgálata külön kérdés, és top-down játékban tipikusan két, eltérő célú formája van:

  • Tömör ütközés (ne sétáljunk át az NPC-n): két AABB átfedés-vizsgálata, és a mozgás blokkolása.
  • Trigger-zóna (elég közel vagyok, hogy beszéljek vele?): átfedés vagy távolság-vizsgálat, ami nem blokkol, csak eseményt vált ki.

Két AABB átfedése egysoros logika — akkor fedik egymást, ha mindkét tengelyen átfednek:

src/collision.rs (folytatás)rust
pub fn aabb_overlap(ca: Vec2, ha: Vec2, cb: Vec2, hb: Vec2) -> bool {
    (ca.x - cb.x).abs() < (ha.x + hb.x) &&
    (ca.y - cb.y).abs() < (ha.y + hb.y)
}

Az NPC-interakcióhoz viszont kör (távolság) alapú trigger a természetes — a játékos „aurája”. Eseményt küldünk, amit egy másik rendszer (a dialógus-UI) dolgoz fel:

src/interact.rsrust
use bevy::prelude::*;
use crate::npc::Npc;
use crate::player::Player;

const INTERACT_RADIUS: f32 = 20.0;

/// Bevy 0.17+ neve: Message (#[derive(Message)], MessageWriter/MessageReader).
/// Bevy 0.16-ban és korábban: #[derive(Event)], EventWriter/EventReader.
#[derive(Message)]
pub struct InteractWithNpc {
    pub npc: Entity,
}

pub fn try_interact(
    keys: Res<ButtonInput<KeyCode>>,
    player_q: Query<&Transform, With<Player>>,
    npc_q: Query<(Entity, &Transform), With<Npc>>,
    mut writer: MessageWriter<InteractWithNpc>,
) {
    if !keys.just_pressed(KeyCode::KeyE) {
        return;
    }
    let Ok(player) = player_q.single() else { return };
    let p = player.translation.truncate();

    // A legközelebbi, sugáron belüli NPC-t keressük.
    let closest = npc_q
        .iter()
        .map(|(e, t)| (e, t.translation.truncate().distance_squared(p)))
        .filter(|(_, d2)| *d2 < INTERACT_RADIUS * INTERACT_RADIUS)
        .min_by(|a, b| a.1.total_cmp(&b.1));

    if let Some((entity, _)) = closest {
        writer.write(InteractWithNpc { npc: entity });
    }
}

pub fn handle_interaction(
    mut reader: MessageReader<InteractWithNpc>,
    npc_q: Query<(&Npc, &Name)>,
) {
    for msg in reader.read() {
        if let Ok((npc, name)) = npc_q.get(msg.npc) {
            info!("{}: {}", name, npc.dialog); // itt jönne a dialógus-UI
        }
    }
}

Üzenetek (Message) és események (Event)

A pufferelt üzenet a rendszerek közti laza csatolás eszköze: az író rendszer nem tud a feldolgozóról. Regisztrálni kell: app.add_message::<InteractWithNpc>(). Az üzenetek két frame-ig élnek, az olvasó saját kurzorral jár (minden olvasó mindent megkap egyszer).

Verzió-térkép: Bevy ≤0.16: Event + EventWriter/EventReader + add_event. Bevy 0.17-től a pufferelt változat átnevezve Message-re, az Event név pedig a megfigyelőknek (observer) maradt: az commands.trigger(..) + app.add_observer(..) páros azonnali, reaktív kézbesítést ad pufferelés helyett. Egyszeri reakciókhoz (láda kinyílt) az observer elegánsabb; frame-enként sokszor előforduló jelzésekhez az üzenet hatékonyabb.

distance_squared? A gyökvonás megspórolható: ha d² < r², akkor d < r. Apróság, de jól mutatja az elvet — a sugaras trigger zónánál soha nincs szükség a tényleges távolságra, csak az összehasonlításra.

És ha az NPC tömör?

Ha az NPC-n nem lehet átsétálni, a tengely-szétválasztós mozgásba az aabb_hits_solid mellé egy második feltétel kerül: „és nem fedek át más Collider-es entitással”. Ehhez a mozgató rendszer egy extra query-vel végigmegy a többi ütközőtesten. Néhány tucat szereplőig a naiv O(n²) is bőven elég; ha száz fölé mész, jön a térbeli rácsozás (spatial hash) — vagy a következő fejezet.

Összefoglalás és következő lépések

A teljes kép, amit felépítettünk — érdemes egyszer fejben végigfuttatni egy frame életét:

  1. Startup: a térképadatból CollisionMap erőforrás készül, a spawn-lista entitásokká válik (Commands).
  2. Update / input: a read_input a billentyűzetből MoveIntent-et ír.
  3. Update / mozgás: az apply_movement a szándékból delta-idős elmozdulást számol, és tengelyenként, AABB-vel a tile-rács ellen validálja.
  4. Update / interakció: távolság-trigger üzenetet küld, másik rendszer feldolgozza.
  5. Update / mélység: a y_sort a lábpozícióból Z-t számol.

Animáció

A MoveIntent-ből irány és „mozog-e” állapot olvasható — ebből vezérelhető a sprite-lap (TextureAtlas) animáció. Aseprite-hoz: bevy_aseprite_ultra.

Kamera-követés

Egy rendszer, ami a kamera Transformját a játékoséhoz simítja (lerp), a pálya széleire clampelve — minden hozzávalót ismersz már hozzá.

NPC-mozgás

Az NPC ugyanazt a MoveIntent-et írja, csak nem inputból, hanem pl. véletlen sétából vagy útkeresésből (A* a tile-rácson — az ütközési térkép ehhez is kész adat).

Pályaváltás

States + a pálya entitásainak közös marker komponense, amit váltáskor despawnolsz — a spawn-from-data felépítés miatt az új pálya betöltése ugyanaz a kódút.
