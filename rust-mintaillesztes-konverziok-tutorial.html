A két téma, ami a Rust-alapok és a haladó fejezetek közt a leggyakrabban előkerül — és amit a legtöbb tananyag épp csak érint. Ez a kiegészítő pontosan ezt a hézagot tölti ki.

0x00

A minták mindenhol ott vannak

A nagy tananyagban a match-et úgy ismerted meg, mint elágazást. De a mintaillesztés ennél jóval több: a Rustban a minta egy önálló nyelvi elem, ami sok helyen felbukkan. Valójában már az első naptól használtad — csak nem hívtad így.

src/main.rs
fn main() {
    // Ez már mintaillesztés: a bal oldal MINTA, nem csak egy név.
    let (x, y) = (10, 20);
    let [a, b, c] = [1, 2, 3];
    let Pont { x: px, y: py } = Pont { x: 3, y: 4 };

    println!("{x} {y} | {a} {b} {c} | {px} {py}");
    osszeg((px, py)); // a függvény-paraméter HELYE is minta
}

struct Pont { x: i32, y: i32 }

// A paraméter helyén egy tuple-mintát bontunk szét két névre:
fn osszeg((a, b): (i32, i32)) {
    println!("összeg: {}", a + b);
}

Két fajta mintát különböztetünk meg, és ez a megkülönböztetés végigkíséri a fejezetet:

  • Cáfolhatatlan (irrefutable) minta — mindig illeszkedik. Ezek mehetnek let bal oldalára, függvény-paraméterbe, for-ciklusba.
  • Cáfolható (refutable) minta — el is bukhat. Ilyen a Some(n): ha az érték None, nem illeszkedik. Ezek a match ágaiba, az if let-be és a while let-be valók.
Miért érdekes ez?

Mert megmagyarázza a fordító hibáit. Ha let Some(n) = opcio;-t írsz, a fordító elutasítja: „refutable pattern in local binding". Egy sima let ugyanis cáfolhatatlan mintát vár — ami sosem bukik el. A megoldás épp a következő fejezet: if let vagy let ... else.

0x01

if let, while let, let else

A match kimerítő: minden ágat le kell írni. Sokszor viszont csak egyetlen eset érdekel. Erre való ez a három tömör forma.

if let — „ha illeszkedik, csináld"

src/main.rs
let beallitas: Option<i32> = Some(5);

// match-csel: a None ágat KÖTELEZŐ kiírni, pedig nem is érdekel.
match beallitas {
    Some(n) => println!("érték: {n}"),
    None => {}
}

// if let: csak az érdekes ágat írjuk le.
if let Some(n) = beallitas {
    println!("érték: {n}");
} else {
    println!("nincs érték"); // az else nem kötelező
}
Hüvelykujjszabály

Ha egy esetre reagálnál, és a többit hagynád — if let. Ha több esetet kezelsz külön, és számít, hogy egyik se maradjon ki — match. Az if let ugyanis nem ellenőrzi a kimerítőséget.

while let — „amíg illeszkedik, ismételd"

Tökéletes gyűjtemények ürítésére: addig fut, amíg a minta illeszkedik, és az első nem-illeszkedésnél magától leáll.

src/main.rs
let mut verem = vec![1, 2, 3, 4, 5];

// A pop() Option-t ad: Some(érték), amíg van elem, utána None.
while let Some(tetel) = verem.pop() {
    println!("kivettem: {tetel}");
}
println!("a verem üres: {verem:?}");
kivettem: 5 kivettem: 4 kivettem: 3 kivettem: 2 kivettem: 1 a verem üres: []

let else — „bontsd ki, vagy lépj ki"

Ez a legfiatalabb a háromból, és a leghasznosabb. Az if let baja, hogy a kibontott érték csak a blokkon belül él. A let ... else fordítva működik: a sikeres ágban kibontott változó a blokk után is látható — az else ágnak viszont ki kell lépnie (return, break, continue vagy panic).

src/main.rs
fn elso_szam(sor: &str) -> i64 {
    // Ha nem szám, az else ág kilép. Ha igen, `n` innentől sima i64.
    let Ok(n) = sor.trim().parse::<i64>() else {
        eprintln!("nem szám: {sor:?}");
        return -1;
    };

    n * 2 // nincs beljebb húzva: `n` itt is él, és nem Result, hanem i64
}
A „pyramid of doom" ellenszere

A let ... else nélkül a hibaágak egyre beljebb tolják a fő logikát (egymásba ágyazott if let-ek). Vele a hibaeseteket sorban „letudod" az elején (early return), és a függvény törzse lapos marad. Ez az egyik leggyakoribb modern Rust-idióma.

Bónusz: let-láncok (Rust 1.88+ / 2024 edition)

Újabb Rusttal több let és feltétel fűzhető össze egyetlen if-ben &&-del:

src/main.rs
let x: Option<i32> = Some(2);
let y: Option<i32> = Some(5);

if let Some(a) = x
    && let Some(b) = y
    && a < b
{
    println!("{a} kisebb, mint {b}");
}

0x02

Őrfeltételek és tartományok

Match-őr (guard): if egy ágon belül

Egy minta után if-fel további feltételt szabhatsz. Az ág csak akkor választódik, ha a minta és a feltétel is teljesül.

src/main.rs
let pont = (0, -3);

match pont {
    (0, 0)          => println!("origó"),
    (0, y) if y < 0 => println!("y-tengely, lefelé: {y}"),
    (0, y)          => println!("y-tengely, felfelé: {y}"),
    (x, 0)          => println!("x-tengely: {x}"),
    _               => println!("általános pont"),
}
Az őr nem számít a kimerítőségbe

A fordító nem tudja kiértékelni az if feltételt, ezért egy őrzött ágat nem tekint teljesnek. Ha minden ágad őrzött, kelleni fog egy minta-fedő (_) ág a végére, különben „non-exhaustive patterns" hibát kapsz.

Tartomány- és vagy-minták

Számokra és karakterekre tartományt (..=) illeszthetsz, több értéket pedig a | („vagy") köt össze egyetlen ágban.

src/main.rs
let jel = 'q';
match jel {
    'a'..='z' => println!("kisbetű"),
    'A'..='Z' => println!("nagybetű"),
    '0'..='9' => println!("számjegy"),
    _         => println!("egyéb karakter"),
}

let n = 2;
match n {
    1 | 2 | 3 => println!("kicsi"),   // vagy-minta: 1, 2 vagy 3
    4..=6     => println!("közepes"), // tartomány: 4-től 6-ig bezárólag
    _         => println!("nagy"),
}

Megoldás
megoldás

0x03

Destrukturálás

A mintaillesztés igazi ereje a mélységi szétbontás: egyetlen mintával belenyúlsz egy beágyazott szerkezetbe, és pont azokat a mezőket kötöd névhez, amik kellenek.

Beágyazott struct és enum egyszerre

src/main.rs
struct Pont { x: i32, y: i32 }

enum Alak {
    Kor { kozep: Pont, sugar: f64 },
    Szakasz(Pont, Pont),
}

fn leiras(alak: &Alak) {
    match alak {
        // a Kör belsejében a Pont-ot is azonnal szétbontjuk x-re és y-ra:
        Alak::Kor { kozep: Pont { x, y }, sugar } =>
            println!("kör, közép ({x}, {y}), sugár {sugar}"),

        Alak::Szakasz(Pont { x: x1, y: y1 }, Pont { x: x2, y: y2 }) =>
            println!("szakasz ({x1},{y1}) → ({x2},{y2})"),
    }
}

A maradék elnyelése: .. és _

A _ egyetlen értéket dob el (és nem köti meg). A .. a maradék mezőket nyeli el — struktúrában, tuple-ben és tömbben is.

src/main.rs
struct Ablak { szelesseg: u32, magassag: u32, cim: String, teljes: bool }

let a = Ablak {
    szelesseg: 1920, magassag: 1080,
    cim: String::from("Játék"), teljes: true,
};

// Csak a méret kell — a többi mezőt a `..` elnyeli:
let Ablak { szelesseg, magassag, .. } = &a;
println!("{szelesseg} x {magassag}");

// Tuple közepének kihagyása:
let szamok = (1, 2, 3, 4, 5);
let (elso, .., utolso) = szamok;
println!("{elso} .. {utolso}"); // 1 .. 5

// Egyetlen pozíció eldobása a _ jellel:
let (_, kozepso, _) = ('a', 'b', 'c');
println!("{kozepso}"); // b
Hol jön ez elő a Bevyben?

Folyton. Egy lekérdezés iterálásakor tuple-t kapsz, amit destrukturálsz: for (transform, sebesseg) in &query { ... }. Az eseménykezelők, a Res/ResMut kicsomagolása, a Vec2 { x, y } szétszedése — mind ez a minta.

0x04

@-kötés és ref

@ — kötés és ellenőrzés egy lépésben

Néha egy értéket egyszerre szeretnél ellenőrizni (illik-e egy tartományba) és megtartani (kelleni fog a névvel). A @ operátor pontosan ezt teszi: a bal oldalon a név, a jobb oldalon a minta, amit közben ellenőriz.

src/main.rs
enum Esemeny {
    Kattintas { x: i32, y: i32 },
}

let e = Esemeny::Kattintas { x: 5, y: 40 };

match e {
    // xkord MEGKAPJA az értéket, ÉS közben ellenőrizzük, hogy 0..=10 közé esik:
    Esemeny::Kattintas { x: xkord @ 0..=10, y } =>
        println!("bal sávban, x={xkord}, y={y}"),

    Esemeny::Kattintas { x, y } =>
        println!("máshol: ({x}, {y})"),
}

ref — amikor régi kódban találkozol vele

A modern Rust („match ergonomics") magától eldönti, hogy referenciát kössön-e: ha &-re illesztesz, a kötések automatikusan referenciák lesznek, és nem mozgatsz el semmit. Régebbi kódban viszont látni fogod a ref kulcsszót, ami ugyanezt kérte kézzel.

src/main.rs
let szoveg = Some(String::from("szia"));

// MODERN: &-re illesztünk → s automatikusan &String lesz, nincs mozgatás.
match &szoveg {
    Some(s) => println!("hossz: {}", s.len()),
    None    => println!("üres"),
}
println!("még megvan: {szoveg:?}"); // nem költöztettük el

// RÉGEBBI kód ugyanerre — a `ref` kézzel kér referenciát az érték elmozgatása helyett:
match szoveg {
    Some(ref s) => println!("hossz: {}", s.len()),
    None        => {}
}
Miért fontos érteni?

A String nem Copy. Ha érték szerint illesztenéd (match szoveg { Some(s) => ... } a & nélkül), akkor a String kiköltözne a szoveg-ből, és utána nem használhatnád. A &szoveg-re illesztés (vagy a régi ref) ezt kerüli meg — kölcsönöz, nem költöztet. Ez a tulajdonjog és a mintaillesztés metszéspontja.

0x05

From és Into

Itt kezdődik a második rész. A Rust egyik legelegánsabb mintája, hogy a típusok közti átalakítást két párba állított trait szabványosítja: a From és az Into. A trükk: csak a From-ot írod meg, az Into-t ingyen kapod.

src/main.rs
struct Celsius(f64);
struct Fahrenheit(f64);

// Csak a From-ot írjuk meg: "Celsius-ból hogyan lesz Fahrenheit".
impl From<Celsius> for Fahrenheit {
    fn from(c: Celsius) -> Self {
        Fahrenheit(c.0 * 9.0 / 5.0 + 32.0)
    }
}

fn main() {
    // explicit, From-mal:
    let f = Fahrenheit::from(Celsius(100.0));
    println!("{}", f.0); // 212

    // ...az Into INGYEN jár — a fordító a From-ból származtatja:
    let f2: Fahrenheit = Celsius(0.0).into();
    println!("{}", f2.0); // 32
}
Aranyszabály

Mindig a From-ot implementáld, soha közvetlenül az Into-t. A standard könyvtárban van egy „blanket impl", ami minden From-ból automatikusan legyártja a megfelelő Into-t. Az .into() cserébe gyakran tömörebb, mert a célt a típuskövetkeztetés adja.

Into a függvény-fejlécben: rugalmas API

Ha egy függvény impl Into<T>-t fogad, akkor a hívó bármilyen típust átadhat, amiből T lesz — így egyetlen függvény több bemeneti típust is kiszolgál.

src/main.rs
// Bármi elfogadható, amiből Fahrenheit lesz — Celsius IS, Fahrenheit IS:
fn melegszik(ho: impl Into<Fahrenheit>) -> f64 {
    ho.into().0
}

fn main() {
    println!("{}", melegszik(Celsius(37.0)));    // 98.6
    println!("{}", melegszik(Fahrenheit(72.0))); // 72  — minden típus konvertálható ÖNMAGÁRA
}

A második hívás azért működik, mert a standard könyvtárban ott a From<T> for T identitás-implementáció: minden típus átalakítható önmagává. Ezért nem kell külön túlterhelni a függvényt.

0x06

TryFrom és TryInto

A From feltételezi, hogy az átalakítás sosem bukhat el. De mi van, ha igen? Egy i32-ből nem mindig lesz érvényes életkor. Ilyenkor a TryFrom a megoldás: ugyanaz, mint a From, csak Result-ot ad vissza.

src/main.rs
struct Eletkor(u8);

impl TryFrom<i32> for Eletkor {
    type Error = String; // mi a hiba típusa, ha nem sikerül

    fn try_from(ertek: i32) -> Result<Self, Self::Error> {
        if (0..=130).contains(&ertek) {
            Ok(Eletkor(ertek as u8))
        } else {
            Err(format!("érvénytelen életkor: {ertek}"))
        }
    }
}

fn main() {
    match Eletkor::try_from(25) {
        Ok(k)  => println!("rendben: {}", k.0),
        Err(e) => println!("hiba: {e}"),
    }

    // TryInto is ingyen jár, a TryFrom-ból:
    let probal: Result<Eletkor, _> = 200.try_into();
    println!("200 érvényes? {}", probal.is_ok()); // false

    // A std is tele van vele: szűkítő szám-átalakítás try_from-mal.
    println!("{:?}", u8::try_from(300_i32)); // Err(TryFromIntError(..))
}
From vagy TryFrom?

Kérdezd meg: „Van olyan bemenet, amire ennek nincs értelmes eredménye?" Ha nincs (pl. i32 → i64, mindig belefér) → From. Ha van (pl. i64 → i32, túlcsordulhat; szöveg → szám, lehet érvénytelen) → TryFrom. A Result rákényszerít, hogy a hívó kezelje a kudarcot.

0x07

A ? és a From: hibakonverzió

Ez a fejezet köti össze a két részt — és ez a leggyakorlatibb. A nagy tananyagból tudod, hogy a ? operátor „továbbpasszolja" a hibát. De hogyan illik egy ParseIntError a saját AppHiba típusodba? A válasz: a ? automatikusan meghívja a From-ot a hibán.

src/main.rs
use std::num::ParseIntError;

#[derive(Debug)]
enum AppHiba {
    UresSor,
    RosszSzam(ParseIntError),
}

// EZ a kulcs: a ? operátor ezt hívja meg, hogy a ParseIntError-ból AppHiba legyen.
impl From<ParseIntError> for AppHiba {
    fn from(e: ParseIntError) -> Self {
        AppHiba::RosszSzam(e)
    }
}

fn elso_szam(sor: &str) -> Result<i32, AppHiba> {
    // ok_or: Option → Result, a SAJÁT hibánkkal, ha None.
    let mezo = sor.split(',').next().ok_or(AppHiba::UresSor)?;
    // a parse() ParseIntError-t ad; a ? a From-mal AppHibá-vá alakítja.
    let n: i32 = mezo.trim().parse()?;
    Ok(n)
}
Mivé bővül a ?

A mezo.trim().parse()? nagyjából ezzé:

koncepció
match mezo.trim().parse() {
    Ok(v)  => v,
    Err(e) => return Err(From::from(e)), // <-- itt hívódik a From!
}

Ezért elég, ha a hibatípusod implementálja a From<ForrásHiba>-t minden olyan hibára, amit a ?-tal továbbítani akarsz.

A gyors út: Box<dyn Error>

Alkalmazásokban (nem könyvtárakban) gyakran nem akarsz saját enumot. A Box<dyn Error> azt jelenti: „bármilyen hiba belefér". A ? ekkor minden szabványos hibát ebbe dobozol.

src/main.rs
use std::error::Error;

fn elso_szam(sor: &str) -> Result<i32, Box<dyn Error>> {
    let mezo = sor.split(',').next().ok_or("üres sor")?; // &str → Box<dyn Error>
    let n: i32 = mezo.trim().parse()?;                   // ParseIntError → Box<dyn Error>
    Ok(n)
}

A profi út: thiserror és anyhow

A gyakorlatban a kézi From-okat egy makró írja meg helyetted. Könyvtárhoz a thiserror (pontos, beszédes hibatípus), alkalmazáshoz az anyhow (egy mindent elnyelő hibatípus + kontextus).

terminál
cargo add thiserror   # könyvtárhoz
cargo add anyhow      # alkalmazáshoz
src/main.rs — thiserror
use thiserror::Error;

#[derive(Error, Debug)]
enum AppHiba {
    #[error("üres sor")]
    UresSor,

    // a #[from] LEGENERÁLJA a From<ParseIntError> impl-t — nem kell kézzel megírni!
    #[error("rossz szám: {0}")]
    RosszSzam(#[from] std::num::ParseIntError),
}
Most kapcsolódott össze minden

A ? a From-ra épül. A #[from] attribútum a From-ot generálja. Vagyis amit a 0x05-ben mint elvont mintát tanultál, az itt válik a mindennapi hibakezelés motorjává. Ez a Rust egyik legszebb „minden összeér" pillanata.

⚙ Gyakorlat 0x07

Készíts enum CliHiba-t két változattal: HianyzoArgumentum és RosszSzam(ParseIntError). Implementáld rá kézzel a From<ParseIntError>-t, majd írj fn parse_kor(args: &[String]) -> Result<u32, CliHiba> függvényt, amely a második argumentumot számmá alakítja a ?-tal (ha nincs ennyi argumentum, HianyzoArgumentum).

Megoldás
megoldás
use std::num::ParseIntError;

#[derive(Debug)]
enum CliHiba {
    HianyzoArgumentum,
    RosszSzam(ParseIntError),
}

impl From<ParseIntError> for CliHiba {
    fn from(e: ParseIntError) -> Self {
        CliHiba::RosszSzam(e)
    }
}

fn parse_kor(args: &[String]) -> Result<u32, CliHiba> {
    let masodik = args.get(1).ok_or(CliHiba::HianyzoArgumentum)?;
    let kor: u32 = masodik.trim().parse()?; // a ? a From-mal CliHibá-vá alakít
    Ok(kor)
}
0x08

AsRef, parse és a String-térkép

AsRef — „kölcsönkérek belőled egy referenciát"

A From/Into tulajdont alakít át (elfogyasztja a bemenetet). Az AsRef ezzel szemben olcsó, kölcsönzött referenciát ad. A leggyakoribb haszna: egy függvény elfogadjon &str-t, String-et és &String-et is, anélkül hogy másolatot kérne.

src/main.rs
// impl AsRef<str>: &str, String, &String — mind átadható, másolás nélkül.
fn betuk_szama(s: impl AsRef<str>) -> usize {
    s.as_ref().chars().count()
}

fn main() {
    println!("{}", betuk_szama("körte"));              // 5
    println!("{}", betuk_szama(String::from("alma"))); // 4
}

Ezért fogad el a std „bármilyen útvonalat" a fájlműveleteknél: a std::fs::read és társai impl AsRef<Path>-et várnak, így a &str, a String és a PathBuf mind működik.

parse — szövegből bármilyen típus

A &str::parse a FromStr trait-re épül. A célt a bal oldali típus vagy egy „turbofish" adja meg.

src/main.rs
fn main() {
    let n: i32   = "42".parse().unwrap();        // típus a bal oldalról
    let m        = "42".parse::<i32>().unwrap();  // vagy turbofish-sel
    let pi: f64  = "3.14".parse().unwrap();
    let ok: bool = "true".parse().unwrap();

    println!("{n} {m} {pi} {ok}");
    // Saját típusra is: ha implementálod a FromStr-t, a .parse() rá is működni fog.
}

to_string vs. to_owned vs. into

&str → String esetén mindegyik jó. A különbség az eredetükben van: a to_owned a „kölcsönzöttből tulajdont" (ToOwned) mintából jön, a to_string a Display-ből, az into a From-ból. Ha bizonytalan vagy: szövegre to_string(), és kész.

A ? operátor a From-ra épül. Ezt az egy összefüggést ha megjegyzed, a Rust hibakezelése egy csapásra logikussá válik — és a thiserror/anyhow sem varázslat többé, csak kényelem ugyanezen elv fölött.

Merre tovább?

Ez a kiegészítő a magyar Rust-tananyag mellé készült. A sorozat következő, tervezett hiánypótló darabjai:

  • String, &str és az iterátor-eszköztár a gyakorlatban — filter_map, flat_map, collect::<Result<…>>, UTF-8 buktatók, és mikor melyik szövegtípus a helyes.
  • „A fordító a barátod" — a borrow checker hibaüzeneteinek olvasása (E0382, E0499, E0502…), és a hibakeresési munkafolyamat (dbg!, tracing, debugger).
