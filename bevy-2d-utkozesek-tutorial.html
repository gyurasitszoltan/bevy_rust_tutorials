00

Miért kell ütközés? Detektálás vs. válasz

Ütközés nélkül a játékos átsétál a falon, az ellenfél egymásba olvad, az érme sosem kerül a zsebbe. De „az ütközés" valójában két, élesen különböző feladat, és a legtöbb zűrzavar abból ered, hogy összemossák őket:

  • Detektálás (broad + narrow phase). Átfed-e A és B? Ez tisztán geometria: egy igen/nem (és néha: mennyivel). A broad phase gyorsan kiszórja a lehetetlen párokat, a narrow phase pontosan eldönti a maradékot.
  • Válasz (response). Mit kezdünk az átfedéssel? Toljuk ki a testet? Csússzon a fal mentén? Pattanjon? Vagy csak jelezzünk (sebződés, pickup), és semmi fizikai? Ez már játéktervezői döntés, nem geometria.

Egy top-down játékban a válasz a fontosabb és a kényesebb. A „falon nem megyek át" könnyű; a „falnak ütközve elcsúszom a mentén, nem ragadok be a sarokba" — ez különbözteti meg a jó kontrollt a bosszantótól. A 06. szakasz erről szól, és ez a lecke centrumdarabja.

A lecke gerince

Végig ezt a két szálat fonjuk: előbb a detektálás primitíveit építjük fel (01–02) és a Bevy bounding-eszköztárat (03–04), aztán a választ rétegezzük rá — kiszorítás (05), csúszás (06), átlós/sarok-esetek (07), statikus/dinamikus szerepek (08), ütközési rétegek (09), szenzorok (10). Utána jön a méretezés: csemperács (11), broad phase (12), gyors testek / CCD (13), majd a Bevy-integráció: üzenetek vs observerek (14), rendszer-sorrend (15) és egy komplett, futtatható minta (16).

Miért kézzel, ha van avian/rapier?

Egy top-down arcade-mozgáshoz a kézzel írt AABB-ütközés sokszor jobb, mint egy teljes fizikai motor: nincs nem kívánt lendület, súrlódás vagy pattogás, teljes kontroll a „mozdulj és csússz" fölött, és kristálytisztán megérted. Ha viszont valódi merevtest-fizika kell (lökhullám, halmozódó dobozok, ízületek), nyúlj avian2d-hez vagy bevy_rapier2d-hez. A döntésről bővebben a 17. szakaszban.

01

A collider mint adat

Az első és legfontosabb szétválasztás: a collider nem a sprite. A megjelenített kép lehet 48×64 pixeles, dekorációkkal teli karakter; az ütköző teste mégis egy szűk, 16×20-as doboz a lába körül. A collider a logikai alak — külön komponens, amit a fizika lát, a renderelő nem.

Top-downban két alak visz el szinte mindent: a tengely-igazított doboz (AABB) és a kör. A doboz a falakhoz, akadályokhoz tökéletes; a kör a karakterekhez ideális, mert nincs „beakadó" sarka — simán siklik a geometrián.

use bevy::prelude::*;

/// A collider a sprite-tól FÜGGETLEN adat: a logikai test alakja.
/// A közepe mindig a Transform; az alak ehhez relatív.
#[derive(Component, Clone, Copy)]
enum Collider {
    /// tengely-igazított doboz: `half` a fél-szélesség és fél-magasság
    Aabb { half: Vec2 },
    /// kör: olcsó teszt, nincs beakadó sarok — karakterekhez
    Circle { radius: f32 },
}

/// Sebesség külön komponensben — az integrálás ezt adja a pozícióhoz.
#[derive(Component, Default)]
struct Velocity(Vec2);

/// Szerep-markerek: a rendszerek ezekkel válogatnak (lásd 05).
#[derive(Component)] struct Mover;   // mozog: játékos, ellenfél, lövedék
#[derive(Component)] struct Solid;   // áll: fal, akadály, határ
A fél-méret (half-extent) a barátod

Tárold a dobozt közép + fél-méret formában (center, half), ne min/max sarkokkal. A közép egybeesik a Transform-mal (ingyen mozog vele), és minden teszt szebb lesz: az AABB-átfedés egyetlen abs() <= ha + hb tengelyenként. A min/max alak csak a rajzolásnál kell.

02

Átfedés-tesztek: a narrow phase primitívei

A detektálás magja három apró függvény. Mindegyik egy bool-t ad — „átfed-e?" —, és mindegyik néhány sor. Bevy ad rájuk beépített primitívet (a bevy::math::bounding modul), de érdemes kézzel is látni őket, mert a válaszhoz a puszta bool kevés lesz.

AABB × AABB

/// Két doboz átfed, ha a középpontjaik távolsága MINDEN tengelyen
/// kisebb-egyenlő a fél-méretek összegénél. Ha egy tengelyen "szét"
/// vannak, akkor van elválasztó vonal — nincs ütközés.
fn aabb_hit(ca: Vec2, ha: Vec2, cb: Vec2, hb: Vec2) -> bool {
    (ca.x - cb.x).abs() <= ha.x + hb.x &&
    (ca.y - cb.y).abs() <= ha.y + hb.y
}

Kör × kör

/// A legolcsóbb teszt: a középpontok távolsága <= a sugarak összege.
/// (Élesben gyökvonás nélkül: a NÉGYZETES távolságot hasonlítsd.)
fn circle_hit(ca: Vec2, ra: f32, cb: Vec2, rb: f32) -> bool {
    ca.distance_squared(cb) <= (ra + rb) * (ra + rb)
}

Kör × AABB

A trükk: keresd meg a dobozon a kör középpontjához legközelebbi pontot (a középpontot a doboz határai közé csípve), és mérd onnan a távolságot. Ez a pont a csúcson, az élen vagy a doboz belsejében lehet — a clamp mindhárom esetet lefedi.

fn circle_aabb_hit(c: Vec2, r: f32, bc: Vec2, bh: Vec2) -> bool {
    let closest = c.clamp(bc - bh, bc + bh);   // legközelebbi pont a dobozon
    c.distance_squared(closest) <= r * r
}
Bevy beépítve: a detektálás készen van

A bevy::math::bounding ugyanezt adja, csiszoltan. A lényeg, hogy az IntersectsVolume trait .intersects()-e egy igen/nem — a detektáláshoz tökéletes, a válaszhoz nem elég (nem mondja meg, mennyivel fednek át):

use bevy::math::bounding::{Aabb2d, BoundingCircle, IntersectsVolume};

let player = Aabb2d::new(player_pos, Vec2::splat(8.0)); // közép + fél-méret
let wall   = Aabb2d::new(wall_pos,   Vec2::new(16.0, 8.0));
if player.intersects(&wall) { /* átfedés — de a MENNYIT magunk számoljuk (05) */ }

// kör is megy, keverve is: kör vs doboz egyetlen hívásban
let body = BoundingCircle::new(enemy_pos, 6.0);
if body.intersects(&wall) { /* ... */ }

03

Bounding volumok: a Bevy eszköztár

A 02 tesztjeit kézzel írtuk meg — érdemes is érteni őket. De Bevy csomagban is adja ugyanezt, és ráadásnak két olyan műveletet, amire később építünk: a tágítást (CCD) és az egyesítést (broad phase). Ez a bevy::math::bounding modul.

Két 2D térfogata van: az Aabb2d (tengely-igazított doboz, belül min/max sarokkal) és a BoundingCircle. Mindkettő olcsón megépíthető közép + méret formából:

use bevy::math::bounding::{Aabb2d, BoundingCircle, BoundingVolume, IntersectsVolume};
use bevy::math::{Isometry2d, Rot2};

// közép + fél-méret, illetve közép + sugár
let a = Aabb2d::new(pos_a, Vec2::new(16.0, 8.0));
let c = BoundingCircle::new(pos_b, 10.0);

// metszés: ugyanaz, mint a 02 — de készen, és KEVERVE is megy (doboz × kör)
if a.intersects(&c) { /* átfed */ }

Honnan jön a térfogat? — Bounded2d

Egy forgó sprite befoglalóját nem kézzel számolod: a primitív (Rectangle, Circle, Capsule2d…) maga adja egy Isometry2d (eltolás + forgatás) mellett. A tanulság látványos: egy forgatott doboz tengely-igazított befoglalója nő, a köré írt kör viszont nem.

use bevy::math::bounding::Bounded2d;
use bevy::math::primitives::Rectangle;

let shape = Rectangle::new(40.0, 16.0);
let iso   = Isometry2d::new(pos, Rot2::radians(angle));
let aabb   = shape.aabb_2d(iso);          // forgásnál NŐ (lélegzik)
let circle = shape.bounding_circle(iso);  // forgástól független, állandó

A két ráadás-művelet: merge és grow

A BoundingVolume trait adja a térfogat-műveleteket. Kettő kulcsfontosságú a következő szakaszokhoz:

// merge: két térfogat KÖZÖS befoglalója — egy entitás-csoport vagy egy
// térkép-chunk EGYETLEN AABB-je a broad phase-hez (12).
let group = aabb_a.merge(&aabb_b);

// grow: Minkowski-tágítás a mozgó fél-méretével — a swept/CCD (13) alapja.
// A kinagyított falba lőtt sugár pont a mozgó ÉRINTKEZÉSÉT adja.
let expanded = wall_aabb.grow(mover_half);

// a szokásos lekérdezések is megvannak
if group.contains(&aabb_a) { /* teljesen benne */ }
let mid  = group.center();
let half = group.half_size();
A térfogat eldobható, ne tárold

A bounding volume egy pillanatkép a test mostani helyzetéről — minden kereten újraszámolod a Transform-ból, nem komponensként tárolod. Egy elavult Aabb2d a tavalyi helyen tesztel. Forgó sprite-nál külön figyelj: a befoglaló minden kereten más.

A detektálás itt is csak igen/nem

Az .intersects() egy bool — a választ (mennyivel, merre = MTV) továbbra is magad számolod, lásd a 05. szakaszt. A bounding modul „sugár-vető" testvéreit (RayCast2d, AabbCast2d) a következő szakasz és a CCD (13) használja.

04

Raycast: sugarak a világban

Az átfedés-teszt azt kérdi: „összeér-e két térfogat?" A raycast mást kérdez: „mit talál el egy sugár először, és milyen messze?" Top-downban ez legalább négy helyen kell:

  • Hitscan lövés. A puska golyója azonnal a legközelebbi falig / ellenfélig ér — nincs repülő lövedék.
  • Látótávolság (line-of-sight). Lát-e az őr a játékosra? Lövünk egy sugarat kettejük közt; ha fal megszakítja, nincs rálátás.
  • Kurzor-kijelölés. Mire mutat az egér? A kurzorból (vagy a kamerából) vetett sugár első találata.
  • MI-érzékelés, lézer-irányzék. Ugyanaz a sugár, más felhasználásban.

Bevy a bounding modulban adja: a RayCast2d egy origóból egy Dir2 irányba, adott max hosszig fut, és megmondja a találat távolságát:

use bevy::math::bounding::{Aabb2d, RayCast2d};
use bevy::math::Dir2;

/// Lát-e az őr a célpontra? Ha BÁRMELYIK fal közelebb metszi a sugarat,
/// mint a célpont távolsága, akkor takarásban van.
fn has_line_of_sight(from: Vec2, to: Vec2, walls: &[Aabb2d]) -> bool {
    let delta = to - from;
    let Ok(dir) = Dir2::new(delta) else { return true; };  // egy helyben: lát
    let ray = RayCast2d::new(from, dir, delta.length());   // max hossz = a célpontig
    // ha a sugár (a célpontig érő hosszán belül) bármit elér, az takar
    !walls.iter().any(|w| ray.aabb_intersection_at(w).is_some())
}

Hitscanhez ugyanez, de a legközelebbi találat kell — a legkisebb t távolságú fal:

/// A legközelebbi fal a sugár mentén (index + távolság), vagy None.
fn raycast_first(origin: Vec2, dir: Dir2, max: f32, walls: &[Aabb2d]) -> Option<(usize, f32)> {
    let ray = RayCast2d::new(origin, dir, max);
    walls.iter().enumerate()
        .filter_map(|(i, w)| ray.aabb_intersection_at(w).map(|t| (i, t)))
        .min_by(|x, y| x.1.total_cmp(&y.1))
}
Kurzor → világ

A képernyő-kurzort a camera.viewport_to_world_2d(cam_tf, cursor) teszi világ-koordinátává (Result-ot ad) — innen indíthatod a pick-sugarat, vagy egy pontszerű Aabb2d-vel közvetlenül .intersects-elsz. Ugyanez a hívás vezérli a kattintás-mozgást a pathfinding leckében.

Vékony és vastag sugár

A RayCast2d egy vékony (nulla vastagságú) sugár. Ha a „sugárnak" mérete van — egy mozgó test söprése —, akkor az AabbCast2d / BoundingCircleCast kell: ugyanaz az ötlet, de egy térfogatot söpör végig. Ez a gyors testek alagutazásának megoldása, lásd a CCD (13) szakaszt.

05

A válasz I.: kiszorítás (penetráció + MTV)

A bool tudja, hogy ütköztünk — de a válaszhoz tudni kell, mennyivel és merre lapolódunk át. Ez a penetrációs vektor, közismert nevén a MTV (Minimum Translation Vector): a legrövidebb eltolás, ami épp szétválasztja a két testet.

Két AABB-nél kiszámítjuk az átfedést mindkét tengelyen, és a kisebbik mentén tolunk ki — ez a legkevésbé feltűnő korrekció. (Ha fentről nyomod a falba, fentre ugraszt vissza; ha oldalról, oldalra.)

/// A penetrációs vektor (MTV) az A testre nézve: None, ha nincs átfedés.
fn aabb_mtv(ca: Vec2, ha: Vec2, cb: Vec2, hb: Vec2) -> Option<Vec2> {
    let d  = cb - ca;                      // A-ból B felé mutat
    let ox = (ha.x + hb.x) - d.x.abs();    // átfedés az X tengelyen
    let oy = (ha.y + hb.y) - d.y.abs();    // átfedés az Y tengelyen
    if ox <= 0.0 || oy <= 0.0 {
        return None;                       // valamelyik tengelyen szét → nincs ütközés
    }
    if ox < oy {
        Some(Vec2::new(if d.x > 0.0 { -ox } else { ox }, 0.0))   // kisebb átfedés: X
    } else {
        Some(Vec2::new(0.0, if d.y > 0.0 { -oy } else { oy }))   // kisebb átfedés: Y
    }
}

// használat: a DINAMIKUS testet toljuk ki, a fal marad
if let Some(push) = aabb_mtv(player_pos, player_half, wall_pos, wall_half) {
    player_pos += push;          // a játékost toljuk, nem a falat
    velocity.0 = Vec2::ZERO;     // a legegyszerűbb: az ütközés "lenyeli" a sebességet
}
A „legkisebb tengely" buktatója

A min-tengelyű MTV egyetlen, mély átfedésnél meglepő irányba tud lökni: ha a test mélyen belóg a falba, és véletlenül az Y-átfedés kisebb, akkor Y-ban tol ki, pedig X-ben mozogtál bele. A megoldás nem a bonyolítás, hanem a megelőzés: ne hagyd mélyre jutni — minden kereten oldd fel, és — ami a top-down kulcsa — oldd fel tengelyenként, a mozgás után rögtön. Pont erről szól a következő szakasz.

06

A válasz II.: csúszás a fal mentén

Itt dől el a top-down kontroll minősége. Képzeld el: a játékos átlósan nekifut egy függőleges falnak. A naiv válasz — „ha a mozgás ütközést okoz, dobd el az egész mozgást" — a testet beragasztja: nem megy a falba (jó), de a fal mentén sem (rossz). A játékos a falhoz tapad, amíg pontosan párhuzamosan nem nyomja az irányt. Idegesítő.

A megoldás meglepően egyszerű, és ez a lecke egyetlen legfontosabb fogása: mozgasd és oldd fel az X és az Y tengelyt KÜLÖN.

/// Egyetlen tengely feloldása: ha a mozgás után átfedés van, told vissza
/// PONTOSAN ezen a tengelyen, épp az átfedés mértékével.
fn resolve_x(pos: &mut Vec2, half: Vec2, wc: Vec2, wh: Vec2) {
    if aabb_hit(*pos, half, wc, wh) {
        let overlap = (half.x + wh.x) - (pos.x - wc.x).abs();
        pos.x += if pos.x < wc.x { -overlap } else { overlap };
    }
}
fn resolve_y(pos: &mut Vec2, half: Vec2, wc: Vec2, wh: Vec2) {
    if aabb_hit(*pos, half, wc, wh) {
        let overlap = (half.y + wh.y) - (pos.y - wc.y).abs();
        pos.y += if pos.y < wc.y { -overlap } else { overlap };
    }
}

/// A "move and slide": előbb X-ben mozgunk és oldunk fel, AZTÁN Y-ban.
fn move_and_slide(pos: &mut Vec2, half: Vec2, vel: Vec2, dt: f32, walls: &[(Vec2, Vec2)]) {
    pos.x += vel.x * dt;                              // 1) X tengely
    for &(wc, wh) in walls { resolve_x(pos, half, wc, wh); }
    pos.y += vel.y * dt;                              // 2) Y tengely
    for &(wc, wh) in walls { resolve_y(pos, half, wc, wh); }
}

Miért működik? Az átlós nekifutásnál a falra merőleges komponens elnyelődik (visszatoljuk azon a tengelyen), a fallal párhuzamos komponens viszont érintetlen marad — így a test tovább halad a fal mentén. A csúszás nem külön funkció: a tengelyenkénti feloldás ajándékként adja.

A sorrend stabil, de nem szent

X-először vagy Y-először: legtöbbször mindegy. Ritka, beékelődő sarokesetnél a kettő picit más eredményt ad — ha tökéletes szimmetria kell, futtasd le mindkét sorrendben és válaszd a kisebb összkorrekciót. A 99%-hoz az egyszerű „X, majd Y" elég.

07

Átlós mozgás & sarok-esetek

A tengelyenkénti csúszás (06) a gerinc, de az átlós irány három finomságot is felszínre hoz. Mindhárom apró, mindhárom érezhető a kézben.

1. Átló-sebesség: ne legyél gyorsabb átlósan

Ha a fel és a jobb gombot együtt nyomod, a nyers irányvektor (1, 1) — a hossza viszont √2 ≈ 1,41, vagyis a játékos átlósan 41%-kal gyorsabb. Klasszikus, árulkodó hiba. A javítás egy sor: normalizáld az irányt, mielőtt a sebességgel szorzod.

let mut dir = Vec2::ZERO;
if up    { dir.y += 1.0; }  if down  { dir.y -= 1.0; }
if left  { dir.x -= 1.0; }  if right { dir.x += 1.0; }
let vel = dir.normalize_or_zero() * SPEED;   // átlósan sem gyorsabb (a nulla is kezelve)

2. Sarok-beakadás és a min-MTV csapdája

Emlékszel a 05-ből: a min-tengelyű MTV a kisebb átfedés mentén tol ki. Ez egy fal lapjánál tökéletes, de egy külső sarok átlós megkerülésekor a választott tengely keretről keretre átbillenhet (egyszer X-re, egyszer Y-ra korrigál) — a test megrándul, beakad, „pattog" a sarkon. A tengelyenkénti feloldás (06) ezt elkerüli, mert minden tengelyt a tényleges azirányú mozgás alapján old fel, nem aszerint, hogy melyik átfedés kisebb. Ezért nyer a per-tengely a karaktermozgásnál — a demó pont ezt a billegést mutatja.

3. Átlós rés: a sarok-átszökés

A legalattomosabb: ahol két tömör csempe csak a sarkával érintkezik, marad köztük egy nulla szélességű átlós rés. Egy gyors átlós lépés átbújhat ezen — a lépés elején és végén a test egyik cellát sem fedi át, közben meg „átugorja" az érintkezési pontot. Ez átlós irányú alagutazás (13), és a teljes megoldása a CCD. Csemperácson viszont van egy olcsó, bevett szabály: tiltsd az átlós lépést, ha mindkét érintett ortogonális szomszéd tömör (a „no corner cutting" szabály — a pathfindingból is ismerős).

/// Átlós lépés tiltása, ha két fal SARKA közt vágna át.
/// Csak akkor engedjük, ha legalább az egyik ortogonális szomszéd szabad.
fn diagonal_blocked(map: &TileMap, from: IVec2, dx: i32, dy: i32) -> bool {
    if dx != 0 && dy != 0 {                       // tényleg átlós lépés?
        map.is_solid(IVec2::new(from.x + dx, from.y))      // vízszintes szomszéd
            && map.is_solid(IVec2::new(from.x, from.y + dy))  // függőleges szomszéd
    } else {
        false
    }
}
A per-tengely már fél-megoldás

Mivel a 06 amúgy is külön mozgatja az X-et és az Y-t, a sarok-átszökés ott ritkább, mint egy „teljes vektort egyszerre" mozgatónál: a köztes (X-mozgás utáni) pozíció gyakran már elkapja a falat. Gyors testeknél azonban ez sem elég — ott a CCD (13) a biztos megoldás.

08

Statikus vs dinamikus testek

Nem minden test egyforma. A statikus testek nem mozdulnak (falak, akadályok, határok); a dinamikusak igen (játékos, ellenfél, lövedék). Ez a megkülönböztetés két dolgot dönt el — és mindkettő spórol:

  • Ki mozdul a válaszban? Dinamikus–statikus ütközésnél csak a dinamikust toljuk ki, a fal marad. Dinamikus–dinamikusnál a kiszorítást szét kell osztani (fele-fele, vagy tömeg szerint), különben az egyik átlöki a másikat.
  • Mit kell egyáltalán tesztelni? Statikus–statikus párt soha: két fal sosem fog ütközni. Ez ingyen feleslegmunka-spórolás a párosításnál.

Bevy-ben a szerepeket marker-komponensek (Mover / Solid) jelölik, és két külön Query válogat. De itt jön a 0.18 (és minden korábbi verzió) egyik leggyakoribb pánikja:

fn collision_system(
    mut movers: Query<(&mut Transform, &Collider), (With<Mover>, Without<Solid>)>,
    walls:      Query<(&Transform, &Collider), With<Solid>>,
) {
    for (mut tf, mc) in &mut movers {
        for (wtf, wc) in &walls {
            // … detektálás + tengelyenkénti kiszorítás (05–06) …
        }
    }
}
A Query-aliasolás csapdája

Két Query, ami ugyanazt a komponenst kéri — és az egyik &mut — indításkor azonnal panicra fut („conflicting access"). A movers a &mut Transform-ot kéri, a walls a &Transform-ot — ütköznek. A javítás a bizonyítottan diszjunkt halmaz: a Without<Solid> a movereken + a With<Solid> a falakon garantálja, hogy egyetlen entitás sincs mindkettőben, így Bevy elengedi. Ez nem stílus — e nélkül el sem indul.

Dinamikus–dinamikus: oszd meg a kiszorítást

Két mozgó test (két ellenfél) ütközésénél a párokon a Query::iter_combinations_mut::<2>() megy végig, és az MTV felét kapja az egyik, felét a másik (ellentétes irányban). Tömeg szerint: a nehezebb kevésbé mozdul. A statikus testet ebből mindig kihagyod — a fal „végtelen tömegű".

09

Ütközési rétegek bonyolultabb helyzetben

Egy igazi játékban a nehéz kérdés nem az, hogy „átfed-e A és B", hanem hogy egyáltalán reagáljon-e a kettő egymásra. Gondold végig egy lövöldözős top-down párosait: a játékos lövedéke eltalálja az ellenfelet és a falat, de nem a játékost és nem a saját társait. Az ellenfelek egymáson átmennek, de a falon nem. A pickup csak a játékost érdekli. Ha ezt páronkénti if-ekkel kódolod, kombinatorikus rémálom lesz.

A tiszta eszköz az ütközési réteg + maszk (bitflag). Minden testnek két mezője van:

  • tagság (is): melyik réteg(ek) ő maga — PLAYER, ENEMY, WALL, P_BULLET…
  • maszk (mask): mely rétegekkel hajlandó ütközni.

Két test akkor hat egymásra, ha kölcsönösen szerepelnek egymás maszkjában — egyetlen bitművelet:

/// Bitflag-rétegek: minden réteg egy bit. (Élesben a `bitflags` crate.)
#[derive(Component, Clone, Copy)]
struct Layers { is: u32, mask: u32 }

mod layer {
    pub const WALL:     u32 = 1 << 0;
    pub const PLAYER:   u32 = 1 << 1;
    pub const ENEMY:    u32 = 1 << 2;
    pub const P_BULLET: u32 = 1 << 3;   // a játékos lövedéke
    pub const E_BULLET: u32 = 1 << 4;   // az ellenfél lövedéke
    pub const PICKUP:   u32 = 1 << 5;
}

/// Akkor ütközik, ha KÖLCSÖNÖSEN benne van a másik a maszkjában.
fn layers_collide(a: Layers, b: Layers) -> bool {
    (a.mask & b.is) != 0 && (b.mask & a.is) != 0
}

A bonyolult viselkedés így pár sornyi adat lesz, nem logika:

use layer::*;

// a játékost fal + ellenfél + ellenséges lövedék + pickup érdekli
Layers { is: PLAYER,   mask: WALL | ENEMY | E_BULLET | PICKUP };

// a játékos lövedéke CSAK falat és ellenfelet talál — magát és társait nem
Layers { is: P_BULLET, mask: WALL | ENEMY };

// az ellenfél: fal + játékos + a játékos lövedéke (de más ellenfélen átmegy)
Layers { is: ENEMY,    mask: WALL | PLAYER | P_BULLET };

// a fal mindenkit megállít, de maga sosem kezdeményez
Layers { is: WALL,     mask: PLAYER | ENEMY | P_BULLET | E_BULLET };

Hova kerül a szűrő?

A réteg-teszt a párosítás után, a drága narrow phase ELŐTT fut — ez az egyetlen kapcsoló, ami a komplex játékot is skálázza. A broad phase (12) kiadja a jelölt-párokat; mindjárt az elején eldobod, ami réteg szerint nem érdekes:

for (i, j) in broadphase(&bodies) {
    if !layers_collide(layer[i], layer[j]) { continue; }   // SZŰRÉS előbb
    if narrow_overlap(i, j) {                               // csak utána a drága teszt
        resolve_or_signal(i, j);
    }
}
A réteg és a szerep merőleges

Ne keverd össze: a réteg azt dönti el, kik hatnak egymásra; a szerep (Mover / Solid) és a Sensor azt, hogy mi történik (kitolás vagy csak jelzés). A kettő együtt: egy „tüskés zóna" lehet ENEMY-maszkú szenzor — érzékel, nem tol ki, és csak az ellenfeleket sebzi.

Motorban készen van

Ha fizikai motort használsz, ne írd újra: az avian2d a CollisionLayers (tagság + szűrő-maszk) komponenst adja, a bevy_rapier2d a CollisionGroups + SolverGroups párost. Pontosan ez a tagság/maszk-modell, csak becsomagolva.

Szenzorok: detektálj, de ne blokkolj

Nem minden ütközés fal. Az érme, a damage-zóna, a kapcsoló-lap, az ajtó-kioldó — ezek detektálnak, de nem tolnak ki. A játékos áthalad rajtuk; a kontaktus csak egy esemény, nem fizikai akadály. Ezt hívják szenzornak (vagy triggernek).

A kód örömteli: ugyanaz az átfedés-teszt, mint mindenütt — csak a válasz marad el. Nincs MTV, nincs kiszorítás; helyette egy hatás (felvétel, sebzés, kapu nyitása).

/// Marker: ez a test érzékel, de nem akadály.
#[derive(Component)] struct Sensor;
#[derive(Resource, Default)] struct Score(u32);

fn collect_pickups(
    player: Query<(&Transform, &Collider), With<Mover>>,
    coins:  Query<(Entity, &Transform, &Collider), With<Sensor>>,
    mut commands: Commands,
    mut score: ResMut<Score>,
) {
    // 0.18: a single() Result-ot ad — kezeljük le tisztán
    let Ok((ptf, _pc)) = player.single() else { return; };
    for (e, ctf, _cc) in &coins {
        if ptf.translation.distance(ctf.translation) < 16.0 {   // egyszerű kör-teszt
            score.0 += 1;
            commands.entity(e).despawn();   // felvettük: eltűnik, de SOSEM tolt ki
        }
    }
}
A szenzor egy harmadik szerep

A Mover/Solid mellé a Sensor a harmadik. A párosító rendszer a sensor-párokat a narrow phase után más ágra küldi: nincs kiszorítás, csak egy esemény (vagy azonnali observer-reakció). Egy entitás lehet egyszerre tömör és szenzor is — pl. egy tüskés akadály blokkol ÉS sebez.

11

Csempe-alapú ütközés

Egy top-down világ teste rendszerint nem szétszórt falak listája, hanem egy csemperács — pont az, amit a procedurális térkép és a biom leckében generáltunk. A rács óriási egyszerűsítést ad: nem kell minden fallal tesztelni, mert tudjuk, hol vannak — egy O(1) indexeléssel megnézhetők.

A kulcs: a mozgó AABB-je csak néhány cellát érint. Csak azokat teszteljük — a cella-tartományt a test min/max sarka adja —, sosem az egész térképet.

const TILE: f32 = 16.0;

/// Csak a mozgó AABB-je által ÉRINTETT cellákon iterálunk.
/// Pár cella tucatnyi vagy ezernyi fal helyett.
fn cells_touching(pos: Vec2, half: Vec2) -> impl Iterator<Item = IVec2> {
    let min = ((pos - half) / TILE).floor().as_ivec2();
    let max = ((pos + half) / TILE).floor().as_ivec2();
    (min.y..=max.y).flat_map(move |y| (min.x..=max.x).map(move |x| IVec2::new(x, y)))
}

/// Tengelyenkénti feloldás a rács ellen — ugyanaz a "move and slide" (06),
/// csak a falak most a tömör cellák, a cella közepe (x+0.5)*TILE.
fn move_on_grid(pos: &mut Vec2, half: Vec2, vel: Vec2, dt: f32, map: &TileMap) {
    let cell_half = Vec2::splat(TILE / 2.0);
    pos.x += vel.x * dt;                                   // X tengely
    for c in cells_touching(*pos, half) {
        if map.is_solid(c) {
            let wc = (c.as_vec2() + 0.5) * TILE;
            resolve_x(pos, half, wc, cell_half);
        }
    }
    pos.y += vel.y * dt;                                   // Y tengely
    for c in cells_touching(*pos, half) {
        if map.is_solid(c) {
            let wc = (c.as_vec2() + 0.5) * TILE;
            resolve_y(pos, half, wc, cell_half);
        }
    }
}
Honnan a „tömör-e?" adat

A map.is_solid(cell) ugyanabból a forrásból olvas, amit a tile-editor lecke LevelData-ja megír és a render kirajzol — egyetlen igazságforrás, amiből a megjelenítés ÉS az ütközés is származik. A klikk-mozgásos, fal-kerülő mesterséges intelligenciát (A* + kör-vs-fal AABB) a pathfinding lecke építi rá ugyanerre a rácsra.

12

Broad phase: a párosítás méretezése

A csemperács azért volt olyan kényelmes, mert a falak helye ismert. De a mozgó testek (50 ellenfél, 200 lövedék) egymással is ütközhetnek — és itt a naiv „mindenkit mindenkivel" megöli a teljesítményt. N testnél ez N·(N−1)/2 párostesztelés: 200 testnél már ~20 000 keretenként, és négyzetesen romlik.

A megoldás a broad phase: egy olcsó előszűrő, ami kihajítja a nyilvánvalóan lehetetlen párokat, hogy a drága narrow phase csak a gyanús párokat lássa. A legegyszerűbb, top-downhoz tökéletes eszköz a spatial hash (térbeli hash-rács): a teret nagy cellákra osztjuk, minden testet a cellájába rakunk, és csak az azonos és szomszédos cellák lakóit párosítjuk.

use std::collections::HashMap;
use bevy::utils::HashMap as _;   // (a prelude-ban is van; std is jó)

const CELL: f32 = 64.0;          // jóval nagyobb a testeknél: pár test / vödör

fn cell_of(p: Vec2) -> (i32, i32) {
    ((p.x / CELL).floor() as i32, (p.y / CELL).floor() as i32)
}

/// Vödrökbe rakjuk a testeket, és csak a 3×3 szomszédságon belül
/// párosítunk. N² helyett ~N·k összehasonlítás (k = a vödör-sűrűség).
fn broadphase(bodies: &[(Entity, Vec2)]) -> Vec<(usize, usize)> {
    let mut grid: HashMap<(i32, i32), Vec<usize>> = HashMap::new();
    for (i, (_, p)) in bodies.iter().enumerate() {
        grid.entry(cell_of(*p)).or_default().push(i);
    }
    let mut pairs = Vec::new();
    for (i, (_, p)) in bodies.iter().enumerate() {
        let (cx, cy) = cell_of(*p);
        for gy in cy - 1..=cy + 1 {
            for gx in cx - 1..=cx + 1 {
                if let Some(bucket) = grid.get(&(gx, gy)) {
                    for &j in bucket {
                        if j > i { pairs.push((i, j)); }   // j > i: minden párt egyszer
                    }
                }
            }
        }
    }
    pairs
}
A cellaméret a hangolható csavar

A CELL legyen nagyobb a legnagyobb testnél (különben egy test több cellát fed, és bonyolódik a könyvelés), de ne túl nagy (különben minden test egy vödörbe esik, és visszakaptuk az N²-et). Hüvelykujjszabály: a tipikus test 2–4-szerese. Egyenletes eloszlásnál a spatial hash veri a quadtree-t; erősen klaszteres jelenethez a quadtree/BVH lehet jobb — de top-down arcade-hoz a hash szinte mindig elég.

▶ Demó: broad phase — ne teszteld mindenkit mindenkivel

13

CCD: gyors mozgás & alagutazás

Van egy hiba, ami csak gyors testeknél bukkan elő, és először mindig megzavar: a lövedék átmegy a falon. Nem hibás az ütközés-kód — a lövedék egyszerűen olyan messzire ugrik egy kereten belül, hogy az előző kockán a fal előtt volt, a következőn már mögötte, és a kettő közt sosem fedte át. Ez az alagutazás (tunneling), és az ellenszere a CCD (folytonos ütközés-detektálás, Continuous Collision Detection): nem pillanatképeket tesztelünk, hanem a folytonos mozgáspályát.

A diszkrét teszt akkor bukik, ha egy test egy kereten belül a saját méreténél többet mozdul. Innen a hüvelykujjszabály: ha vel * dt hossza nagyobb a test fél-méreténél, kell CCD. Három szint van, egyre pontosabb (és drágább):

1. szint — aldarabolás (sub-stepping)

A legbutább, mégis hatékony: ha a lépés túl nagy, bontsd több kicsire, és mindegyik után fuss le a sima diszkrét feloldás (06). Pár sor, és a legtöbb lövedéket megfogja.

/// A mozgást a test méretéhez igazított allépésekre bontjuk.
fn move_substepped(pos: &mut Vec2, half: Vec2, vel: Vec2, dt: f32, walls: &[(Vec2, Vec2)]) {
    let dist     = (vel * dt).length();
    let max_step = half.min_element() * 0.5;            // fél test-méretnyi lépések
    let steps    = (dist / max_step).ceil().max(1.0) as u32;
    let sub_dt   = dt / steps as f32;
    for _ in 0..steps {
        move_and_slide(pos, half, vel, sub_dt, walls);  // a 06-os feloldás, kicsiben
    }
}

2. szint — swept (söprő) teszt

A pontos megoldás: végigsöpörjük a test útját, és megkeressük a találat idejét (egy t a 0..1 között). A trükk a Minkowski-tágítás (innen a 03-as grow): nagyítsd a falat a mozgó fél-méretével, és lődj egy vékony sugarat a mozgó közepéből. Ahol a sugár metszi a felnagyított falat, ott a mozgó épp érinti — a teljes test-átfedés egyetlen sugár-teszttel.

use bevy::math::bounding::{Aabb2d, AabbCast2d, RayCast2d, BoundingVolume};
use bevy::math::Dir2;

// söpörj egy MOZGÓ AABB-t a fal felé, és kérdezd a találat idejét
let Ok(dir) = Dir2::new(vel) else { return; };
let cast = AabbCast2d::new(mover_aabb, origin, dir, max_dist);
if let Some(t) = cast.aabb_collision_at(wall_aabb) {

3. szint — spekulatív kontaktusok

A motorok kedvence: a broad phase befoglalót kicsit kitágítjuk a mozgás irányába (grow a sebességgel), így a megoldó már a becsapódás előtt „látja" a közelgő falat, és lefékez, mielőtt áthatolna. Nincs külön swept-ág, jól illeszkedik az iteratív megoldóhoz — cserébe finomhangolást kíván.

Motorban egy kapcsoló

Ha fizikai motort viszel, a CCD egy komponens: az avian2d-ben a SweptCcd, a bevy_rapier2d-ben a Ccd::enabled(). Rakd rá a gyors testekre (lövedék, dash-elő játékos) — cserébe drágább, ezért ne tedd mindenre.

Top-downban ritkán kell mindenhol

Normál karaktersebességnél a diszkrét + tengelyenkénti feloldás (06) bőven elég; CCD-t csak a tényleg gyorsakra tegyél. A legolcsóbb gyakorlat: alapból sub-stepping a lövedékeknek, és csak ha az is kevés, akkor swept.

▶ Demó: alagutazás — diszkrét vs. CCD
Egy lövedék ismételten nekirepül egy vékony falnak. Diszkrét módban, nagy sebességnél a lövedék átszökik a falon (a két keret közt sosem fedte át). CCD (swept) módban a söprő teszt mindig elkapja, és a lövedék a falnál áll meg. Told fel a sebességet, amíg a diszkrét el nem kezd alagutazni — a CCD sosem fog.

Ütközés-jelzés: üzenet vagy observer?

A detektálás megtörtént, a kiszorítás kész — de az ütközésnek következménye is van: hang, sebzés, részecske, pickup. A kérdés: hogyan jut el ez a jelzés a feloldó rendszertől a reagálókig? Bevy 0.18-ban két, élesen különböző eszköz van rá — és érdemes tudni, melyik mire való.

Pufferelt jelzés → Message

Ha az ütközések egy keret alatt gyűlnek, és kötegben dolgozod fel őket (sok-a-sokhoz), a Message a kézenfekvő. (0.18-ban ez a régi Event/EventWriter új neve — részletek a üzenetek leckében.)

/// Pufferelt, kötegelt jelzés — frame-ritmusú.
#[derive(Message)]
struct Collision { a: Entity, b: Entity, normal: Vec2 }

fn resolve_collisions(/* queries */, mut hits: MessageWriter<Collision>) {
    // … ütközéskor:
    hits.write(Collision { a, b, normal });
}

fn react_to_hits(mut hits: MessageReader<Collision>) {
    for c in hits.read() {
        // hangeffekt, sebződés, részecske a c.normal mentén …
    }
}
// a main()-ben:  app.add_message::<Collision>();

Azonnali, entitásra célzott reakció → observer

Ha egy konkrét entitásra azonnal (még ugyanabban a parancs-folyamban) reagálni akarsz — pl. egy érme felvétele —, az observer a tisztább. (0.18: az observer első paramétere kötelezően On<E> — nem Trigger<E>. Bővebben az observer leckében.)

/// Entitásra célzott esemény: az `entity` mező a célpont.
#[derive(EntityEvent)]
struct Picked { entity: Entity }

fn setup(mut commands: Commands) {
    commands.spawn((/* érme komponensei */,))
        // az observer az entitáshoz tapad — vele együtt el is tűnik
        .observe(|pick: On<Picked>, mut commands: Commands| {
            commands.entity(pick.entity).despawn();   // On<E> deref-el az eseményre
        });
}

// ütközéskor a feloldó rendszer csak ennyit tesz:
//   commands.trigger(Picked { entity: coin });
Melyiket mikor?

Message: sok ütközés, kötegelt feldolgozás, laza csatolás (a hang-rendszernek nem kell tudnia, ki lőtt). Observer: ritka, célzott, „itt és most" reakció egy konkrét entitásra (pickup, halál, kapu). A kettő nem versengő — egy igazi játék mindkettőt használja: a fizikai ütközések üzenetként folynak, a játéklogikai triggerek observerként csattannak.

15

Bevy integráció: a rendszer-sorrend

A darabok megvannak; most fűzzük láncba. Az ütközésnél a sorrend nem ízlés kérdése — a feloldás minőségét dönti el. A helyes ritmus mindig ugyanaz:

  1. Bemenet → szándék. A billentyűzet/AI beállítja a Velocity-t.
  2. Mozgás + feloldás. Integráljuk a sebességet, és azonnal feloldjuk az ütközést (tengelyenként). Ez egy rendszer, hogy a kettő közé semmi ne ékelődjön.
  3. Jelzés. A feloldás eredménye Message/observer formában megy a reagálókhoz.
  4. Reakció. Hang, sebzés, részecske — ezek már a feloldott állapotot látják.
#[derive(States, Default, Clone, PartialEq, Eq, Hash, Debug)]
enum AppState { #[default] Loading, Playing }

fn main() {
    App::new()
        .add_plugins(DefaultPlugins)
        .add_message::<Collision>()
        .init_state::<AppState>()
        .add_systems(Update, (
            read_input,            // 1) bemenet -> Velocity
            move_and_collide,      // 2) integrálás + tengelyenkénti feloldás
            collect_pickups,       // 3) szenzorok (nem blokkol)
            react_to_hits,         // 4) reakció a Collision üzenetekre
        ).chain().run_if(in_state(AppState::Playing)))
        .run();
}
A mozgást és a feloldást ne válaszd szét

Csábító külön rendszerbe tenni az integrálást és a kiszorítást — de ha közéjük fut bármi, ami pozíciót olvas (kamera-követés, AI-érzékelés), az egy félig feloldott állapotot lát: a test még a falban van. Tartsd a kettőt egy rendszerben (vagy egy explicit .chain()-ben, barrier nélkül), és a kamerát mindig a feloldás után futtasd. A rendszer-sorrendről bővebben az időzítés lecke.

16

Komplett minta — összerakva

Egy minimál, de teljes, futtatható top-down ütközés: WASD-del mozgó játékos egy falakból álló szobában, tengelyenkénti csúszással. Ez a csontváz — a csemperács, a broad phase és a jelzés mind erre épül rá.

use bevy::prelude::*;

const SPEED: f32 = 160.0;

#[derive(Component)] struct Player;
#[derive(Component)] struct Solid;
#[derive(Component, Clone, Copy)] struct Aabb { half: Vec2 }
#[derive(Component, Default)]      struct Velocity(Vec2);

fn main() {
    App::new()
        .add_plugins(DefaultPlugins)
        .add_systems(Startup, setup)
        .add_systems(Update, (read_input, move_and_slide).chain())
        .run();
}

fn setup(mut commands: Commands) {
    commands.spawn(Camera2d);
    // játékos
    commands.spawn((
        Player, Velocity::default(),
        Aabb { half: Vec2::splat(12.0) },
        Sprite::from_color(Color::srgb(0.34, 0.83, 0.77), Vec2::splat(24.0)),
        Transform::from_xyz(0.0, 0.0, 1.0),
    ));
    // egy szoba: négy fal + egy belső pillér  (közép, fél-méret)
    let walls = [
        (Vec2::new(0.0,  140.0), Vec2::new(240.0, 12.0)),
        (Vec2::new(0.0, -140.0), Vec2::new(240.0, 12.0)),
        (Vec2::new(-228.0, 0.0), Vec2::new(12.0, 140.0)),
        (Vec2::new( 228.0, 0.0), Vec2::new(12.0, 140.0)),
        (Vec2::new(70.0, -30.0), Vec2::new(40.0, 40.0)),
    ];
    for (c, h) in walls {
        commands.spawn((
            Solid, Aabb { half: h },
            Sprite::from_color(Color::srgb(0.23, 0.23, 0.32), h * 2.0),
            Transform::from_xyz(c.x, c.y, 0.0),
        ));
    }
}

fn read_input(keys: Res<ButtonInput<KeyCode>>, mut q: Query<&mut Velocity, With<Player>>) {
    let mut dir = Vec2::ZERO;
    if keys.pressed(KeyCode::KeyW) { dir.y += 1.0; }
    if keys.pressed(KeyCode::KeyS) { dir.y -= 1.0; }
    if keys.pressed(KeyCode::KeyA) { dir.x -= 1.0; }
    if keys.pressed(KeyCode::KeyD) { dir.x += 1.0; }
    if let Ok(mut v) = q.single_mut() {
        v.0 = dir.normalize_or_zero() * SPEED;     // átlósan se gyorsabb
    }
}

fn move_and_slide(
    time: Res<Time>,
    mut movers: Query<(&mut Transform, &Aabb, &Velocity), (With<Player>, Without<Solid>)>,
    walls: Query<(&Transform, &Aabb), With<Solid>>,
) {
    let dt = time.delta_secs();
    let Ok((mut tf, ab, vel)) = movers.single_mut() else { return; };
    let half = ab.half;

    // 1) X tengely: mozdulj, majd oldd fel minden fal ellen X-ben
    tf.translation.x += vel.0.x * dt;
    for (wtf, wb) in &walls {
        let (p, w) = (tf.translation.truncate(), wtf.translation.truncate());
        if hit(p, half, w, wb.half) {
            let overlap = (half.x + wb.half.x) - (p.x - w.x).abs();
            tf.translation.x += if p.x < w.x { -overlap } else { overlap };
        }
    }
    // 2) Y tengely: ugyanez Y-ban → a csúszás ingyen jön
    tf.translation.y += vel.0.y * dt;
    for (wtf, wb) in &walls {
        let (p, w) = (tf.translation.truncate(), wtf.translation.truncate());
        if hit(p, half, w, wb.half) {
            let overlap = (half.y + wb.half.y) - (p.y - w.y).abs();
            tf.translation.y += if p.y < w.y { -overlap } else { overlap };
        }
    }
}

fn hit(ca: Vec2, ha: Vec2, cb: Vec2, hb: Vec2) -> bool {
    (ca.x - cb.x).abs() < ha.x + hb.x && (ca.y - cb.y).abs() < ha.y + hb.y
}
Skálázás

Ez a „minden mozgót minden fallal" lekérdezés egy szobához kiváló. Egy nagy pályához cseréld a falakat csemperácsra (11) — csak az érintett cellákat teszteled —, a sok mozgó testet pedig broad phase-szel (12) párosítsd. Az adatmodell (Aabb + tengelyenkénti feloldás) változatlan marad; csak a párkeresés okosodik.

17

Gyakorlati tippek & buktatók

Tedd

Válaszd szét a collidert a sprite-tól. A test legyen a játékmenethez igazítva (szűk doboz a láb körül), ne a rajzhoz. Külön komponens, közép + fél-méret formában.

Tedd

Oldd fel tengelyenként (X, majd Y). Ez az egyetlen fogás, ami a fal menti csúszást ingyen adja — a top-down kontroll lelke. Ha csak egyet viszel el a leckéből, ez legyen az.

Ne

Ne dobd el az egész mozgást ütközéskor („ha ütközik, ne mozdulj"). A test beragad a falba, és nem tud elcsúszni. A tengelyenkénti feloldás a javítás.

Vigyázz

Két Query, ami ugyanazt a komponenst kéri (egyik &mut) → indítási panic. Tedd diszjunkttá With/Without párral (08). E nélkül el sem indul a játék.

Vigyázz

Gyors testek átalagutaznak a vékony falon. Ha bármi gyorsabb a saját méreténél keretenként (lövedék), kell swept teszt vagy sub-stepping (13).

Tedd

Méretezd a párosítást: csemperács a statikus világhoz (11), spatial hash a mozgó tömeghez (12). A naiv N² 100+ testnél már fáj.

Mikor nyúlj fizikai motorhoz?

Maradj a kézzel írt AABB-nél, ha a mozgás arcade — azonnali gyorsulás/megállás, fal menti csúszás, nulla nem kívánt lendület (a legtöbb top-down: zelda-szerű, twin-stick, roguelike). Nyúlj avian2d-hez vagy bevy_rapier2d-hez, ha valódi merevtest-fizika kell: halmozódó/lökődő dobozok, ízületek, rugók, realisztikus pattogás, vagy ha egyszerűen nem akarsz magad bajlódni a broad/narrow phase-szel. A két világ keverhető: kézi karaktermozgás + motor a környezeti tárgyakhoz.

TL;DR

A lényeg egy kártyán

  • Két feladat: detektálás (átfed-e? — geometria) és válasz (mit teszünk vele? — tervezői döntés). Ne mosd össze.
  • Collider ≠ sprite: külön komponens, közép + fél-méret; a játékmenethez igazítva, nem a rajzhoz.
  • Három teszt visz el mindent: AABB×AABB (abs() <= h+h tengelyenként), kör×kör (négyzetes táv), kör×AABB (legközelebbi pont clamp-pal).
  • Bevy beépítve: bevy::math::bounding — Aabb2d, BoundingCircle, .intersects(). De ez csak bool; az MTV-t magad számolod.
  • Bounding-eszköztár: Bounded2d forgatott alak befoglalóját adja (a kör stabil, az AABB lélegzik); merge → broad phase, grow → CCD.
  • Raycast: „mit talál el a sugár először?" — hitscan, látótávolság (LoS), kurzor-pick. RayCast2d::aabb_intersection_at; vastag sugár = AabbCast2d.
  • Kiszorítás (MTV): a kisebb átfedés tengelye mentén told ki a dinamikus testet; a fal marad.
  • ★ Csúszás: mozgasd és oldd fel az X-et és az Y-t KÜLÖN. A fal menti csúszás így ingyen jön — ez a top-down kontroll lelke.
  • Átlós: normalizáld az irányt (átlósan se légy gyorsabb); a per-tengely a saroknál is stabilabb, mint az egyszeri min-MTV; sarok-átszökés ellen „no corner cutting".
  • Statikus vs dinamikus: csak a mozgót told; statikus–statikust sose teszteld; a Query-ket tedd diszjunktté (With/Without), különben panic.
  • Ütközési rétegek: tagság + maszk bitflag; (a.mask & b.is) != 0 && (b.mask & a.is) != 0; szűrj a narrow phase ELŐTT. A réteg = kik; a szerep = mi történik.
  • Szenzor: detektálj, de ne blokkolj (érme, zóna). Ugyanaz a teszt, elmaradó válasz.
  • Csemperács: csak a mozgó AABB-je által érintett cellákat teszteld, tengelyenkénti feloldással.
  • Broad phase: N² helyett spatial hash — csak az azonos + szomszédos cellák lakóit párosítsd.
  • CCD (gyors mozgás): ha a lépés > a test mérete, alagutazik. Megoldás növekvő sorrendben: sub-stepping → swept (AabbCast2d / Minkowski-grow + sugár) → spekulatív kontaktus; motorban SweptCcd / Ccd.
  • Jelzés: kötegelt → Message (MessageWriter/Reader); azonnali, célzott → observer (On<E>, commands.trigger).
  • Sorrend: bemenet → mozgás+feloldás (együtt!) → jelzés → reakció; a kamera a feloldás UTÁN.
