Bevy 0.18 · Rust · proc-gen útvonalak

Utak a procedurális térképen
— biomok és tárgyak összekötése

A biomok-lecke megmondta, hol terül el a sivatag és a hegy; a procedurális térkép és a biom-átmenetek széppé is tették. Mégis hiányzik valami: a térkép néma. Nincs benne irány, nincs benne történet. Egy út, ami a sivatag pereméről felkígyózik a hegyekbe, egyszerre ad struktúrát (a játékos tudja, merre induljon) és narratívát (valaki járt itt előtte). Ez a lecke arról szól, hogyan generálsz ilyen utakat egy már kész zaj-térkép tetején — kétféleképpen: biom-utakkal (két biom-régiót kötnek össze) és tárgy-utakkal (kitüntetett pontokat, pl. bányát és kutat). Hat interaktív demóval.

út = külön réteg költségmező A* súlyozott gráfokon legközelebbi cellapár = kapu MST hálózat út-autotiling (maszk → csempe)
00

A probléma: a néma térkép

Futtass egy zajt, oszd biomokra, rajzold ki. Az eredmény szép — és teljesen érzéketlen. A játékos egyenrangú foltokat lát: sivatag, rét, hegy, mindegyik egyforma jelentőségű. Nincs semmi, ami azt mondaná: „innen indulj, és menj arra." A valóságban a táj nem ilyen: az emberek (és az állatok) csapásokat taposnak, ösvények kötik össze a vizet a táborral, a bányát a kikötővel. Ezek az utak egyben irányjelzések is — elárulják a játékosnak, hová érdemes mennie.

Egy út három dolgot csinál egyszerre, és ez a három határozza meg, mit kell tudnia a generálásnak:

  • Struktúra. Két távoli pontot egy folytonos, követhető vonallal köt össze — a játékos szeme azonnal ráakad.
  • Narratíva. Aki az utat rakta (vagy taposta), szándékkal ment A-ból B-be. Az út egy történetet mesél: miért pont ez a két végpont?
  • Összenövés. Egy jó út nem metszi a tájat, hanem követi: kikerüli a vizet, megmássza a hágót, a síkságon egyenes. Ez a költségmező feladata (04).

Ennek a leckének egyetlen konkrét feladata van: egy már elkészült biom-térkép fölé utakat generálni. Nem generálunk új terepet, nem rajzolunk új csempéket — a terep adott, az utak csak egy réteget tesznek rá. Pontosan úgy, ahogy a biomok-lecke a zaj fölé épült, ez a biomok fölé.

▶ A néma térkép vs. az utas térkép
Ugyanaz a biom-térkép, ugyanazzal a seeddel. Bal: csak a biomok — szép, de iránytalan. Jobb: ugyanez a térkép, fölötte egy biom-út (sivatag → hegy) és három tárgy-út, ami a kitüntetett pontokat (bánya, kút, tábor) fűzi fel. A térkép adata nem változott — csak egy réteg jött rá. Ezt a réteget építjük fel lépésről lépésre.
Hol vagyunk a sorozatban?

Ez a biomok-lecke közvetlen folytatása. Ott a kérdés makró volt: melyik biom kerül a térkép melyik régiójába. Itt a biomok már készek — mi csak összekötjük őket. Két korábbi leckére támaszkodunk: a pathfinding tutorial A*-jára (az út = egy keresett útvonal a rácson) és a procedurális térkép kitüntetett objektumok szakaszára (a tárgyak, amiket összekötünk).

01

Az út mint adat — külön réteg

A legfontosabb (és leggyakrabban elrontott) döntés: az út nem tile. Az út egy külön réteg, ami a tile-grid fölé kerül — egy halmaz cella, amin „út van", plusz metaadat (melyik úthoz tartozik, milyen típusú). Ha az utat a Tile-ba írod be (pl. Tile::Road felülírja a Tile::Grass-t), akkor két bajod lesz egyszerre: az út eltereli a fát/bokrot a celláról (játékmenet), és az út csempéje összekeveredik a terepével (megjelenítés).

A helyes modell három, egymástól független réteg — pontosan az az elválasztás, amit a proc-gen lecke 01. szakasza és a biomok-lecke is követ:

RétegTípusMit tárolKi olvassa
terep (Tile)Tilemi van a cellán (fű, víz, kő)ütközés, autotiling
út (RoadLayer)RoadCellvan-e út, melyik úté, típusút-megjelenítés, sebesség
tárgy (Landmark)Landmarkkitüntetett pont (bánya, kút)játékmenet, útvégpontok

Rustban ez egy egyszerű, lazán csatolt adat-struktúra. Az út-réteg egy cella-indexelt tömb, ahol None = nincs út, Some(id) = az id-jű út cellája. A tárgyak külön Vec, mert egy cellán ritkán van több terep, de utak gyakran keresztezik egymást:

use bevy::prelude::*;

const W: i32 = 64;
const H: i32 = 48;

/// 1 · A TERRA réteg — „mi van a cellán". Az út ide NEM ír.
#[derive(Clone, Copy, PartialEq)]
enum Biome { Water, Desert, Plains, Forest, Mountain, Snow }

#[derive(Resource)]
struct BiomeMap { cells: Vec<Biome> }     // [y*W + x]

/// 2 · Az ÚT réteg — külön. `None` = nincs út, `Some(id)` = az id-jű út cellája.
#[derive(Resource, Default)]
struct RoadLayer {
    cells: Vec<Option<u32>>,                 // Option, mert egy cella lehet út nélküli
    roads: Vec<Road>,                       // maguk az utak (id → metainfo)
}

#[derive(Clone, Copy, PartialEq)]
enum RoadKind { BiomeLink, Landmark }        // a két típus a 02-ből

struct Road { id: u32, kind: RoadKind, path: Vec<IVec2> }   // a cellák sorrendje

impl RoadLayer {
    fn stamp(&mut self, path: Vec<IVec2>, kind: RoadKind) -> u32 {
        let id = self.roads.len() as u32;
        for &p in &path {
            let i = (p.y * W + p.x) as usize;
            if self.cells[i].is_none() { self.cells[i] = Some(id); }   // első út nyer
        }
        self.roads.push(Road { id, kind, path });
        id
    }
    fn has(&self, x: i32, y: i32) -> bool {
        self.cells[(y * W + x) as usize].is_some()
    }
}

/// 3 · A TÁRGYAK réteg — a kitüntetett pontok (03/07 végső pontjai).
#[derive(Component)]
struct Landmark { pos: IVec2, tag: LandmarkTag }
#[derive(Clone, Copy, PartialEq)]
enum LandmarkTag { Mine, Well, Camp, Shrine }
Miért Option, nem bool?

Mert egy cellán több út is átmehet (kereszteződés!). A bool csak azt mondja, „van-e út" — de az autotiling (08) tudni akarja, melyik utak kereszteződnek itt, hogy a megfelelő csempét válassza. Az Option<u32> megőrzi, melyik út „nyert" (elsőként ráírt); a kereszteződéseket külön számoljuk (07). Ha csak „van/nincs" kell, a bool is elég — de akkor a junction-t nem tudod szépen rajzolni.

A proc-gen lecke szabálya visszaköszön

„Ne keverd az adatot és a megjelenítést." Itt ez kétszeresen igaz: az út-réteg sem nem terep, sem nem csempe. A játékmenet (sebesség az úton, üldözés) az út-réteget kérdezi; a megjelenítés (melyik út-csempe) is az út-réteget — a terep Tile-t egyik sem írja át.

02

A két út-típus — és miért más

Nem minden utat ugyanúgy generálsz. Két, határozottan különböző típus van, és a különbség a végpontok kiválasztásában rejlik — maga a nyomkeresés (A*) mindkettőnél ugyanaz. Ezt a különbséget érdemes már az adatmodellben is élesen tartani (lásd a RoadKind enumot fent):

Biom-út (BiomeLink)Tárgy-út (Landmark)
végpontokbiom-régiók (sivatagblokk → hegyvonulat)konkrét pontok (bánya, kút)

Biom-út — „a sivatagból felvezet a hegyekbe"

Egy biom-út két különböző biom régióját köti össze. A végpontok nincsenek előre megadva — a generálásnak kell kitalálnia, hogy a sivatag melyik cellájából induljon, és a hegy melyik cellájába érkezzen. A kulcs-felismerés: nem a közepeket kötjük össze (az gyakran értelmetlen, mert a hegy közepén lehet járhatatlan csúcs), hanem a két régió legközelebbi cellapárját — az úgynevezett kaput. Ezzel a 03 foglalkozik.

A biom-út narratívája: valaki a sivatag pereméről indult, és a hágón ment át a hegyekbe. Ez a gondolat rögtön megszabja, hogy a költségmező (04) a hegyet drágának (de nem járhatatlannak) kell hogy tartsa — az út megy fel, csak keresi a legolcsóbb (legkisebb szintkülönbségű) átjárót.

Tárgy-út — „a bányától a kútig"

Egy tárgy-út konkrét, kitüntetett pontokat köt össze. A végpontok ismertek: a proc-gen lecke 09. szakasza már elhelyezte a bányákat, kutakat, táborokat reject-samplinggel — mi most ezek közül veszünk párokat, és utat húzunk köztük. Ha több tárgy van, nem egyetlen utat, hanem egy hálózatot generálunk: a 07-ben egy minimális feszítőfával (MST) összekötjük az összeset a lehető legkevesebb úttal.

A tárgy-út narratívája: gazdasági/stratégiai — az érc a bányából a kikötőbe (kút) kerül. Ezért a tárgy-utak rövidebbek, egyenesebbek, és kevésbé kanyargósak — a költségmező itt a meglévő utakat részesíti előnyben (hogy a hálózat összeforrjon), nem a táj legfőbb értékeit követi.

A két típus nem kizárja egymást

Egy valós térképen mindkettő jelen van, és össze is érnek: egy biom-út felér a hegy lábához, ott egy bánya van, és onnan egy tárgy-út megy tovább a kútig. A RoadLayer mindkettőt ugyanabban a rétegben tartja — a kereszteződéseket (07) és az autotilinget (08) egységesen kezeli. A különbség csak a végpont-kiválasztásban és a költségmező súlyozásában van.

03

Végpontok & kapuk kiválasztása

A tárgy-út végpontjai adottak (maguk a tárgyak). A biom-út végpontjait nekünk kell megtalálnunk — és ez a lépés dönti el, hogy az út értelmes-e vagy sem. Három, egyre jobb stratégia; a legutolsó az, amit használni fogjuk.

1 · A régió centroidja — rossz

A legnyilvánvalóbb ötlet: vedd a sivatag összes cellájának átlagát (centroid), és a hegy összes cellájának átlagát, és kösd össze őket. A baj: a centroid lehet a víz alatt, vagy a hegy legmagasabb, járhatatlan csúcsán — mert a centroid egy matematikai középpont, nem egy létező cella. Ráadásul a centroid gyakran belül van a régióban, így az út a sivatag mélyéről indul — nincs narratíva, „nincs perem".

2 · A perem egy véletlen cellája — jobb, de érzékeny

A következő ötlet: vedd a régió peremét (azokat a cellákat, amiknek van nem-sivatag szomszédja), és válassz közülük egyet determinisztikusan (hash-szel, nem rand()-dal!). Ez már ad egy „perem"-érzést — de a melyik peremcellát választod, nagyban befolyásolja, hova megy az út. Ha a két régió pereméből rossz párt választasz, az út egy hatalmas kerülőt tesz meg.

3 · A legközelebbi cellapár — a kapu (nyerő)

A helyes megoldás: a két régió kapuja az a cellapár (a, b), ahol a az egyik régióban, b a másikban, és a távolságuk minimális. Ez a természetes átjáró: a sivatag azon cellája, ami legközelebb van a hegyhez — éppen az, ahonnan egy gyalogos is elindulna. Az út rövid lesz, és pontosan ott lépi át a határt, ahol a két biom tényleg szomszédos. Két régió közt a kapu egyértelmű és jól definiált.

use bevy::math::IVec2;

/// A két régió legközelebbi cellapárja = a KAPU. `region_a` és `region_b`
/// a cellák halmaza (pl. minden cella, ahol a biome == Desert / Mountain).
/// O(n·m), de a régiók cellaszáma ezres nagyságrend — másodpercek alatt megvan.
fn gate(region_a: &[IVec2], region_b: &[IVec2]) -> Option<(IVec2, IVec2)> {
    let mut best = (i32::MAX, IVec2::ZERO, IVec2::ZERO);
    for &a in region_a {
        for &b in region_b {
            // négyzetes táv (a gyökölés felesleges, csak az összehasonlítás kell)
            let d2 = (a.x - b.x).pow(2) + (a.y - b.y).pow(2);
            if d2 < best.0 { best = (d2, a, b); }
        }
    }
    (best.0 != i32::MAX).then_some((best.1, best.2))
}
O(n·m) — de vigyázz a partvonalra

Egy 64×48-as tanulótérképen ez pillanatok alatt lefut. Egy nagy nyílt világnál (tízezres régiók) az O(n·m) már drága lehet. Trükk: csak a perem-cellákat hasonlítsd össze (amiknek van szomszédja a másik régióban) — a kapu mindenképp a peremen van, a belső cellák sosem nyernek. Ezzel a cellaszám drasztikusan csökken, és az algoritmus egy nagy térképen is másodpercek alatt lefut.

A kapu = a proc-gen lecke „legközelebbi régió"-ötlete

Ez ugyanaz a gondolat, mint a flood-fill összefüggőség-ellenőrzés: a kettő közti határt a legközelebbi cellapár írja le. Ott az volt a kérdés, hogy „összeér-e"; itt az, hogy „hol ér össze". A kapu ott van, ahol a biomok-lecke domain warpingje a határt organikussá tette — az út ezt az organikus határt használja ki.

▶ Demó: végpont-stratégiák
Ugyanaz a sivatag→hegy régiópár, három végpont-stratégiával, ugyanazzal a költségmezővel és A*-lal. Centroid: a matematikai közép — gyakran rossz cellán. Perem-cellák: determinisztikus peremválasztás — érzékeny a szerencsére. Kapu: legközelebbi cellapár — a természetes átjáró. Kapcsold be a költségmezőt is, és lásd, hogyan kerüli a vizet.

04

A költségmező — az út lelke

Eddig csak a végpontokról beszéltünk — de ami az utat úttá teszi (és nem csak egy egyenes vonallá), az a költségmező: minden cellához egy szám, ami azt mondja, mennyibe kerül átjárni. Az A* (05) ez alapján keresi a legolcsóbb utat — és mivel minden biom máshogy „drága", az út természetesen követi a tájat: síkságon egyenesen megy, kikerüli a vizet, a hágón megy át a hegyen.

A költségmező három tényezőből áll — érdemes őket külön elkódolni, mert külön is hangolhatók:

use bevy::math::IVec2;

#[derive(Resource)]
struct BiomeMap { cells: Vec<Biome>, elev: Vec<f32> }   // elev: a zaj magassága 0..1

/// A költségmező három rétege:
///  1) TEREP-költség: biomonként — víz drága/járhatatlan, síkság olcsó.
///  2) LEJTŐ-büntetés: a magasságkülönbség — hegyi út kerüli a meredeket.
///  3) (hálózatnál) MÁR-ÉPÜLT-út bónusz: a meglévő utak olcsók → összeér az út.
fn cost(map: &BiomeMap, from: IVec2, to: IVec2) -> f32 {
    let b = map.cells[(to.y * W + to.x) as usize];
    let base = match b {
        Biome::Plains   => 1.0,    // a síkság a természetes útfelület
        Biome::Desert   => 1.4,    // homok: lassabb
        Biome::Forest   => 1.8,    // erdő: vágni kell
        Biome::Snow     => 2.4,    // hó: lassú, de járható
        Biome::Mountain => 3.5,    // hegy: drága, de NEM járhatatlan!
        Biome::Water    => 1e6,    // víz: gyakorlatilag tilos (komp = külön eset)
    };
    // (2) LEJTŐ: a szomszéd cella magasságkülönbsége. Meredek = drága.
    let e_from = map.elev[(from.y * W + from.x) as usize];
    let e_to   = map.elev[(to.y   * W + to.x  ) as usize];
    let slope  = (e_to - e_from).abs() * 6.0;          // hangolható súly
    base + slope
}

Miért NEM járhatatlan a hegy?

Ez a legfontosabb design-döntés. Ha a hegyet f32::INFINITY-re teszed (mint a vizet), az A* soha nem megy fel — a biom-út („a sivatagból fel a hegyekbe") értelmét veszti, mert a végpont maga a hegyben van. A hegyet drágának, de járhatónak kell tartani: az út tényleg felmegy, csak a legkisebb szintkülönbségű (legolcsóbb) hágót keresi. A víz ezzel szemben sosem cél — azt tényleg tilosnak vesszük, hacsak nem akarsz hidat/kompot generálni (ami külön tutorial).

A víz = járhatatlan, de a „1e6" trükk

A 1e6 (nem INFINITY) apró trükk: az A* elméletileg képes átmenni rajta, ha nincs más út (pl. egy vékony vízcsík, amit megér egy híd ára). Így a generálás sosem akad el „nincs út" hibával — csak akkor megy át a vízen, ha tényleg muszáj. Ha viszont sosem akarsz víz-átkelést, használj INFINITY-t és kezeld az „elérhetetlen" esetet külön (03 végén: a kapu nem jó, ha nem ér el).

A lejtő-súly a titkos fűszerező

A terep-költség önmagában egy vázlát ad: az út kikerüli a vizet, de a síkságon belül még mindig túl egyenes lehet. A lejtő-súly (a magasságkülönbség büntetése) az, ami az utat kígyózóvá teszi: a zaj finom domborzatát követi, a gerinceken halad, a völgyeket kedveli. Ezt a demóban külön fel tudod tolni — és látni fogod, ahogy az út hirtelen elkezd kanyarogni.

▶ Demó: a költségmező hatása az útra
Ugyanaz a térkép, ugyanaz a kapu (sivatag → hegy), a költségmező három súlyával játszva. Terep-költség: ha 1-re állítod (minden cella egyforma), az út egy egyenes vonal lesz — nincs tájkövetés. Feljebb tolva az út kikerüli a vizet és erdőt. Lejtő-súly: a magasságkülönbség büntetése — feljebb tolva az út a völgyeket/gerinceket követi, kígyózó lesz. Síkság-preferencia: a síkságokhoz adott bónusz csökkentése — egyenessé teszi az utat a nyílt terepen.
A költségmező = a biomok-lecke „nehézség-mezője" egy szinttel lejjebb

A biomok-lecke nehézség-mezője a játékos számára mondta meg, mennyire veszélyes egy cella. Itt ugyanez a gondolat, csak az út-építő számára: a költségmező mondja meg, mennyire „drága" egy cellán átjárni. Ugyanaz az adat (a biom + a magasság), két különböző olvasatban — és ezért van külön rétegben (01).

05

A* — az útvonal megkeresése

Most már megvan a két végpont (03) és a költségmező (04). A kettő közé kell húzni az utat — és erre az A* a klasszikus eszköz. A pathfinding tutorial ezt teljesen kifejti (BFS, Dijkstra, A*, heurisztikák); itt csak annyit veszünk át, amennyi az utakhoz kell. A különbség a karakter-mozgáshoz képest: az út-generálás egyszer fut le, tervezéskor, nem képkockánként — így megengedhetünk egy picit drágább, de szép utat.

A legfontosabb beállítások, amik az utakat másokká teszik, mint a karakter-útkövetést:

  • 8-irányú mozgás. A 4-irányú (Manhattan) út lépcsőzetes, „lépcsőházas" — csúnya. A 8-irányú megengedi az átlós lépést, így az út ferde szakaszokat is tud. Vigyázz: az átlós lépés ne vágjon át sarkon (a két oldalsó szomszéd mindkettő ne járhatatlan legyen) — lásd lent.
  • Súlyozott lépésköltség. Az átlós lépés √2 ≈ 1.41-szerese az egyenesnek (Pythagorasz), különben az A* az átlóst túlzottan preferálja. A lépésköltség maga a cost() a 04-ből.
  • Heurisztika: oktáli-távolság. A Manhattan-távolság 8-irányú mozgásnál nem engedi meg az optimális megoldást (felülbecsli). Az oktáli-távolság (a 8 irány közti max + a maradék) a helyes választás.
use std::collections::BinaryHeap;
use bevy::math::IVec2;

/// A* a költségmezőn, 8-irányú mozgással. Visszaadja a cellák sorrendjét
/// (starttal és céllal együtt), vagy `None`-t, ha nem ér el.
/// `cost(from, to)` a 04-es költségmező; `passable(to)` a víz-szabály.
fn astar<C: Fn(IVec2, IVec2) -> f32, P: Fn(IVec2) -> bool>(
    start: IVec2, goal: IVec2, cost: C, passable: P,
) -> Option<Vec<IVec2>> {
    let mut open: BinaryHeap<Node> = BinaryHeap::new();
    let mut g = std::collections::HashMap::<IVec2, f32>::new();
    let mut parent = std::collections::HashMap::<IVec2, IVec2>::new();
    g.insert(start, 0.0);
    open.push(Node { pos: start, f: octile(start, goal) });

    while let Some(Node { pos, .. }) = open.pop() {
        if pos == goal { return Some(reconstruct(&parent, goal)); }
        let g_pos = *g.get(&pos)?;
        for (n, diag) in neighbors8(pos) {
            if !passable(n) { continue; }
            // NE VÁGJON SARKON: két oldalsó szomszéd ne legyen egyszerre tilos.
            if diag && (!passable(IVec2::new(pos.x, n.y)) || !passable(IVec2::new(n.x, pos.y))) {
                continue;
            }
            let step = cost(pos, n) * if diag { 1.41421 } else { 1.0 };
            let tentative = g_pos + step;
            if tentative < *g.get(&n).unwrap_or(&f32::INFINITY) {
                parent.insert(n, pos);
                g.insert(n, tentative);
                open.push(Node { pos: n, f: tentative + octile(n, goal) });
            }
        }
    }
    None
}

/// Oktáli-távolság: 8-irányú mozgásra a helyes (admisszibilis) heurisztika.
fn octile(a: IVec2, b: IVec2) -> f32 {
    let dx = (a.x - b.x).abs() as f32;
    let dy = (a.y - b.y).abs() as f32;
    (dx + dy) + (1.41421 - 2.0) * dx.min(dy)   // = 1.41*min + (max-min)
}
fn neighbors8(p: IVec2) -> impl Iterator<Item = (IVec2, bool)> {
    const DIRS: [(IVec2, bool); 8] = [
        (IVec2::new(1,0),false),(IVec2::new(-1,0),false),
        (IVec2::new(0,1),false),(IVec2::new(0,-1),false),
        (IVec2::new(1,1),true),(IVec2::new(-1,-1),true),
        (IVec2::new(1,-1),true),(IVec2::new(-1,1),true),
    ];
    DIRS.into_iter().map(move |(d, diag)| (p + d, diag))
}
#[derive(PartialEq)] struct Node { pos: IVec2, f: f32 }
impl Eq for Node {}
impl Ord for Node { fn cmp(&self, o: &Self) -> std::cmp::Ordering { o.f.partial_cmp(&self.f).unwrap() } }
impl PartialOrd for Node { fn partial_cmp(&self, o: &Self) -> Option<std::cmp::Ordering> { Some(self.cmp(o)) } }
A sarkon-vágás szabály

A diag feltétel kritikus: ha átlósan lépsz, és a két oldalsó szomszéd (a „sarkok") közül valamelyik járhatatlan, akkor az út átmegy a falon. Ez a pathfinding tutorial 3. fejezete „sarkon-vágás" problémája. Utaknál ez nemcsak csúnya, hanem narratíva-romboló is: egy út, ami átmegy a hegy gerincén, értelmetlen.

Determinizmus

Az A* a seedelt zajból számolt költségmezővel determinisztikus — ugyanaz a seed ugyanazt az utat adja. SOHA ne vigyél be rand()-ot az A*-ba (pl. a nyitott halmaz rendezésébe) — az út minden indításkor más lesz, és nem reprodukálható a hiba. Ha „kicsit véletlenszerűbb" utat akarsz, a költségmezőbe tegyél hash-jittert (mint a biomok domain warpingje), nem a keresőbe.

06

Simítás & szélesítés — a nyers A* csúnya

Az A* egy cella-listát ad vissza — és az nyersen csúnya. Két baja van: (1) a 8-irányú mozgás „lépcsős" átlókat hoz létre (mert az átlós lépés valójában egyetlen cella-ugrás, nem folytonos vonal), és (2) az út túl kígyózó lehet, ha a költségmező finom zajjal van tele. Mindkettőt egy utófeldolgozó lépés javítja — miután az A* lefutott, és anélkül, hogy az útvonalat elhagyná a terepről.

1 · String-pulling — a felesleges kanyarok elhagyása

Az A* gyakran tesz felesleges köztes lépéseket: az út megy A→B→C, de A-ból C is egyenesen látható (nincs drága cella köztük). A string-pulling (vagy line-of-sight-simplification) ezeket a felesleges pontokat eldobja: minden pontra megnézi, a legmesszebbi olyan pont elérhető-e egyenes vonalban, és ha igen, a közteseket kihagyja. Az eredmény: rövidebb, egyenesebb, természetesebb út.

/// String-pulling: elhagyja a felesleges köztes pontokat. Az A* útvonalából
/// csak azokat a pontokat tartja meg, amik közt NINCS egyenes vonalú átjárás.
/// `visible(a, b)` = az a→b egyenes (Bresenham) mentén minden cella járható.
fn simplify<V: Fn(IVec2, IVec2) -> bool>(path: &[IVec2], visible: V) -> Vec<IVec2> {
    if path.len() <= 2 { return path.to_vec(); }
    let mut out = vec![path[0]];
    let mut anchor = 0;
    for i in 1..path.len() {
        // ha az anchor→i+1 már nem látható, az i a legutolsó látható pont
        if i + 1 == path.len() || !visible(path[anchor], path[i + 1]) {
            out.push(path[i]);
            anchor = i;
        }
    }
    out
}

2 · Út-szélesítés — a „csapásból út"

Egy 1-cella széles út egy csapás, nem út. A valós utak 1–2 cella szélesek. A szélesítés egyszerű: a simított út minden cellájának oldalsó szomszédját is megjelöljük útnak — de csak akkor, ha az nem víz/nem járhatatlan. Ez ad egy „szélességet", anélkül, hogy az út elhagyná a terepet. A demóban külön bekapcsolható.

/// Út-szélesítés: a simított út mindkét oldalán 1 cellát hozzáadunk,
/// ha az járható. A `normal(p)` az út iránya az előző pontból (a merőlegeshez).
fn widen(layer: &mut RoadLayer, path: &[IVec2], map: &BiomeMap) {
    for w in path.windows(2) {
        let dir = w[1] - w[0];              // iránya (pl. (1,0))
        // a két merőleges irány: (−dy, dx) és (dy, −dx)
        let perp = IVec2::new(-dir.y, dir.x);
        for &p in &[w[0], w[1]] {
            for n in [p + perp, p - perp] {
                if in_bounds(n) && map.cells[..].get(idx(n)).map_or(false, |b| *b != Biome::Water) {
                    let i = idx(n);
                    if layer.cells[i].is_none() { layer.cells[i] = Some(u32::MAX); } // széles
                }
            }
        }
    }
}
A szélesítés felülírhatja a terepet?

NEM. A szélesítés csak az út-rétegbe ír (01) — a Tile-t nem módosítja. Az út-csempe (08) megjeleníti az „út-tal" a fűt, de az alatt a fű még mindig fű marad a terep-rétegben. Ezért nem tör el semmit, ha később az utat eltolod vagy elhagyod. Soha ne végy le egy fát csak azért, mert ott út van — az egy külön (játékmeneti) döntés.

Ugyanaz az út, három állapotban. Nyers A*: a 8-irányú lépések „lépcsős" átlókat adnak, és a finom költségmező kígyózást. + Simítás: string-pulling elhagyja a felesleges kanyarokat — egyenesebb, természetesebb. + Szélesítés: az út 1 celláról 3-ra nő (oldalsó szomszédok) — ez már „út", nem „csapás".

07

Hálózat — MST & elágazások

Eddig egyetlen utat generáltunk: A-ból B-be. De egy térképen ritkán egy út van — több tárgyat kell összekötni, és azokat egy hálózatként, nem független utakkal. Ha minden tárgyat mindennel összekötsz, a térkép aszfalt-tenger lesz (a teljes gráf N²/2 út). Ha viszont csak egy vonalat húzol (lánc), a két szélső tárgy egymástól nagyon messze kerül. A helyes megoldás a minimális feszítőfa (MST): a lehető legkevesebb út, ami még minden tárgyat elér.

A recept: MST az út-végpontokon, majd A* minden élen

A folyamat két lépésből áll — és fontos, hogy a kettő külön maradjon. Először a tárgyakon (mint csúcsokon) felépítesz egy távolsággráfot: minden tárgypárhoz a költségmező szerinti távolságot (nem a légvonalat!) számolod. Ezen az MST-t keressed (Prim vagy Kruskal). Az MST élei mondják meg, melyik két tárgyat kösd össze — és csak ezután, minden élre külön lefuttatod az A*-t (05).

use std::collections::HashSet;

/// A tárgyakon minimális feszítőfát (MST) építünk, hogy a lehető
/// LEGKEVESEBB úttal kapcsoljunk össze mindent. Az élek súlya = a
/// tárgyak közti költségmező-távolság (nem légvonal!).
fn mst_edges(landmarks: &[IVec2], dist: &dyn Fn(IVec2, IVec2) -> f32) -> Vec<(usize, usize)> {
    let n = landmarks.len();
    if n < 2 { return vec![]; }
    // Prim-algoritmus: egy csúcsból indulunk, mindig a legolcsóbb új élt vesszük.
    let mut in_tree = vec![false; n];
    let mut min_edge = vec![(f32::INFINITY, 0); n];  // (költség, honnan)
    in_tree[0] = true;
    let mut edges = vec![];
    for _ in 1..n {
        for j in 0..n {
            if !in_tree[j] {
                let d = dist(landmarks[0/*...függ a fától*/], landmarks[j]);
                // (teljes impl-ben az éppen vizsgált fában lévőkhöz mérünk)
            }
        }
        // válaszd a legkisebb min_edge-et, vedd be a fába, rögzítsd az élt
        // (rövidített — a Prim klasszikus, lásd a pathfinding tutorial gráf-fejezetét)
    }
    edges
}

/// A teljes hálózat-generálás: MST → minden élre A* → simítás → rétegbe.
fn build_landmark_network(
    landmarks: &[IVec2],
    map: &BiomeMap,
    layer: &mut RoadLayer,
) {
    // 1) távolságfüggvény: A* költsége (nem légvonal!) — cache-elve, mert N²
    let dist = |a: IVec2, b: IVec2| {
        astar(a, b, |f, t| cost(map, f, t), |p| map.get(p) != Some(Biome::Water))
            .map(|p| path_cost(&p, map)).unwrap_or(f32::INFINITY)
    };
    // 2) MST megmondja, mely tárgypárokat kössük össze
    for (i, j) in mst_edges(landmarks, &dist) {
        // 3) minden MST-élen A* → simítás → stamp
        if let Some(path) = astar(landmarks[i], landmarks[j],
            |f, t| cost(map, f, t), |p| map.get(p) != Some(Biome::Water)) {
            let simple = simplify(&path, |a, b| los_clear(a, b, map));
            layer.stamp(simple, RoadKind::Landmark);   // 01-es réteg
        }
    }
    // 4) a MÁR-ÉPÜLT utakat olcsóvá tesszük (cost-bónusz) → a későbbi utak rájuk futnak,
    //    és a hálózat ÖSSZEÉR ahelyett, hogy párhuzamos szálakat húzna.
}

A „már-épült út bónusz" — miért forr össze a hálózat

Egy apró, de varázslatos trükk: amikor a második út generálódik, a költségmezőbe belevesszük, hogy a már lerakott út cellái nagyon olcsók (pl. 0.2). Az eredmény: a második út, ahelyett hogy párhuzamosan menne az első mellett, rátalál az első útra, és azon megy tovább. Ez hozza létre a elágazásokat és a kereszteződéseket — anélkül, hogy explicit módon „elágazást" generálnál. A hálózat önszerveződővé válik: az út mindig a legolcsóbbat keresi, és a legolcsóbb a már meglévő út.

A bónusz = a „hálózat" eleme

Ez a trükk az, ami miatt az út-generálás nem egyszerű „A* sokszor". A cost() függvény (04) iteratívan újraszámolódik: miután az 1. út lekerült, a 2. út költségmezője már tartalmazza azt. Így a 7. tárgy-út automatikusan ráakad a korábbi utak hálózatára. Ez a „már-épült út bónusz" a 04-es költségmező harmadik (hálózati) rétege.

▶ Demó: teljes gráf vs. MST vs. + bónusz
Három különböző hálózat-stratégia ugyanazon a tárgyhalmazon. Teljes gráf: minden tárgyat mindennel összeköt — aszfalt-tenger. MST: minimális feszítőfa — legkevesebb út, de még párhuzamos szálak (nincs bónusz). MST + bónusz: a már-épült utak olcsóak → az utak rátalálnak a korábbiakra, elágaznak, összeérnek. Kapcsold be a biom-utat is, és lásd, ahogy a tárgy-hálózat rácsatlakozik a biom-út végére.

08

Út-csempék — autotiling a maszk alapján

Az út-réteg (01) eddig csak adat: „van-e út a cellán". Ebből kell csempévé válnia — és pont úgy, mint a terep autotilingja (proc-gen 10. szakasz), a csempe választása a szomszédokból történik. A különbség: az út egy vékony csík, így a maszk nem „mi van körülöttem", hanem „melyik szomszédokon van út". Ebből a maszkból indexeljük az út-csempét: egyenes, sarok, T, kereszt, csonk.

A 4-bites maszk — 16 csempe elég

Minden út-cellához megnézzük a 4 fő-irányú szomszédot (É/K/D/Ny), és felépítünk egy 4-bites maszkot: bit i = 1, ha az i-edik irányban út van. Ez a maszk (0..15) indexel egy út-csempét. A 16 maszkból azonban csak 6 eset hordoz jelentést — a többi szimmetria-változat:

/// 4-bites szomszéd-maszk: bit 0=É, 1=K, 2=D, 3=Ny. 1, ha ott út van.
fn road_mask(layer: &RoadLayer, x: i32, y: i32) -> u8 {
    let n = |dx, dy| layer.has(x + dx, y + dy) as u8;
    (n(0,-1) << 0) | (n(1,0) << 1) | (n(0,1) << 2) | (n(-1,0) << 3)
}

/// A maszk → csempe-típus. A forgatást (melyik irányba nyílik) a Sprite
/// `rotation`-ja oldja meg, így mindössze 6 CSEMPÉT kell rajzolni.
#[derive(Clone, Copy)]
enum RoadTile { Dot, Stub, Straight, Corner, TJunction, Cross }

fn road_tile(mask: u8) -> RoadTile {
    let count = mask.count_ones();
    match count {
        0 => RoadTile::Dot,
        1 => RoadTile::Stub,
        2 => if mask == 0b0101 || mask == 0b1010 { RoadTile::Straight } else { RoadTile::Corner },
        3 => RoadTile::TJunction,
        _ => RoadTile::Cross,        // count == 4
    }
}
6 csempe + forgatás = 16 maszk

Ez ugyanaz a trükk, mint a proc-gen autotiling: a maszk mondja meg mit, a forgatás mondja meg melyik irányba. A 16 maszkból így elég 6 csempét megrajzolni (pont, csonk, egyenes, sarok, T, kereszt), és a maradék 10 a forgatással előáll. A csempe-bitmaszkolás lecke ezt teljesen kifejti (8-bites sarkokkal, 47/256 redukcióval) — ha a 4-bites sima vonalak túl „geometriaiak", az a mélyebb verzió.

Hova kerül az út-csempe?

Az út-csempe külön entitás a terep-csempe fölött (kisebb Z-értékkel, hogy látszódjon). Soha ne írd felül a terep-csempét az út-csempével — a kettő különböző réteg (01). A megjelenítő rendszer csak akkor spawnol út-csempét, ha a RoadLayer szerint van ott út; ha az utat megszünteted (pl. egy quest hatására), csak az út-entitást tünteted el, a fű ott marad. Pont az az elválasztás, amit a biomok-lecke 10. szakasza is követ.

A teljes recept — minden együtt

Rakjuk össze. Egy teljes út-réteg generálása öt lépésből áll, és a lépések sorrendje számít — az 1. lépés kimenete a 2. lépés bemenete:

  1. Végpontok. Biom-utakra: a kapu (legközelebbi cellapár, 03). Tárgy-utakra: maguk a tárgyak (proc-gen 09).
  2. Költségmező. Terep-költség + lejtő-súly (+ már-épült út bónusz a hálózathoz, 04).
  3. Útvonal. A* 8-irányban, oktáli-heurisztikával, sarkon-vágás-szabállyal (05).
  4. Utófeldolgozás. String-pulling (simítás) + szélesítés (06).
  5. Rétegbe + csempe. RoadLayer.stamp() (01) → maszk → út-csempe (08).

10

Bevy integráció

A generálás egyetlen system, ami a biom-térkép elkészülte után fut (a biomok-lecke 10. szakaszának generálási fázisa után). A kimenet egy RoadLayer resource és a út-csempe entitások. A logika (út-réteg) és a megjelenítés (csempe) külön — ezért a stratégiát (költségmező súlyok) cserélheted a megjelenítő rendszer átírása nélkül.

use bevy::prelude::*;

const W: i32 = 64;
const H: i32 = 48;
const TILE: f32 = 32.0;

#[derive(Component)]
struct RoadTileMarker { x: i32, y: i32 }

/// A generálási rendszer: biomok után fut, az út-réteget építi.
/// A költségmező és az A* a 04/05-ből; itt csak a vezérlés látszik.
fn generate_roads(
    biomes: Res<BiomeMap>,
    landmarks: Query<&Landmark>,
    mut commands: Commands,
) {
    if !biomes.is_changed() { return; }      // csak ha (újra)generálódott a térkép

    let mut layer = RoadLayer::default();
    layer.cells = vec![None; (W * H) as usize];

    // 1 · BIOM-ÚT: sivatag → hegy kapu (03), A* (05), simítás (06), stamp (01)
    let desert: Vec<IVec2> = cells_of(&biomes, Biome::Desert);
    let mountain: Vec<IVec2> = cells_of(&biomes, Biome::Mountain);
    if let Some((a, b)) = gate(&desert, &mountain) {
        let cost = |f, t| cost(&biomes, f, t);
        let pass = |p| biomes.get(p) != Some(Biome::Water);
        if let Some(path) = astar(a, b, cost, pass) {
            let simple = simplify(&path, |x, y| los_clear(x, y, &biomes));
            layer.stamp(simple, RoadKind::BiomeLink);
        }
    }

    // 2 · TÁRGY-HÁLÓZAT: MST a tárgyakon (07), minden élen A* + bónusz.
    let pts: Vec<IVec2> = landmarks.iter().map(|l| l.pos).collect();
    build_landmark_network(&pts, &biomes, &mut layer);   // 07

    // 3 · MEGJELENÍTÉS: minden út-cellára spawnolunk egy út-csempe-entitást.
    for y in 0..H { for x in 0..W {
        if layer.has(x, y) {
            let mask = road_mask(&layer, x, y);
            let tile = road_tile(mask);
            let rot = rotation_for(mask);     // a forgatás a maszk alapján
            commands.spawn((
                Sprite {
                    image: road_atlas.clone(),         // 6 csempe (08)
                    texture_atlas: Some(TextureAtlas {
                        layout: road_layout.clone(),
                        index: tile as usize,
                    }),
                    custom_size: Some(Vec2::splat(TILE)),
                    color: road_tint_for(layer.cells[idx]),  // biom-út vs. tárgy-út szín
                    ..default()
                },
                Transform::from_xyz(x as f32 * TILE, y as f32 * TILE, 1.0), // a fű felett
                RoadTileMarker { x, y },
            ));
        }
    }}

    commands.insert_resource(layer);
}

Ha az út futásidőben változik (a játékos felépít egy újat, vagy egy quest nyit egy utat), nem kell újra-spawnolni a teljes réteget — csak a megváltozott cellákon frissíted a maszkot és a csempét. A Z-érték (1.0) biztosítja, hogy az út-csempe a terep fölött, de a tárgyak alatt jelenjen meg:

/// Futásidejű frissítés: ha a RoadLayer változik, az érintett cellák
/// maszkját és csempéjét újra számoljuk. Nem spawnolunk újra mindent.
fn refresh_road_tiles(
    layer: Res<RoadLayer>,
    mut tiles: Query<(&RoadTileMarker, &mut Sprite)>,
) {
    if !layer.is_changed() { return; }
    for (m, mut sprite) in &mut tiles {
        let mask = road_mask(&layer, m.x, m.y);
        let tile = road_tile(mask);
        if let Some(atlas) = &mut sprite.texture_atlas {
            atlas.index = tile as usize;     // csempe-típus
        }
        sprite.rotation = rotation_for(mask); // forgatás
    }
}
Skálázás

Cellánként egy út-entitás egy 64×48-as térképen kiváló tanulni, de egy nagy nyílt világhoz bevy_ecs_tilemap kell (chunkolt mesh) — ott a TileTextureIndex-et írod a maszk alapján. Az út-generálás logikája (költségmező, A*, MST) változatlan; csak a megjelenítő réteg cserélődik. Telepítés előtt nézd meg, melyik Bevy-verziót támogat a crate — lásd a biomok-lecke skálázási tippjét.

Mikor fut a generálás?

Az út-generálás egyszer fut, a biom-térkép generálása után. Ha a térkép statikus (egy dungeon, egy pálya), ez indításkor megtörténik. Ha a térkép folyamatosan generálódik (chunkok, végtelen világ), akkor a generálás chunkonként fut — és az utak át a chunk-határokon csak akkor folytatódnak, ha a költségmező a szomszéd chunk adatait is látja. Ez egy külön (haladó) téma; a gyakorlatban az utakat egy regionális léptékben tervezik meg, és csak a chunkokra „vágják”.

Gyakorlati tippek & buktatók

Tedd

Tartsd az utat külön rétegben (01) — soha ne írd az útat a Tile-ba. A terep, az út és a tárgy három független réteg; a megjelenítés és a játékmenet mindegyiket külön olvassa.

Ne

Ne használj centroidot biom-út végpontnak (03). A matematikai közép lehet a víz alatt vagy a hegy csúcsán. A kapu (legközelebbi cellapár) a helyes — ez adja a természetes átjárót.

Ne

Ne tedd a hegyet járhatatlannak (04). A hegy drága legyen, de járható — különben a „sivatagból fel a hegyekbe" biom-út értelmét veszti, mert a végpont maga a hegyben van. Csak a víz legyen tényleg tilos (és ott is érdemes 1e6, nem INFINITY).

Tedd

A „már-épült út bónusz" (07) az, ami a hálózatot összeforró varázslattá teszi. Kapcsold be: az utak rátalálnak a korábbiakra, elágazások és kereszteződések automatikusan jönnek létre — nem kell külön „elágazás"-logika.

Ne

Ne felejtsd el a sarkon-vágás szabályt (05). 8-irányú A* nélküle átmegy a hegy gerincén — narratíva-romboló. A két oldalsó szomszéd közül valamelyik járhatatlan → az átlós lépés tilos.

Vigyázz

A lejtő-súly (04) a legerősebb fűszer — de túl felhúzva az út túl kígyózó lesz, és elveszti az „út" karakterét. Kezdd 0.3 körül, és finomhangold a demóban.

Tedd

A biom-utakat előbb generáld, mint a tárgy-utakat (09). A biom-út a táj nagy szerkezetét adja; a tárgy-hálózat (a bónusszal) automatikusan rácsatlakozik. Így a két típus összenő, nem keresztezi egymást értelmetlenül.

A 6 csempe elég

A 4-bites maszkból 6 csempe + forgatás elég (08). Ha a 4-bites „lépcsőzetes" túl geometriai, a 8-bites bitmaszkolás vagy a dual-grid simábbat ad — de a 4-bites a legtöbb úthálózatnál bőven elég. Ne túlbonyolítsd.

Tedd

A generálást determinisztikusan végezd a seedelt zajból. Soha ne vigyél be rand()-ot az A*-ba vagy a végpont-választásba — a koordináta hash (a biomok-lecke mintája) a helyes, különben az út minden indításkor más.

TL;DR

A lényeg egy kártyán

  • Az út = külön réteg (01): RoadLayer (Option<u32> cellánként) a terep-réteg fölött. Soha ne írd az útat a Tile-ba — a terep, az út és a tárgy három független réteg.
  • Két út-típus: biom-út (két biom-régiót köt, a végpontokat választani kell) és tárgy-út (konkrét kitüntetett pontokat). A különbség a végpont-kiválasztásban és a költségmező súlyozásában van; maga a nyomkeresés közös.
  • Végpontok (03): a tárgy-út végpontjai adottak; a biom-út végpontja a kapu = a két régió legközelebbi cellapárja. Ne centroidot használj — az gyakran rossz cellán van.
  • Költségmező (04): terep-költség (víz tilos, síkság olcsó, hegy drága de járható) + lejtő-súly (magasságkülönbség-büntetés) + már-épült út bónusz (hálózat). Ez teszi az utat tájkövetővé.
  • A* (05): 8-irányú, oktáli-heurisztika, sarkon-vágás-szabály, √2-es átlós lépésköltség. Determinisztikus — sose rand().
  • Simítás & szélesítés (06): string-pulling (felesleges kanyarok elhagyása, line-of-sight-tal) + oldalsó szomszédok megjelölése (1 cella → 3 cella széles).
  • Hálózat (07): MST a tárgyakon (minimális út, ami mindent elér), minden élen A*. A „már-épült út bónusz" miatt az utak rátalálnak a korábbiakra → elágazások automatikusan jönnek.
  • Út-csempék (08): 4-bites szomszéd-maszk → 6 csempe-típus (pont, csonk, egyenes, sarok, T, kereszt) + forgatás. A junction-öt a maszk automatikusan felismeri.
  • Sorrend (09): biom-út előbb, mint a tárgy-hálózat; a bónusz miatt a tárgy-utak rácsatlakoznak a biom-útra. Előbb adat, aztán kép.
  • Bevy (10): egy generálási rendszer a biomok után; az út-csempe külön entitás Z=1-en a fű felett. Futásidejű változás: csak a megváltozott cellák maszkját/csempéjét frissítsd. Nagy világ: bevy_ecs_tilemap.
