Observerek mélyebben: Event, EntityEvent és a propagáció

Az üzenetek (Message) türelmesen várnak a pufferben, amíg valaki ki nem olvassa őket. Az observerek a türelmetlen testvérek: a trigger pillanatában azonnal lefutnak — entitásra célozva, a hierarchián felfelé buborékolva, akár láncreakcióban. Ez a tutorial a Bevy 0.18 observer-rendszerét veszi végig az első add_observer-től a propagáció finomhangolásáig.

1. rész — Alapok

Mi az observer?

Az előző részben láttuk a pull-modellt: az író üzenetet tesz egy pufferbe, és az olvasó rendszer akkor dolgozza fel, amikor az ütemezésben sorra kerül. Az observer ennek a tükörképe, a push-modell: amikor valahol megtörténik egy trigger, a Bevy azonnal, helyben lefuttatja az összes érintett observert — még mielőtt a triggerelő kód folytatódna a következő utasítással.

Az observer tehát egy callback-rendszer: ugyanúgy system-paraméterekkel dolgozik (Query, Res, Commands…), mint egy normál rendszer, de nem egy ütemezőben él, hanem eseményekre van „felakasztva”. Ha használtad már a picking rendszert (Pointer<Click> és társai), akkor már írtál observert — most megnézzük, mi van a motorháztető alatt.

  • Azonnaliság: nincs puffer, nincs frame-késés — és nincs „lejárt, elveszett” üzenet sem.
  • Célozhatóság: az esemény szólhat az egész világnak, vagy egyetlen konkrét entitásnak — ez a Message-eknél nem létezik.
  • Buborékolás: az entitás-események a hierarchián (pl. ChildOf) felfelé tovább tudnak terjedni, mint a DOM-eventek a böngészőben.
  • Ára: nincs kötegelés — minden trigger külön observer-futás. Ezért a gyakori, tömeges jelzésekre továbbra is a Message a jó eszköz.
Bevy fogalom

Observer — eseményre kötött rendszer

Egy függvény, amelynek első paramétere kötelezően On<E> (vagy életciklus-eseménynél On<E, B>), a többi paramétere pedig bármilyen szokásos system-paraméter lehet. A trigger(E) hívás minden illeszkedő observert szinkron lefuttat.

A sandboxban egy kis entitás-hierarchián nézheted majd élőben, ahogy egy találat-esemény végigbuborékol a csontváz-boss testrészein, és útközben sorra elsülnek az observerek.

1. rész — Alapok

Event és EntityEvent — a két eseményfajta

A 0.17-es nagy átnevezés óta (lásd a 2. rész fordítótáblázatát) az Event szó kizárólag az observer-világé. Ezen belül a 0.18-ban két derive közül választasz:

#[derive(Event)]#[derive(EntityEvent)]
Célpont az egész világ („broadcast”) egy konkrét entitás — az eseménystruct egy Entity mezőben hordozza
Kit futtat? a globális (app-szintű) observereket a globális observereket és a célentitásra akasztott entity-observereket
Propagáció nincs (nincs minek mentén) opcionális — pl. ChildOf mentén felfelé
Tipikus példa WaveStarted, GamePaused Hit { entity, damage }, Pointer<Click>
events.rsrust
use bevy::prelude::*;

/// Globális esemény: nincs címzettje, "a világnak szól".
#[derive(Event)]
struct WaveStarted {
    wave: u32,
}

/// Entitás-esemény: a struct maga hordozza, KIRŐL szól.
/// Az `entity` nevű mezőt a derive automatikusan célpontnak tekinti.
#[derive(EntityEvent)]
struct Hit {
    entity: Entity,
    damage: u32,
}
Bevy fogalom

A Trigger asszociált típus

Az Event trait belül egy Trigger típust hordoz, ami azt írja le, hogyan kell az eseményt kézbesíteni: a sima Event globális triggert kap (csak broadcast observerek), az EntityEvent entitás-triggert (broadcast + entity-observerek + opcionális propagáció). Ezt a derive intézi — a lényeg, hogy a típusrendszer garantálja: globális eseményt nem tudsz véletlenül entitásra célozni, és fordítva.

Regisztráció? Az observer-eseményeknél nincs add_message-szerű lépés — nem kell (és nem is lehet) add_event-tel regisztrálni őket. A derive + az első observer felakasztása minden, ami kell. Ez is jelzi: itt nincs puffer, amit karban kéne tartani.
1. rész — Alapok

Az első observer — a minimális példa

Három lépés: (1) eseménytípus, (2) observer felakasztása, (3) trigger. Itt a fordítható minimum egy globális eseménnyel:

main.rsrust
use bevy::prelude::*;

#[derive(Event)]
struct WaveStarted {
    wave: u32,
}

fn main() {
    App::new()
        .add_plugins(DefaultPlugins)
        // (2) globális observer: minden WaveStarted-re lefut
        .add_observer(on_wave_started)
        .add_systems(Update, start_wave_on_key)
        .run();
}

/// Az observer: első paraméter KÖTELEZŐEN On<E>,
/// utána bármilyen system-paraméter jöhet.
fn on_wave_started(
    wave: On<WaveStarted>,
    mut commands: Commands,
    assets: Res<AssetServer>,
) {
    info!("Indul a(z) {}. hullám!", wave.wave);
    // pl. spawnolhatunk ellenfeleket, UI-feliratot...
}

/// (3) Trigger: bárhonnan, ahol van Commands.
fn start_wave_on_key(
    keys: Res<ButtonInput<KeyCode>>,
    mut commands: Commands,
) {
    if keys.just_pressed(KeyCode::Enter) {
        commands.trigger(WaveStarted { wave: 1 });
    }
}
Bevy API

App::add_observer / commands.trigger

Az add_observer(rendszer) egy globális (broadcast) observert regisztrál: a megadott függvény minden olyan triggerre lefut, amelynek eseménytípusa egyezik az On<E> paraméterével. A commands.trigger(esemény) pedig kiváltja az eseményt — globálisat és entitás-eseményt ugyanazzal a hívással (utóbbinál a célpont a struct mezőjében utazik).

Az On<E> deref-elhető az eseményre, ezért írhattunk wave.wave-et wave.event().wave helyett.

2. rész — A gépezet

Mikor fut pontosan az observer?

A „azonnal lefut” kétféle azonnalt jelenthet, attól függően, honnan triggerelsz:

  • commands.trigger(...) — a trigger egy parancs, tehát a soron következő command flush-nál fut le (tipikusan a rendszered lefutása után, még ugyanabban a frame-ben). Ugyanúgy halasztott, mint a commands.spawn.
  • world.trigger(...) — exkluzív világ-hozzáférésnél (exclusive system, observer belseje Commands nélkül stb.) az observerek szó szerint a hívás helyén, szinkron futnak le, és csak utánuk folytatódik a kódod.

Láncreakciók

Mivel az observer maga is kaphat Commands-ot, simán triggerelhet újabb eseményt — amire megint observerek futnak, és így tovább. Ez nagyon kifejező (dominó-logika: Hit → Died → LootDropped), de két dologra figyelj:

1. Végtelen lánc: ha A esemény observere B-t triggerel, B observere pedig A-t, az végtelen rekurzió — a Bevy nem véd meg tőle automatikusan.
2. Sorrend az observerek közt: ha ugyanarra az eseményre több observer is fel van akasztva, a futási sorrendjük nem garantált — nincs .before()/.after() observerekre. Ha a kezelőid sorrendje számít, az erős jel, hogy ott inkább Message + rendezett rendszerek valók.
2. rész — A gépezet

Az On<E, B> anatómiája és az életciklus-események

Az observer első paramétere az On, aminek két generikus paramétere van — és a második rendszeresen összezavarja a kezdőket:

bevy_ecs (egyszerűsítve)rust
pub struct On<'w, E, B: Bundle = ()> {
    // belül: referencia az eseményre + metaadatok
}
  • E — az eseménytípus. A saját eseményeidnél csak ezt használod: On<Hit>.
  • B — komponens-szűrő, ami gyakorlatilag csak a beépített életciklus-eseményeknél értelmes: On<Add, Enemy> = „amikor egy entitásra Enemy komponens kerül”.

A beépített életciklus-események

A Bevy minden komponens-műveletre eseményt vált ki — ezekre observerrel reagálni az egyik legtisztább minta arra, hogy a „mi történjen, amikor megjelenik/eltűnik valami” logika egy helyen lakjon:

EseményMikor sül el?
Addamikor a komponens rákerül egy entitásra, amelyiken eddig nem volt
Insertminden beszúrásnál — akkor is, ha már volt és felülíródik
Replaceamikor egy meglévő komponens értéke lecserélődik
Removeamikor a komponens lekerül az entitásról (még elérhető az értéke)
Despawnamikor az entitást despawnolják
életciklus-observerrust
#[derive(Component)]
struct Enemy;

/// Lefut, valahányszor BÁRMELY entitásra Enemy komponens kerül.
fn on_enemy_added(
    added: On<Add, Enemy>,
    positions: Query<&Transform>,
) {
    // az életciklus-esemény entitás-esemény: tudjuk, KIRŐL szól
    if let Ok(tf) = positions.get(added.entity) {
        info!("új ellenfél itt: {:?}", tf.translation);
    }
}

fn main() {
    App::new()
        .add_plugins(DefaultPlugins)
        .add_observer(on_enemy_added)
        .run();
}
A B szűrő VAGY-logikájú! Az On<Add, (Enemy, Chest)> akkor sül el, ha Enemy vagy Chest kerül egy entitásra — nem akkor, ha mindkettő. Ez eltér a Query-szűrők megszokott ÉS-logikájától, és klasszikus meglepetésforrás.
Bevy fogalom

Életciklus-observer vs. Added/RemovedComponents

Ugyanerre az információra létezik pull-oldali eszköz is: az Added<T> Query-szűrő és a RemovedComponents<T> paraméter, amelyek frame-enként, kötegelve adják a változásokat. Az ökölszabály ugyanaz, mint mindig: ha azonnali, eseti reakció kell (pl. felszereléskor azonnal számoljuk újra a statokat) → observer; ha frame-enként, tömegesen dolgozod fel → maradj a Query-szűrőknél.

2. rész — A gépezet

Entity-observerek: esemény egyetlen entitásnak

Az igazi erő ott kezdődik, hogy observert nemcsak globálisan, hanem egy konkrét entitásra is felakaszthatsz az EntityCommands::observe-val. Ilyenkor az observer csak akkor fut le, ha az eseményt pont arra az entitásra triggerelték:

combat.rsrust
#[derive(EntityEvent)]
struct Hit {
    entity: Entity,   // a célpont — a derive ezt a mezőt használja
    damage: u32,
}

#[derive(Component)]
struct Health(u32);

fn spawn_boss(mut commands: Commands) {
    commands
        .spawn((Name::new("Csontváz boss"), Health(100)))
        // entity-observer: CSAK ezen az entitáson elsülő Hit-ekre fut
        .observe(on_boss_hit);
}

fn on_boss_hit(
    hit: On<Hit>,
    mut healths: Query<&mut Health>,
) {
    if let Ok(mut hp) = healths.get_mut(hit.entity) {
        hp.0 = hp.0.saturating_sub(hit.damage);
        info!("Boss megsérült! Maradt: {} HP", hp.0);
    }
}

/// Trigger ugyanúgy, mint a globálisnál — a címzett a structban utazik:
fn sword_attack(mut commands: Commands, boss: Entity) {
    commands.trigger(Hit { entity: boss, damage: 12 });
}

Egy EntityEvent triggerelésekor a sorrend tehát: lefutnak a célentitásra akasztott entity-observerek, és a globális (app-szintű) observerek is — utóbbiak minden ilyen eseményt látnak, entitástól függetlenül. Így építhető „helyi viselkedés + központi könyvelés” páros: a boss a saját observerével kezeli a sebzést, egy globális observer pedig pl. harci naplót ír.

Bevy fogalom

Az observer maga is entitás!

Az observe(...) és az add_observer(...) a háttérben egy entitást spawnol, rajta egy Observer komponenssel — az observer tehát ugyanolyan polgára a World-nek, mint bármi más. Ebből két hasznos dolog következik: a megfigyelt entitás despawnolásakor a rá akasztott observerek is automatikusan eltakarodnak, és egy observer kézzel is leszedhető (despawnolható), ha már nincs rá szükség.

Honnan tudja a derive, melyik mező a célpont? Alapesetben az entity nevű mezőt keresi. Ha máshogy hívod (pl. ship), jelöld meg a #[event_target] attribútummal — a következő fejezet propagációs példájában pont ezt látod majd.
3. rész — Haladó minták

Propagáció: buborékolás a hierarchián

Ha dolgoztál már böngészős JS-sel, ismerős lesz: egy gombra kattintva az esemény „felbuborékol” a szülő elemekre is. A Bevy entitás-eseményei ugyanezt tudják — alapból a ChildOf kapcsolat mentén, felfelé. Ez akkor aranyat ér, amikor az esemény fizikailag egy részen történik (a kard pengéjén landol a kattintás, a boss kezét éri a nyíl), de a logika a egészhez tartozik (a bossnak kell sebződnie).

propagáló eseményrust
/// auto_propagate: kezelés után magától megy tovább a szülőre.
/// propagate = ...: MILYEN kapcsolat mentén — itt a beépített ChildOf.
#[derive(EntityEvent)]
#[entity_event(auto_propagate, propagate = &'static ChildOf)]
struct Hit {
    #[event_target]      // nem "entity" a neve, ezért jelöljük
    part: Entity,
    damage: u32,
}

fn spawn_boss(mut commands: Commands) {
    commands
        .spawn(Name::new("Boss"))
        .observe(on_boss_hit)            // a gyökéren ül a logika
        .with_children(|parent| {
            parent.spawn(Name::new("Kar"))
                  .with_children(|kar| {
                      kar.spawn(Name::new("Kard"));
                  });
        });
}

/// A Kardra triggerelt Hit felbuborékol: Kard → Kar → Boss,
/// és a Boss observere is megkapja.
fn on_boss_hit(hit: On<Hit>, names: Query<&Name>) {
    // hit.part            — az AKTUÁLIS célpont (ahol most jár a buborék)
    // hit.original_target() — ahol az esemény EREDETILEG keletkezett (a Kard)
    info!("találat ide érkezett: {:?}", names.get(hit.original_target()));
}

A buborék megállítása

Az observer menet közben dönthet úgy, hogy elnyeli az eseményt — például egy pajzs-testrész kivédi a találatot, és nem engedi tovább a boss felé:

propagate(false)rust
fn on_shield_hit(mut hit: On<Hit>) {
    info!("A pajzs kivédte a találatot!");
    hit.propagate(false);   // a buborék itt megáll, a szülők már nem kapják meg
}

Fordítva is működik: ha nem kérsz auto_propagate-et, az esemény alapból csak a célentitáson sül el, és az observer propagate(true)-val kézzel küldheti tovább — így csak ott buborékol, ahol kifejezetten akarod.

Bevy fogalom

Miért pont a picking miatt fontos ez?

A beépített picking-események (Pointer<Click>, Pointer<Over>…) pontosan így működnek: a kattintás a konkrét meshen/sprite-on keletkezik, majd felbuborékol a hierarchián, így a kezelő observert elég a logikai gyökérre (gomb, kártya, egység) akasztani. Amikor a saját eseményeidnél propagációt tervezel, ezt a mintát másolod.

Nem csak ChildOf! A propagate = &'static ChildOf tetszőleges relationship-komponensre cserélhető — ha a 4. (Relationships) témából saját kapcsolatot építettél (pl. RészeEnnek), az események amentén is tudnak utazni.
3. rész — Haladó minták

0.18-as változás: immutábilis EntityEvent

Ez az a pont, ahol a 0.18 ténylegesen hozzányúlt az observer-világhoz. A 0.17-ben az EntityEvent trait tartalmazott két mutáló metódust (from és event_target_mut), amelyekkel az esemény célpontja menet közben átírható volt. A 0.18 ezeket kiszervezte egy külön traitbe, a SetEntityEventTarget-be — így minden EntityEvent alapértelmezésben immutábilis.

  • Miért jó ez? A célpont átírása a propagáció motorja — de ha bárki bárhol átírhatja, az nehezen követhető hibákat szül. A 0.18-ban a típusrendszer mondja ki: egy sima entitás-esemény célpontja a trigger után nem mozog.
  • Ki kapja meg automatikusan? A propagáló események — pl. amin #[entity_event(propagate)] vagy auto_propagate van — automatikusan implementálják a SetEntityEventTarget-et, hiszen a buborékolásnak tényleg át kell írnia a célpontot lépésenként.
  • Mikor találkozol vele? Ha generikus kódot írsz, ami eseményeket továbbít/átcéloz, a 0.18-ban E: EntityEvent helyett E: EntityEvent + SetEntityEventTarget megkötés kell. A hétköznapi observer-kódot a változás nem érinti.
0.17 → 0.18rust
// 0.17 — a mutáló metódusok az EntityEvent részei voltak:
fn retarget<E: EntityEvent>(ev: &mut E, new_target: Entity) {
    *ev.event_target_mut() = new_target;
}

// 0.18 — külön trait kell hozzá:
fn retarget<E: EntityEvent + SetEntityEventTarget>(ev: &mut E, new_target: Entity) {
    *ev.event_target_mut() = new_target;
}
A többi 0.18-as apróság az observer-környéken: az AnimationEventTrigger félrevezető animation_player mezője target-re lett átnevezve, a picking DragEnter eseménye pedig már a húzás indulásakor is elsül a már hoverelt entitáson. A core observer-API (On, add_observer, observe, trigger) változatlan a 0.17-hez képest.
3. rész — Haladó minták

Korlátok és buktatók

1. Nincs sorrend az observerek között

Ugyanarra az eseményre akasztott observerek futási sorrendje nem definiált, és nincs rá rendező API. Ha azon kapod magad, hogy „előbb fusson a sebzés-observer, aztán a halál-ellenőrző”, akkor vagy vond össze őket egy observerbe, vagy láncold őket eseményekkel (Hit observere triggereli a Died-et), vagy térj át Message-re, ahol az olvasó rendszerek szabadon rendezhetők.

2. Nincs kötegelés

Ezer trigger = ezer külön observer-futás, futásonként paraméter-előkészítéssel. A 2. rész refresh_display-mintája — „gyűjtsd össze a frame összes változását, és dolgozd fel egy menetben” — observerrel nem kifejezhető. Gyakori, tömeges jelzésekhez (cellaváltozások, sebzés-tickek, részecske-triggerek) maradj a Message-nél.

3. A „mindent observerrel” csapda

Az observer csábítóan kényelmes, és könnyű mindent rá építeni — de minden trigger-lánc egy darab futási folyamat, amit az ütemező nem lát: nem párhuzamosítja, nem rendezi, a profilerben is nehezebben követhető. Az egészséges arány a legtöbb játékban: a folyamatos játéklogika rendszerekben + Message-ekkel él, az observer pedig a ritka, pontszerű, azonnali reakciók eszköze (felvettem egy tárgyat, meghalt valami, rákattintottak valamire, komponens került fel/le).

Apró, de fájdalmas: az observer csak akkor fut le, ha a trigger pillanatában már létezik. Ha egy rendszer előbb triggerel, és az observer csak utána kerül felakasztásra (akár ugyanabban a frame-ben), az az esemény végleg kimaradt — itt nincs kettős puffer, ami megmentené. Az observereket ezért tipikusan Startup-ban / spawn-kor érdemes regisztrálni.
3. rész — Haladó minták

Event vagy Message? — a teljes döntési térkép

A 2. rész táblázatának kibővített, observer-szemszögű változata — ez a két tutorial közös gyorsreferenciája:

Event + Observer (push)Message (pull)
Ideális gyakoriság ritka, pontszerű gyakori, tömeges
Feldolgozás eseményenként, azonnal kötegelve, az olvasó ütemezésekor
Latencia 0 (szinkron) 0–1 frame
Célozhatóság világ VAGY konkrét entitás csak világ
Propagáció van (hierarchián buborékol) nincs
Kezelők sorrendje nem definiált rendszer-rendezéssel pontosan szabályozható
Elveszhet? nem (de: csak a már létező observerek kapják meg) igen, 2 frame után, ha senki sem olvassa
Tipikus felhasználás picking, életciklus-reakciók, tárgyfelvétel, halál, UI cellaváltozások, sebzés-tickek, input-szándékok, naplózás

És a hüvelykujjszabály egy mondatban: ha a kérdésed „mi történjen, amikor EZZEL történik valami?” — observer; ha „mi történt ebben a frame-ben?” — Message.

4. rész — Gyakorlat

Sandbox: buborékolás élőben

A csontváz-boss három testrészből áll, ChildOf hierarchiában: a gyökéren ül a Boss entity-observere (ő könyveli a sebzést), a Pajzs-on egy védekező observer, a Kard-on semmi. Kattints bármelyik testrészre: oda triggerelődik egy Hit { damage: 12 } entitás-esemény, és — ha az auto_propagate be van kapcsolva — végigbuborékol felfelé. A napló minden observer-futást mutat, sorrendben.

  • Kattints a Kardra: nincs saját observere, de a buborék a Karon át eljut a Bossig — a sebzés mégis „megérkezik”. Pont ezért szeretjük a propagációt.
  • Kattints a Pajzsra bekapcsolt propagate(false) mellett: a pajzs observere lefut, de a buborékot elnyeli — a Boss HP-ja nem csökken.
  • Kapcsold ki az auto_propagate-et, és kattints a Kardra: az esemény csak ott sül el — observere viszont nincs, így (a harci naplón kívül) semmi sem történik.
  • A + Rúnakő gombbal spawnolj egy új testrészt a Kard alá: az On<Add, Rúnakő> életciklus-observer azonnal jelez — despawnnál az On<Remove>.

Összefoglalás

A teljes kép, hat mondatban:

  1. Az observer push-modellű eseménykezelő: a trigger pillanatában (command flush-kor, vagy world.trigger-nél szó szerint azonnal) szinkron lefut — puffer és frame-késés nélkül.
  2. Két eseményfajta van: #[derive(Event)] a globális, #[derive(EntityEvent)] az entitásra célzott eseményekhez — utóbbi a célpontot egy Entity mezőben hordozza (entity név vagy #[event_target]).
  3. Observert globálisan az add_observer-rel, entitásra az EntityCommands::observe-val akasztasz fel; az első paraméter mindig On<E>, az életciklus-eseményeknél On<E, B> (és a B szűrő VAGY-logikájú!).
  4. A beépített életciklus-események (Add, Insert, Replace, Remove, Despawn) observerrel kezelve adják a legtisztább „spawn/despawn-hoz kötött logika” mintát.
  5. Az entitás-események a #[entity_event(auto_propagate, propagate = ...)] attribútumokkal a hierarchián felfelé buborékolnak; az observer propagate(false)-szal elnyelheti, az original_target()-tel pedig mindig tudja, honnan indult.
  6. A 0.18 fő újdonsága: az EntityEvent alapból immutábilis, a célpont-átírás külön traitbe (SetEntityEventTarget) került — a hétköznapi kódot ez nem érinti, a generikus event-továbbítókat igen.
