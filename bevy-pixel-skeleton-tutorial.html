01

Mit építünk?

A célunk egy skeleton — egy minimális, de jól strukturált játékváz, amire később bármilyen topdown pixel art játék (farm szimulátor, survival, RPG) ráépíthető. A skeleton funkciói:

  • Topdown nézet, pixel art rendereléssel (éles, nem elmosott pixelek)
  • Zöld mező mint scene, csempékből (tile-okból) felépítve
  • Elszórt sprite-ok a mezőn (bokrok, kövek, virágok)
  • Scrollozás: a kamera mozgatható a mezőn (WASD / nyilak)
  • Idő-panel a jobb felső sarokban: ingame óra, napszak, eltelt napok, évszak
  • Inventory sáv alul 4 slottal — a felső panellel együtt mindig látható, a kamerától függetlenül

Miért skeleton, és miért nem „kész játék”?

Egy játékprojekt legnagyobb kockázata az elején van: ha az alaprendszerek (renderelés, idő, UI, input) rosszul strukturáltak, minden későbbi feature ezekkel fog harcolni. A profi workflow ezért mindig egy vertikális szelettel indul: minden réteg (világ, kamera, idő, UI) megvan minimális formában, és már az első naptól futtatható. Tartalmat hozzáadni könnyű — architektúrát utólag javítani fájdalmas.

Mire lesz szükséged

  • Telepített Rust toolchain (rustup, stabil csatorna) — rustup.rs
  • Egy editor Rust-támogatással (VS Code + rust-analyzer, vagy RustRover)
  • Git a verziókezeléshez
  • Rust alapismeret (ownership, traitek, enumok) — Bevy-ismeret nem kell, azt itt tanuljuk

A tutorial a Bevy 0.16-ra épül. A Bevy gyorsan fejlődik és a minor verziók között törő API-változások vannak — a 12. fejezetben megnézzük, hogyan kezeld ezt a workflow részeként.

02

A workflow madártávlatból

Mielőtt egyetlen sor kódot írnánk, nézzük meg, hogyan dolgozik az ember egy ilyen skeletonon. A fejezetek sorrendje nem véletlen — ez maga a workflow.

1. lépés — Feature-bontás papíron

A követelménylistát (lásd 1. fejezet) független funkciókra bontjuk. A jó bontás ismérve: minden funkció önállóan fejleszthető és tesztelhető, és minimális a függés köztük.

FunkcióFelelősségeMitől függ?
cameraCamera2d létrehozása, mozgatás, pixel art beállításokworld (térképméret a clampoláshoz)
worldMező (tile-ok) és elszórt sprite-ok spawnolása—
game_timeIngame idő léptetése; óra/napszak/nap/évszak kiszámítása—
hudFelső idő-panel kirajzolása és frissítésegame_time (olvassa)
inventoryInventory állapot + alsó 4 slotos sáv—

Figyeld meg: a game_time nem tud a HUD-ról. Az idő-logika tisztán adat; a HUD csak olvassa. Ez a szétválasztás (logika vs. megjelenítés) később ingyen ad tesztelhetőséget és cserélhető UI-t.

2. lépés — A bontás leképezése Bevy-fogalmakra

A Bevy egy ECS (Entity–Component–System) engine. A fenti táblázat oszlopai szinte egy-az-egyben megfeleltethetők Bevy-fogalmaknak:

  • Entity — egy „dolog” a játékban: egy fűcsempe, egy bokor, a kamera, egy UI-slot. Önmagában csak egy azonosító.
  • Component — adat, amit egy entityre ragasztunk: Transform (pozíció), Sprite (kinézet), vagy saját markerek, pl. Prop.
  • System — egy sima Rust függvény, ami componenseken/erőforrásokon dolgozik: pl. „minden frame-ben mozgasd a kamerát az input szerint”.
  • Resource — globális singleton adat: pl. a GameTime vagy az Inventory.
  • Plugin — a csomagolás egysége: egy plugin összefogja egy funkció összes systemjét, resource-át. Egy funkció = egy plugin = egy file. Ez lesz a projektstruktúránk gerince.

Ökölszabály

Ha egy új feature-ön gondolkodsz, mindig ebben a sorrendben kérdezz: Milyen adat kell? (component vagy resource) → Mikor és mi módosítja? (systemek) → Hova csomagolom? (plugin). A Bevy-fejlesztés 80%-a ez a három kérdés.

3. lépés — Az iterációs ciklus

A fejlesztés nem „megírom az egészet, aztán futtatom”, hanem rövid ciklusok sorozata:

  1. Egy plugin / egy kis funkció megírása (15–45 perc munka)
  2. cargo run — vizuális ellenőrzés: azt látom-e, amit vártam?
  3. Ha igen: git commit. Ha nem: debug (11. fejezet), javítás, újra.

Minden fejezet végén lesz egy Ellenőrzőpont doboz: pontosan leírja, mit kell látnod a képernyőn, és mit érdemes commitolni. Így ha valami elromlik, mindig tudod, melyik lépésnél.

4. lépés — Placeholder-first grafika

Pixel art játéknál csábító rögtön asseteket rajzolni. Ne tedd. A skeleton fázisban minden grafika egyszínű négyzet (a Bevy ezt natívan tudja, asset nélkül): a fű zöld csempe, a bokor sötétzöld kocka, az item színes négyzet a slotban. Amikor a rendszerek működnek, a placeholder lecserélése egy PNG-re egyetlen sor módosítása — ezt minden érintett helyen kommentben jelölöm is.

Miért éri ez meg?

1) A logikai hibák színes kockákkal ugyanúgy látszanak, mint kész grafikával. 2) Nem kötöd magad korai art-döntésekhez (tile-méret, paletta). 3) A git history tiszta marad: a kód és az assetek külön életet élnek.

03

Projekt setup

Hozzuk létre a projektet, és azonnal állítsuk be azt a két dolgot, ami a Bevy-fejlesztés komfortját meghatározza: a gyors újrafordítást és a könyvtárstruktúrát.

Projekt létrehozása

terminál
cargo new pixel-skeleton
cd pixel-skeleton
cargo add bevy@0.16
cargo add rand@0.8

A rand crate-et a sprite-ok véletlenszerű szétszórásához használjuk majd a 6. fejezetben.

Cargo.toml — gyors fordítás beállítása

A Bevy nagy függőség, és Rustban az alapértelmezett dev-fordítás lassú futásidejű kódot ad. A bevett gyakorlat: a saját kódunkat enyhén optimalizáljuk, a függőségeket (köztük a Bevy-t) pedig teljesen — ezeket úgyis csak egyszer fordítja le a cargo, utána cache-eli.

Cargo.toml
[package]
name = "pixel-skeleton"
version = "0.1.0"
edition = "2021"

[dependencies]
bevy = "0.16"
rand = "0.8"

# A sajat kodunk: enyhe optimalizacio, gyors fordulas
[profile.dev]
opt-level = 1

# A fuggosegek (Bevy is): teljes optimalizacio.
# Egyszer fordul le, utana cache-bol jon — de a jatek
# dev modban is jatszhato sebessegu lesz.
[profile.dev.package."*"]
opt-level = 3

Még gyorsabb iteráció (opcionális)

A cargo run --features bevy/dynamic_linking a Bevy-t dinamikus libként linkeli, ami drasztikusan gyorsítja az újrafordítást fejlesztés közben. Release buildbe soha ne kerüljön — ezért flag-ként használd, ne a Cargo.toml-ba írd. Linuxon ehhez kellhet a mold vagy lld linker is; a Bevy hivatalos setup oldala platformonként leírja.

04

Az app váza — main.rs

Az első kód, amit megírunk, a main.rs — és ez az utolsó alkalom, hogy hozzányúlunk érdemben. A main.rs egyetlen feladata: összerakni az appot a pluginekből. Most még csak az „üres” vázat írjuk meg; a saját pluginjeink sorai kommentben várakoznak, és fejezetről fejezetre kommentezzük ki őket. Így a projekt minden fejezet végén fordul és fut.

src/main.rs
use bevy::prelude::*;

// A modulok mar leteznek (ures file-okkent), itt regisztraljuk oket.
// Fejezetrol fejezetre kommentezzuk ki a hozzajuk tartozo plugint lent.
mod camera;
mod game_time;
mod hud;
mod inventory;
mod world;

fn main() {
    App::new()
        .add_plugins(
            DefaultPlugins
                // PIXEL ART KULCSBEALLITAS: a texturak nagyitasa
                // "nearest neighbor" modban tortenjen, ne linearisan.
                // Enelkul minden sprite elmosodott lenne.
                .set(ImagePlugin::default_nearest())
                .set(WindowPlugin {
                    primary_window: Some(Window {
                        title: "Pixel Skeleton".into(),
                        resolution: (1280.0, 720.0).into(),
                        ..default()
                    }),
                    ..default()
                }),
        )
        // Hatterszin arra az esetre, ha valahol kilatszana a "vilag moge"
        .insert_resource(ClearColor(Color::srgb(0.08, 0.10, 0.08)))
        .add_plugins((
            camera::CameraPlugin,      // 05. fejezet
            // world::WorldPlugin,     // 06. fejezet
            // game_time::GameTimePlugin, // 08. fejezet
            // hud::HudPlugin,         // 09. fejezet
            // inventory::InventoryPlugin, // 10. fejezet
        ))
        .run();
}

Mit csinál itt mi?

  • App::new() — a Bevy app builder. Mindent (plugin, resource, system) erre fűzünk fel.
  • DefaultPlugins — a Bevy „batteries included” csomagja: ablak, renderelés, input, asset-kezelés, idő, hang… A .set(...) hívásokkal egy-egy beépített plugint konfigurálunk át.
  • ImagePlugin::default_nearest() — a pixel art beállítás. A GPU alapból lineárisan interpolálja a felnagyított textúrákat (ez fotóknál szép, pixel artnál katasztrófa). A nearest mód minden képpontot éles négyzetként nagyít.
  • ClearColor — az első saját resource-unk: a Bevy minden frame elején erre a színre törli a képernyőt.

Gyakori hiba

Ha a mod sorok bent vannak, de a file-ok üresek, a fordító még nem szól. Amint viszont egy pluginre hivatkozol, ami még nincs megírva, cannot find ... in module hibát kapsz. Ezért tartjuk kommentben a még meg nem írt plugineket — a „mindig forduló main” a workflow része.

Ellenőrzőpont

A camera::CameraPlugin sort még kommentezd vissza (az 5. fejezetben írjuk meg), majd cargo run: egy 1280×720-as, sötét, üres ablakot kapsz „Pixel Skeleton” címmel. Ez az első futó build — git commit -m "feat: app vaz, ablak, pixel art image sampling"

05

Kamera és pixel art renderelés

Az első igazi plugin. Egyben ez a minta minden további pluginhez: struct + Plugin trait impl + systemek, mindez egyetlen file-ban. A mozgatást a 7. fejezetben adjuk hozzá; most csak a spawnolás és a pixel art szempontból fontos beállítások jönnek.

src/camera.rs
use bevy::prelude::*;

pub struct CameraPlugin;

impl Plugin for CameraPlugin {
    fn build(&self, app: &mut App) {
        // Startup schedule: egyszer fut le, az app indulasakor
        app.add_systems(Startup, spawn_camera);
    }
}

fn spawn_camera(mut commands: Commands) {
    commands.spawn((
        // A Camera2d "required components" mechanizmussal automatikusan
        // maga melle huzza a Camera, Transform, Projection stb. komponenseket.
        Camera2d,
        // Pixel artnal az elsimitas (MSAA) csak artana az eles eleknek:
        Msaa::Off,
        // A projekcio skalaja = zoom. 0.75 -> kicsit kozelebbi nezet,
        // a pixelek lathatoan nagyobbak. (1.0 = 1 vilag-egyseg = 1 kepernyopixel)
        Projection::from(OrthographicProjection {
            scale: 0.75,
            ..OrthographicProjection::default_2d()
        }),
    ));
}

És élesítsük a main.rs-ben — ez mostantól minden fejezet végén egy egysoros mozdulat lesz:

src/main.rs (módosítás)
        .add_plugins((
            camera::CameraPlugin,          // <-- kikommentezve
            // world::WorldPlugin,
            ...
        ))

A plugin-minta anatómiája

  • A Plugin trait build() metódusa kapja meg az App-ot — ide regisztrálunk mindent, ami a funkcióhoz tartozik. A main.rs-nek nem kell tudnia, mi van benne.
  • A spawn_camera egy system: paraméterei (itt: Commands) mondják meg a Bevy-nek, mihez kér hozzáférést. A Commands-szal entityket hozunk létre/törlünk.
  • A commands.spawn((A, B, C)) egy entityt hoz létre a tuple összes komponensével. Ez a tuple-spawnolás a Bevy leggyakoribb mozdulata.

Pixel art döntés: zoom a projekcióval

Topdown pixel art játéknál a „mennyire látunk rá a világra” kérdést a kamera Projection skálájával felelhetjük meg: kisebb scale = nagyobb pixelek. Az alternatíva (minden sprite-ot Transformmal felskálázni) szétzilálná a koordinátarendszert. Ha később tökéletesen pixelhű megjelenítés kell (minden világ-pixel pontosan N képernyő-pixel), a bevett technika az alacsony felbontású render target felskálázása — a 12. fejezetben adok hozzá kiindulópontot. A skeletonhoz a nearest sampling + Msaa::Off + fix zoom bőven elég.

Ellenőrzőpont

cargo run: még mindig üres, sötét ablak — de a logban nincs hiba, és a kamera már létezik. git commit -m "feat: 2d kamera pixel art beallitasokkal"

06

A mező és a sprite-ok

Most kap látványt a játék. Két startup-systemet írunk: az egyik felépíti a zöld mezőt csempékből, a másik elszórja a sprite-okat (bokrok, kövek, virágok). Előtte azonban — és ez a workflow-lecke — rögzítjük a koordinátarendszert, mert erre minden későbbi rendszer (kamera-clamp, később ütközés, item-lerakás) építeni fog.

Tervezési döntések, kód előtt

  • Tile-méret: 32 világ-egység. Pixel artnál tipikus a 16 vagy 32 px-es tile; a 32 jó alap.
  • Térképméret: 60×40 tile = 1920×1280 világ-egység — nagyobb, mint az ablak, hogy legyen mit scrollozni.
  • Origó: a világ közepe a (0,0). A Bevy 2D-ben az Y felfelé nő. A kamera (0,0)-n indul, így rögtön a mező közepét látjuk.
  • Z-rétegek: fű z=0, propok z=1. 2D-ben a Z dönti el a kirajzolási sorrendet.
src/world.rs
use bevy::prelude::*;
use rand::prelude::*;

pub struct WorldPlugin;

impl Plugin for WorldPlugin {
    fn build(&self, app: &mut App) {
        app.add_systems(Startup, (spawn_field, scatter_props));
    }
}

// ---- A vilag "alkotmanya": ezekre az ertekekre epit minden mas modul ----

pub const TILE_SIZE: f32 = 32.0;
pub const MAP_WIDTH: u32 = 60; // csempeben
pub const MAP_HEIGHT: u32 = 40;

pub const WORLD_WIDTH: f32 = MAP_WIDTH as f32 * TILE_SIZE;
pub const WORLD_HEIGHT: f32 = MAP_HEIGHT as f32 * TILE_SIZE;

// Ket fuszin: a sakktablas valtakozas olcso trukk, amitol a szem
// erzekeli a kamera mozgasat egyszinu mezo helyett.
const GRASS_A: Color = Color::srgb(0.36, 0.55, 0.25);
const GRASS_B: Color = Color::srgb(0.33, 0.51, 0.23);

/// Marker component: ezzel jeloljuk a mezon elszort objektumokat.
/// Most ures; kesobb barmely system lekerdezheti vele a propokat.
#[derive(Component)]
pub struct Prop;

/// Tile racskoordinata -> vilagkoordinata.
/// A (0,0) tile a vilag bal also sarka; a vilag kozepe az origo.
fn tile_position(x: u32, y: u32) -> Vec2 {
    Vec2::new(
        x as f32 * TILE_SIZE - WORLD_WIDTH / 2.0 + TILE_SIZE / 2.0,
        y as f32 * TILE_SIZE - WORLD_HEIGHT / 2.0 + TILE_SIZE / 2.0,
    )
}

fn spawn_field(mut commands: Commands) {
    for y in 0..MAP_HEIGHT {
        for x in 0..MAP_WIDTH {
            let color = if (x + y) % 2 == 0 { GRASS_A } else { GRASS_B };
            commands.spawn((
                // PLACEHOLDER: egyszinu negyzet. Csere kesz assetre:
                // Sprite::from_image(asset_server.load("sprites/grass.png"))
                Sprite::from_color(color, Vec2::splat(TILE_SIZE)),
                Transform::from_translation(tile_position(x, y).extend(0.0)),
            ));
        }
    }
}

fn scatter_props(mut commands: Commands) {
    // Fix seed: minden inditasnal ugyanaz az elrendezes.
    // Fejlesztes kozben ez ANY, mert reprodukalhatova teszi, amit latsz.
    let mut rng = StdRng::seed_from_u64(42);

    for _ in 0..120 {
        let x = rng.gen_range((-WORLD_WIDTH / 2.0 + TILE_SIZE)..(WORLD_WIDTH / 2.0 - TILE_SIZE));
        let y = rng.gen_range((-WORLD_HEIGHT / 2.0 + TILE_SIZE)..(WORLD_HEIGHT / 2.0 - TILE_SIZE));

        // PLACEHOLDER propok: (szin, meret) parok.
        // Kesobb: Sprite::from_image(asset_server.load("sprites/bush.png")) stb.
        let (color, size) = match rng.gen_range(0..3) {
            0 => (Color::srgb(0.18, 0.38, 0.16), Vec2::new(20.0, 20.0)), // bokor
            1 => (Color::srgb(0.50, 0.50, 0.52), Vec2::new(14.0, 10.0)), // ko
            _ => (Color::srgb(0.91, 0.55, 0.69), Vec2::new(6.0, 6.0)),   // virag
        };

        commands.spawn((
            Prop,
            Sprite::from_color(color, size),
            Transform::from_translation(Vec3::new(x, y, 1.0)), // z=1: a fu FELETT
        ));
    }
}

Élesítés: a main.rs-ben kommentezd ki a world::WorldPlugin, sort.

Workflow-megjegyzések

  • A konstansok publikusak. A WORLD_WIDTH/WORLD_HEIGHT a modul „szerződése” a külvilág felé — a kamera-clamp (7. fejezet) ezekre épül. Ha a térkép nő, egyetlen helyen kell átírni.
  • Determinisztikus random. A fix seed fejlesztés közben hatalmas segítség: a „azon a bokron furcsa valami” típusú hibák reprodukálhatók. Release-ben cserélheted rand::thread_rng()-re.
  • 2400 tile-entity sok? Egy skeletonnak nem — a Bevy renderer az azonos kinézetű sprite-okat batch-eli. Ha majd 1000×1000-es térkép kell, a bevy_ecs_tilemap crate a megoldás (12. fejezet); a koordinátarendszerünk változatlan marad, pont ezért éri meg most rögzíteni.

Ellenőrzőpont

cargo run: zöld, finoman sakktáblás mező tölti ki a képernyőt, rajta elszórt sötétzöld/szürke/rózsaszín foltok. A mező szélét még nem látod — nagyobb, mint az ablak. git commit -m "feat: mezo tile-okbol + elszort prop sprite-ok"

07

Scrollozás a térképen

Itt jelenik meg először a Bevy futásidejű lelke: az Update systemek, amik minden frame-ben futnak. A scrollozást úgy valósítjuk meg, hogy a kamerát mozgatjuk a világ felett — a világ áll, a nézőpont megy. Visszatérünk a camera.rs-be, és bővítjük (a teljes, végleges file-t mutatom):

src/camera.rs (végleges)
use bevy::prelude::*;

use crate::world::{WORLD_HEIGHT, WORLD_WIDTH};

pub struct CameraPlugin;

impl Plugin for CameraPlugin {
    fn build(&self, app: &mut App) {
        app.add_systems(Startup, spawn_camera)
            .add_systems(Update, move_camera); // UJ: minden frame-ben fut
    }
}

/// Kamera sebessege, vilag-egyseg / masodperc.
const CAMERA_SPEED: f32 = 400.0;

fn spawn_camera(mut commands: Commands) {
    commands.spawn((
        Camera2d,
        Msaa::Off,
        Projection::from(OrthographicProjection {
            scale: 0.75,
            ..OrthographicProjection::default_2d()
        }),
    ));
}

fn move_camera(
    keys: Res<ButtonInput<KeyCode>>,
    time: Res<Time>,
    mut camera: Query<&mut Transform, With<Camera2d>>,
) {
    // Pontosan egy kamerat varunk; ha (meg) nincs, csendben kilepunk.
    let Ok(mut transform) = camera.single_mut() else {
        return;
    };

    let mut direction = Vec2::ZERO;
    if keys.pressed(KeyCode::KeyW) || keys.pressed(KeyCode::ArrowUp)    { direction.y += 1.0; }
    if keys.pressed(KeyCode::KeyS) || keys.pressed(KeyCode::ArrowDown)  { direction.y -= 1.0; }
    if keys.pressed(KeyCode::KeyA) || keys.pressed(KeyCode::ArrowLeft)  { direction.x -= 1.0; }
    if keys.pressed(KeyCode::KeyD) || keys.pressed(KeyCode::ArrowRight) { direction.x += 1.0; }

    // normalize_or_zero: atloban se legyen gyorsabb (es a nullvektort is turi)
    let delta = direction.normalize_or_zero() * CAMERA_SPEED * time.delta_secs();

    // Clamp: a kamera kozeppontja nem hagyhatja el a vilagot.
    let half_w = WORLD_WIDTH / 2.0;
    let half_h = WORLD_HEIGHT / 2.0;
    transform.translation.x = (transform.translation.x + delta.x).clamp(-half_w, half_w);
    transform.translation.y = (transform.translation.y + delta.y).clamp(-half_h, half_h);
}

A system-paraméterek nyelve

A move_camera szignatúrája maga a dokumentáció — a Bevy ebből tudja, mit adjon át, és mit futtathat párhuzamosan mivel:

  • Res<ButtonInput<KeyCode>> — csak olvasott resource: az aktuális billentyűállapot. A .pressed() a nyomva tartást, a .just_pressed() az adott frame-beli lenyomást figyeli.
  • Res<Time> — a frame-idő. A time.delta_secs()-szel szorzás teszi a mozgást framerate-függetlenné: 60 és 240 FPS-en is ugyanolyan gyors a scroll.
  • Query<&mut Transform, With<Camera2d>> — „add ide az összes entity Transformját, amelyiken van Camera2d”. A With<> csak szűr, az adatát nem kérjük. A single_mut() Result-ot ad — a let Ok(...) else { return } minta a Bevy-ben bevett védekező írásmód.

Miért a kamera mozog, és miért clampelünk?

Topdown játékban a „scroll” szinte mindig kameramozgás: a világ entityk Transformja stabil marad, így minden világbeli logika (ütközés, lerakás, mentés) egyszerű. A clamp pedig az első olyan pont, ahol kifizetődik, hogy a world.rs publikus konstansokban rögzítette a méreteket: a camera modul függ a worldtől, fordítva soha — pont, ahogy a 2. fejezet függőségi táblázata előírta.

Finomítási ötlet (most ne)

A clamp jelenleg a kamera középpontját tartja a világon belül, így a széleken túl is kilátunk kicsit. Ha „edge-perfect” határt akarsz, a fél képernyőméretet (ablakméret × projection scale / 2) is le kell vonni a határból. Skeleton fázisban ez tudatosan elhagyott finomítás — jegyezd fel a TODO-listára. A scope-fegyelem is workflow.

Játékidő: óra, napszak, évszak

A HUD-on négy adatot kell mutatnunk: óra, napszak, eltelt napok, évszak. A kezdő hiba az lenne, hogy ezt a négy dolgot négy külön változóként tároljuk és külön léptetjük. Ehelyett egyetlen forrásértéket tárolunk — az eltelt játékbeli perceket —, és minden mást abból számolunk. Így az idő nem tud inkonzisztens állapotba kerülni (pl. „4. nap, de még Tavasz 1. hete”).

Tervezés: az idő paraméterei

  • Időskála: 1 valós másodperc = 12 játékperc → egy teljes nap 2 valós perc. Skeletonnál szándékosan gyors, hogy tesztelés közben lásd váltani a napszakokat. Egyetlen konstans, később hangolod.
  • Évszak: 7 nap. Négy évszak: Tavasz → Nyár → Ősz → Tél, körbe.
  • Napszakok: Hajnal 5–8, Nappal 8–18, Este 18–22, Éjszaka 22–5.
src/game_time.rs
use bevy::prelude::*;

pub struct GameTimePlugin;

impl Plugin for GameTimePlugin {
    fn build(&self, app: &mut App) {
        app.init_resource::<GameTime>()
            .add_systems(Update, tick_game_time);
    }
}

// ---- Hangolhato parameterek ----

/// 1 valos masodperc = ennyi jatekbeli perc. (12 -> egy nap = 2 valos perc)
pub const MINUTES_PER_REAL_SECOND: f32 = 12.0;
pub const MINUTES_PER_DAY: f32 = 24.0 * 60.0;
pub const DAYS_PER_SEASON: u32 = 7;

// ---- Az allapot: EGYETLEN forrasertek ----

#[derive(Resource, Default)]
pub struct GameTime {
    /// Osszes eltelt jatekbeli perc a jatek kezdete ota.
    /// MINDEN mas (ora, nap, evszak) ebbol szarmaztatott ertek.
    pub total_minutes: f32,
}

#[derive(Debug, Clone, Copy, PartialEq, Eq)]
pub enum Season {
    Tavasz,
    Nyar,
    Osz,
    Tel,
}

#[derive(Debug, Clone, Copy, PartialEq, Eq)]
pub enum DayPart {
    Hajnal,
    Nappal,
    Este,
    Ejszaka,
}

impl GameTime {
    /// Hanyadik napon jarunk (1-tol indul).
    pub fn day(&self) -> u32 {
        (self.total_minutes / MINUTES_PER_DAY) as u32 + 1
    }

    /// A napon beluli ora (0..=23).
    pub fn hour(&self) -> u32 {
        ((self.total_minutes % MINUTES_PER_DAY) / 60.0) as u32
    }

    /// Az oran beluli perc (0..=59).
    pub fn minute(&self) -> u32 {
        (self.total_minutes % 60.0) as u32
    }

    pub fn season(&self) -> Season {
        match ((self.day() - 1) / DAYS_PER_SEASON) % 4 {
            0 => Season::Tavasz,
            1 => Season::Nyar,
            2 => Season::Osz,
            _ => Season::Tel,
        }
    }

    pub fn day_part(&self) -> DayPart {
        match self.hour() {
            5..=7 => DayPart::Hajnal,
            8..=17 => DayPart::Nappal,
            18..=21 => DayPart::Este,
            _ => DayPart::Ejszaka,
        }
    }
}

impl Season {
    pub fn label(self) -> &'static str {
        match self {
            Season::Tavasz => "Tavasz",
            Season::Nyar => "Nyár",
            Season::Osz => "Ősz",
            Season::Tel => "Tél",
        }
    }
}

impl DayPart {
    pub fn label(self) -> &'static str {
        match self {
            DayPart::Hajnal => "Hajnal",
            DayPart::Nappal => "Nappal",
            DayPart::Este => "Este",
            DayPart::Ejszaka => "Éjszaka",
        }
    }
}

/// Az egyetlen system, ami IRJA az idot.
fn tick_game_time(time: Res<Time>, mut game_time: ResMut<GameTime>) {
    game_time.total_minutes += time.delta_secs() * MINUTES_PER_REAL_SECOND;
}

    #[test]
    fn evszakok_korbe_fordulnak() {
        // 29. nap = 4 evszak (28 nap) utan ujra Tavasz
        let gt = GameTime { total_minutes: 28.0 * MINUTES_PER_DAY + 1.0 };
        assert_eq!(gt.day(), 29);
        assert_eq!(gt.season(), Season::Tavasz);

Workflow-megjegyzések

  • Derivált értékek metódusként. A day(), season() stb. nem tárolt mezők, hanem számítások. Az állapotgép így nem tud „szétcsúszni”, és a HUD-nak (következő fejezet) kész, formázható API-t adunk.
  • Egy író, sok olvasó. A GameTime-ot kizárólag a tick_game_time módosítja. Minden más rendszer Res<GameTime>-ként (csak olvasásra) kéri. Ez a fegyelem ECS-ben aranyat ér: a Bevy a csak-olvasó rendszereket párhuzamosan futtathatja.
  • Unit tesztek — mert lehet! Mivel az idő-logika tiszta számítás (nincs benne renderelés, entity, semmi Bevy-futtatókörnyezet), cargo test-tel azonnal tesztelhető. A napszak-határok (pont 8:00 = már Nappal?) tipikus off-by-one terep — a teszt megírása olcsóbb, mint kivárni az ingame 8 órát. Ez az igazi oka a logika/megjelenítés szétválasztásának.

HUD panel (jobb felső sarok)

Most lépünk be a Bevy UI világába. Kulcsfontosságú megérteni: a Bevy UI screen-space — a képernyőhöz van rögzítve, nem a világhoz. Ezért lesz a panel (és a 10. fejezet inventoryja) automatikusan „mindig látható”: hiába scrollozod a kamerát a mező felett, az UI nem mozdul. Semmi extra trükk nem kell hozzá.

A UI gondolkodásmódja

  • A UI is entitykből áll, de a Transform helyett Node component írja le őket — ez lényegében egy flexbox (ha írtál CSS-t, otthon leszel: flex_direction, padding, row_gap…).
  • PositionType::Absolute + top/right = a panel a képernyő jobb felső sarkához tűzve.
  • A szöveg a Text komponens; a TextFont és TextColor a formázás.

A frissítés megtervezése — egy tipikus Bevy-csapda kikerülése

Négy szövegsort kell frissíteni. A naiv megoldás: négy marker komponens (ClockText, DayText…) és négy Query<&mut Text, With<...>> paraméter egy systemben. Ez futásidőben pánikol (B0001 hiba): a Bevy nem tudja bizonyítani, hogy a négy mutable Text-query diszjunkt entityhalmazokat érint. Az elegáns megoldás: egyetlen enum-marker komponens, egyetlen query, és a mezőtípus szerint formázunk:

src/hud.rs
use bevy::prelude::*;

use crate::game_time::GameTime;

pub struct HudPlugin;

impl Plugin for HudPlugin {
    fn build(&self, app: &mut App) {
        app.add_systems(Startup, spawn_time_panel)
            .add_systems(Update, update_time_panel);
    }
}

/// EGY marker komponens, ami megmondja, melyik sor mit mutat.
/// Igy egyetlen query-vel, konfliktus nelkul frissitheto mind a negy sor.
#[derive(Component)]
enum HudField {
    Clock,
    DayPart,
    Day,
    Season,
}

const PANEL_BG: Color = Color::srgba(0.04, 0.07, 0.04, 0.78);

fn spawn_time_panel(mut commands: Commands) {
    commands.spawn((
        Name::new("TimePanel"),
        Node {
            // Screen-space rogzites a jobb felso sarokba:
            position_type: PositionType::Absolute,
            top: Val::Px(12.0),
            right: Val::Px(12.0),
            // A sorok fuggolegesen, kis kozzel:
            flex_direction: FlexDirection::Column,
            row_gap: Val::Px(2.0),
            padding: UiRect::axes(Val::Px(14.0), Val::Px(10.0)),
            ..default()
        },
        BackgroundColor(PANEL_BG),
        // children!: a panel negy gyerek-entityt (szovegsort) kap.
        children![
            time_row("--:--", 22.0, HudField::Clock),
            time_row("Nappal", 14.0, HudField::DayPart),
            time_row("1. nap", 14.0, HudField::Day),
            time_row("Tavasz", 14.0, HudField::Season),
        ],
    ));
}

/// Segedfuggveny: egy szovegsor bundle-je.
/// Pixel fonthoz kesobb: TextFont { font: asset_server.load("fonts/PressStart2P.ttf"), .. }
fn time_row(initial: &str, size: f32, field: HudField) -> impl Bundle {
    (
        Text::new(initial),
        TextFont {
            font_size: size,
            ..default()
        },
        TextColor(Color::WHITE),
        field,
    )
}

fn update_time_panel(
    game_time: Res<GameTime>,
    mut rows: Query<(&HudField, &mut Text)>,
) {
    for (field, mut text) in &mut rows {
        // A Text egy String korul newtype: text.0 maga a String.
        text.0 = match field {
            HudField::Clock => {
                format!("{:02}:{:02}", game_time.hour(), game_time.minute())
            }
            HudField::DayPart => game_time.day_part().label().to_string(),
            HudField::Day => format!("{}. nap", game_time.day()),
            HudField::Season => game_time.season().label().to_string(),
        };
    }
}

Workflow-megjegyzések

  • Name komponens. Ingyen van, és minden debug-eszközben (log, inspector — 11. fejezet) olvasható nevet ad az entitynek. Szokj rá minden „gyökér” entitynél.
  • children![] makró. Deklaratívan írja le a UI-fát: a panel és a sorai egy helyen, egy spawn-hívásban. Az alternatíva a .with_children(|parent| ...) closure — azt a 10. fejezetben látod majd, mert ott ciklusból spawnolunk.
  • Frissítés minden frame-ben? Itt igen, és ez rendben van — 4 string formázása semmi. Ha optimalizálni akarsz: a system elejére if !game_time.is_changed() { return; }. Az inventorynál (ahol az állapot ritkán változik) ezt meg is lépjük.

Inventory sáv (alul, 4 slot)

Az utolsó funkcionális plugin, és egyben a skeleton legteljesebb mintája: állapot (resource) + UI (spawn) + input (kiválasztás) + szinkronizáció (állapot → UI). Ugyanez a négyes minden későbbi játékrendszerben (crafting, párbeszéd, bolt…) visszaköszön majd.

Tervezés

  • Az inventory igazsága egy resource-ban él: 4 slot, mindegyik Option<ItemKind>, plusz a kiválasztott index. Az UI ennek csak a vetülete.
  • A sáv alul, vízszintesen középre igazítva, screen-space — a HUD-hoz hasonlóan magától „mindig látható”.
  • Kiválasztás az 1–4 billentyűkkel; a kiválasztott slot kerete arany.
  • Az itemek placeholderként színes négyzetek a slotban — a kommentek jelzik, hol cserélheted képre.
src/inventory.rs
use bevy::prelude::*;

pub struct InventoryPlugin;

impl Plugin for InventoryPlugin {
    fn build(&self, app: &mut App) {
        app.init_resource::<Inventory>()
            .add_systems(Startup, spawn_inventory_bar)
            .add_systems(Update, (select_slot_by_key, refresh_slots).chain());
    }
}

pub const SLOT_COUNT: usize = 4;

// ---- Allapot ----

#[derive(Debug, Clone, Copy, PartialEq, Eq)]
pub enum ItemKind {
    Kapa,
    Mag,
    Fejsze,
}

impl ItemKind {
    /// PLACEHOLDER megjelenes: szin. Kesobb itt adhatsz vissza
    /// asset utvonalat: "sprites/items/kapa.png", es a slot-ikon
    /// BackgroundColor helyett ImageNode-ot kap.
    fn color(self) -> Color {
        match self {
            ItemKind::Kapa => Color::srgb(0.66, 0.45, 0.25),
            ItemKind::Mag => Color::srgb(0.95, 0.83, 0.36),
            ItemKind::Fejsze => Color::srgb(0.60, 0.63, 0.67),
        }
    }
}

#[derive(Resource)]
pub struct Inventory {
    pub slots: [Option<ItemKind>; SLOT_COUNT],
    pub selected: usize,
}

impl Default for Inventory {
    fn default() -> Self {
        Self {
            // Indulo demo-tartalom, hogy legyen mit nezni:
            slots: [
                Some(ItemKind::Kapa),
                Some(ItemKind::Mag),
                None,
                Some(ItemKind::Fejsze),
            ],
            selected: 0,
        }
    }
}

// ---- UI markerek ----

/// A slot kerete (ezen valt a kijeloles szine).
#[derive(Component)]
struct SlotIndex(usize);

/// A slot belso ikonja (ezen valt az item szine/kepe).
#[derive(Component)]
struct SlotItemIcon(usize);

const SLOT_BG: Color = Color::srgba(0.04, 0.07, 0.04, 0.78);
const BORDER_NORMAL: Color = Color::srgb(0.25, 0.32, 0.24);
const BORDER_SELECTED: Color = Color::srgb(0.95, 0.76, 0.31);

// ---- UI felepites ----

fn spawn_inventory_bar(mut commands: Commands) {
    commands
        .spawn((
            Name::new("InventoryBar"),
            Node {
                // Teljes szelessegu, lathatatlan sav a kepernyo aljan,
                // ami kozepre rendezi a slotokat:
                position_type: PositionType::Absolute,
                bottom: Val::Px(12.0),
                left: Val::Px(0.0),
                right: Val::Px(0.0),
                justify_content: JustifyContent::Center,
                column_gap: Val::Px(8.0),
                ..default()
            },
        ))
        .with_children(|bar| {
            for i in 0..SLOT_COUNT {
                bar.spawn((
                    SlotIndex(i),
                    Node {
                        width: Val::Px(56.0),
                        height: Val::Px(56.0),
                        border: UiRect::all(Val::Px(3.0)),
                        justify_content: JustifyContent::Center,
                        align_items: AlignItems::Center,
                        ..default()
                    },
                    BackgroundColor(SLOT_BG),
                    BorderColor(BORDER_NORMAL),
                ))
                .with_children(|slot| {
                    // A belso ikon: ures slotnal atlatszo.
                    slot.spawn((
                        SlotItemIcon(i),
                        Node {
                            width: Val::Px(32.0),
                            height: Val::Px(32.0),
                            ..default()
                        },
                        BackgroundColor(Color::NONE),
                        // Kep-assethez csere:
                        // ImageNode::new(asset_server.load("sprites/items/kapa.png"))
                    ));
                });
            }
        });
}

// ---- Input ----

fn select_slot_by_key(keys: Res<ButtonInput<KeyCode>>, mut inventory: ResMut<Inventory>) {
    const SLOT_KEYS: [KeyCode; SLOT_COUNT] = [
        KeyCode::Digit1,
        KeyCode::Digit2,
        KeyCode::Digit3,
        KeyCode::Digit4,
    ];
    for (i, key) in SLOT_KEYS.iter().enumerate() {
        if keys.just_pressed(*key) {
            inventory.selected = i;
        }
    }
}

// ---- Allapot -> UI szinkron ----

fn refresh_slots(
    inventory: Res<Inventory>,
    mut borders: Query<(&SlotIndex, &mut BorderColor)>,
    mut icons: Query<(&SlotItemIcon, &mut BackgroundColor)>,
) {
    // Change detection: csak akkor dolgozunk, ha az inventory valtozott.
    // (Az elso frame-ben az init_resource miatt "valtozott" — pont jo:
    // akkor rajzoljuk ki az indulo allapotot.)
    if !inventory.is_changed() {
        return;
    }

    for (slot, mut border) in &mut borders {
        border.0 = if slot.0 == inventory.selected {
            BORDER_SELECTED
        } else {
            BORDER_NORMAL
        };
    }

    for (icon, mut bg) in &mut icons {
        bg.0 = match inventory.slots[icon.0] {
            Some(item) => item.color(),
            None => Color::NONE,
        };
    }
}

Workflow-megjegyzések

  • .chain() — a két Update-system (input, majd szinkron) garantáltan ebben a sorrendben fut. Enélkül a Bevy szabadon ütemezné őket, és a kijelölés egy frame késéssel jelenhetne meg. Apróság, de a determinisztikus sorrend olcsó biztosíték.

Inspector: belelátni a futó játékba

A bevy-inspector-egui crate egy fejlesztői panelt ad, amiben futás közben látod az összes entityt (a Name komponensek itt fizetnek!), és élőben átírhatod a komponensértékeket — kamerapozíciót, színeket, akár a GameTime-ot is. Hangolásnál (mennyi legyen a CAMERA_SPEED?) ez a leggyorsabb visszacsatolás:

terminál
cargo add bevy-inspector-egui
# A Bevy 0.16-hoz illo verziot a crate doksija jelzi —
# a fo verzioszamuk a Bevy verziojat koveti.
src/main.rs (csak fejlesztéshez)
use bevy_inspector_egui::quick::WorldInspectorPlugin;

// az add_plugins lancba, idiglenesen:
.add_plugins(WorldInspectorPlugin::new())

Workflow-szokás: a hangolható értékek útja

1) Az érték konstansként születik a modul tetején (CAMERA_SPEED, MINUTES_PER_REAL_SECOND — mind ott van). 2) Ha sokat hangolod, resource-ba költözik, és az inspectorból élőben állítod. 3) Ha a játékosnak is állíthatónak kell lennie, config file-ba kerül. A skeletonunk az 1. lépcsőn áll — szándékosan.

Verziókezelési ritmus

Ahogy a fejezetek végig csinálták: egy működő funkció = egy commit. A commit-üzenet a funkcióról szól (feat:/fix:/chore:), nem a file-okról. Mielőtt nagyobb refaktorba kezdesz (pl. a tile-rendszer cseréje bevy_ecs_tilemap-re), branchelj — a működő skeleton a legértékesebb vagyonod.

Merre tovább?

A skeleton mostantól hordozó rakéta. Néhány irány, mindegyikhez azzal, hogy hova illeszkedik a meglévő struktúrában:

Következő lépésHova épül?
Igazi pixel art assetek (Aseprite → PNG az assets/sprites-be)A kommentekkel jelölt Sprite::from_color → Sprite::from_image cserék a world.rs-ben és inventory.rs-ben. Pixel font a HUD-ra: TextFont.font.
Játékos-karakter, kamera követésselÚj player.rs plugin; a move_camera input-logikája átköltözik a playerre, a kamera pedig a player Transformját követi (a clamp marad).
Nagy térképekA spawn_field cseréje bevy_ecs_tilemap-re — a TILE_SIZE/WORLD_* konstansok és minden más modul változatlan.
Napszak szerinti fényÚj system a game_time.rs-ben (vagy külön lighting.rs): a day_part() alapján egy képernyőt fedő, félig átlátszó sötétkék sprite alfáját animálja. A GameTime API-ja már most kész erre.
Item felvétel/lerakásAz Inventory resource-on műveletek (add_item, remove_selected) + egy system, ami a Prop entityk és a kamera/player pozícióját veti össze. A refresh_slots a change detection miatt magától követi.
Game state-ek (menü / játék / pause)Bevy States: a systemek run_if(in_state(...)) feltételt kapnak — pluginonként egy sor.
Pixelhű renderelés (integer scaling)Alacsony felbontású render target + felskálázó kamera; a hivatalos Bevy-példák közt a pixel grid snapping a kiindulópont.

Bevy-verziók kezelése — a workflow utolsó pillére

A Bevy ~3–4 havonta ad ki minor verziót, törő változásokkal (pl. a 0.17 átalakította a BorderColor-t oldalankéntire, a 0.18 tovább formálja az API-t). A fenntartható hozzáállás:

  • Pinneld a verziót (mi a 0.16-on vagyunk), és ne frissíts reflexből.
  • Frissítéskor a hivatalos migration guide-ot kövesd (minden release-hez készül a bevy.org-on), egy dedikált branchen, fejezetnyi lépésekben — pont úgy, ahogy építkeztünk.
  • A plugin-per-file struktúra itt is fizet: a törő változások modulonként, izoláltan javíthatók, és a cargo check modulonként mutatja a tennivalót.
