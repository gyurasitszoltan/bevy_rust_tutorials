Írd le, ne építsd — BSN

A Bevy Scene Notation (BSN) a 0.19 zászlóshajó-újdonsága: egy deklaratív jelölés, amivel komponens-fákat írsz le ahelyett, hogy kézzel spawnolnád őket. Kompozálható, patchelhető és függőség-tudatos — csak azt adod meg, ami érdekel, a többit a motor kitölti és betölti. Nézzük meg, miért ez a Bevy UI/scene jövője.

↑ Élő minta: egy bsn! jelenet balról — a motor jobbra felépíti az entitás-fát. Egy deklaratív leírás → kész hierarchia, függőségekkel együtt.
█ I. rész — Miért BSN?
1. fejezet

A régi fájdalom & az ígéret

A scene-ek (jelenetek) Bevyben hagyományosan fájdalmasak voltak. Egy összetett entitás spawnolása azt jelentette, hogy kézzel felsoroltad minden komponensét és minden függőségét, a helyes sorrendben. Egy egyszerű gomb is így nézett ki:

a „régi" út — kézi spawn, sok boilerplate
commands.spawn((
    Button,
    Node {
        width: Val::Px(150.0),
        height: Val::Px(65.0),
        justify_content: JustifyContent::Center,
        align_items: AlignItems::Center,
        ..default()
    },
    BackgroundColor(Color::srgb(0.15, 0.15, 0.15)),
    children![(
        Text::new("Ok"),
        TextColor(Color::srgb(0.9, 0.9, 0.9)),
    )],
));

Ugyanez BSN-ben — ugyanaz a fa, kevesebb zaj, és ami a lényeg: kompozálható darabokból:

a „BSN" út — deklaratív, kompozálható
bsn! {
    Button
    Node {
        width: px(150),
        height: px(65),
        justify_content: JustifyContent::Center,
        align_items: AlignItems::Center,
    }
    BackgroundColor(Color::srgb(0.15, 0.15, 0.15))
    Children [(
        Text("Ok")
        TextColor(Color::srgb(0.9, 0.9, 0.9))
    )]
}
⚙ Mit old meg a BSN?
  • Defaultok: csak a számodra fontos mezőket írod ki; a többi a Default-ot kapja.
  • Kompozíció: kis jeleneteket nagyobbakká fűzöl össze, duplikáció nélkül.
  • Patch / override: egyetlen mezőt felülírsz anélkül, hogy az egészet újraírnád.
  • Függőség-tudatosság: nem kell kézzel behúzni a komponens- és asset-függőségeket — a motor kitalálja, mit kell betölteni és milyen sorrendben.

A négy szuperképességet a II. rész veszi sorra. De előbb a makró maga.

2. fejezet

A bsn! makró alapjai

A bsn! { ... } egy egyetlen entitást ír le. A komponenseket szóközzel (új sorral) választod el. A mezőket struct-szintaxissal adod meg — és csak azt írod ki, amit állítasz; a többi mező a Default értékét kapja.

komponensek & mezők
// csak a típusnév -> minden mező default:
bsn! { Player }

// néhány mező megadva, a többi default:
bsn! {
    Player { score: 100 }
    Node { width: px(100) }
}

// tetszőleges Rust-kifejezés is mehet { } között:
bsn! { Player { score: { current_points + 10 } } }

A px(100) és percent(100) kényelmi segéd-függvények (a Val::Px / Val::Percent helyett). A tuple-struct komponensek a természetes zárójeles formában mennek: BackgroundColor(Color::WHITE), Text("Helló").

▶ JÁTSZÓTÉR — „csak amit megadsz"kapcsold a mezőket
Mely mezőket adod meg? (a többi default)
a generált bsn!
az eredmény (Node)
Node

3 / 8 mező megadva

A többi mező nem jelenik meg a kódban — a motor a Default-ot teszi be. Minél kevesebbet írsz, annál olvashatóbb a jelenet.

█ II. rész — A négy szuperképesség
3. fejezet

Hierarchia: Children

A gyerek-entitásokat a Children [ ... ] reláció írja le. A vessző külön gyerekeket választ el; egy gyereken belül a komponenseket szóköz választja el. Az átláthatóságért egy gyerek komponenseit ( ... ) zárójelbe tesszük:

Children — vessző = új gyerek, szóköz = ugyanazon
bsn! {
    Node { /* a konténer */ }
    Children [
        ( Button  Text("Ok") ),       // 1. gyerek: Button + Text egy entitáson
        ( Button  Text("Mégse") ),    // 2. gyerek
    ]
}
▲ Vessző vs. szóköz
Children [ A B ] = egy gyerek, rajta A és B komponens. Children [ A, B ] = két gyerek. A ( ... ) csak egyértelműsít — a vessző a döntő.

Kattints lent a kód egy sorára — kiemelődik a hozzá tartozó csomópont az entitás-fában:

▶ JÁTSZÓTÉR — kód ↔ entitás-fakattints egy sorra
bsn! forrás
a felépülő entitás-fa

Minden gyökér-sor egy entitás; a Children [ ] beágyazott entitásokat hoz létre. A relációkat (Children) a Bevy automatikusan beköti.

4. fejezet

Kompozíció: scene-függvények

Itt kezd igazán erős lenni a BSN. Egy bsn!-t visszaadó függvény típusa impl Scene — és ezt a függvényt beágyazhatod egy másik bsn!-be, mintha komponens volna. Így kis, újrahasználható darabokból építed a nagyot:

scene-függvények egymásba ágyazva
// egy újrahasználható gomb-jelenet:
fn button(label: &str) -> impl Scene {
    bsn! {
        Button
        Node { width: px(150), height: px(65) }
        BackgroundColor(Color::srgb(0.15, 0.15, 0.15))
        Children [( Text(label) )]
    }
}

// egy dialógus, ami KÉT button()-t komponál:
fn dialog() -> impl Scene {
    bsn! {
        Node { column_gap: px(5) }
        Children [
            ( button("Ok") ),
            ( button("Mégse") ),
        ]
    }
}
⚙ Bevy fogalom — Scene & SceneList

A bsn! { ... } egy impl Scene (egyetlen gyökér-entitás sablonja). Ha több gyökér-entitást akarsz, a bsn_list![ ... ] ad egy impl SceneList-et. A scene-ek tehát értékek, amiket függvények adnak vissza, változókba tehetők, és — ahogy itt — egymásba fűzhetők.

5. fejezet

Patch & override — „last write wins"

A jelenetek patchek: ha egy beágyazott jelenet után további komponenseket vagy mezőket írsz, azok rárétegeződnek. Az átfedő mezőknél az utolsó írás nyer — a többi marad. Így egy kész darabot „testre szabhatsz" anélkül, hogy újraírnád:

a button() testreszabása patcheléssel
fn danger_button() -> impl Scene {
    bsn! {
        button("Törlés")                              // alap: szürke gomb, 150x65
        BackgroundColor(Color::srgb(0.4, 0.15, 0.15)) // PATCH: piros háttér felülír
        Node { height: px(80) }                       // PATCH: csak a height változik,
                                                       //        a width (150) marad!
    }
}

Told a kapcsolókat — a base réteg (a button()) változatlan, a patch-réteg felülír, és látod az eredő gombot:

▶ JÁTSZÓTÉR — patch-rétegezéslast write wins
① base: button()
width: px(150) height: px(65) background: szürke
② patch — mit írsz felül?
= eredmény (last write wins)
Törlés

A width a base-ből jön; a patch csak a megadott mezőket írja felül.

6. fejezet

Függőségek & assetek

A BSN függőség-tudatos: ha egy komponensnek Handle<T> mezője van (pl. egy Sprite képe), egyszerűen string elérési utat adsz meg, és a motor betölti az assetet, majd behelyettesíti a handle-t:

asset-handle string elérési útból
bsn! {
    Sprite { image: "player.png" }   // "player.png" -> Handle<Image> automatikusan
}

Ezért van kétféle spawnolás: az azonnali, és az asset-függőségekre váró:

HívásViselkedés
commands.spawn_scene(..)Azonnal spawnol.
commands.queue_spawn_scene(..)Megvárja a függő assetek betöltését, csak utána spawnol — így nincs „egy frame-ig nincs textúra" villanás.
▶ JÁTSZÓTÉR — függőség-betöltésspawn → várakozás → kész

Egy Sprite { image: "player.png" } spawnolása. Figyeld a fázisokat: a motor felismeri az asset-függőséget, betölti, majd behelyettesíti a handle-t.

█ III. rész — A gyakorlatban
7. fejezet

Spawnolás

Háromféleképp keltheted életre a jelenetet. A leggyakoribb: egy impl Scene-t visszaadó függvényt közvetlenül rendszerré alakítasz a .spawn()-nal:

main.rs — a scene-függvény mint Startup-rendszer
fn main() {
    App::new()
        .add_plugins(DefaultPlugins)
        .add_systems(Startup, scene.spawn())   // a `scene` fn -> rendszer
        .run();
}

fn scene() -> impl SceneList {
    bsn_list![ Camera2d, ui() ]                // több gyökér-entitás
}

Vagy egy meglévő rendszeren belül, a Commands-on keresztül:

spawnolás rendszeren belül
fn setup(mut commands: Commands) {
    commands.spawn_scene(bsn! { Player Ship });
}
8. fejezet

Observerek: on()

A jelenetbe közvetlenül esemény-figyelőket (observer) is írhatsz az on(...) segítségével. Egy gomb így rögtön „magával hozza" a kattintás-kezelőjét — nincs külön bekötés:

observer a jelenetben
fn button(label: &str) -> impl Scene {
    bsn! {
        Button
        Node { width: px(150), height: px(65) }
        Children [( Text(label) )]
        on(|_press: On<Pointer<Press>>| {
            info!("megnyomták: {}", "gomb");
        })
    }
}
⚙ Bevy fogalom — On<E> observer

Az on(closure) egy observert csatol az entitáshoz. A closure első paramétere a kiváltó esemény, On<Pointer<Press>> formában (a 0.15+ observer-API). A jelenet így önállóan viselkedik: a vizuál és a reakciója egy helyen, egy darabban utazik.

9. fejezet

Teljes példa: egy UI dialógus

Tegyük össze: a hivatalos bsn példa egy két-gombos UI-t épít — kompozícióval, patcheléssel és observerekkel együtt. Ez a teljes kép:

examples/scene/bsn.rs (Bevy 0.19) — kivonat
fn scene() -> impl SceneList {
    bsn_list![ Camera2d, ui() ]
}

fn ui() -> impl Scene {
    bsn! {
        Node {
            width: percent(100),
            height: percent(100),
            align_items: AlignItems::Center,
            justify_content: JustifyContent::Center,
            column_gap: px(5),
        }
        Children [
            (
                button("Ok")
                on(|_event: On<Pointer<Press>>| println!("Ok pressed!"))
            ),
            (
                button("Cancel")
                on(|_event: On<Pointer<Press>>| println!("Cancel pressed!"))
                BackgroundColor(Color::srgb(0.4, 0.15, 0.15))   // patch: piros
            ),
        ]
    }
}

Figyeld meg: a Cancel gomb ugyanaz a button(), mint az Ok — csak rárétegez egy observert és egy piros BackgroundColor patchet. Egy darab, kétféleképp testre szabva. Ez a BSN lényege egyetlen képben.

⚠ Státusz a 0.19-ben — friss, és van pár éle

A 0.19 a kódból írt (bsn!) munkafolyamatra fókuszál. A .bsn asset-betöltő (jelenetek fájlból, hot-reloaddal) még nincs kész — egy későbbi kiadásra tervezik, ahogy a glTF BSN-re portolását is. A BSN „frissen jött ki", így a Bevy csapata szerint is várhatók durva élek és hiányzó funkciók — de már most bőven használható.

⚙ Megjegyzés — haladó BSN

Van még mit felfedezni: a #Név előtaggal nevet adsz egy entitásnak és hivatkozol rá (relációkhoz); a @ előtag SceneComponent-et spawnol a saját beágyazott jelenetével; a Template / FromTemplate trait pedig a „okos konstruktor" a háttérben (ez alakítja a "player.png" stringet Handle-lé). Ezek a hivatalos dokumentációban mélyülnek el.

▸ Merre tovább?
  • Scene-architektúra: a scene-váltás (States, fade, plugin-elrendezés) lecke — a BSN ezt kiegészíti (mit spawnolj), nem váltja le (mikor/melyik scene aktív).
  • UI: a Bevy UI alapok — a BSN a UI-fák leírásában a legerősebb.
  • Hivatalos: a Bevy 0.19 bejelentés és a examples/scene/bsn.rs példa.
