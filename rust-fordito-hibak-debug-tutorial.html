A fordító a barátod

A Rust-magabiztosság igazi próbája nem a szintaxis, hanem hogy elolvasol egy fordítóhibát — és tudod a választ. Plusz: a hibakeresés teljes eszköztára a dbg!-tól a debuggerig.

10 fejezet ~30 kód- és hibapélda Szint: középhaladó Épít rá: tulajdonjog, referenciák, élettartamok
0x00

Hogyan olvass egy hibát

A Rust hírhedt arról, hogy „veszekszik veled". Pedig a fordító a legszigorúbb, de legkorrektebb code reviewer, akivel valaha dolgozni fogsz: nem fordít le hibás kódot, de cserébe pontosan megmondja, mi a baj, és gyakran azt is, hogyan javítsd. A magabiztosság kulcsa nem a hibák elkerülése — hanem hogy gyorsan elolvasod és megérted őket.

Nézzünk egy egyszerű hibát, és boncoljuk fel:

src/main.rs
fn main() {
    let x = 5;
    x = 6; // ❌
}
error[E0384]: cannot assign twice to immutable variable `x` --> src/main.rs:3:5 | 2 | let x = 5; | - first assignment to `x` 3 | x = 6; | ^^^^^ cannot assign twice to immutable variable | help: consider making this binding mutable | 2 | let mut x = 5; | +++

Egy Rust-hiba mindig ugyanazt a felépítést követi. Olvasd fentről lefelé:

  • error[E0384] — a hiba kódja. Ezzel kereshetsz: rustc --explain E0384 egész oldalas magyarázatot ad példákkal.
  • A kód utáni mondat a tömör üzenet: „nem adhatsz kétszer értéket egy változatlan változónak".
  • --> a helyszín: fájl, sor, oszlop.
  • A | oszlop a forráskódot mutatja, a ^^^ és - jelek pontosan rámutatnak a hibás és a kapcsolódó részekre.
  • A help: és note: sorok a megoldást kínálják — itt épp a +++ azt jelzi, hová írj mut-ot.
Három szokás, ami sokat segít

1. Mindig az első hibát javítsd, majd fordíts újra — a többi gyakran csak a láncreakció. 2. Használd a cargo check-et a leggyorsabb visszajelzésért (nem generál gépi kódot). 3. A cargo clippy a hibákon túl stílus- és logikai figyelmeztetéseket is ad. A rust-analyzer pedig mindezt gépelés közben, az editorban mutatja.

0x01

E0382: az elmozgatott érték

Ez a kezdők első számú hibája, és egyben a tulajdonjogi modell lényege. Nézd:

src/main.rs
fn main() {
    let s = String::from("szia");
    let s2 = s;       // s MOVE-olt s2-be
    println!("{s}");  // ❌ s már nem érvényes
}
error[E0382]: borrow of moved value: `s` --> src/main.rs:4:15 | 2 | let s = String::from("szia"); | - move occurs because `s` has type `String`, which does not implement the `Copy` trait 3 | let s2 = s; | - value moved here 4 | println!("{s}"); | ^^^ value borrowed here after move

A kulcsmondat: „does not implement the Copy trait". A String a heap-en tárol adatot, ezért az értékadás elköltözteti (move) a tulajdont — nem másol. A régi név (s) ezután érvénytelen. (Az olcsó típusok — egészek, bool, char, &T — Copy-k, azoknál ez nem történik meg.)

Három javítás — a szándékod szerint

1. Ha tényleg két független adat kell — klónozz:

src/main.rs
let s = String::from("szia");
let s2 = s.clone();    // ÚJ, független másolat — két tulajdonos
println!("{s} / {s2}"); // mindkettő él

2. Ha csak olvasni akarod — kölcsönözz:

src/main.rs
let s = String::from("szia");
let s2 = &s;          // csak KÖLCSÖN — nincs mozgatás
println!("{s} / {s2}"); // mindkettő olvasható
Ne klónozz reflexből!

A .clone() a leggyorsabb mód, hogy egy E0382 eltűnjön — de gyakran felesleges másolás. Először kérdezd meg: tényleg kell két példány, vagy csak olvasni akarom? Ha olvasni — &. A reflexszerű klónozás működő, de pazarló kódot szül.

⚙ Gyakorlat 0x01

Az alábbi kód E0382-vel elszáll. Írd át úgy, hogy lefusson, és mindkét println! működjön (mindkét névnek a saját adata kell).

hibás kód
fn main() {
    let nev = String::from("Anna");
    let masolat = nev;
    println!("{nev}");
    println!("{masolat}");
}
Megoldás
megoldás
fn main() {
    let nev = String::from("Anna");
    let masolat = nev.clone(); // klón: két független String
    println!("{nev}");
    println!("{masolat}");
}
0x02

Kölcsönzési ütközések

A borrow checker aranyszabálya: egy adatra egyszerre vagy egyetlen módosító (&mut) kölcsön lehet, vagy tetszőleges sok olvasó (&) kölcsön — de a kettő együtt soha. Ez a két leggyakoribb ütközés.

E0502 — olvasó és módosító egyszerre

src/main.rs
fn main() {
    let mut v = vec![1, 2, 3];
    let elso = &v[0];   // olvasó (immutable) kölcsön
    v.push(4);          // ❌ módosító kölcsön, amíg `elso` él
    println!("{elso}");
}
error[E0502]: cannot borrow `v` as mutable because it is also borrowed as immutable --> src/main.rs:4:5 | 3 | let elso = &v[0]; | -- immutable borrow occurs here 4 | v.push(4); | ^^^^^^^^^ mutable borrow occurs here 5 | println!("{elso}"); | ------ immutable borrow later used here

Ez nem szőrszálhasogatás: a push újraallokálhatja a vektort, így elso egy felszabadított memóriára mutatna — pontosan ezt a hibát (dangling pointer) előzi meg a fordító.

A megoldás kulcsa: a kölcsön az UTOLSÓ használatáig él (NLL)

A „non-lexical lifetimes" óta a kölcsön nem a blokk végéig, hanem az utolsó felhasználásáig tart. Ha az elso-t a push előtt használod fel, az ütközés eltűnik:

src/main.rs
fn main() {
    let mut v = vec![1, 2, 3];
    let elso = &v[0];
    println!("{elso}"); // `elso` utolsó használata — a kölcsön ITT véget ér
    v.push(4);          // ✅ most már szabad módosítani
}

E0499 — két módosító kölcsön

src/main.rs
fn main() {
    let mut s = String::from("hello");
    let r1 = &mut s;
    let r2 = &mut s; // ❌ második módosító kölcsön egyszerre
    println!("{r1} {r2}");
}
error[E0499]: cannot borrow `s` as mutable more than once at a time --> src/main.rs:4:14 | 3 | let r1 = &mut s; | ------ first mutable borrow occurs here 4 | let r2 = &mut s; | ^^^^^^ second mutable borrow occurs here 5 | println!("{r1} {r2}"); | ---- first borrow later used here

A javítás ugyanaz a gondolat: ne éljen egyszerre két módosító kölcsön. Használd az egyiket, fejezd be, aztán jöhet a másik — vagy gondold át, tényleg kell-e két párhuzamos hozzáférés.

0x03

Lógó referenciák, élettartamok

A Rustban lehetetlen lógó (dangling) referenciát létrehozni — a fordító nem engedi. Két tipikus hibában találkozol ezzel.

E0515 — referencia egy lokális változóra

src/main.rs
fn koszonto(nev: &str) -> &str {
    let s = format!("Szia, {nev}!");
    &s // ❌ s a függvény végén megsemmisül — a referencia lógna
}
error[E0515]: cannot return reference to local variable `s` --> src/main.rs:3:5 | 3 | &s | ^^ returns a reference to data owned by the current function

A megoldás: ne referenciát adj vissza egy helyi adatra, hanem a tulajdont magát:

src/main.rs
fn koszonto(nev: &str) -> String {
    format!("Szia, {nev}!") // ✅ tulajdont adunk vissza, nem referenciát
}

E0106 — hiányzó élettartam

Ha egy függvény referenciát ad vissza, de több bemenete is van, a fordító nem tudja, melyikből származik a kimenet élettartama:

src/main.rs
fn leghosszabb(a: &str, b: &str) -> &str {
    if a.len() > b.len() { a } else { b } // ❌
}
error[E0106]: missing lifetime specifier --> src/main.rs:1:37 | 1 | fn leghosszabb(a: &str, b: &str) -> &str { | ---- ---- ^ expected named lifetime parameter | = help: this function's return type contains a borrowed value, but the signature does not say whether it is borrowed from `a` or `b` help: consider introducing a named lifetime parameter | 1 | fn leghosszabb<'a>(a: &'a str, b: &'a str) -> &'a str { | ++++ ++ ++ ++

A help: szó szerint megírja a javítást — vezess be egy 'a élettartamot, és kösd hozzá a bemeneteket és a kimenetet:

src/main.rs
fn leghosszabb<'a>(a: &'a str, b: &'a str) -> &'a str {
    if a.len() > b.len() { a } else { b }
}
Miért nem kell mindig kiírni?

Egyetlen bemeneti referenciánál a fordító kitalálja (élettartam-elízió), ezért az fn elso_szo(s: &str) -> &str kiírás nélkül is fordul. Csak akkor kér tőled 'a-t, ha tényleg többértelmű — pont mint fent.

⚙ Gyakorlat 0x03

Ez a függvény E0515-tel nem fordul. Javítsd ki a legkisebb változtatással, hogy működjön.

hibás kód
fn cimke(szam: u32) -> &str {
    let s = format!("#{szam}");
    &s
}
Megoldás
megoldás
fn cimke(szam: u32) -> String {
    format!("#{szam}") // tulajdont adunk vissza
}
0x04

Típushibák: E0308, E0277

E0308 — nem egyező típusok

A legtöbbet látott hiba. A fordító pontosan megmondja, mit várt és mit kapott.

src/main.rs
fn main() {
    let x: i32 = "42"; // ❌ ez &str, nem i32
    println!("{x}");
}
error[E0308]: mismatched types --> src/main.rs:2:18 | 2 | let x: i32 = "42"; | --- ^^^^ expected `i32`, found `&str` | | | expected due to this

A javítás: alakítsd át a megfelelő típusra (itt .parse()-szal):

src/main.rs
let x: i32 = "42".parse().unwrap(); // &str → i32

E0277 — nem teljesül egy trait-feltétel

Akkor jön, ha egy típus nem tud valamit, amit elvársz tőle — például kiíratni:

src/main.rs
struct Pont { x: i32, y: i32 }

fn main() {
    let p = Pont { x: 1, y: 2 };
    println!("{}", p); // ❌ a Pont nem implementál Display-t
}
error[E0277]: `Pont` doesn't implement `std::fmt::Display` --> src/main.rs:5:20 | 5 | println!("{}", p); | ^ `Pont` cannot be formatted with the default formatter | = help: the trait `std::fmt::Display` is not implemented for `Pont` = note: in format strings you may be able to use `{:?}` (or {:#?} for pretty-print) instead
Olvasd el a note: sort — gyakran kész a válasz

A fenti note: szó szerint megmondja: használj {:?}-t. Ahhoz a típusnak Debug-ot kell tudnia, amit egyetlen #[derive(Debug)] sorral megkapsz:

src/main.rs
#[derive(Debug)] // legenerálja a Debug implementációt
struct Pont { x: i32, y: i32 }

fn main() {
    let p = Pont { x: 1, y: 2 };
    println!("{p:?}"); // Pont { x: 1, y: 2 }
}
⚙ Gyakorlat 0x04

Az alábbi kód E0277-tel nem fordul, mert a Szin nem írható ki. Javítsd ki úgy, hogy lefusson és kiírja a struktúrát (a {:?} formátum is megengedett).

hibás kód
struct Szin { r: u8, g: u8, b: u8 }

fn main() {
    let c = Szin { r: 255, g: 100, b: 0 };
    println!("{}", c);
}
Megoldás
megoldás
#[derive(Debug)]
struct Szin { r: u8, g: u8, b: u8 }

fn main() {
    let c = Szin { r: 255, g: 100, b: 0 };
    println!("{c:?}"); // Szin { r: 255, g: 100, b: 0 }
}
0x05

dbg! és eprintln!

Itt kezdődik a második rész: ha a kód lefordul, de nem azt csinálja, amit vársz. A leggyorsabb eszköz a dbg! makró — sokkal jobb, mint a sima println!.

src/main.rs
fn main() {
    let v = vec![1, 2, 3];
    let osszeg: i32 = v.iter().sum();

    dbg!(&v);                    // kiírja a helyet, a kifejezést ÉS az értéket
    let dupla = dbg!(osszeg * 2); // a dbg! VISSZAADJA az értéket → láncba illeszthető

    eprintln!("a duplája: {dupla}"); // eprintln! a STDERR-re megy
}
[src/main.rs:5:5] &v = [ 1, 2, 3, ] [src/main.rs:6:17] osszeg * 2 = 12 a duplája: 12

Miért jobb, mint a println!? Mert (1) kiírja a fájlt és sort, (2) kiírja magát a kifejezést is, nemcsak az értékét, (3) szép, többsoros ({:#?}) formában mutatja, és (4) visszaadja az értéket, így bármilyen részkifejezést körbecsomagolhatsz vele a logika módosítása nélkül.

stdout vs. stderr

Az eprintln! a standard hibakimenetre (stderr) ír, a println! a standard kimenetre (stdout). Diagnosztikát mindig eprintln!-nel írj ki — így nem keveredik a program „valódi" kimenetével, és külön átirányítható.

A Debug derive — enélkül nincs {:?}

src/main.rs
#[derive(Debug)] // KELL, hogy a {:?} és a dbg! működjön a saját típusodra
struct Jatekos { nev: String, pont: u32 }

fn main() {
    let j = Jatekos { nev: "Anna".into(), pont: 100 };
    println!("{j:?}");  // Jatekos { nev: "Anna", pont: 100 }
    println!("{j:#?}"); // ugyanez, szépen tördelve több sorba
}
0x06

Pánik, unwrap és backtrace

A pánik a program szabályozott összeomlása futásidőben. Az üzenete elárulja, hol és miért történt.

src/main.rs
fn main() {
    let v = vec![1, 2, 3];
    let x = v[10]; // ❌ futásidejű pánik: nincs 10. index
    println!("{x}");
}
thread 'main' panicked at src/main.rs:3:13: index out of bounds: the len is 3 but the index is 10 note: run with `RUST_BACKTRACE=1` environment variable to display a backtrace

Az üzenet megadja a helyet (src/main.rs:3:13) és az okot. A hívási lánc megtekintéséhez állítsd be a RUST_BACKTRACE környezeti változót:

terminál
RUST_BACKTRACE=1 cargo run
thread 'main' panicked at src/main.rs:3:13: index out of bounds: the len is 3 but the index is 10 stack backtrace: 0: rust_begin_unwind 2: core::panicking::panic_bounds_check 3: myapp::main 4: core::ops::function::FnOnce::call_once note: Some details are omitted, run with `RUST_BACKTRACE=full` for a verbose backtrace.

unwrap vs. expect

Az unwrap() pánikol, ha az érték None vagy Err — de néma. Az expect("...") ugyanezt teszi, csak beszédes üzenettel. Mindig expect-et használj, és írd le, miért bízol benne, hogy nem hibázhat:

src/main.rs
fn main() {
    // ha mégis hibázik, az üzenet azonnal elárulja a feltételezésedet:
    let kor: i32 = "30".parse().expect("a kor mezőnek számnak kell lennie");
    println!("{kor}");
}
Pánik vagy Result?

A pánik a „ez sosem fordulhat elő, és ha mégis, az programhiba" eseteknek való (pl. egy invariáns sérülése). A várható hibákat (hiányzó fájl, rossz felhasználói input) ne pánikkal kezeld, hanem Result-tal és a ? operátorral — lásd a mintaillesztés + konverziók modult.

0x07

Strukturált naplózás

A dbg! gyors helyi piszkálásra való — kiveszed, mielőtt commitolsz. Ami a kódban marad, az legyen rendes napló, szintekkel. A bevett megoldás a log homlokzat (facade) + egy backend, pl. az env_logger.

terminál
cargo add log env_logger
src/main.rs
use log::{info, warn, error, debug};

fn main() {
    env_logger::init(); // a RUST_LOG környezeti változó vezérli, mi látszik

    info!("alkalmazás indul");
    debug!("részletes adat: {}", 42);
    warn!("valami gyanús");
    error!("hiba történt");
}

A szintet a RUST_LOG állítja — újrafordítás nélkül:

terminál
RUST_LOG=debug cargo run
[2026-06-23T10:00:00Z INFO myapp] alkalmazás indul [2026-06-23T10:00:00Z DEBUG myapp] részletes adat: 42 [2026-06-23T10:00:00Z WARN myapp] valami gyanús [2026-06-23T10:00:00Z ERROR myapp] hiba történt
És az async / nagyobb projektek? — tracing

Aszinkron kódhoz és komolyabb alkalmazásokhoz a modern választás a tracing (+ tracing-subscriber): nemcsak üzeneteket, hanem strukturált mezőket és időzített „span"-eket is kezel. A Bevy is ezt használja belül, így ismerős lesz:

src/main.rs
tracing::info!(jatekos = "Anna", pont = 100, "szint teljesítve");
0x08

Igazi debugger

Amikor a dbg! kevés — lépésenként akarsz haladni, megállni egy soron, megnézni minden változót — előveszel egy debuggert. A Rust az lldb-vel és a gdb-vel is megy, és szállít hozzájuk Rust-barát wrappert (rust-lldb, rust-gdb), ami szépen jeleníti meg a std típusokat.

terminál — lldb a parancssorból
# előbb sima (debug) fordítás kell — a --release optimalizál és kioptimalizálja a változókat!
cargo build
rust-lldb target/debug/myapp
# (lldb) b main.rs:10   # töréspont a 10. sorra
# (lldb) run            # indítás
# (lldb) next           # következő sor (rövidítve: n)
# (lldb) print x        # változó értéke (p x)
# (lldb) continue       # tovább a következő töréspontig (c)

VS Code + CodeLLDB — a kényelmes út

A legtöbben editorból debugolnak. Telepítsd a CodeLLDB bővítményt, tegyél egy töréspontot a sor mellé kattintva, és indíts F5-tel. A változókat egy panel mutatja, lépni a vezérlősorral tudsz. Egy minimális indító-konfiguráció:

.vscode/launch.json
{
  "type": "lldb",
  "request": "launch",
  "name": "Debug (cargo)",
  "cargo": { "args": ["build", "--bin=myapp"] },
  "args": []
}
Mindig debug-módban debugolj

A --release build optimalizál: változókat összevon vagy eltüntet, függvényeket beágyaz (inline), így a debugger zavaros képet ad. A sima cargo build (debug profil) megőrzi a debug-információt — ezt használd hibakereséshez.

A trilógia vége — és a kezdet

Ez volt a három hiánypótló modul utolsó darabja: mintaillesztés + konverziók, String + iterátorok, és most a fordítóhibák + hibakeresés. Együtt a nagy Rust-tananyaggal ezek lefedik a magabiztos mozgáshoz szükséges hétköznapi réteget.

Merre tovább?

Innen már nem olvasni kell, hanem csinálni:

  • rustlings — csináld végig; pont a fordítóhibákon keresztül tanít, ahogy ez a modul.
  • Saját kis projektek — egy CLI-eszköz clap-pel és anyhow-val, egy parser, egy kis játéklogika.
  • Vissza a Bevyhez — a játékfejlesztéshez a teherhordó képesség a tulajdonjog és kölcsönzés folyékony kezelése (ez vesz elő a system-ekben). Ha a 0x01–0x02 hibák már elsőre érthetők, készen állsz.

A fordító nem ellenfél. Néhány hét után azon kapod magad, hogy már az ő szabályai szerint tervezel — és onnantól más nyelveken is jobb kódot írsz. 🦀
